## Section 1: Epic 仕様承認

### 前提条件（`/aidd-next` が自動検証）

- [ ] `/aidd-new-epic` のブリーフィングが実行済み（スキル内蔵のブリーフィングステップで完了していること）

### チェック項目（Epic ごとに確認）

- [ ] ユーザーストーリー一覧が Epic 内で定義されている（Phase の機能意図から AI が詳細化し、ユーザーが承認済み）
- [ ] 全ストーリーに AC が定義されている（ユーザー視点の E2E シナリオとして記述されており、実装詳細を含まない）
- [ ] 全 AC に AC-ID（`AC-ENNN-NN` 形式）が付与されている
- [ ] AC → ユーザーストーリーのトレーサビリティが確保されている（各 AC がどのストーリーに由来するか追跡可能）
- [ ] 正常系・異常系のレスポンスが定義されている
- [ ] バリデーションルールが網羅されている
- [ ] ステータス遷移が図示されている（該当する場合）
- [ ] 権限（RBAC）が各操作で明記されている
- [ ] 関連する ADR が参照されている
- [ ] 非機能要件が定義されている
- [ ] 他 Epic への依存・影響が明記されている
- [ ] 未決定事項が明示されている
- [ ] 外部依存コンポーネント（外部 API、Helm chart、コンテナイメージ、SDK 等）が特定され、アクセス可能であることが確認されている
- [ ] 所属 BC が記載され、BC キャンバス（`docs/domain/bc-*.md`）が存在する
- [ ] デリバリーする価値が明記されている（対象ユーザー・提供価値・デモシナリオ）
- [ ] E2E 検証計画が定義されている（検証シナリオ・検証環境・前提条件）
- [ ] 完了時に意味のあるデモが可能である — 技術基盤 Epic の場合は検証シナリオで代替
- [ ] 1人のペルソナの1つの業務目的を完結させる — 技術基盤 Epic の場合は「誰の何を可能にするか」で判断

### AC の品質基準（各 AC が全て満たすこと）

| 基準 | 定義 |
|------|------|
| **テスト可能** | 合否を自動テストで判定できる。曖昧な表現を含まない |
| **境界明確** | 入力値・状態・条件の境界が数値や列挙で定義されている |
| **独立** | 他の AC の合否に依存せず、単体で検証できる |
| **完全** | 正常系と異常系の両方をカバーしている |
| **観測可能** | システムの外部インターフェースで結果を観測できる |
| **E2E 検証可能** | 実コンポーネントを用いた E2E テストまたはデモで合否を判定できる |

### 横断チェック（全 Epic 完了後）

- [ ] Epic 間の整合性が確認されている（共通エンティティ、ステータス遷移の整合）
- [ ] 横断的な要件が横断仕様書として整理されている
- [ ] Phase 定義書の未定義事項が全て解決済み
- [ ] Spec Validation: 各 AC のテストスケルトン（テスト名 + 検証観点）が生成可能であることを確認済み（生成できない AC は曖昧さがあるため改善する）
