## 前提条件（`/aidd-next` が自動検証）

## チェック項目

### Part 1: スコープ・Epic 構成

- [ ] ユーザーストーリー一覧が完成している（全ストーリーが識別・記述済み）
- [ ] 全ストーリーが Epic に構造化され、ストーリー → Epic のトレーサビリティが確保されている
- [ ] 全 Epic の目的とスコープが明確
- [ ] MoSCoW 優先度に異論がない
- [ ] 成功基準がビジネス成果指標として定義されている（計測可能・期限付き・合意済み）
- [ ] Impact Mapping で成功基準と Epic の関係が整理されている
- [ ] Epic 間の依存関係が図示されている
- [ ] 横断的な技術課題が特定されている
- [ ] Won't Have が明示されている（スコープクリープ防止）
- [ ] 未定義事項のリストが完全
- [ ] 前 Phase の ADR・feedback Issue・レトロスペクティブの改善 Issue を確認し、本 Phase への影響を評価した

### Part 2: ドメイン分析

- [ ] BC（境界づけられたコンテキスト）が識別され、`docs/domain/` に BC キャンバスが配置されている
- [ ] コンテキストマップ（`docs/domain/context-map.md`）が作成され、BC 間の統合パターンが定義されている
- [ ] サブドメイン分類（コア/支援/汎用）が Phase 定義書に記載され、分類理由が明記されている
- [ ] ドメインイベント一覧（概要）（`docs/domain/domain-events.md`）が作成されている
- [ ] 用語集が BC セクション形式で更新され、各 BC のユビキタス言語が定義されている
- [ ] Epic が BC 内に閉じる原則に従って構成されている（BC をまたぐ Epic がある場合、その理由が記載されている）

## Phase 定義時の判断事項

以下の事項について明示的に判断し、Phase 定義書に記録されていること:

| # | 判断事項 | 判断者 | 記録先 |
|---|---------|--------|--------|
| 1 | この Phase で新たに導入するドメイン概念は何か | PO + TL | Phase 定義書 Epic 一覧 |
| 2 | この Phase のアーキテクチャへの影響は何か | TL | ADR（必要な場合） |
| 3 | 非機能要件の目標値はいくつか | TL | Phase 定義書 成功基準 |
| 4 | 技術基盤の追加・変更が必要か | TL | ADR（必要な場合） |
| 5 | この Phase で意図的にスコープ外とするものは何か | PO | Phase 定義書 Won't Have |
| 6 | 他の Phase への影響・依存は何か | PO | Phase 定義書 未定義事項 |
| 7 | Phase の成功をどう検証するか。計測可能なビジネス成果指標は何か | PO + TL | Phase 定義書 成功基準 |
| 8 | 成功基準と Epic の関係は明確か（Impact Mapping） | PO + TL | Phase 定義書 |
