# G3: 設計承認マイルストーン

| 項目 | 内容 |
|------|------|
| 旧ゲート | G2.5 + G2.7 |
| 判断者 | TL |
| タイミング | 設計完了時 |
| 問い | **設計が実装可能で、Epic 仕様との整合が取れているか?** |

> **マイルストーンの役割:** 品質は各スキルが実行時に保証済み。このマイルストーンは、設計成果物が揃い、Epic 仕様との整合が取れているかを統合確認する。

---

## Section 1: 基盤設計承認（旧 G2.5）

### 前提条件（`/aidd-next` が自動検証）

- [ ] `/aidd-new-epic` のブリーフィングが実行済み（設計に関するスキル内蔵のブリーフィングステップで完了していること）
- [ ] G2（Epic 仕様承認）が通過済み

### チェック項目

- [ ] プロジェクト種別に応じた必須設計成果物が全て作成されている
- [ ] Epic 仕様書の全 AC が設計でカバーされている（AC → 設計要素の対応が追跡可能）
- [ ] ADR の決定事項と矛盾していない
- [ ] この設計で新規 ADR が必要な判断が行われていないか確認した（アーキテクチャ選択・ライブラリ採用・設計原則の逸脱・重要なトレードオフ → 該当する場合は ADR を作成してからこのチェックをマーク）
- [ ] 他 Epic の設計成果物との整合性が確認されている
- [ ] ドメインモデルがビジネスルールを正しく表現している（集約境界、トランザクション境界が適切）

#### ドメインロジック網羅性（/aidd-epic-design 実行時）

- [ ] AC-ドメイン要素マッピングマトリクスが作成されており、全 AC がマップされている（`docs/domain/[epic-slug]-ac-mapping.md`）
- [ ] 全エンティティに不変条件（Invariants）が 1 件以上定義されている
- [ ] ステートフルエンティティの全状態遷移にガード条件が明記されている（「常に可」含む）
- [ ] 「〜によって決まる」パターンの AC に計算式または決定表（Domain Logic）が定義されている

#### サブドメイン種別に応じた設計深度

- [ ] コアドメインの Epic: リッチドメインモデルが定義されている（集約ルート、値オブジェクト、不変条件が詳細）
- [ ] コアドメインの Epic: 集約設計の 4 原則が適用されている（真の不変条件のみ境界内 / 小さな集約 / ID 参照 / 結果整合性）
- [ ] 支援サブドメインの Epic: 標準モデル（エンティティ、関係、主要ビジネスルール）が定義されている
- [ ] 汎用サブドメインの Epic: 連携設計（ACL / アダプタの I/F）が定義されている

#### ドメインイベント・BC 間統合

- [ ] ドメインイベント詳細仕様（ペイロード、順序保証、冪等性）が定義されている（該当イベントがある場合）
- [ ] BC 間統合仕様（`docs/design/integration-*.md`）が作成されている（BC 間連携がある場合）
- [ ] ドメインイベント詳細仕様が概要一覧（`docs/domain/domain-events.md`）と整合している

- [ ] パフォーマンス上の懸念が識別され、対策が検討されている
- [ ] セキュリティ上の懸念が識別され、対策が検討されている
- [ ] 外部依存コンポーネント（外部 API、Helm chart、コンテナイメージ、SDK 等）のアクセス確認が完了し、実際のスキーマ・インターフェースおよび動作仕様（命名規則、ライフサイクル、デフォルト設定等）と設計の前提が一致している
- [ ] DB スキーマの変更がある場合、マスタドキュメント（DB 設計）の更新案が含まれている

### 設計成果物間の整合性チェック

| チェック | 対象 |
|---------|------|
| ドメインモデルの属性 = DB スキーマのカラム（マッピングに漏れがない） | ドメインモデル ↔ DB スキーマ |
| API のリクエスト/レスポンスフィールド = ドメインモデルの属性のサブセット | ドメインモデル ↔ API spec |
| 画面に表示するデータ = API レスポンスに含まれる（画面系のみ: Section 2 で検証） | API spec ↔ 画面設計 |
| 状態遷移図の状態 = DB スキーマのステータス型 | ドメインロジック設計 ↔ DB スキーマ |
| バッチのスキーママッピング = DB スキーマのカラム | データフロー設計 ↔ DB スキーマ |

### 「AI が実装できるレベル」の判断基準

設計成果物を読んだ AI が以下に「設計を見ればわかる」と答えられれば十分:

1. このエンティティにはどの属性があるか?
2. この操作でどのビジネスルールが適用されるか?
3. このデータはどこにどう保存されるか?
4. このインターフェースはどの入力を受け、どの出力を返すか?
5. この操作が失敗したとき、何が起きるか?

## Section 2: 画面設計承認（画面系のみ）（旧 G2.7）

> 画面（UI）を含まないプロジェクト（API / バッチ / CLI のみ）はこのセクション全体をスキップする。

### 前提条件（`/aidd-next` が自動検証）

- [ ] `/aidd-new-epic` のブリーフィングが実行済み（画面設計に関するスキル内蔵のブリーフィングステップで完了していること）

### モック

- [ ] 全画面の実働モック（`.tsx`）が構築されている
- [ ] 全モックが PO のレビューを受け、合意を得ている
- [ ] モックがデザインシステム定義に準拠している（CSS 変数を使用、色のハードコードなし）
- [ ] モックが shadcn/ui コンポーネントをベースにしている
- [ ] レスポンシブ表示（モバイル / デスクトップ）が全画面で確認されている
- [ ] ダークモードが全画面で確認されている

### 画面仕様書

- [ ] 画面設計テンプレート（`templates/design/screen-design.md`）を使用して画面仕様書が作成されている
- [ ] 全画面にモック参照（ファイルパス + スクリーンショット）が含まれている
- [ ] 画面一覧テーブルが全画面を網羅している
- [ ] 画面遷移図が全画面・全遷移パスを網羅している

### コンポーネント・インタラクション

- [ ] 全画面のコンポーネント構成が定義されている
- [ ] 全コンポーネントの API 呼び出し先が明記されている（API 仕様書との整合性）
- [ ] 全インタラクションにトリガー・アクション・結果（正常系 + エラー系）が定義されている
- [ ] フォーム入力がある画面でバリデーション定義（制約、エラーメッセージ、タイミング）が記載されている
- [ ] 画面固有の条件分岐（空状態、ローディング、エラー）が全画面で定義されている
- [ ] 状態管理（スコープ、初期値、更新トリガー）が全画面で定義されている

### 整合性

- [ ] コンポーネントがデザインシステムのコンポーネントカタログと整合している
- [ ] 画面に表示するデータが API レスポンスに含まれている（API 仕様書との整合性）
- [ ] Epic 仕様書の全 AC が AC カバレッジでカバーされている
- [ ] 「AI が迷うポイント」の全項目にプロジェクトの方針が記入されている
- [ ] ADR の決定事項と矛盾していない
- [ ] この設計で新規 ADR が必要な判断が行われていないか確認した（該当する場合は ADR 作成後にチェック）

### 共通要素

- [ ] 共通コンポーネント（ヘッダー、フッター、サイドバー等）が定義されている
- [ ] 画面間のデータフロー（URL パラメータ、state、cache）が定義されている

### 「AI が実装できるレベル」の判断基準

画面設計を読んだ AI が以下に「設計を見ればわかる」と答えられれば十分:

1. この画面にどのコンポーネントがあるか?
2. 各コンポーネントはどの API を呼ぶか?
3. ユーザーがこの操作をしたとき、何が起きるか?
4. データが 0 件 / ローディング中 / エラー時に何が表示されるか?
5. この画面から他のどの画面に遷移できるか?
