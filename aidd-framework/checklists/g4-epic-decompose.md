# G4: Epic 分解承認（Task 定義）

| 項目 | 内容 |
|------|------|
| 旧ゲート | G3（Task 定義承認） |
| 判断者 | TL |
| タイミング | Epic の Task 分解完了時 |
| 問い | **AC → Task のトレーサビリティが完全で、実装着手可能か？** |

> 品質は各スキルが実行時に保証済み。マイルストーンは統合確認。

## 確認事項

### Task 分解の品質
- [ ] 全 Task が実装単位（Epic ブランチ上のコミット単位）に分解されている
- [ ] 各 Task に対応 AC が明記されている（AC-E001-01 等）
- [ ] 全 AC が少なくとも 1 つの Task でカバーされている（カバレッジ 100%）
- [ ] Task 間の依存関係が整理され、循環依存がない
- [ ] E2E 検証 Task が Epic の最後に配置されている

### トレーサビリティの完全性
- [ ] ストーリー → Epic → AC → Task の全チェーンが追跡可能
- [ ] 各 Task から上位の AC、Epic、ストーリーへの逆引きが可能
- [ ] AI 補完で追加された AC にも Task が割り当てられている

### 実装準備
- [ ] 設計成果物（G3 承認済み）との整合性がある
- [ ] 各 Task の完了条件が明確
- [ ] 必要な前提条件（外部 API、データ、環境）が Task に記載されている
- [ ] GitHub Issue が全 Task について作成されている

#### 設計判断（要検討マーク解決）

- [ ] 設計判断クラス（アルゴリズム選択・ライブラリ選定・エラー戦略・キャッシュ戦略・並行制御）に該当する委ねられた詳細が全て `<!-- 設計判断確定 -->` コメントに解決されている
- [ ] 設計判断の選択肢・根拠・トレードオフがユーザーに提示・承認されている
- [ ] 未解決の `<!-- 要検討 -->` マークが Task ファイルに残っていない
