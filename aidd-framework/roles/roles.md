# ロールと AI モード詳細

## 3 つのロール

### PO (Product Owner)

| 項目 | 内容 |
|------|------|
| **種別** | 人間 |
| **責務** | ビジネス意図の入力。ドメイン知識の補完。最終承認 |
| **判断領域** | ビジネス要件の優先度・スコープ、ドメインの正確性（AC の内容が正しいか）、リリース判断 |

**各段階での関与:**

| 段階 | PO の役割 |
|------|----------|
| Phase 定義 | スコープ・優先度を決定。成功基準を定義。Phase 定義書を承認 |
| Epic 定義 | AC を承認。ビジネスルールの正しさを判断 |
| 設計 | — |
| Task 定義 | — |
| 実装 | — |
| Phase 完了 | E2E シナリオの受入確認。ビジネス成果指標の達成判断 |

### TL (Tech Lead)

| 項目 | 内容 |
|------|------|
| **種別** | 人間 |
| **責務** | 技術的判断の最終決定。アーキテクチャ承認。PR 最終承認 |
| **判断領域** | アーキテクチャ判断（集約境界、技術選定）、設計の妥当性、実装品質 |

**各段階での関与:**

| 段階 | TL の役割 |
|------|----------|
| Phase 定義 | 技術的実現可能性を評価。アーキテクチャ方針を提示。非機能要件を定義 |
| Epic 定義 | バリデーション・エラーケースを設計。技術的網羅性を判断 |
| 設計 | 設計を承認。ドメインモデルの妥当性を判断 |
| Task 定義 | Task を分解・設計。粒度の妥当性を判断 |
| 実装 | PR を最終承認。マージ可否を判断 |
| Phase 完了 | 技術品質の最終確認 |

> **チーム規模に応じてロールを兼務する。** 小規模チームでは1人が PO と TL を兼務する。その場合でも、各ロールの視点で意識的にチェックすることで品質を担保する。

## AI の 4 つのモード

AI エージェント群は 4 つのモードで動作する。これらは概念的な役割分担であり、1 つの AI セッションが複数モードを兼務してもよい。

### 生成モード（旧ドラフトモード）

| 項目 | 内容 |
|------|------|
| **目的** | AI が品質責任を負い、完全な成果物を生成する |
| **入力** | 人間の意図入力（不完全でよい）+ 上位ドキュメント + コンテキスト情報 |
| **出力** | Phase 定義書、Epic 仕様書、設計成果物、Task 定義 |
| **AI の裁量** | 不完全な入力の検知・補完。情報収集・質問・推論による補完 |
| **品質責任** | **AI が品質責任を負う。** 自己レビューを経て人間に提示する。人間は方向性を確認する |
| **対応スキル** | `/aidd-new-phase`, `/aidd-new-epic`, `/aidd-decompose-epic` |

**旧ドラフトモードとの違い:**

| 旧ドラフトモード | 新・生成モード |
|----------------|--------------|
| 人間が全面的に品質責任を負う | **AI が品質責任を負う** |
| 叩き台を作り、人間が全面検証 | AI が完全な成果物を生成し、人間は方向性を確認 |
| 人間のレビューなしに承認してはいけない | AI の自己レビュー後に人間にブリーフィング |

### 実装モード

| 項目 | 内容 |
|------|------|
| **目的** | 合意済み仕様に基づきコード・テストを生成する |
| **入力** | 合意済みドキュメント（AC、設計成果物、規約） |
| **出力** | コード + テスト（PR） |
| **AI の裁量** | 実装アプローチの選択（ADR・規約の範囲内） |
| **品質責任** | AC・設計・規約に準拠した実装を生成。G5 レビューで検証 |
| **対応スキル** | `/aidd-impl` |

### レビューモード

| 項目 | 内容 |
|------|------|
| **目的** | 成果物の全問題を検出する |
| **入力** | 検証対象の成果物 + チェックリスト + 規約 |
| **出力** | 構造化された指摘（MUST FIX / SHOULD FIX / CONSIDER） |
| **AI の裁量** | 規約・チェックリストに基づく機械的判定。**設計判断・ビジネス判断は行わない** |
| **品質責任** | **全問題検出を保証する。** 検出漏れがあれば AI の改善課題とする |
| **対応スキル** | `/aidd-epic-review`, `/aidd-phase-review`（レビュー機能は各スキルに内蔵） |

**旧レビューモードとの違い:**

| 旧レビューモード | 新・レビューモード |
|----------------|------------------|
| 検出漏れのリスクあり。人間の最終判断が必須 | **全問題検出を保証する。** 検出漏れは AI の不具合 |
| 一次検証の位置づけ | AI が品質保証者として機能 |

### オーケストレーションモード

| 項目 | 内容 |
|------|------|
| **目的** | プロセス進行を管理する |
| **入力** | プロジェクト状態（Issue、ドキュメント、コードベース） |
| **出力** | 次のアクション提案、ブロッカー検知、進捗レポート |
| **AI の裁量** | 3 条件のチェック: (1) 前提条件が揃っているか (2) 次のアクションは何か (3) ブロッカーはないか |
| **品質責任** | 調整ミスのリスクあり。人間が計画を承認 |
| **対応スキル** | `/aidd-next`, `/aidd-status` |

## 各段階での AI モード対応

| 段階 | PO | TL | AI モード |
|------|----|----|----------|
| Phase 定義 | スコープ・優先度を決定 | 技術的実現可能性を評価 | **生成:** Phase 定義書を生成 |
| Epic 定義 | AC を承認 | 技術的網羅性を判断 | **生成:** Epic 仕様書を生成 |
| 設計 | — | 設計を承認 | **生成:** 設計ドラフトを生成。**レビュー:** AC カバレッジ・規約整合性を検証 |
| Task 定義 | — | 粒度の妥当性を判断 | **生成:** Task を生成。**オーケストレーション:** 依存関係の整理 |
| 実装 | — | PR を最終承認 | **実装:** コード・テストを実装。**レビュー:** 規約・AC・設計整合性の検証 |
| Phase 完了 | ビジネス成果を判断 | 技術品質を確認 | **オーケストレーション:** 全 AC 検証状況の集約 |

## エージェント間のコンテキスト共有

複数の AI エージェントが並行して動作する場合、以下のメカニズムでコンテキストを共有する:

| 共有メカニズム | 内容 |
|-------------|------|
| **リポジトリ（Single Source of Truth）** | コード、ドキュメント、設計成果物、規約。全エージェントがリポジトリを起点に情報を取得する |
| **用語集（BC セクション形式）** | BC ごとのユビキタス言語。コード・ドキュメントの命名を統一する |
| **AI エージェント指示ファイル（CLAUDE.md 等）** | プロジェクト固有の規約・禁止事項。全エージェントの行動を統一する |
| **Issue / PR（進捗状態）** | 各 Task の進捗、レビュー状態、ブロッカー |

> **原則: エージェント間の通信は「リポジトリへの書き込み」を介して行う。** エージェント A がコードをコミットし、エージェント B がそれを参照する、というリポジトリベースの間接的な情報共有を基本とする。

## スケーリング戦略: BC 単位の並行実行

複数の Epic を AI エージェント群で並行実行する場合、並行実行の基本単位は**境界づけられたコンテキスト（BC）**である。

**並行実行の判定基準:**

| 条件 | 並行可否 | 理由 |
|------|:-------:|------|
| 異なる BC に属する Epic 同士 | ○ 並行可 | BC 境界で分離 |
| 同一 BC 内の独立した Task 同士 | ○ 並行可 | ファイルレベルでの競合がない |
| 同一集約への変更を含む Task 同士 | 直列必須 | コンフリクトが確実に発生 |
| BC 間統合 Epic | 依存 Epic 完了後 | 統合対象の BC が安定していないと統合テストが無意味 |

実践的な運用手順は [スケーリングガイド](../guides/scaling-guide.md) を参照。
