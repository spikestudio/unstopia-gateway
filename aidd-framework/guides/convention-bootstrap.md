# 規約ドキュメント初期構築ガイド

G0 通過に必要な規約ドキュメント7項目を効率的に構築するためのガイド。

## AI が参照しやすい規約の書き方

規約ドキュメントは人間だけでなく AI が直接参照する。以下の原則に従うことで、AI の規約遵守率が向上する。

### 1. 宣言的に書く

```
❌ 変数名にはキャメルケースを使うようにしてください
✅ 変数名: camelCase（例: userId, orderCount）
```

### 2. 具体例を必ず付ける（正例と誤例のペア）

```
❌ 適切な命名を使うこと
✅ 正: getUserById(id: string): Promise<User>
   誤: getUser(s: string): Promise<any>
```

### 3. 箇条書き・テーブル形式を優先する

AI は構造化された情報を正確に読み取る。自然言語の段落より、箇条書きやテーブルの方が遵守率が高い。

### 4. MUST / SHOULD / MAY で強度を明示する

- MUST: 必ず従う。違反は PR を差し戻す
- SHOULD: 原則従う。例外には理由をコメントで記載する
- MAY: 推奨。チームの判断で採否を決めてよい

### 5. 1ファイル200行以内を目安にする

長すぎると AI は後半の規約を無視する傾向がある。超えたらファイル分割を検討する。

### 6. 禁止事項は代替案とセットで書く

```
❌ any 型を使用禁止
✅ any 型を使わない。代わりに unknown を使い、型ガードで絞り込む
   正: function parse(input: unknown): Result { ... }
   誤: function parse(input: any): any { ... }
```

## 7項目の最小限定義（G0 通過レベル）

各項目で最低限含めるべき内容を示す。

### 1. 命名規則（目安: 30〜50行）

以下の対象について、ルール・正例・誤例をテーブルで定義する:

- 変数・引数
- 関数・メソッド
- クラス・型・インターフェース
- ファイル名・ディレクトリ名
- 定数
- DB テーブル名・カラム名
- Boolean 変数（is/has/can 接頭辞等）

### 2. ディレクトリ構造（目安: 30〜60行）

以下を含める:

- プロジェクトルートのディレクトリツリー（各ディレクトリに1行説明）
- 新規ファイルの配置判断基準（3〜5ルール）

### 3. レイヤー間のルール（目安: 30〜50行）

以下を含める:

- 依存方向の図示（例: `interface → application → domain ← infrastructure`）
- 各レイヤーが import できる範囲
- 禁止される依存方向
- レイヤーをまたぐデータの受け渡し方（DTO 等）

### 4. エラーハンドリング方針（目安: 40〜60行）

以下を含める:

- エラーの分類（ドメインエラー / アプリケーションエラー / インフラエラー等）
- エラーの伝播方法（各レイヤーでどう処理するか）
- エラーレスポンスの共通フォーマット（JSON 形式の例）
- ログ出力のルール（レベル、フォーマット）
- 空 catch の禁止

### 5. テスト規約（目安: 40〜60行）

以下を含める:

- テストフレームワーク
- テストファイルの配置ルール
- テストの命名パターン
- テスト構造（Arrange-Act-Assert 等）
- レイヤー別のテスト方針（どこをモックするか）
- カバレッジ基準（数値）

### 6. 禁止事項（目安: 20〜40行）

最低5項目以上。各項目にテーブル形式で以下を含める:

- 禁止事項
- 理由
- 代替手段

### 7. Git 運用ルール（目安: 30〜50行）

以下を含める:

- ブランチ戦略（GitHub Flow 等）
- ブランチ命名規則
- コミットメッセージ規約（Conventional Commits 等）
- マージ方式（squash merge 等）
- main への直接 push の禁止

## 8. 汎用原則の具体化ガイド

SOLID、DRY、KISS 等の汎用プログラミング原則は AI が既に知っている。しかし「SOLID に従う」と書いても AI の行動は変わらない。**規約として機能させるには、プロジェクトのアーキテクチャに合わせて具体化する必要がある。**

### 一般論のまま書いてはいけない理由

- AI は一般論を「既知の情報」として無視する傾向がある
- 抽象的な原則は解釈の幅が広く、AI が自己流に適用する
- 具体的なルール（正例・誤例付き）の方が遵守率が高い

### 具体化の基準

一般論を規約に変換するときは、以下の3点を含める:

1. **このプロジェクトでの適用対象** — どのレイヤー・モジュールに適用するか
2. **正例・誤例のペア** — コード例で示す
3. **強度（MUST / SHOULD / MAY）** — 違反時の対応を明確にする

### 原則と具体化テンプレート

各原則について、「一般論（書いても効かない）」と「具体化（効く）」の対比で示す。`/aidd-setup conventions` スキルはこのテンプレートに従って規約を生成する。

> **注記:** 以下のコード例は擬似コードである。言語固有の DI パターン（TypeScript: constructor injection、Go: interface + constructor function、Python: Protocol + dependency-injector 等）は `/aidd-setup conventions` が技術スタックに応じて生成する。

#### テスタビリティ / 依存性逆転（DIP）

→ 対象ファイル: `layer-rules.md`, `testing.md`

```
❌ 依存性逆転の原則に従う
✅ 外部依存（DB, API, ファイルシステム）はインターフェースで抽象化し、DI で注入する（MUST）
   正: UserService(repo: UserRepository)  ← インターフェース経由で注入
   誤: UserService() { this.db = new DatabaseClient() }  ← 直接インスタンス化
   DI なしでインスタンス化できないクラスは設計不良として差し戻す
   「テストが書きづらい」は設計を直す理由であり、テストをスキップする理由ではない
```

#### 単一責任の原則（SRP）

→ 対象ファイル: `layer-rules.md`, `directory-structure.md`

```
❌ 各クラスは単一の責任を持つこと
✅ 1ファイル1パブリッククラス / 1関数1目的。以下の兆候があれば分割する（SHOULD）:
   - ファイルが 300 行を超える
   - 3つ以上の外部依存を import している
   - メソッド名に And / Or が含まれる（例: validateAndSave）
   正: UserValidator, UserRepository（責務ごとに分離）
   誤: UserService（バリデーション + 永続化 + 通知を全て担当）
```

#### Fail Fast

→ 対象ファイル: `error-handling.md`

```
❌ エラーは早期に検出する
✅ バリデーションはエントリポイント（API ハンドラ / コマンドハンドラ）で即座に行う（MUST）
   不正入力をドメイン層まで伝播させない
   正: handler で zod スキーマ検証 → エラーなら即 400 返却
   誤: ドメインロジック内部で初めて不正値に気づく
```

#### YAGNI（You Aren't Gonna Need It）

→ 対象ファイル: `prohibitions.md`

```
❌ 不要な機能を実装しない
✅ Task 定義の AC に含まれない機能・抽象化を実装しない（MUST）
   - 「将来使うかもしれない」インターフェースの事前作成を禁止
   - 呼び出し元が1箇所しかないヘルパー関数の抽出を禁止
   - 設定値のハードコードは、変更要件が AC に含まれるまでそのままにする
```

#### DRY（Don't Repeat Yourself）

→ 対象ファイル: `prohibitions.md`, `layer-rules.md`

```
❌ コードの重複を避ける
✅ 3回以上繰り返されるロジックを共通化する。ただし以下に注意（SHOULD）:
   - 2回の重複で早まった抽象化をしない（3回目で初めて共通化を検討）
   - 異なるコンテキストの偶然の一致は共通化しない
   正: 3箇所で同じバリデーションロジック → 共通バリデータを抽出
   誤: User と Product のバリデーションが偶然似ている → 無理に共通化
```

#### テストピラミッド

→ 対象ファイル: `testing.md`

```
❌ 適切なテストを書く
✅ テスト種別の比率を以下の目安で維持する（SHOULD）:
   - 単体テスト: 多（ドメインロジック、ユーティリティ）
   - 統合テスト: 中（API エンドポイント、DB アクセス）
   - E2E テスト: 少（主要ユーザーフロー）
   ドメインロジックは単体テスト必須（MUST）。モックは外部依存のみに限定し、
   テスト対象のロジック自体はモックしない
```

#### 防御的プログラミング

→ 対象ファイル: `error-handling.md`, `prohibitions.md`

```
❌ 防御的にプログラミングする
✅ 外部境界（API 入力、ファイル読み取り、外部 API レスポンス）は常に検証する（MUST）
   内部境界（自モジュール内の関数呼び出し）は型で保証し、冗長な検証をしない
   正: API ハンドラで入力スキーマ検証 → ドメイン層では型を信頼
   誤: 全関数の先頭で null チェック、全レイヤーで同じバリデーション
```

#### 命名の意図明確化

→ 対象ファイル: `naming.md`

```
❌ わかりやすい名前を付ける
✅ 名前は「何を」「なぜ」を表現する。型情報の繰り返しや略語を避ける（MUST）
   正: isEligibleForDiscount, fetchActiveUsers, MAX_RETRY_COUNT
   誤: flag, data, tmp, usr, cnt, process(), handle(), doSomething()
   Boolean は is/has/can/should で始める。関数は動詞で始める
```

### PM/PdM ベストプラクティスの扱い

PM/PdM のベストプラクティスは規約ドキュメントではなく、既存の FW テンプレートのコメント・ガイドとして組み込む:

| 原則 | 組み込み先 | 具体化 |
|------|----------|--------|
| ユーザーストーリー形式 | Epic 仕様書テンプレート | AC は Given-When-Then 形式で記述する |
| AC の粒度 | Epic 仕様書テンプレート | 1 AC = 1テストケースで検証可能な粒度にする |
| 作業分割 | Task 定義テンプレート | 1 Task = 1 PR。粒度は複雑度レベル（S/M/L）で判断する（→ FRAMEWORK.md） |
| リスク管理 | Phase 定義書テンプレート | リスク一覧に発生確率・影響度・対策を記載する |

> これらは FRAMEWORK.md のプロセス定義と重複しない。プロセスは FW が定義し、各テンプレートのコメントで具体的な書き方をガイドする。
