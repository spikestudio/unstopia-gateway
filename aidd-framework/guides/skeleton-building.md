# Skeleton 構築ガイド

## Skeleton の目的

Skeleton 構築は単なるサンプル実装ではない。以下の3つを同時に検証する。

- **規約検証** — 規約ドキュメント7項目が現実の実装と整合するか
- **AI パターン検証** — AI が規約を参照して規約通りのコードを生成するか
- **CI 検証** — CI パイプラインが正しく動作し、品質を自動チェックするか

## 「最小限の機能」の選定基準

以下の全てを含む1つの集約を選定する。

- [ ] **1つの集約ルート** — ドメインモデルの中心となるエンティティ
- [ ] **CRUD 操作** — 作成・取得（一覧 + 詳細）・更新・削除の4操作
- [ ] **バリデーション** — 入力値の検証（正常系 + 異常系）
- [ ] **エラーハンドリング** — ドメインエラー、バリデーションエラー、NotFound の3種類以上
- [ ] **全レイヤー貫通** — domain → application → infrastructure → interface の全層
- [ ] **永続化** — DB への保存と読み出し（マイグレーション含む）
- [ ] **テスト** — ユニットテスト + 統合テスト（最低限）

**選定のヒント:**
- Phase 1 の最初の Epic に含まれる集約から選ぶと、成果物がそのまま使える
- 外部サービス連携は含めない（内部アーキテクチャの検証が目的）
- 認証のような横断的関心事は含めない

## 完了の判定基準

- [ ] 1集約の CRUD が E2E で動作する（DB → ドメイン → 外部インターフェース → テスト）
- [ ] 実装が規約ドキュメント7項目全てに準拠している
- [ ] CI パイプラインが動作し、lint・テスト・ビルドが自動実行される
- [ ] 規約ドキュメントと実装の乖離がない（乖離があれば規約を修正済み）
- [ ] CLAUDE.md が最新化されている
