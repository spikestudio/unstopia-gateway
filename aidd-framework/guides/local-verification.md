# ローカル確認プロセスガイド

すべての動作確認はローカル環境で行い、PASS 後にデプロイする。このガイドはその原則と具体的な手順を定義する。

---

## 1. 確認マトリクス

Task（コミット単位）・Epic（PR 単位）・Phase（マイルストーン）の各レベルで何を確認するかを定義する。

| 確認種別 | Task（コミット前） | Epic（PR 作成前） | Phase（G6 判定前） |
|---------|------------------|-----------------|------------------|
| **ビルド** | 静的チェック（型/lint） | フルビルド（`task build`） | 本番ビルド（`task build`） |
| **起動** | — | `task dev` が起動すること | `task infra:up` + `task dev`（フルスタック） |
| **実働** | ユニットテスト（対象 AC） | Tier 判定による動作確認 → [§2](#2-tier-判定epic-実働) | ユースケース通し確認 → [§3](#3-phase-実働確認フロー) |

---

## 2. Tier 判定（Epic 実働）

Epic レビュー時の実働確認は、変更ファイルのパターンから Tier を自動判定し最高 Tier を適用する。

| Tier | 判定条件（変更ファイルのパターン） | AI が実行すること |
|------|--------------------------------|----------------|
| **0** | `*.md` / `docs/**` / `skills/**` のみ | 実働確認スキップ |
| **1** | API コード変更 | ローカル dev server にリクエストを送信し期待レスポンスを確認 |
| **2** | UI コード変更 | agent-browser でローカル dev server の画面を確認 |
| **3** | IaC / DB スキーマ変更 | plan / migration diff を AI が分析・提示。apply は人間が実行 |

複数 Tier に該当する場合は最高 Tier を適用する。

> **詳細手順:** `skills/aidd-review/references/step2-review-code.md` の「実行時検証」セクションを参照。

---

## 3. Phase 実働確認フロー

Phase 実働確認は時間がかかるため、AI が実行前にユーザーの承認を得る。

```
① AI: 実行計画を提示し、ユーザー承認を待つ
   ─────────────────────────────────────────────
   「以下を実行します。時間がかかります。進めてよいですか？」

   起動:
     - task infra:up（データストア起動）
     - task dev（アプリ起動）

   確認するユースケース（Phase 定義書の主要ユースケースから取得）:
     - [UC-1: ...]
     - [UC-2: ...]
   ─────────────────────────────────────────────

② ユーザー: 承認

③ AI: 実行
   → task infra:up
   → task dev（起動確認）
   → 各ユースケースを順に実行
     API 変更: http コマンドでリクエスト確認
     UI 変更:  agent-browser で画面確認

④ AI: 結果レポートを提示

   | ユースケース | 結果 | 備考 |
   |------------|------|------|
   | [UC-1]     | ✅   |      |
   | [UC-2]     | ❌   | [エラー内容] |

⑤ ユーザー: 結果を確認し G6 通過を判断
```

---

## 4. 実行主体・承認タイミング

| レベル | AI が実行すること | ユーザーが承認するタイミング |
|-------|-----------------|--------------------------|
| **Task** | 静的チェック（型/lint）+ ユニットテスト | コミット前（内容確認） |
| **Epic** | フルビルド + dev server 起動 + Tier 確認 | PR 作成前（内容確認） |
| **Phase** | フルスタック起動 + ユースケース実行 | **実行前**（時間がかかるため）+ G6 通過前（結果確認） |

---

## 5. 原則・例外

### 原則

- **全確認はローカル環境で行う**（Tier 3 の apply 操作を除く）
- **ローカルで PASS してから push / PR 作成 / G6 判定に進む**
- **デプロイ前にローカル確認を完了する** — CI/CD や dev 環境を動作確認の場として使ってはならない

### 例外

ローカル再現が物理的に不可能な環境依存問題（クラウド IAM・マネージドサービス固有の挙動等）に限り、dev 環境での直接確認を許容する。

**例外を適用する場合は必ず:**

1. PR コメントに例外理由を明記する
2. 「ローカル再現不可」の根拠を示す

### アンチパターン

| アンチパターン | 正しい順序 |
|-------------|-----------|
| PR 作成 → CI 結果を待って修正 | ローカルテスト PASS → PR 作成 |
| マージ → dev 環境で動作確認 | ローカル確認 PASS → マージ → デプロイ |
| デプロイして原因特定 → 修正 → 再デプロイ | ローカルで再現・修正・確認 → デプロイ |
