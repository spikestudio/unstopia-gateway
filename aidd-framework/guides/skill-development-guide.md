## リサーチ結果: Anthropic スキル構造ベストプラクティス

### 1. スキルの構造

**推奨ディレクトリ構成:**

```
skill-name/
├── SKILL.md           # 必須: メタデータ + 指示
├── scripts/           # 任意: 決定論的に実行すべきコード
├── references/        # 任意: コンテキストにロードされる参照ドキュメント
├── assets/            # 任意: 出力に使用されるファイル（コンテキスト非ロード）
└── examples/          # 任意: 動作サンプル
```

**Frontmatter フィールド（Agent Skills 標準 + Claude Code 拡張）:**

| フィールド | 必須 | 出典 | 説明 |
|-----------|------|------|------|
| `name` | Yes | 標準 | 最大 64 文字。小文字英数字+ハイフン。ディレクトリ名と一致必須 |
| `description` | Yes | 標準 | 最大 1024 文字。用途と使用タイミングを記述 |
| `context` | No | Claude Code | `fork` でサブエージェント実行。省略時はインライン実行 |
| `argument-hint` | No | Claude Code | オートコンプリート時のヒント表示 |
| `agent` | No | Claude Code | `context: fork` 時のサブエージェントタイプ |
| `model` | No | Claude Code | スキル実行時のモデル指定 |
| `effort` | No | Claude Code | `low` / `medium` / `high` / `max` |
| `user-invocable` | No | Claude Code | `false` で `/` メニューから非表示 |
| `disable-model-invocation` | No | Claude Code | `true` で自動呼び出し無効化 |
| `allowed-tools` | No | 標準 | 事前承認ツールリスト |
| `hooks` | No | Claude Code | スキルスコープのフック定義 |
| `paths` | No | Claude Code | glob パターンで自動アクティベーション対象を制限 |
| `license` | No | 標準 | ライセンス名 |
| `compatibility` | No | 標準 | 環境要件 |
| `metadata` | No | 標準 | 任意のキーバリューマッピング |

**出典:**

- [Agent Skills Specification](https://agentskills.io/specification)
- [Claude Code Skills ドキュメント](https://code.claude.com/docs/en/skills)

### 2. description の書き方

**基本ルール（Anthropic 公式）:**

- **三人称で書く**: "Processes Excel files..." / "This skill should be used when..."
- **具体的なトリガーフレーズを含める**: ユーザーが実際に言いそうな文言を列挙
- **用途と使用タイミングの両方を記述**: 何をするか + いつ使うか
- **先頭に最重要ユースケースを配置**: 一覧表示時に 250 文字で切り詰められる

**良い例:**

```yaml
description: >
  This skill should be used when the user asks to "create a hook",
  "add a PreToolUse hook", or mentions hook events (PreToolUse, PostToolUse, Stop).
```

**悪い例:**

```yaml
description: Helps with documents.  # 曖昧すぎる
description: Use this skill when working with hooks.  # 二人称、曖昧
```

**バジェット制約:** スキル description はコンテキストウィンドウの 1% バジェットでロード。フォールバック 8,000 文字。`SLASH_COMMAND_TOOL_CHAR_BUDGET` で調整可能。

**出典:**

- [Anthropic 公式ベストプラクティス](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)
- [Claude Code Skills ドキュメント](https://code.claude.com/docs/en/skills)

### 3. ステップ構成

**推奨パターン（Anthropic 公式）:**

1. **チェックリスト付きワークフロー**: 複雑なタスクは連番ステップに分解し、進捗追跡チェックリストを提供
2. **フィードバックループ**: 「実行→検証→修正→再実行」のバリデーションループが品質に大きく寄与
3. **条件分岐**: 判断ポイントでは明確に分岐（"Creating new? → Creation workflow / Editing? → Editing workflow"）
4. **自由度の適切な設定**:
   - 高自由度（テキスト指示のみ）: 複数アプローチが有効
   - 中自由度（パラメータ付き疑似コード）: 推奨パターンがある
   - 低自由度（具体的スクリプト）: 操作が脆弱でエラーが起きやすい

**出典:**

- [Anthropic 公式ベストプラクティス](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)

### 4. 参照ドキュメント管理（プログレッシブ・ディスクロージャー）

**三段階のロード戦略:**

| レベル | 内容 | ロードタイミング | トークン消費 |
|-------|------|---------------|------------|
| メタデータ | name + description | 起動時に常時 | 約 100 トークン |
| SKILL.md 本文 | コア指示 | トリガー時 | 推奨 5,000 トークン未満 |
| バンドルリソース | references/, scripts/ | 必要時にオンデマンド | 実質無制限 |

**references/ の活用ルール:**

- SKILL.md はリーンに保つ（500 行以下、理想は 1,500〜2,000 語）
- 詳細は references/ に移動し、SKILL.md から参照
- 10,000 語超の参照ファイルには grep 検索パターンを SKILL.md に含める
- **情報の重複禁止**: SKILL.md か references/ のどちらか一方にのみ配置
- **参照は 1 階層まで**: references/ から別の references/ へのネスト参照は避ける
- 100 行超の参照ファイルには目次を付ける

**出典:**

- [Agent Skills Specification](https://agentskills.io/specification)
- [Anthropic 公式ベストプラクティス](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)

### 5. 命名規則

**name フィールドの制約（Agent Skills 標準）:**

- 最大 64 文字
- 小文字英数字（a-z, 0-9）とハイフン（-）のみ
- ハイフンで開始/終了不可、連続ハイフン不可
- ディレクトリ名と一致必須
- 予約語（"anthropic", "claude"）を含めない

**推奨命名パターン:**

- 動名詞形が推奨: `processing-pdfs`, `analyzing-spreadsheets`
- 名詞句も許容: `pdf-processing`, `spreadsheet-analysis`
- 動作指向も許容: `process-pdfs`, `analyze-spreadsheets`
- 避けるべき: `helper`, `utils`, `tools`（曖昧すぎる）

**関連ファイルの命名:**

- ファイル名: snake_case（例: `rotate_pdf.py`）
- ディレクトリ名: kebab-case（例: `hello-world/`）

**出典:**

- [Agent Skills Specification](https://agentskills.io/specification)
- [Anthropic 公式ベストプラクティス](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)

### 6. 品質基準

**Anthropic 公式チェックリスト:**

コア品質:

- [ ] description が具体的でキーワードを含む
- [ ] description に用途と使用タイミングの両方を記述
- [ ] SKILL.md 本文が 500 行以下
- [ ] 詳細は別ファイルに分離
- [ ] 時間依存の情報を含まない
- [ ] 用語が一貫している
- [ ] 例が具体的（抽象的でない）
- [ ] ファイル参照が 1 階層
- [ ] プログレッシブ・ディスクロージャーを適切に使用
- [ ] ワークフローのステップが明確

コード・スクリプト:

- [ ] スクリプトが問題を解決する（Claude に丸投げしない）
- [ ] エラーハンドリングが明示的
- [ ] マジックナンバーがない
- [ ] 必要パッケージが指示に明記

テスト:

- [ ] 少なくとも 3 つの評価シナリオを作成
- [ ] Haiku, Sonnet, Opus でテスト
- [ ] 実際の使用シナリオでテスト

**記述スタイル:**

- 命令形/不定詞形を使用: "To create a hook, define the event type."
- 二人称を避ける: "You should..." / "You need to..." は使わない
- Claude が既知の説明は省略する

**反復開発プロセス:**

1. スキルなしで Claude にタスクを実行させ、失敗箇所を特定
2. 最低 3 つの評価シナリオを作成
3. ベースライン（スキルなし）のパフォーマンスを測定
4. 最小限の指示を書いてギャップを埋める
5. 評価を実行し、ベースラインと比較

**出典:**

- [Anthropic 公式ベストプラクティス](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)

### 7. コンテキスト管理

**`context: fork` の動作:**

- サブエージェントコンテキストで実行（会話履歴にアクセス不可）
- SKILL.md の内容がサブエージェントのプロンプトになる
- CLAUDE.md はロードされる
- 実行完了後、結果がメイン会話に要約されて返る

**使い分けガイド:**

| 条件 | 推奨 |
|------|------|
| 明確なタスク指示がある + 大量の参照資料 | `context: fork` |
| ガイドラインや規約のみ（タスクなし） | fork 不適切（インライン） |
| 会話の文脈が必要 | インライン（デフォルト） |
| メインコンテキストを汚したくない | `context: fork` |

**注意:** `context: inherit` は Claude Code 公式ドキュメントに未記載。デフォルト動作（インライン実行）がこれに該当する。

**出典:**

- [Claude Code Skills ドキュメント](https://code.claude.com/docs/en/skills)

### 乖離点の詳細と改善指針

#### 乖離 1: `type` フィールド（軽微）

**現状:** v2 スキルの frontmatter に `type: skill` を記載。

**Anthropic 標準:** Agent Skills 標準の frontmatter reference に `type` フィールドは存在しない。

**改善指針:** `type` フィールドはプロジェクト独自拡張として継続可能。ただし、他プロジェクトとの互換性を考慮する場合は `metadata.type` への移行を検討。現時点では破壊的変更の必要なし。

#### 乖離 2: description の記述スタイル（中程度）

**現状:** v2 スキルの description は日本語で用途を記述しているが、形式が不統一。

- `/aidd-new-phase`: 「ユーザーストーリーを入力として Phase 定義書とドメイン分析成果物を生成する。...」
- `/aidd-impl`: 「Task定義のACに従ってコードを実装する。...」

**Anthropic 推奨:** 三人称 + 具体的なトリガーフレーズ + 用途とタイミングの両方。

**改善指針:** 日本語スキルのため三人称の直接適用は不自然。代わりに以下の統一形式を推奨:

```
[何をするか]。[いつ使うか（トリガー条件）]。
```

例: 「ユーザーストーリーを入力として Phase 定義書を生成する。ユーザーが新しい Phase を定義したい場合は常にこのスキルを呼び出す。」

v2 の `/aidd-new-phase` は既にこの形式に近い。他スキルへの展開を E2 で実施。

#### 乖離 3: スキルテストの仕組み（中程度）

**現状:** v2 にスキルの評価・テストの仕組みがない。

**Anthropic 推奨:** 最低 3 評価シナリオ + マルチモデルテスト + ベースライン比較。

**改善指針:** フレームワーク本体のスキルは Claude Code 上でしか実行できず、自動テストの仕組み構築は Phase 4.1 のスコープ外。ガイドラインに「推奨プラクティス」として記載し、将来の検討事項とする。

#### 乖離 4: 品質チェックリストのカバー範囲（軽微）

**現状:** CLAUDE.md のスキル開発規約は 5 項目。Anthropic 公式は 10+ 項目。

**Anthropic 推奨チェック項目で v2 未カバー:**

- SKILL.md 500 行以下
- 時間依存の情報を含まない
- ファイル参照が 1 階層
- プログレッシブ・ディスクロージャーの使用

**改善指針:** TASK-106 のガイドライン策定で Anthropic 公式チェックリストを統合した完全版を作成。

### 整合している点（v2 の強み）

| v2 パターン | Anthropic との整合 | 付加価値 |
|------------|-------------------|---------|
| `context: fork` + references/ | プログレッシブ・ディスクロージャーの実践 | 明確な分離が実現済み |
| ブリーフィング→承認ゲート | フィードバックループの発展形 | 人間の意思決定を構造的に組み込み |

## 設計原則

### スキルは「AI が迷わず動ける指示書」である

スキルの目的は AI に正しい行動を取らせること。ルールを列挙するだけでは不十分で、**AI がどの情報をいつ参照し、どの順序で何を判断すべきか** を構造化する必要がある。

#### 目的定義: 1 文で書けないスキルは分割する

- スキルの責務は SKILL.md 冒頭の 1〜2 文で完結すること
- 「A をして、さらに B もする」なら A と B を別スキルにするか、capability として分離する
- **根拠:** 責務が曖昧だと AI が「どこまでやるか」を判断できず、過剰実装やスコープ逸脱が起きる

#### スコープ: 「やらないこと」を明示する

- 「対象外」を書くことで AI のスコープ逸脱を防ぐ
- 隣接スキルとの境界を具体的に記述する（「xxx は /aidd-yyy のスコープ」）
- **根拠:** AI は善意で改善しようとするため、明示的な境界がないと隣接領域に手を出す

#### 入出力: 具体例で示す

- 入力と出力の形式を抽象的に書かない。具体的なファイルパス・フォーマットで示す
- **悪い例:** 「ドキュメントを生成する」
- **良い例:** 「`docs/requirements/ES-NNN-slug.md` を Epic 仕様書テンプレートから作成する」
- **根拠:** 抽象的な指示は AI の解釈の幅を広げ、出力のばらつきを生む

#### 前提条件と後続スキル: パイプラインの中での位置を明確にする

- 前提条件テーブル（前提 / 確認方法 / 未完了時の案内）で事前検証を強制する
- 後続スキルテーブルで「次に何をすべきか」を案内する
- **根拠:** AI は「今やるべきこと」に集中させ、前後の工程は構造で保証する

#### 命名規則

| 種別 | パターン | 例 |
|------|---------|-----|
| パイプラインスキル | `aidd-{動詞}-{対象}` | `aidd-new-phase`, `aidd-new-epic` |
| 横断スキル | `aidd-{動作}` | `aidd-status`, `aidd-doctor` |
| capability | `aidd-{動作}` | ~~`aidd-review`, `aidd-autofix`~~（廃止済み） |
| セットアップ系 | `aidd-setup [カテゴリ]` | `/aidd-setup env`, `/aidd-setup ci`（統合済み） |

- 小文字英数字 + ハイフンのみ（Agent Skills 標準準拠）
- ディレクトリ名と `name` フィールドを一致させる
- `aidd-` プレフィックスで本フレームワークのスキルを識別する

**出典:** [Agent Skills Specification §naming](https://agentskills.io/specification), v2 実践知見

## 構造基準

### SKILL.md: AI が最初に読むドキュメント

#### frontmatter 必須フィールド

```yaml
---
name: aidd-xxx          # 必須。ディレクトリ名と一致
description: >          # 必須。性能に直結する最重要フィールド（後述）
  ...
context: fork           # パイプラインスキルは fork 必須。横断スキルは省略（インライン）
argument-hint: "[...]"  # 引数を取るスキルのみ
---
```

**`type` フィールドについて:** Anthropic 標準には存在しないプロジェクト独自フィールド。スキル種別の識別に使用しているが、`description` の記述で代替可能。現状維持で問題ないが、新規スキルでは省略可。

#### context 選択指針: fork か インラインか

| 条件 | 選択 | 理由 |
|------|------|------|
| 大量の references/ を持つ | `context: fork` | メインコンテキストを汚さない |
| 明確なステップ指示がある | `context: fork` | サブエージェントが指示に集中できる |
| 会話の文脈が必要 | 省略（インライン） | fork では会話履歴にアクセス不可 |
| ガイドライン提供のみ（タスクなし） | 省略（インライン） | fork だとサブエージェントが何をすべきか不明 |
| capability（他スキルから呼ばれる） | 省略（インライン） | 呼び出し元の文脈が必要 |

**性能への影響:** `context: fork` はコンテキストを分離するため、**スキルの指示が唯一の情報源** になる。指示が不十分だと AI が何をすべきか分からず性能が低下する。逆に、十分な指示があれば会話のノイズが排除されて性能が向上する。

**出典:** [Claude Code Skills ドキュメント](https://code.claude.com/docs/en/skills), v2 整合分析

#### ステップ構成の規則: AI の判断ポイントを減らす

**原則: 自由度は段階的に絞る**

Anthropic は自由度を 3 段階で使い分けることを推奨している:

| 自由度 | 指示方法 | 使用場面 | 例 |
|--------|---------|---------|-----|
| 高 | テキスト指示のみ | 複数アプローチが有効 | ブリーフィングの説明方法 |
| 中 | パラメータ付き疑似コード | 推奨パターンがある | テンプレートからのファイル生成 |
| 低 | 具体的コマンド/コード | エラーが起きやすい操作 | git コマンド、GitHub CLI |

**守らせるテクニック:**

1. **チェックリストで進捗を追跡させる**: AI に「チェックリストをコピーして進捗を追跡」と指示すると、ステップの飛ばしが減る
2. **フィードバックループを組み込む**: 「生成→検証→修正」のサイクルを明示すると品質が大幅に向上する（Anthropic 公式推奨）
3. **条件分岐を明確にする**: 判断ポイントで「A の場合→X / B の場合→Y」と分岐を書く。AI に判断を任せると不安定になる
4. **1 ステップ 1 責務**: 複数のことを 1 ステップに詰め込まない。AI は長い指示の後半を忘れやすい

**性能への影響:** ステップが明確なスキルは、曖昧なスキルに比べて AC 充足率が高い。特に `context: fork` スキルでは、ステップ構成がそのまま AI の行動計画になる。

#### references/ 配置ルール: 情報を必要なタイミングで渡す

**プログレッシブ・ディスクロージャー（段階的開示）:**

```
起動時に常時ロード          → frontmatter（name + description）
スキルトリガー時にロード     → SKILL.md 本文（500 行以下推奨）
必要時にオンデマンドロード   → references/, scripts/
```

**ルール:**

| ルール | 理由 |
|--------|------|
| SKILL.md は 500 行以下 | コンテキスト消費を抑え、指示の遵守率を維持する |
| 詳細は references/ に分離 | AI が「今のステップに必要な情報」だけを読める |
| 情報の重複禁止 | 同じ情報が 2 箇所にあると AI が矛盾した参照をする |
| 参照は 1 階層まで | references/ 内のファイルから別の references/ を参照しない。AI が部分的にしか読まない |
| 100 行超のファイルには目次 | AI がセクション単位で読めるようにする |

**命名パターン:**

- `references/briefing-spec.md` — ブリーフィング仕様
- `references/step1-xxx.md` — ステップ詳細（ステップ番号 + 内容）
- `references/xxx-template.md` — テンプレート定義

**出典:** [Anthropic 公式ベストプラクティス](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices), [Agent Skills Specification](https://agentskills.io/specification)

## 品質基準

### description: スキル性能の最重要レバー

description は **AI がスキルを選択するかどうかの判断基準** であり、性能に最も直結するフィールド。

#### 書き方の公式

```
[何をするか]。[いつ使うか（トリガー条件）]。
```

**テンプレート:**

```yaml
description: >
  {成果物}を{入力}から生成する。
  {N}ステップ（{ステップ概要}）で段階的に進め、各ステップでブリーフィング→承認ゲートを設ける。
  {トリガー条件}場合は常にこのスキルを呼び出す。
```

**実例（v2 スキルの良い例）:**

```yaml
# aidd-new-phase
description: >
  ユーザーストーリーを入力として Phase 定義書とドメイン分析成果物を生成する。
  4 ステップ（ストーリー補完→Epic構造化→BC識別→詳細分析）で段階的に進め、
  各ステップでブリーフィング→承認ゲートを設ける。
  ユーザーが新しい Phase を定義したい場合は常にこのスキルを呼び出す。
```

**なぜこの形式が効くか:**

1. **「何をするか」が先頭**: description は 250 文字で切り詰められるため、最重要情報を先頭に
2. **ステップ概要**: AI が「このスキルは何ステップで何をするか」を事前把握でき、適切な選択につながる
3. **トリガー条件が明示的**: 「〜場合は常にこのスキルを呼び出す」で曖昧さを排除

#### トリガー条件の明確さ: 誤発火を防ぐ

- **「常に」で強制**: 「〜場合は **常に** このスキルを呼び出す」と書くと、AI が他の手段を選ぶ余地を潰せる
- **否定条件も書く**: 紛らわしいスキルがある場合、「〜の場合はこのスキルを使わない」と明示
- **具体的なキーワードを列挙**: ユーザーが実際に使う言葉をトリガーフレーズとして含める

**出典:** [Anthropic 公式ベストプラクティス](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices), [Claude Code Skills ドキュメント](https://code.claude.com/docs/en/skills)

### 重要なルールセクション: AI にルールを守らせる技術

全スキルに「重要なルール」セクションを設ける。これは Anthropic 標準にはないが、**v2 で最も効果的だったパターン** の一つ。

#### 必須要素

```markdown
## 重要なルール

- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う** — 優先順序 1〜5 に該当する問題の見送り・後回しを推奨しない
- **ブリーフィング（共通ループ Step 5）は省略禁止** — 成果物生成後、ブリーフィングを必ず実行する
- [スキル固有のルール]
- コミットはユーザーの判断に委ねる
```

#### ルールを守らせる 5 つのテクニック

**1. ルールは SKILL.md の末尾に配置する**

AI は SKILL.md の末尾を「最後に読む情報」として重視する傾向がある。ルールを末尾に置くことで、実行直前に強化される。

**2. 禁止事項は「何をしない」だけでなく「代わりに何をする」を書く**

- **悪い例:** 「承認なしに次のステップに進まない」
- **良い例:** 「承認なしに次のステップに進まない。承認待ちの間はブリーフィングを実行し、ユーザーの判断を待つ」

禁止だけだと AI が行動を止められず、代替行動が必要。

**3. 「なぜ」を添えるとルール遵守率が上がる**

- **悪い例:** 「コミットはユーザーの判断に委ねる」
- **良い例:** 「コミットはユーザーの判断に委ねる — AI が勝手にコミットすると、ユーザーが差分を確認できない」

理由があると AI が「このルールは重要だ」と判断し、遵守率が上がる。

**4. ステップ内にガードチェックを埋め込む**

ルールセクションに書くだけでなく、**ルール違反が起きやすいステップ内にもチェックを入れる**:

```markdown
## Step 3: 完了処理

<!-- ガードチェック: ブリーフィングを実行したか確認 -->
**ブリーフィングを実行しましたか？** 実行していない場合は Step 2 のブリーフィングに戻ってください。
```

**5. CLAUDE.md とスキルの役割分担**

| 配置先 | 書く内容 | 理由 |
|--------|---------|------|
| CLAUDE.md | プロジェクト横断の 100% 強制ルール | 全会話で常時ロードされる |
| SKILL.md 重要なルール | スキル固有のルール | スキル実行時のみロードで十分 |
| references/ | ルールの詳細・根拠 | 必要時にオンデマンド参照 |

CLAUDE.md に書きすぎるとコンテキストを消費し、逆にルール遵守率が下がる。スキル固有のルールはスキル内に閉じ込める。

### ブリーフィング仕様: 人間のレビュー効率を最大化する

ブリーフィングは「AI が何をしたか」の報告ではなく、「**人間が何を判断すべきか**」を構造化して提示するもの。

#### 5 要素ブリーフィング

| # | 要素 | 目的 |
|---|------|------|
| 1 | 成果物の全体像 | 何が作られたか |
| 2 | AC との対応 | 完了条件は満たされているか |
| 3 | AI の判断と根拠 | AI が自律的に決めたことと、その理由 |
| 4 | AI 補完の明示 | ユーザーが入力していないが AI が追加した内容の一覧 |
| 5 | 不確実性の表明 | AI-UNCERTAIN マーク付き箇所の説明 |

**仕様の配置先:** `references/briefing-spec.md` にスキル固有のブリーフィング仕様を定義。

**性能への影響:** ブリーフィングが構造化されていると、人間のレビューが効率化され、フィードバックの質が上がる。結果として成果物の品質が向上するサイクルが回る。

### スキル開発の反復プロセス

Anthropic はスキルの品質を反復的に改善するプロセスを推奨している:

1. **ベースライン測定**: スキルなしで AI にタスクを実行させ、失敗箇所を特定する
2. **評価シナリオ作成**: 最低 3 つの評価シナリオ（正常系・異常系・エッジケース）を作成
3. **最小限の指示**: ベースラインの失敗箇所を埋める最小限の指示を書く
4. **評価と比較**: スキルありで実行し、ベースラインと比較して改善を確認
5. **反復**: 失敗が残れば指示を追加。改善が確認されるまで繰り返す

**重要:** 指示を追加するときは **最小限** にする。過剰な指示はコンテキストを消費し、他の重要な指示の遵守率を下げる。

**出典:** [Anthropic 公式ベストプラクティス](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)
