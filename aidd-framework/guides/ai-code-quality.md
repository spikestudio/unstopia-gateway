# AI 生成コード品質ガイド

AI が生成するコードには、人間が書くコードとは異なるリスクプロファイルがある。このガイドでは、AI 生成コードの品質を担保するための観点を整理する。

## AI 生成コード特有の品質リスク

| リスク分類 | 具体例 | 検出方法 |
|-----------|--------|---------|
| **もっともらしい誤り** | 構文的に正しく動作もするが、仕様と微妙に異なるロジック（境界値のズレ、条件の過不足） | AC との突合。境界値テストの確認 |
| **過剰実装** | 指示されていない機能の追加、不要な抽象化、使われないユーティリティ関数の生成 | Task スコープとの突合。YAGNI チェック |
| **規約からの逸脱** | 規約ドキュメント群の命名規則・パターンと異なる実装。学習データに引きずられた一般的なパターンの適用 | 規約ドキュメントとの突合。リンターで検出可能な部分は自動化 |
| **ハルシネーション** | 存在しない API・ライブラリ・設定の参照。架空のメソッド呼び出し | ビルド・型チェックで検出。外部依存の場合は手動確認 |
| **セキュリティの見落とし** | 入力値検証の不足、認可チェックの漏れ、機密情報のログ出力 | セキュリティチェックリストとの突合 |
| **テストの品質不足** | テストが通るが、仕様ではなく実装をテストしている（実装に合わせた assertion） | AC に基づいてテストが書かれているかの確認 |

## PR レビューの構造化

### Tier 1: 全 PR で必須

| 観点 | 確認内容 |
|------|---------|
| **AC 準拠** | Task の完了条件と AC に対して、実装が正確に対応しているか |
| **テスト存在** | 各 AC に対応するテストが存在し、CI を通過しているか |
| **スコープ遵守** | Task のスコープ外の変更が含まれていないか（過剰実装の検出） |
| **規約準拠** | 規約ドキュメント群の命名規則・実装パターンに従っているか |

### Tier 2: ドメインロジック・スキーマ変更・契約変更を含む PR

| 観点 | 確認内容 |
|------|---------|
| **境界値** | バリデーションの境界値が Epic 仕様書の定義と一致しているか |
| **ステータス遷移** | 状態遷移が Epic 仕様書の遷移図と一致しているか |
| **エラーハンドリング** | Epic 仕様書のエラーケース定義が全て実装されているか |
| **Contract 整合** | API spec / DB スキーマとの整合が取れているか |

### Tier 3: セキュリティ・データ整合性に関わる PR

| 観点 | 確認内容 |
|------|---------|
| **認可** | 権限が Epic 仕様書の定義通りに実装されているか |
| **入力検証** | 外部入力に対するサニタイズ・バリデーションが実装されているか |
| **データ整合性** | トランザクション境界、同時実行制御が適切か |
| **機密情報** | ログ・レスポンスに機密情報が含まれていないか |

## 自動化すべき品質チェック

CI で自動化し、PR マージの必須条件とする:

- **静的解析**: リンター、型チェック
- **テスト**: 全テスト通過
- **テストカバレッジ**: ドメインロジックは高カバレッジ（目安はプロジェクトごとに定める）、ハンドラ層は中程度
- **ビルド**: 正常にビルドが完了する
- **Contract 整合**: Design First の場合、仕様とコードの乖離を検出する仕組み

## 品質改善のフィードバックループ

AI 生成コードのレビューで発見したパターンは、以下にフィードバックする:

1. **規約ドキュメント群**: AI が間違えやすいパターンを禁止事項・規約として追記
2. **CLAUDE.md**: AI ツール固有の補足として追記
3. **テンプレート**: Task テンプレートのチェック項目に追加
4. **レトロスペクティブ**: Phase 完了時に初回パス率として計測
