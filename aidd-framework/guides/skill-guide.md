## クイックチェックリスト（新規スキル作成時）

- [ ] `skills/aidd-<name>/SKILL.md` を作成
- [ ] frontmatter に `name`・`description` を記載
- [ ] `context` は原則省略（インライン実行）
- [ ] description が「何をするか + いつ使うか」形式
- [ ] トリガー条件に「常にこのスキルを呼び出す」を含む
- [ ] 引数を取る場合、3 パターン（あり / なし+自動推定 / 推定失敗）を定義
- [ ] SKILL.md が 500 行以下（詳細は references/ に分離）
- [ ] 「重要なルール」セクションに意思決定基準・日本語対話・ブリーフィング省略禁止を記載
- [ ] 完了メッセージに次のアクションへの導線（`→ /aidd-next` 等）を含む
- [ ] パイプライン上のスキルなら FRAMEWORK.md / doctor / チェックリストを更新

## ファイル構造

```
skills/
  aidd-<name>/
    SKILL.md           # 必須: メタデータ + 指示（500 行以下）
    references/        # 任意: ステップ詳細・テンプレート（オンデマンドロード）
    scripts/           # 任意: 決定論的に実行するコード
    assets/            # 任意: 出力ファイル（コンテキスト非ロード）
```

**命名規則:**

| 種別 | パターン | 例 |
|------|---------|-----|
| パイプラインスキル | `aidd-{動詞}-{対象}` | `aidd-new-phase`, `aidd-new-epic` |
| 横断スキル | `aidd-{動作}` | `aidd-status`, `aidd-doctor` |
| capability | `aidd-{動作}` | `aidd-research` |

- 小文字英数字 + ハイフンのみ。ディレクトリ名と `name` フィールドを必ず一致させる

## frontmatter

```yaml
---
name: aidd-xxx          # 必須。ディレクトリ名と一致。最大 64 文字
description: >          # 必須。性能に最も直結するフィールド（後述）
  ...
# context: fork        # 原則省略（インライン）。完全自己完結の単発タスクのみ使用
argument-hint: "[...]"  # 引数を取るスキルのみ
---
```

### context 選択指針

| 条件 | 選択 | 理由 |
|------|------|------|
| 多段階ユーザー対話が必要 | **省略（インライン）** | 自然な対話。fork では回避策が必要になり複雑化 |
| 内部でサブエージェントを起動する | **省略（インライン）** | 不要なネスト防止。起動するエージェントを depth 1 に保つ |
| 会話の文脈・履歴が必要 | **省略（インライン）** | fork では会話履歴にアクセス不可 |
| capability（他スキルから呼ばれる） | **省略（インライン）** | 呼び出し元の文脈が必要 |
| 完全自己完結の単発タスク | `context: fork` | `/aidd-research` のような例外的ケースのみ |

> **Anthropic 公式:** `context: fork` only makes sense for skills with explicit instructions. 推奨: Research・Analysis・Self-contained tasks のみ。

その他の主要フィールド:

| フィールド | 説明 |
|-----------|------|
| `model` | スキル実行時のモデル指定 |
| `effort` | `low` / `medium` / `high` / `max` |
| `allowed-tools` | 事前承認ツールリスト |
| `user-invocable` | `false` で `/` メニューから非表示 |

## description の書き方

description は **AI がスキルを選択するかどうかの判断基準** であり、性能に最も直結するフィールド。

### 書き方の公式

```
[何をするか]。[いつ使うか（トリガー条件）]。
```

**テンプレート:**

```yaml
description: >
  {成果物}を{入力}から生成する。
  {N}ステップ（{ステップ概要}）で段階的に進め、各ステップでブリーフィング→承認ゲートを設ける。
  {トリガー条件}場合は常にこのスキルを呼び出す。
```

**ルール:**

- description は 250 文字で切り詰められる → 最重要情報を先頭に
- 「常にこのスキルを呼び出す」でトリガーを強制する
- 具体的なキーワードを列挙する（ユーザーが実際に使う言葉）
- バジェット制約: 1% コンテキストバジェット（フォールバック 8,000 文字）

## ステップ設計

### 基本原則

1. **1 ステップ 1 責務** — 複数のことを詰め込まない。AI は長い指示の後半を忘れやすい
2. **条件分岐を明確に** — 「A の場合→X / B の場合→Y」と書く。AI に判断を任せない
3. **フィードバックループ** — 「生成→検証→修正」のサイクルを明示すると品質が向上する
4. **自由度を段階的に絞る**:

| 自由度 | 指示方法 | 使用場面 |
|--------|---------|---------|
| 高 | テキスト指示のみ | 複数アプローチが有効 |
| 中 | パラメータ付き疑似コード | 推奨パターンがある |
| 低 | 具体的コマンド/コード | エラーが起きやすい操作 |

### 引数ハンドリング（統一パターン）

引数を取るスキルは Step 1 で以下の 3 パターンを全て定義すること:

```markdown
**引数あり（例: `/aidd-xxx ES-003`）:**
- 引数から対象を特定する

**引数なし:**
- GitHub Issues からプロジェクト状態を分析し、自動推定
- 推定結果をユーザーに確認してから実行

**推定できない場合:**
```
対象を自動推定できませんでした。以下のいずれかを指定してください:
- `/aidd-xxx ES-003` — [説明]
```
```

### 承認ゲートのパターン

ステップ間の承認はインライン実行なので自然な対話で実現できる（AskUserQuestion 不要）:

```markdown
## Step N: [処理]
...
ブリーフィングを実行し、承認を得てから Step N+1 に進む。
```

### 完了メッセージ

```markdown
次のステップ:
→ `/aidd-next` で次のアクションを確認
→ `/aidd-xxx [対象]` で [次の処理]
```

## references/ の使い方

**プログレッシブ・ディスクロージャー（段階的開示）:**

```
起動時に常時ロード          → frontmatter（name + description）
スキルトリガー時にロード     → SKILL.md 本文（500 行以下推奨）
必要時にオンデマンドロード   → references/
```

**ルール:**

| ルール | 理由 |
|--------|------|
| SKILL.md は 500 行以下 | コンテキスト消費を抑え、指示の遵守率を維持 |
| 詳細は references/ に分離 | AI が「今のステップに必要な情報」だけを読める |
| 情報の重複禁止 | 同じ情報が 2 箇所にあると AI が矛盾した参照をする |
| 参照は 1 階層まで | references/ 内のファイルからさらに references/ を参照しない |
| 100 行超のファイルには目次 | AI がセクション単位で読めるようにする |

**命名パターン:**

- `references/briefing-spec.md` — ブリーフィング仕様
- `references/step1-xxx.md` — ステップ詳細（ステップ番号 + 内容）
- `references/xxx-template.md` — テンプレート定義

## サブエージェント（Agent()）の使い方

### 基本原則

> **Anthropic 公式:** サブエージェントは副タスクがメイン会話に大量のデータ（検索結果・ログ・ファイル内容）を返す場合に使用する。独自のコンテキストで作業し、サマリーのみを返す。

**役割分担:**

| 層 | 実行方法 | 責務 |
|----|---------|------|
| スキル本体 | **インライン実行** | ユーザーとの多段階対話・承認ゲート・会話履歴の参照 |
| 孤立した作業 | **`Agent()` 起動** | 並列分析・大量ファイル処理・専門ペルソナによる評価 |

### Agent() を使うべきケース vs 使わないケース

| 処理 | 推奨 | 理由 |
|------|------|------|
| ユーザーとの対話・承認 | インライン直接実行 | 会話履歴アクセスが必要 |
| 複数観点の並列分析 | `Agent()` 並行起動 | コンテキスト分離・並列実行 |
| 大量ファイル処理・リサーチ | `Agent()` | メインコンテキストを汚さない |
| 専門ペルソナによる評価 | `Agent(subagent_type=...)` | ペルソナの専門性を活用 |
| 単純なファイル操作・検索 | インライン直接実行 | Agent 起動のオーバーヘッドが不要 |

### ネスト深度の制限

```
✅ 推奨: depth 0（インラインスキル）→ depth 1（Agent）
❌ 禁止: depth 0 → depth 1（Agent）→ depth 2（Agent 内で Agent）
```

Agent 内でさらに Agent を起動する二重ネストは原則禁止。

### ペルソナエージェント一覧

| ペルソナ | subagent_type | 専門性 |
|---------|---------------|--------|
| aidd-analyst | `aidd-analyst` | 業務・ドメイン分析 |
| aidd-architect | `aidd-architect` | 設計判断・トレードオフ評価 |
| aidd-builder | `aidd-builder` | 実装・構築 |

### 並行起動のルール

```
# 1 メッセージ内の複数 Agent 呼び出しは並行実行される
Agent(subagent_type="aidd-analyst", prompt="業務視点で分析...")
Agent(subagent_type="aidd-architect", prompt="設計視点で分析...")
Agent(subagent_type="aidd-builder", prompt="実装視点で分析...")
```

- コンテキスト非共有のため、必要情報は `prompt` パラメータに全て埋め込む
- 実用的な同時起動数は **3〜5 程度**
- ペルソナは読み取り専門（Read/Grep/Glob）で運用。書き込みはオーケストレーターが集約後に実行

### スキル委譲パターン

別スキルの処理を Agent で委譲できる。委譲先は **自己完結していること**（会話履歴不要）が条件:

```
Agent(prompt="/aidd-inception-mock を実行してください。機能意図: [X]、リサーチ結果: [Y]")
```

`context: fork` は不要。Agent() で起動すれば自動的にコンテキスト分離される。

### フォールバック

`subagent_type` 非対応環境ではプレーンな Agent 呼び出しで代替:

```
Agent(prompt="あなたは aidd-analyst（業務・ドメイン分析の専門家）です。以下を分析してください。...")
```

## 「重要なルール」セクション

全スキルに以下を含めること:

```markdown
## 重要なルール

- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う** — 優先順序 1〜5 に該当する問題の見送り・後回しを推奨しない
- **ブリーフィングは省略禁止** — 成果物生成後、ブリーフィングを必ず実行する。省略すると人間がレビューすべき判断ポイントを見落とす
- [スキル固有のルール]
- 成果物生成・実装完了時にコミットを自動実行する。push と PR 作成はユーザーの確認を待つ
```

**ルールを守らせるテクニック:**

1. **SKILL.md の末尾に配置** — AI は末尾を「実行直前に読む情報」として重視する
2. **禁止は「代わりに何をする」をセットで書く** — 禁止だけだと AI が行動を止められない
3. **「なぜ」を添える** — 理由があると遵守率が上がる
4. **ガードチェックをステップ内に埋め込む** — ルール違反が起きやすい箇所に直接書く

## ブリーフィング仕様

ブリーフィングは「AI が何をしたか」の報告ではなく、「**人間が何を判断すべきか**」を提示するもの。

### 5 要素ブリーフィング

| # | 要素 | 目的 |
|---|------|------|
| 1 | 成果物の全体像 | 何が作られたか |
| 2 | AC との対応 | 完了条件は満たされているか |
| 3 | AI の判断と根拠 | AI が自律的に決めたことと、その理由 |
| 4 | AI 補完の明示 | ユーザーが入力していないが AI が追加した内容 |
| 5 | 不確実性の表明 | AI-UNCERTAIN マーク付き箇所の説明 |

仕様の詳細は `references/briefing-spec.md` にスキル固有で定義する。

## コンテキスト管理

> **コアインサイト（コミュニティ 650+ 件調査）:** Claude Code 障害のほぼすべての根本原因はコンテキスト管理。コンテキストを肥大化させるスキル・使い果たすサブエージェント・肥大した CLAUDE.md — すべて同じ制約に行き着く。

### コンテキスト劣化のタイミング

出力品質はコンテキストウィンドウの **20〜40%** で劣化し始める（95% の自動コンパクション閾値ではない）。

劣化のシグナル: 自己矛盾・すでに見つけた情報を再発見・回答が冗長になりヘッジが増える・関数名がわずかに崩れる。

### 6 つのアンチパターン

| アンチパターン | 症状 | 対策 |
|-------------|------|------|
| **キッチンシンクセッション** | 1 つのタスクで始め無関係なことを聞く | タスク間で `/clear` |
| **過剰指定の CLAUDE.md** | ある開発者は 847 行 → 127 行に削減でコンプライアンスが劇的改善 | **200 行以下** に保つ |
| **`@ファイル` インポート** | `@path/to/docs.md` で毎回全体が埋め込まれる | パスを参照として書く（読み込まない） |
| **繰り返し修正** | 同じ修正を何度も依頼 | 2 回失敗したら `/clear` して初期プロンプトを書き直す |
| **MCP サーバー過負荷** | 各サーバーがツールスキーマを永続的に追加 | 実用上限は **5〜8 サーバー** |
| **否定形のルール** | 「〜を使うな」はその概念をかえって活性化させる | 肯定的な指示で書く |

### Document & Clear パターン

長時間セッションのリセット手順:

1. Claude に主要な決定事項と進捗を markdown ファイルに書かせる
2. `/clear` でフレッシュなコンテキストを開始
3. そのファイルを参照して継続

### CLAUDE.md のサイズ上限

**200 行以下**（コミュニティのコンセンサス）。これを超えるとコンプライアンスが目に見えて低下。Claude Code のシステムプロンプトがすでに約 50 指示を含むため、モデルが確実に従える合計は約 150〜200 指示。

## フック（Hooks）

フックは SKILL.md のアドバイザリーな性質を補完する **決定論的な** 強制メカニズム。

| 特性 | CLAUDE.md / SKILL.md ルール | フック |
|------|--------------------------|------|
| 遵守率 | 約 80%（アドバイザリー） | **100%（決定論的）** |
| 用途 | ガイドライン・推奨 | ポリシー強制・品質ゲート |

### 主なフックイベント

| イベント | タイミング | 主な用途 |
|---------|----------|---------|
| `PreToolUse` | ツール実行前 | 危険なコマンドのブロック・承認 |
| `PostToolUse` | ツール実行後 | コード自動フォーマット・検証 |
| `Stop` | Claude が停止時 | 作業完了の検証を促す |
| `SubagentStart/Stop` | サブエージェントのライフサイクル | 監視・ログ |
| `UserPromptSubmit` | ユーザーメッセージ送信時 | メッセージバスチェック・前処理 |

終了コード: `0`=許可, `1`=警告（続行）, `2`=ブロック。`PreToolUse` で `deny` を返すと `bypassPermissions` モードでもブロックされる。

## サブエージェントの実行モードとコスト最適化

### 3 つの実行モード

| モード | 動作 | 用途 |
|-------|------|------|
| **フォアグラウンド** | 完了まで待機 | シーケンシャルチェーン |
| **バックグラウンド** | メイン会話と並行実行 | 長時間実行タスク |
| **パラレル** | 独立した複数タスクを同時実行 | 並列リサーチ・分析 |

### コスト最適化パターン

メインセッションは複雑な推論のために上位モデルで実行し、サブエージェントは集中したタスクを軽量モデルで処理:

```
メインエージェント: Opus（複雑な推論・意思決定）
サブエージェント: Sonnet / Haiku（リサーチ・ファイル処理）
```

`CLAUDE_CODE_SUBAGENT_MODEL` 環境変数でデフォルトのサブエージェントモデルを設定可能。

### 5 つのオーケストレーションパターン（公式）

| パターン | 用途 |
|---------|------|
| **シーケンシャルワークフロー** | デプロイパイプライン・データ処理（順序依存） |
| **マルチ MCP コーディネーション** | GitHub/Slack/DB 等複数サービスにまたがるワークフロー |
| **反復的リファインメント** | 品質基準を満たすまでループ（停止条件を必ず定義） |
| **コンテキスト認識型ツール選択** | 環境に応じて異なるツールを選択・フォールバック |
| **ドメイン特化型インテリジェンス** | コンプライアンスチェック・監査証跡・専門知識の組み込み |

## トラブルシューティング

### スキルがトリガーされない

**description 予算の超過:** 全スキルの description 合計が **コンテキストの 1%（約 8,000 文字）** を超えるとスキルが警告なしに消える。

**アクティベーション率:** ベースラインは約 **50%**（Ivan Seleznov の 650 回の制御実験）。改善策:

- description に明示的なトリガー言語を含める（「常にこのスキルを呼び出す」）
- CLAUDE.md でスキルを参照する → **+15 ポイント**改善
- frontmatter へのキーワード追加は **効果ゼロ**

### サイレント障害チェックリスト

- `SKILL.md` のファイル名が大文字小文字含め正確か（`skill.md` や `SKILL.MD` は不可）
- スキルフォルダ名がケバブケースか（スペース・アンダースコア・大文字を含まない）
- スキルフォルダ内に `README.md` を置いていないか（スキルのロードが壊れる）
- `version`、`author`、`category` 等の非サポートフロントマターキーを追加していないか
- Prettier が YAML フロントマターを複数行 description にリフォーマットしていないか（スキル認識不能になる）
- description が 250 文字以内に重要情報を含んでいるか（一覧表示で切り詰められる）
