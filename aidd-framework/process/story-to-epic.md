# ストーリー起点のトレーサビリティ

全ての成果物は人間の意図入力（ストーリー）から導出される。AI が各段階で構造化・補完するが、人間の意図との整合性は常に確認する。

## トレーサビリティチェーン

```
ストーリー（人間入力）     「配信者が広告枠を登録して、メディアに配信できるようにしたい」
  │
  ▼ AI が構造化
Epic（AI 構造化）          ES-002: 広告枠管理
  │                        - ストーリー: 配信者による広告枠の CRUD
  │                        - ストーリー: メディアへの広告枠の紐付け
  │
  ▼ AI が導出
AC（AI 導出）              AC-E002-01: 配信者が広告枠を作成すると、枠名・サイズ・料金が保存される
  │                        AC-E002-02: 配信者が既存の広告枠を編集すると、変更が反映される
  │                        AC-E002-03: 配信者が広告枠を削除すると、関連する配信が停止される
  │
  ▼ AI が分解
Task（AI 分解）            TASK-005: 広告枠 CRUD API 実装（AC-E002-01, AC-E002-02）
  │                        TASK-006: 広告枠削除と配信停止の連携（AC-E002-03）
  │
  ▼ AI が実装（各 Task = コミット、Epic = PR）
<!-- レビュー指摘: PR body で Epic と Task の参照方法が混在していた -->
PR（AI 実装）              PR #12: feat: ES-002 広告枠管理（Epic: #4, Closes #5, Closes #6）
                             ├─ commit: feat: 広告枠 CRUD API（TASK-005）
                             └─ commit: feat: 広告枠削除時の配信停止連携（TASK-006）
```

## 各段階での AI の補完

### ストーリー → Epic

**人間の入力:** ビジネス意図を自然言語で表現する。不完全でよい。

**AI の補完:**
- ストーリーをドメイン概念に基づいてグルーピングする
- 暗黙の前提条件を検出し、質問する（例: 「削除時に関連データはどうなりますか？」）
- 非機能要件の検討を促す（例: 「同時操作のケースは考慮しますか？」）

### Epic → AC

**人間の入力:** Epic のストーリー（AI が構造化済み）を承認する。

**AI の補完:**
- 各ストーリーから AC を導出する
- 正常系だけでなく、異常系・境界値・エッジケースを網羅する
- AC の品質基準（テスト可能・境界明確・独立・完全・観測可能・E2E 検証可能）を検証する

### AC → Task

**人間の入力:** AC と設計成果物を承認する。

**AI の補完:**
- AC を実装単位の Task に分解する（各 Task は Epic ブランチ上のコミットになる）
- Task 間の依存関係を整理する
- 各 Task の AI コンテキスト（参照すべきドキュメント）を特定する

### Task → PR

**人間の入力:** Task 定義を承認する。

**AI の補完:**
- Task 定義 + 規約に基づいてコード・テストを実装する（各 Task をコミット）
- AC に対応するテストケースを生成する
- Epic 全 Task 完了後、PR description に `Epic: #<Epic Issue番号>` と全 Task Issue の `Closes #xxx` を記載する

## 逆方向の追跡

任意の成果物から、その根拠となる人間の意図まで遡れること:

```
PR #12（Epic PR）
  → Epic Issue: #4（PR description の `Epic: #4`）
    → Task: TASK-005, TASK-006（PR description の Closes #5, #6）
      → AC: AC-E002-01, AC-E002-02, AC-E002-03（各 Task のスコープ）
        → Epic: ES-002（AC の所属 Epic）
          → ストーリー: 「配信者が広告枠を登録して...」（Epic のストーリー）
            → Phase: PD-001（Epic の所属 Phase）
              → CHARTER: ビジネスゴール（Phase の所属）
```

**推奨:** テスト名に AC-ID（`AC-ENNN-NN`）を含めること。これにより、CI で AC-ID カバレッジ（全 AC-ID にテストが存在するか）を機械的に検証できる。
