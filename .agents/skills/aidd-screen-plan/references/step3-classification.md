# Step 3: 種別分類・対話フロー — 詳細手順

AC-E052-05（種別自動推定・確認・変更できる対話フロー）および AC-E052-06（種別不明の場合「未分類」として扱い手動設定できる）を満たす詳細手順を定義する。

## 入力

- Step 1・Step 2 で確定した全画面候補リスト（画面名・導出元 AC）

## 3-1. 種別の自動推定（AC-E052-05）

確定した各画面に対して、画面名・導出元 AC のキーワードから種別を自動推定する。

### 推定ルール

以下の優先順序でキーワードを照合し、最初にマッチした種別を採用する。マッチしない場合は「未分類」とする（AC-E052-06）。

| 優先度 | 種別 | キーワード（画面名 or 導出元 AC に含む） |
|-------|------|--------------------------------------|
| 1 | ウィザード | ウィザード、wizard、ステップ、step、設定ウィザード |
| 2 | ダッシュボード | ダッシュボード、dashboard、サマリー、概要、top、ホーム |
| 3 | エラー | エラー、error、404、403、500、not found、not_found |
| 4 | モーダル | モーダル、modal、確認ダイアログ、削除確認、dialog、ポップアップ |
| 5 | フォーム | 作成、登録、編集、更新、form、入力、申請、submit |
| 6 | 詳細 | 詳細、detail、view、show |
| 7 | 一覧 | 一覧、list、index、検索、search |
| 8 | 未分類 | （上記いずれにもマッチしない場合） |

### マッチング方法

1. 画面名と導出元 AC のテキストを結合した文字列を対象にする
2. 上記テーブルの優先度が高いキーワードから順に前方一致・部分一致で検索する
3. 最初にマッチした種別を「推定種別」として記録する
4. いずれのキーワードにもマッチしない場合は「未分類」とする（AC-E052-06）

### 推定結果の内部記録

各画面について以下を記録する:

```
{
  画面名: "[画面名]",
  推定種別: "[種別 or 未分類]",
  推定根拠: "[マッチしたキーワード]",
  確定種別: null  // ユーザー確認後に更新
}
```

## 3-2. 種別分類の提示（AC-E052-05）

推定結果を以下の形式でユーザーに提示する。

```markdown
## Step 3: 種別分類の確認

各画面の推定種別を確認してください。変更したい場合は「N番はモーダルに変更」のように指示してください。

### 種別一覧
- 一覧 / 詳細 / フォーム / ダッシュボード / モーダル / ウィザード / エラー / 未分類

| # | 画面名 | 推定種別 | 推定根拠 |
|---|--------|---------|---------|
| 1 | [画面名] | 一覧 | キーワード「一覧」 |
| 2 | [画面名] | フォーム | キーワード「作成」 |
| 3 | [画面名] | 未分類 | — |

変更したい種別があれば指示してください（例: 「3番はモーダルに変更」）。
問題なければ「OK」または「進めて」で確定します。
```

## 3-3. 対話操作の受け付け（AC-E052-05・AC-E052-06）

ユーザーからの変更指示を受け付け、種別を更新する。

### 種別変更

```
例: 「3番はモーダルに変更」
例: 「削除確認画面をモーダルにしてください」
例: 「すべての未分類をフォームにしてください」
```

処理:

1. 指定された画面の確定種別を更新する
2. 更新後のリストを再提示する

### 未分類の手動設定（AC-E052-06）

「未分類」の画面については、以下のプロンプトで手動設定を促す:

```markdown
### 未分類の画面があります

以下の画面は種別を自動推定できませんでした。種別を手動で設定してください。

| # | 画面名 | 導出元 AC |
|---|--------|----------|
| N | [画面名] | [AC-ID] |

種別を設定してください（例: 「N番はダッシュボード」）。
スキップする場合は「未分類のままにする」と指示してください。
```

未分類のままにすることも許容する。その場合は確定種別を「未分類」として記録する。

### 操作後の再提示

操作ごとに更新後の全リストを再提示し、さらに変更があるか確認する。

```markdown
## 更新後の種別一覧（N 件）

| # | 画面名 | 種別 | 変更 |
|---|--------|------|------|
| 1 | ... | 一覧 | — |
| 2 | ... | モーダル | [変更] |
| 3 | ... | フォーム | — |

他に変更はありますか？なければ「OK」または「進めて」で確定します。
```

## 3-4. 確定・承認

ユーザーから「OK」「進めて」「承認」等の明示的な承認を受けた時点で確定する。

確定後:

- 各画面の確定種別（「未分類」を含む）を全画面候補リストに追加して記録する
- Step 4 に進む

## エラーケース

| ケース | 条件 | 対応 |
|--------|------|------|
| 全画面が未分類 | いずれの画面もキーワードにマッチしない | 「全画面が未分類です。種別を手動で設定してください。」と通知し、未分類の手動設定フローに入る |
| 存在しない種別が指定された | ユーザーが 7 種別 + 未分類 以外の種別を入力した | 「指定された種別は存在しません。以下から選択してください: 一覧 / 詳細 / フォーム / ダッシュボード / モーダル / ウィザード / エラー / 未分類」と案内 |
| 存在しない番号が指定された | ユーザーが範囲外の番号を指定した | 「指定された番号が存在しません。N番はリストにありません。」と案内し、再入力を促す |

## 完了時

以下の情報を整理して Step 4 に渡す:

- 全画面候補リスト（画面名・導出元 AC・確定種別）
- 種別ごとの集計（一覧: N件 / 詳細: N件 / ... / 未分類: N件）
- 未分類のまま確定した画面の一覧（あれば）
