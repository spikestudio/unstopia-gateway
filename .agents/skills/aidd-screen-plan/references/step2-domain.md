# Step 2: ドメインモデル補完 — 詳細手順

AC-E052-03（ドメインモデル起点の CRUD 画面候補の自動検出・追加提案）および AC-E052-04（ドメインモデルが存在しない場合の通知・継続）を満たす詳細手順を定義する。

## 入力

- Step 1 で確定した画面候補リスト（画面名・導出元 AC）
- `docs/domain/` 配下のドメインモデルファイル（存在する場合）

## 2-1. ドメインモデルファイルの確認（AC-E052-04）

### ファイルの存在確認

```bash
ls docs/domain/
```

**ドメインモデルが存在しない場合（AC-E052-04）:**

以下のメッセージを表示してこのステップをスキップし、Step 3 に進む。

```
`docs/domain/` が見つかりませんでした。ドメインモデルによる補完をスキップし、Step 3 に進みます。
（AC起点の画面候補のみで続行します）
```

- エラーで停止しない。AC 起点の画面候補のみで続行する。

**ドメインモデルが存在する場合（AC-E052-03）:**

`docs/domain/` 配下のファイルを全て読み込み、2-2 に進む。

## 2-2. 集約・エンティティの抽出（AC-E052-03）

ドメインモデルファイルから集約（Aggregate）・エンティティを特定する。

### 検出パターン

以下のパターンを含む定義を集約として扱う:

| パターン | 例 |
|---------|---|
| `## [名称]集約` / `### [名称]集約` | `## ユーザー集約` |
| `Aggregate Root:` / `集約ルート:` | `Aggregate Root: User` |
| `- **[名称]**（集約）` | `- **注文**（集約）` |
| `class [名称]` + `@AggregateRoot` / `isAggregateRoot` 記載 | TypeScript / YAML 形式のモデル定義 |

キーワード「集約」「Aggregate」がファイルに存在しない場合は、最上位の見出し（`##` レベル）を集約名として扱う。

### 抽出結果の記録

```
検出された集約: [集約名1], [集約名2], ...
```

## 2-3. CRUD 画面候補の生成（AC-E052-03）

検出した各集約に対して、以下の 5 種類の CRUD 画面候補を生成する。

| 操作 | 生成する画面名 | 種別（推定） |
|------|-------------|------------|
| 一覧取得 | [集約名]一覧画面 | 一覧 |
| 詳細取得 | [集約名]詳細画面 | 詳細 |
| 作成 | [集約名]作成画面 | フォーム |
| 更新 | [集約名]編集画面 | フォーム |
| 削除 | [集約名]削除確認画面 | モーダル |

例: 集約名が「ユーザー」の場合:

- ユーザー一覧画面
- ユーザー詳細画面
- ユーザー作成画面
- ユーザー編集画面
- ユーザー削除確認画面

## 2-4. 重複除外

Step 1 で確定した画面候補（AC 起点）と生成した CRUD 画面候補を照合し、重複するものを除外する。

### 重複判定ルール

以下のいずれかに該当する場合を重複とみなす:

1. 画面名が完全一致する
2. 画面名が正規化後（「画面」「Screen」等のサフィックスを除去）に一致する
3. 同一集約の同一操作（一覧/詳細/作成/編集/削除）に対応する画面が既に存在する

重複する場合は CRUD 画面候補をリストに追加しない（Step 1 の画面候補を優先する）。

## 2-5. 追加候補の提示

重複除外後に残った CRUD 画面候補を「ドメインモデル起点の追加候補」としてユーザーに提示する。

```markdown
## Step 2: ドメインモデル補完

Step 1 の画面候補に加えて、ドメインモデルから以下の追加候補を検出しました。

### ドメインモデル起点の追加候補（N 件）

| # | 画面名（案） | 対応集約 | 種別 | 重複判定 |
|---|------------|---------|------|----------|
| 1 | [集約名]一覧画面 | [集約名]集約 | 一覧 | 新規 |
| 2 | [集約名]詳細画面 | [集約名]集約 | 詳細 | 新規 |
| 3 | [集約名]作成画面 | [集約名]集約 | フォーム | 新規 |

（Step 1 で既に追加済みの画面は除外済みです）

追加・除外の操作があれば指示してください。
変更がなければ「OK」または「進めて」で確定します。
```

追加候補が 0 件の場合（全て重複で除外された場合）:

```
ドメインモデルから検出した画面候補は全て Step 1 の画面一覧と重複していました。
追加候補はありません。Step 3 に進みます。
```

## 2-6. 対話操作の受け付け

ユーザーからの操作指示を受け付け、追加候補リストを更新する。

### 追加（手動追加）

ユーザーが追加候補に含まれない CRUD 画面を追加したい場合:

1. ユーザーが指定した画面名で新しい候補を追加する
2. 対応集約は「手動追加」として記録する
3. 更新後のリストを再提示する

### 除外

ユーザーが特定の追加候補を除外したい場合:

1. 指定された候補をリストから除外する
2. 更新後のリストを再提示する

### 操作後の再提示

操作ごとに更新後のリストを再提示し、さらに変更があるか確認する。

```markdown
## 更新後のドメインモデル起点の追加候補（N 件）

| # | 画面名 | 対応集約 | 種別 |
|---|--------|---------|------|
| 1 | ... | ... | ... |

他に変更はありますか？なければ「OK」または「進めて」で確定します。
```

## 2-7. 確定・承認

ユーザーから「OK」「進めて」「承認」等の明示的な承認を受けた時点で確定する。

確定後:

- Step 1 の画面候補 + Step 2 で追加した候補を結合した「全画面候補リスト」を内部に記録する
- Step 3 に進む

## エラーケース

| ケース | 条件 | 対応 |
|--------|------|------|
| 集約が検出できない | ドメインモデルファイルに集約・エンティティ定義が見当たらない | 「集約を検出できませんでした。ファイル形式を確認するか、手動で集約名を入力してください。」と案内 |
| 全候補が重複 | 生成した全 CRUD 候補が Step 1 と重複 | 「追加候補はありません」と通知して Step 3 へ進む |

## 完了時

以下の情報を整理して Step 3 に渡す:

- 全画面候補リスト（Step 1 確定分 + Step 2 追加分）
- Step 2 で追加した画面数と対応集約名
- 検出した集約一覧
