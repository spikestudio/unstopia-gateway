# /aidd-screen-plan — AC・ドメインモデル起点の画面洗い出し

Epic の AC とドメインモデルを入力として、画面の洗い出し・種別分類・ID付与・優先度設定・Epic 紐付けを対話的に実行し、画面一覧ドキュメント（`docs/design/screen-list-[epic-slug].md`）を生成する。

> **このスキルの目的は「AC の抜け漏れなく画面候補を整理し、後続の画面仕様書作成・実装に接続できる標準フォーマットで出力する」こと。**

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| Epic 仕様書が作成済み | `docs/requirements/ES-NNN-*.md` の存在確認 | `/aidd-new-epic` を先に実行 |
| Epic が G2 通過済み | Epic Issue に G2 通過コメントが存在する（`gh issue view <Epic番号> --comments`） | `/aidd-new-epic` を先に実行 |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| Epic ID | 引数 | 任意 | 指定なしなら未処理の Epic を自動推定 |

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| /aidd-research | Step 1 | ドメイン知識の補完（必要時のみ） |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| 画面一覧ドキュメント | Markdown | `docs/design/screen-list-[epic-slug].md` |
| AC カバレッジサマリー | テーブル | 全 AC 数・画面として表現された AC 数・未表現の AC 数 |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-screen-spec` | 画面一覧承認後、各画面の詳細仕様を作成する場合 |
| `/aidd-next` | G3 マイルストーン判定 |

## Step 0: 引数受付・Epic 特定

`$ARGUMENTS` から Epic ID を取得する。指定がなければ自動推定し、推定結果をユーザーに確認してから進行する。

### 0-1. Epic の特定

- **引数あり**（例: `/aidd-screen-plan ES-001`）: `docs/requirements/ES-001-*.md` を読み込む
- **引数なし**: `docs/requirements/ES-*.md` から画面一覧ドキュメントが未作成の Epic を推定し、ユーザーに確認する
- **推定失敗時**:

```
対象の Epic を特定できませんでした。以下のいずれかで指定してください:
- `/aidd-screen-plan ES-NNN` — Epic ID を指定して実行
- Epic 仕様書が存在しない場合は `/aidd-new-epic` を先に実行してください
```

### 0-2. コンテキスト読み込み

| 入力 | 取得元 | 用途 |
|------|--------|------|
| Epic 仕様書 | `docs/requirements/ES-NNN-*.md` | AC 一覧・バリデーション・エラーケース |
| ドメインモデル | `docs/domain/`（存在する場合） | 集約・エンティティから CRUD 画面を補完 |
| 既存画面一覧 | `docs/design/screen-list-*.md`（存在する場合） | 画面 ID 採番の重複回避 |

`docs/domain/` が存在しない・空の場合は、その旨を通知し AC 起点のみで続行する。

詳細手順は `references/step0-input.md` を参照して実行する。

## Step 1: AC 解析・画面候補提示

Epic 仕様書の AC セクションを解析し、UI 操作（「表示」「入力」「一覧」「選択」「確認」等）を含む AC から画面候補を自動抽出する。

抽出結果を以下の形式でユーザーに提示し、「追加」「除外」「統合」の対話的な操作を受け付ける。

```
## Step 1: 画面候補の提示

### AC 起点の画面候補（N 件）

| # | 画面名（案） | 導出元 AC | 備考 |
|---|------------|----------|------|
| 1 | [画面名] | AC-ENNN-NN | [補足] |
| 2 | [画面名] | AC-ENNN-NN | [補足] |

上記の内容を確認してください。追加・除外・統合の操作があれば指示してください。
```

ユーザーの明示的な承認を得てから Step 2 に進む。

詳細手順は `references/step1-ac-analysis.md` を参照して実行する。

## Step 2: ドメインモデル補完

ドメインモデルファイルが存在する場合、Step 1 で提案した画面候補に加え、未カバーの集約に対応する CRUD 画面を追加提案する。Step 1 で既に提案済みの画面との重複は除外する。

ドメインモデルファイルが存在しない場合はその旨を通知し、このステップをスキップして Step 3 に進む。

ユーザーの明示的な承認を得てから Step 3 に進む。

詳細手順は `references/step2-domain.md` を参照して実行する。

## Step 3: 種別分類・対話フロー

確定した画面候補に対して、スキルが種別（一覧/詳細/フォーム/ダッシュボード/モーダル/ウィザード/エラー）を自動推定し、ユーザーが確認・変更できる対話フローを提供する。種別が不明な画面は「未分類」として扱い、ユーザーが手動で設定できる。

### 種別推定ロジック

画面名・導出元 AC のキーワードから以下の優先順序で種別を推定する（上位から順に照合）:

| 優先度 | 種別 | 判定キーワード |
|-------|------|--------------|
| 1 | ウィザード | ウィザード、wizard、ステップ、step |
| 2 | ダッシュボード | ダッシュボード、dashboard、サマリー、概要、top、ホーム |
| 3 | エラー | エラー、error、404、403、500、not found |
| 4 | モーダル | モーダル、modal、確認ダイアログ、削除確認、dialog |
| 5 | フォーム | 作成、登録、編集、更新、form、入力、申請 |
| 6 | 詳細 | 詳細、detail、view、show |
| 7 | 一覧 | 一覧、list、index、検索、search |
| 8 | 未分類 | （上記いずれにもマッチしない場合） |

### 対話フロー

推定結果をテーブルで提示し、変更を受け付ける:

```
## Step 3: 種別分類の確認

### 種別一覧
- 一覧 / 詳細 / フォーム / ダッシュボード / モーダル / ウィザード / エラー / 未分類

| # | 画面名 | 推定種別 | 推定根拠 |
|---|--------|---------|---------|
| 1 | [画面名] | 一覧 | キーワード「一覧」 |
| 2 | [画面名] | フォーム | キーワード「作成」 |
| 3 | [画面名] | 未分類 | — |

変更したい種別があれば指示してください（例: 「3 番はモーダルに変更」）。
問題なければ「OK」または「進めて」で確定します。
```

「未分類」の画面が存在する場合は手動設定を促す。未分類のまま確定することも許容する。

ユーザーの明示的な承認を得てから Step 4 に進む。

詳細手順は `references/step3-classification.md` を参照して実行する。

## Step 4: 画面 ID 付与・優先度・MVP スコープ・Epic 紐付け・ドキュメント生成

確定した画面一覧に対して以下を順に実行する:

### 4-1. 画面 ID 付与・AC トレーサビリティ

各画面に `SCR-NNN`（3 桁ゼロ埋め連番）形式の一意の ID を付与する。既存の `docs/design/screen-list-*.md` が存在する場合は最大 ID を確認して重複を回避する。各画面の導出元 AC-ID（`AC-ENNN-NN`）を記録する。同一 AC から複数画面が導出される場合も全て記録する。AC カバレッジサマリー（全 AC 数・画面として表現された AC 数・未表現の AC 数）を生成する。

詳細手順は `references/step4a-id.md` を参照して実行する。

### 4-2. 優先度・MVP スコープ・Epic 紐付け

各画面の優先度（MUST/SHOULD/COULD）をユーザーと対話的に設定する（デフォルト: MUST）。MVP スコープ（in/out）を設定する。担当 Epic（`ES-NNN`）を紐付ける。未割当の場合は「未割当」として記録する。

### 4-3. ドキュメント生成

`references/screen-list-template.md` に従って `docs/design/screen-list-[epic-slug].md` を生成する。AC カバレッジサマリー（全 AC 数・画面として表現された AC 数・未表現の AC 数）を含める。

ユーザーの明示的な承認を得てから Step 5 に進む。

詳細手順は `references/step4b-doc.md` を参照して実行する。

## Step 5: ブリーフィング・完了

`references/briefing-spec.md` に従ってブリーフィングを実行する。

**重点項目:**

- 生成した画面数と AC カバレッジ結果
- 優先度・MVP スコープの設定サマリー
- 未分類・未割当の画面（あれば）
- 後続スキルへの接続フォーマットの確認

ユーザーの承認後、完了メッセージを表示する。

```
## 画面一覧生成完了

**Epic:** [Epic 名]
**出力ファイル:** docs/design/screen-list-[epic-slug].md
**画面数:** N 件（MUST: N / SHOULD: N / COULD: N）
**AC カバレッジ:** N/N AC が画面として表現済み

### 次のステップ
- `/aidd-screen-spec` で各画面の詳細仕様を作成する
- `/aidd-next` で G3 マイルストーン判定を進める
```

詳細手順は `references/briefing-spec.md` を参照して実行する。

## 重要なルール

- **日本語で対話する** — フレームワーク全体の対話言語が日本語であるため
- **FRAMEWORK.md の意思決定基準に従う**
- **各ステップでユーザーの確認を得てから次に進む** — 自動的に全ステップを実行しない。ユーザーの意図と成果物のずれを早期に検出するため
- **ユーザーの明示的な承認なしに次ステップへ遷移しない** — 「OK」「進めて」「承認」等の明確な意思表示が必要
- **ドメインモデルが存在しない場合はエラーで停止しない** — AC 起点のみで続行し、その旨を通知する
- **画面 ID は一意を保証する** — 既存の screen-list ファイルを確認して重複を防ぐ
- **AC カバレッジを可視化する** — 未表現の AC がある場合は明示する
- **ブリーフィング省略禁止** — Step 5 完了時に `references/briefing-spec.md` に従ったブリーフィングを必ず実行する
- **各ステップの詳細手順は `references/` を読んで実行する** — SKILL.md 内に手順を複製しない
