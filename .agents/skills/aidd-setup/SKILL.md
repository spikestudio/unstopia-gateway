# /aidd-setup — プロジェクトセットアップ統合

`/aidd-doctor`（診断）の対として機能するセットアップスキル。7 つのカテゴリを統合し、プロジェクトの状態に応じて必要なセットアップを対話的に実行する。

## インターフェース

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| カテゴリ | 引数 | 任意 | `fw`, `project`, `env`, `conventions`, `ci`, `branch-protection`, `mocks` |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| セットアップ結果 | ブリーフィング | 実行したカテゴリと結果 |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-doctor` | セットアップ後の確認 |
| `/aidd-next` | G0 マイルストーン判定 |

## カテゴリ一覧と前提条件

| カテゴリ | 内容 | 前提条件 |
|---------|------|---------|
| `fw` | フレームワーク導入・更新 | なし |
| `project` | プロジェクト初期化（GitHub, ディレクトリ, ドキュメント雛形） | fw 完了 |
| `env` | 開発基盤ツール（mise, direnv, Docker, Taskfile 等） | 技術スタック決定後 |
| `conventions` | コーディング規約（正例・誤例付き） | 技術スタック決定後 |
| `ci` | ローカル CI フック（lefthook） | env 完了後 |
| `branch-protection` | GitHub ブランチ保護ルール | ci 完了後 |
| `mocks` | モック環境（Next.js + shadcn/ui + Storybook） | 画面系プロジェクトのみ |

## Step 0: 入力受付・カテゴリ判定

### 引数ありの場合

`$ARGUMENTS` からカテゴリを特定する。無効なカテゴリの場合:

```
無効なカテゴリです: [入力値]
有効なカテゴリ: fw, project, env, conventions, ci, branch-protection, mocks
```

### 引数なしの場合

1. `/aidd-doctor` の直近の診断結果を確認する（会話コンテキスト内）
2. 診断結果がある場合: ❌ の項目に対応するカテゴリを優先順位順に提案する
3. 診断結果がない場合: プロジェクト状態を簡易診断し、未完了のカテゴリを推定する

```
プロジェクト状態を確認しました。以下のセットアップが未完了です:
1. [カテゴリ] — [理由]
2. [カテゴリ] — [理由]

どのカテゴリから始めますか？（推奨: [最優先カテゴリ]）
```

### 前提条件チェック

選択されたカテゴリの前提条件を検証する。未充足の場合:

```
[カテゴリ] の前提条件が未充足です:
- [未充足の前提条件]

先に以下を実行してください:
→ `/aidd-setup [前提カテゴリ]`
```

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| `/aidd-research` | Step 1（conventions カテゴリ） | 技術スタック固有の規約リサーチ（必要時のみ） |

## Step 1: カテゴリ実行

カテゴリに対応する `references/` ファイルを読み込んで実行する。

| カテゴリ | 参照ファイル |
|---------|------------|
| `fw` | `references/cat-fw.md` |
| `project` | `references/cat-project.md` |
| `env` | `references/cat-env.md` |
| `conventions` | `references/cat-conventions.md` |
| `ci` | `references/cat-ci.md` |
| `branch-protection` | `references/cat-branch-protection.md` |
| `mocks` | `references/cat-mocks.md` |

## Step 2: セルフレビュー

セルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）。スキル固有の追加観点:

- セットアップ結果がカテゴリの完了条件を全て満たしているか
- 生成されたファイルが規約・テンプレートに準拠しているか
- 前提条件が正しく検証されたか

## Step 3: ブリーフィング・次のカテゴリ案内

`references/briefing-spec.md` に従ってブリーフィングを実行し、承認を得る。

ブリーフィング後:

```
[カテゴリ] のセットアップが完了しました。

次のステップ:
→ `/aidd-setup [次のカテゴリ]`（前提条件が充足された場合）
→ `/aidd-doctor` で全体確認
→ `/aidd-next` で G0 マイルストーン判定
```

## 重要なルール

- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う**
- **ブリーフィング省略禁止** — 各カテゴリの完了後にブリーフィングを実行する
- **前提条件の検証を省略しない** — 前提条件未充足のままセットアップを開始しない
- **対話的に進める** — 各ステップでユーザーに確認を求める。勝手に進めない
- **冪等性を保つ** — 同じカテゴリを複数回実行しても安全であること
- 新しい外部ライブラリを人間の確認なしに追加してはならない
- 成果物生成・実装完了時にコミットを自動実行する。ユーザーの指摘反映後・ステップ承認後などの状態遷移ごとにもコミットする。push と PR 作成はユーザーの確認を待つ
