# ブリーフィング仕様 — /aidd-setup

各カテゴリのセットアップ完了時（Step 3）にこの仕様に従ってブリーフィングを実行する。5 要素ブリーフィングを前提とし、`/aidd-setup` スキル固有の品質基準を追加定義する。

## 5 要素チェックリスト

ブリーフィングは以下の 5 要素を**全て**含むこと。1 つでも欠けている場合、ブリーフィングは不完全とみなす。

| # | 要素 | 内容 | 欠落時のリスク |
|---|------|------|-------------|
| 1 | 成果物の全内容の説明 | 実行したカテゴリで作成・変更したファイル・設定を網羅的に説明する。件数・パス・主要な設定値を省略しない | 人間が何が変わったかを把握できない |
| 2 | AI が行った判断の根拠と代替案 | 「なぜこの設定にしたか」「他にどんな選択肢があったか」を明示する | 判断の妥当性を評価できない |
| 3 | 暗黙の前提・行間の説明 | 成果物に直接書かれていないが、前提としている知識・制約（例: 既存ファイルの上書き方針、冪等性の保証範囲）を説明する | 前提の食い違いに気づけない |
| 4 | 未決定事項と論点の洗い出し | 決めきれなかったこと、後続カテゴリで決定が必要なことを列挙する | 未決定事項が放置される |
| 5 | 人間に求める意思決定事項の明示 | 「何を確認・判断してほしいか」を具体的に問いかける | 承認が形骸化する |

## 品質基準

**「この説明だけ読めばカテゴリ完了の承認可否を判断できる」品質を担保すること。**

具体的には:

- 作成・変更されたファイルが全件列挙され、その内容が把握できる
- プロジェクト固有の判断（既存ファイルとの差分対応・設定値の選択等）が明示されている
- 冪等実行した場合に変更がなかったファイルは「変更なし」と明示する
- 次のカテゴリへの前提条件が満たされたかどうかを明示する
- 自信度が低い箇所（`<!-- AI-UNCERTAIN -->` マーク付き）が重点的に説明されている

## 出力フォーマット

```markdown
## ブリーフィング: [カテゴリ名] セットアップ完了

### 実行内容の全体像
- カテゴリ: [fw / project / env / conventions / ci / branch-protection / mocks]
- 作成・変更ファイル: N 件
  - `[パス]` — [内容の概要]
  - `[パス]` — [内容の概要]
  - ...
- 変更なし（冪等）: [該当ファイルがあれば列挙、なければ省略]

### AI が行った判断と根拠
[主要な設定値・方針の選択理由。代替案があれば比較]

### 暗黙の前提
[成果物に書かれていないが前提とした事項（例: 既存ファイルの扱い、スキップした手順）]

### 未決定事項・次カテゴリへの引き継ぎ
[後続で決定が必要な事項、次カテゴリの前提条件が満たされたかどうか]

### AI 自信度
[自信度が低い箇所の説明。全箇所に AI-UNCERTAIN マークを付与済みであること]

### 確認事項
[人間に判断を求める具体的な問いかけ]

この内容で承認しますか？承認後、次のカテゴリに進みます。
```

## カテゴリ別の重点確認項目

ブリーフィングの「確認事項」には、カテゴリに応じた以下の重点項目を必ず含める。

| カテゴリ | 重点確認項目 |
|---------|-----------|
| `fw` | `aidd-framework/` の配置が正しいか。CLAUDE.md / AGENTS.md の `@` インクルードが機能するか |
| `project` | GitHub Labels・ディレクトリ・雛形ドキュメントが全て作成されたか。CHARTER の内容が意図通りか |
| `env` | 言語・ツールの検出結果が正しいか。設定ファイルの内容がプロジェクトの方針と合っているか |
| `conventions` | 生成された規約ドキュメント 7 項目の内容が実態を反映しているか |
| `ci` | lefthook がインストールされ、pre-commit / pre-push フックが動作するか |
| `branch-protection` | ブランチ保護ルールが GitHub に正しく反映されているか |
| `mocks` | モック環境の dev server が起動するか（`mocks/` で起動コマンドを実行）。shadcn/ui のテーマ設定が意図通りか |
