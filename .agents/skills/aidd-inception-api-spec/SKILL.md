# /aidd-inception-api-spec — API仕様 + ストーリードラフト生成

G1 後・`/aidd-new-epic` 実行前に任意で使用するスキル。機能意図を受け取り、AI が対話的にリサーチ・補完を行いながら API エンドポイント仕様と主要ストーリードラフトを共同制作する。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| G1 通過済み（Phase 定義書が承認済み） | `docs/requirements/PD-*.md` にステータス「承認済み」が存在する | `/aidd-new-phase` を先に実行 |

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| `/aidd-research` | Step 1 | 機能意図に関連する類似 API・業界標準・設計パターン |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| 機能意図 または 機能名 | 引数 | 任意 | 省略時はユーザーに確認 |
| 技術スタック | 会話内 | 任意 | REST / GraphQL / gRPC 等。未指定時は機能意図から推定 |
| 参考資料 | 会話内 | 任意 | 類似 API ドキュメント・OpenAPI 定義等 |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| API エンドポイント一覧 | Markdown テーブル | エンドポイント・メソッド・概要・対応機能意図 |
| リクエスト/レスポンス例 | Markdown + JSON | 主要エンドポイントのサンプル（エラーケース含む） |
| ストーリードラフト | Markdown | API 操作から逆算した主要ストーリー一覧 |
| 引き渡しメモ | Markdown | `/aidd-new-epic` Step 0 インプット互換フォーマット |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-new-epic` | API 仕様・ストーリードラフト完成後、成果物を引き渡して Epic 定義に進む |

## Step 0: 入力受付

1. 機能意図が未指定の場合はユーザーに確認する:

   ```
   /aidd-inception-api-spec を開始しますか？
   - 対象機能意図または機能名を入力してください
   ```

2. 技術スタック（REST/GraphQL/gRPC）を確認する。詳細な推定手順は `references/tech-stack-detection.md` を参照。推定できない場合はユーザーに直接確認する。セッション開始を宣言する:

   ```
   🔌 inception-api-spec セッション開始
   対象機能意図: [機能意図]
   技術スタック: [推定または指定された技術スタック]

   まず類似 API をリサーチします...
   ```

## Step 1: 入力収集・AI リサーチ

**AC-E066-01: リサーチと見落とし補完:**

1. `/aidd-research` を呼び出して以下を調査する:
   - 類似 API の設計パターン・命名規則
   - 業界標準（RESTful 設計原則・OpenAPI 規約等）
   - ドメイン固有の規制・制約

2. リサーチ結果を以下の形式でまとめて提示し、**Step 2 でのエンドポイント設計に使用する**（AC-E070-03）:

   ```
   📚 リサーチ結果サマリー

   類似 API の傾向: [主要な設計パターン]
   業界標準: [参考にすべき設計原則]
   ドメイン固有の注意点: [法的・規制上の制約等]
   補完候補（Step 2 で提案するエンドポイント）: [認証・Rate Limit・Webhook 等]

   情報源: [URL または文書名]
   ```

   > このサマリーは Step 2 のエンドポイント設計に直接使用する。
   > 特に「補完候補」セクションを Step 2 の AI 補完提案の根拠とする。

**AC-E066-02: 見落としがちなエンドポイントの補完提案:**

リサーチ結果をもとに以下のカテゴリのエンドポイントを自動提案する:

- 認証・認可（ログイン・トークンリフレッシュ・ログアウト等）
- Rate Limiting・スロットリング対応
- Webhook・イベント通知
- バルク操作（一括登録・一括削除等）
- ページネーション・ソート・フィルタリング
- エラーレスポンス標準化

## Step 2: エンドポイント設計・対話確定

**AC-E066-03: エンドポイント候補の承認フロー:**

1. 機能意図 + リサーチ結果からエンドポイント候補を列挙する:

   ```
   🔌 エンドポイント候補一覧（N エンドポイント）

   | # | メソッド | パス | 概要 | 対応機能意図 |
   |---|---------|------|------|------------|
   | 1 | GET | /[リソース] | [一覧取得] | [機能意図] |
   | 2 | POST | /[リソース] | [新規作成] | [機能意図] |
   ...

   AIが補完提案したエンドポイント:
   - [認証エンドポイント]: [理由]
   - [エラーハンドリング]: [理由]
   ...

   RESTful 設計チェック（AC-E070-10）:
   - 命名: リソース名が複数形・スネークケースか
   - メソッド: CRUD 操作に適切なメソッドを使用しているか
   - 問題があれば候補提示時に指摘し、修正してから承認を取る

   この構成でよいですか？追加・削除があればお知らせください。
   ```

2. ユーザーが承認するまで次ステップに進まない

## Step 3: リクエスト/レスポンス例生成

詳細テンプレートは `references/endpoint-template.md` を参照して実行する。

**AC-E066-04: サンプル生成:**

確定した主要エンドポイントのリクエスト/レスポンス例を生成する。
フォーマットは `references/endpoint-template.md` に従うこと。

**AC-E066-05: エラーケース・バリデーション付記:**

各エンドポイントにエラーレスポンス例とバリデーションルールを付記する:

- 400 Bad Request（バリデーションエラー）
- 401 Unauthorized（認証エラー）
- 403 Forbidden（権限エラー）
- 404 Not Found（リソース未存在）
- 429 Too Many Requests（レート制限）

**AC-E066-09: RESTful 設計指摘:**

生成後、以下の観点で設計品質フィードバックを提示する:

```
💡 設計品質フィードバック

- 命名規則: [リソース名の一貫性確認]
- HTTP メソッド: [適切なメソッド使用確認]
- ステータスコード: [適切なコード使用確認]
- エラーレスポンス: [標準化確認]
- 改善提案: [具体的な提案]
```

## Step 4: ストーリードラフト導出

詳細フォーマットは `references/story-draft-format.md` を参照して実行する。

**AC-E066-06: ストーリー逆算:**

確定した API 操作からストーリーを逆算する:

```
📝 ストーリードラフト

機能意図から導出したストーリー:
- S1: As a [ペルソナ], I want to [行動], so that [理由].
  根拠: [エンドポイント名] ([メソッド] [パス])

AIが補完提案したストーリー:
- S2: As a [ペルソナ], I want to [行動], so that [理由].
  根拠: [補完理由: 認証フロー / エラーハンドリング 等]
```

## Step 5: 引き渡し準備

詳細フォーマットは `references/handoff-format.md` を参照して実行する。

**AC-E066-07: /aidd-new-epic 互換フォーマットで出力:**

API 仕様・ストーリードラフト・未決定事項を `/aidd-new-epic` Step 0 インプット互換フォーマットで出力する。

出力後:

```
🎉 inception-api-spec セッション完了

成果物を /aidd-new-epic に引き渡す準備ができました。

次のステップ:
→ /aidd-new-epic [Epic名] を実行してください。
→ Step 0（ストーリー詳細化）で上記の成果物を参照します。
```

## 重要なルール

- **発散フェーズ**: このスキルは発散フェーズに属する。アイデア・ストーリー候補・制約・対立点を積極的に増やすことが目的。成果物は「叩き台」であり、`/aidd-new-epic` の収束フェーズで精緻化される
- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う**
- **ユーザー確認なしの自動実行禁止** — エンドポイント候補の承認は必ずユーザーに委ねる
- **RESTful 設計原則に従う** — 独自の命名規則や非標準のパターンを推奨しない
- **情報源付記必須** — リサーチ結果には必ず参照元を付記する
- ブリーフィング省略禁止
