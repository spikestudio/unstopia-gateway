# /aidd-mob — Mob 式要件深掘りセッション

G1 後・`/aidd-new-epic` 実行前に任意で使用するスキル。複数の専門家ペルソナが自律リサーチ＋継続的な議論ラウンドを通じて、人間1人では思いつけない量・精度のストーリー候補・制約・前提条件を導出する。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| G1 通過済み（Phase 定義書が承認済み） | `docs/requirements/PD-*.md` にステータス「承認済み」が存在する | `/aidd-new-phase` を先に実行 |

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| `/aidd-research` | Step 2 | 機能意図に関連する業界・ドメイン・競合情報 |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| Epic 名 または 機能意図 | 引数 | 任意 | 省略時はユーザーに確認 |
| `--skip` | 引数 | 任意 | セッションをスキップして即終了 |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| mob セッション成果物 | 対話出力 | ストーリー候補・制約・前提条件（`/aidd-new-epic` Step 0-2 互換フォーマット） |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-new-epic` | mob セッション完了後、成果物を引き渡して Epic 定義に進む |

## Step 0: 入力受付・スキップ判定

1. `$ARGUMENTS` に `--skip` が含まれる場合:

   ```
   mob セッションをスキップしました。
   → /aidd-new-epic で Epic 定義に進んでください。
   ```

   と表示して正常終了する

2. Epic 名・機能意図が指定されていない場合はユーザーに確認する（スキップを選択できる）:

   ```
   mob セッションを開始しますか？
   - 対象 Epic 名または機能意図を入力してください
   - スキップする場合は「skip」と入力してください
   ```

   「skip」が入力された場合は `--skip` 指定と同様にスキップを明示通知して正常終了する
3. Phase 定義書から対象 Epic の機能意図を取得する（見つからない場合はユーザーが入力した内容をそのまま使用）
4. セッション開始を宣言する:

   ```
   🎯 mob セッション開始
   対象機能意図: [機能意図]

   まずペルソナ構成を確定します...
   ```

## Step 1: ペルソナ構成

詳細手順は `references/persona-catalog.md` を参照して実行する。

**AC-E064-08: 固定ペルソナの読み込み（優先順位順）:**

1. `.aidd-mob-config` が存在する場合 → `fixed_personas` を読み込む
2. CLAUDE.md に `mob_fixed_personas:` セクションが存在する場合 → そこから読み込む
3. 上記いずれも未設定の場合 → デフォルト固定ペルソナ（ビジネスアナリスト）を適用する

**AC-E064-09: 追加ペルソナの自動選択:**

機能意図を分析し、`references/persona-catalog.md` のカタログから追加ペルソナを最低3名選択する。各ペルソナの種別・役割・選択理由を提示する。自動選択手順は `references/persona-catalog.md` の「ペルソナ自動選択の手順」セクションに従う。

**AC-E064-10: ユーザー承認フロー:**

提示したペルソナ構成をユーザーが承認・差し替え・追加できる。**承認なしにセッション（Step 2）を開始しない。**

- 「変更なし」「OK」「承認」→ そのまま Step 2 へ
- 「〇〇を追加して」→ カタログから該当ペルソナを追加して再提示
- 「〇〇を××に差し替えて」→ 指定ペルソナを差し替えて再提示
- 再提示後も承認待ちを継続する

**追加ペルソナ最低人数の自動補完:**

承認時点で追加ペルソナが1名以下の場合、`references/persona-catalog.md` から機能意図に適したペルソナを自動選択して補完し、3名以上にしてから Step 2 を開始する。ユーザーへの確認は不要（多様性を保証するための自動補完）。

**提示フォーマット:**

```
📋 ペルソナ構成案

固定参加（常時）:
  🔵 ビジネスアナリスト — ビジネス要件・ROI・ステークホルダー調整の視点

AI が選択した追加ペルソナ（理由付き）:
  🟢 業界専門家（[業界名]） — [選択理由]
  🟢 UX デザイナー — [選択理由]
  🟢 セキュリティエンジニア — [選択理由]

この構成でよいですか？変更があればお知らせください。
（例: 「財務アナリストを追加して」「UX デザイナーをユーザーリサーチャーに変えて」）
```

## Step 2: 自律リサーチ

**AC-E064-01: 並行起動と初回発言（重要）:**

Step 1 で承認されたペルソナ構成の**全ペルソナを同時に（並行で）起動**する。
1ペルソナずつ順番に発言するのではなく、全ペルソナが揃って発言することで多角的な視点を同時に得る。

**AC-E070-02: 並行起動の具体的な実装:**

1 メッセージ内で全ペルソナを Agent() で同時起動する:

```python
# 全ペルソナを1メッセージ内で並行起動（順次ではない）
results = [
    Agent(prompt=f"""あなたは {persona['name']}（{persona['role']}）です。
機能意図「{feature_intent}」について、あなたの専門領域でリサーチを行い、
初期見解・問い・懸念点を発言してください。
情報源（URL・文書名）を必ず付記すること。""")
    for persona in approved_personas
]
# 全ペルソナの発言を収集してユーザーに一括提示
```

各ペルソナは独立した担当領域でリサーチを行い、独立した初回発言を行う。

**AC-E064-02: 自律リサーチと情報源付記:**

各ペルソナが `/aidd-research` を呼び出して機能意図に関連する情報を収集する。

**リサーチ対象（ペルソナごとに担当領域を分担）:**

- 関連業界・市場トレンド
- 競合プロダクトの実装事例
- 規制・法的制約
- 技術的前提・制約
- ユーザー行動パターン

**AC-E064-13: 情報源付記:**

リサーチ完了後、各ペルソナが初回発言を行う。発言には必ずリサーチで参照した情報源（URL・文書名）を付記する。情報源のない主張は禁止する。

**初回発言フォーマット:**

```
**[ペルソナ名]（[役割]）:**

[リサーチ結果を踏まえた初期見解・問い・懸念点]

情報源: [URL または文書名]
```

> 初回発言はラウンドとしてカウントしない。初回発言完了後に Step 3（ラウンド1）を開始する。

## Step 3: 議論ラウンド

詳細手順は `references/discussion-round.md` を参照して実行する。

**AC-E064-03: ラウンドの定義と最低2ラウンド実行（最重要）:**

- **最低2ラウンド実行する**（1ラウンドで収束させない。内容が豊かでも必ず2ラウンド実行する）
- ラウンド1終了時にユーザーが「B) 収束」を選択した場合、以下のメッセージを表示して**収束を拒否し、ラウンド2を強制実行する**:

  ```
  ⚠️ 最低2ラウンド実行が必要です
  ラウンド1で収束させることはできません。ラウンド2を実行してから収束を選択してください。

  🔄 ラウンド 2 開始
  ```

**AC-E064-04: 前ラウンド参照の義務:**

- 1ラウンド = 全ペルソナが**前ラウンドの発言を参照して**反論・補足・深掘りを行うサイクル
- 各ペルソナは前ラウンドの自分の発言を繰り返さず、議論を進展させる内容を発言する

**AC-E064-05・06: ラウンド終了時の論点サマリーとユーザー判断:**

- 各ラウンド終了後に論点サマリーを提示し、ユーザーが続行／収束を明示するまで次のラウンドに進まない

**AC-E064-07: ユーザー介入:**

- ユーザーはいつでも発言を投入でき、次ラウンドで全ペルソナが参照する

**AC-E064-14: 堂々巡り検出:**

- 同一論点が2ラウンド以上継続した場合は堂々巡りを明示して新しい問いを立て直す

**ラウンド終了時の論点サマリーフォーマット:**

```
📊 ラウンド N 論点サマリー

✅ 合意点:
  - [全ペルソナが同意している内容]
  - [議論で確認できた前提・制約]

❓ 未決定事項:
  - [まだ答えが出ていない重要な問い]
  - [ペルソナが提起したが深掘り不足の論点]

⚡ 対立点:
  - [ペルソナ間で意見が分かれている点と各立場]

---
続行しますか？

A) 続行（ラウンド N+1 へ）
B) 収束（成果物を出力する）※ ラウンド2以降のみ選択可
C) 特定の論点に絞って続行（論点を指定してください）
```

## Step 4: 成果物出力

**AC-E064-11: 収束宣言後に `/aidd-new-epic` Step 0-2 互換フォーマットで出力する:**

ユーザーが収束を宣言した後、全ラウンドの議論を統合して以下の 6 セクションを出力する。
フォーマットの詳細は `references/output-format.md` に従うこと。

| セクション | 内容 | /aidd-new-epic での使用場所 |
|-----------|------|--------------------------|
| ストーリー候補 | 人間定義分 + mob 発見分を区別して列挙 | Step 0-2: ストーリー詳細化の入力 |
| 制約一覧 | 技術的・法的・ビジネス的制約（根拠付き） | Step 1: AC 導出時の制約条件 |
| 前提条件一覧 | Epic 成立に必要な前提（根拠付き） | Step 1: AC の前提条件セクション |
| 未決定事項 | セッションで答えが出なかった問い | Step 1〜3: 設計判断・ADR が必要な箇所の特定 |
| 対立点・トレードオフ | ペルソナ間の意見の相違 | Step 3: `/aidd-adr` での意思決定入力 |
| 参照情報源 | リサーチ結果の URL・文書名 | Step 0-2: 根拠の確認 |

**出力後のアクション:**

```
🎉 mob セッション完了

次のステップ:
→ /aidd-inception を実行してください（画面モック・API仕様・CLI仕様を成果物として生成）。
   mob の成果物（ストーリー候補・制約・対立点）を /aidd-inception に引き渡してください。
   引き渡し方法: references/output-format.md の「/aidd-inception への引き渡し方」を参照。
```

## 重要なルール

- **発散フェーズ**: このスキルは発散フェーズに属する。アイデア・ストーリー候補・制約・対立点を積極的に増やすことが目的。成果物は「叩き台」であり、`/aidd-new-epic` の収束フェーズで精緻化される
- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う**
- **ユーザー確認なしの自動実行禁止** — ペルソナ構成の承認、各ラウンドの続行判断は必ずユーザーに委ねる
- **短絡的な収束を禁止** — 最低2ラウンドを実行し、1ラウンドで「結論が出た」とみなしてはならない
- **前ラウンド参照の義務** — 各ペルソナは前ラウンドの発言を参照して議論を深化させる。初回発言の繰り返しを禁止する
- ブリーフィング省略禁止
