# /aidd-inception-cli — CLIコマンド仕様 + ストーリードラフト生成

G1 後・`/aidd-new-epic` 実行前に任意で使用するスキル。機能意図を受け取り、AI が対話的にリサーチ・補完を行いながら CLI コマンド仕様と主要ストーリードラフトを共同制作する。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| G1 通過済み（Phase 定義書が承認済み） | `docs/requirements/PD-*.md` にステータス「承認済み」が存在する | `/aidd-new-phase` を先に実行 |

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| `/aidd-research` | Step 1 | 機能意図に関連する類似 CLI ツール・UX パターン・Unix 哲学 |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| 機能意図 または 機能名 | 引数 | 任意 | 省略時はユーザーに確認 |
| CLI フレームワーク | 会話内 | 任意 | cobra / click / yargs 等。未指定時は言語から推定 |
| 参考資料 | 会話内 | 任意 | 類似 CLI ツール URL 等 |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| コマンド一覧 | Markdown テーブル | コマンド名・サブコマンド・概要・対応機能意図 |
| フラグ定義 | Markdown テーブル | フラグ名・型・デフォルト値・説明 |
| 使用例 | コードブロック | 主要ユースケースのコマンド例 |
| ストーリードラフト | Markdown | CLI 操作から逆算した主要ストーリー一覧 |
| 引き渡しメモ | Markdown | `/aidd-new-epic` Step 0 インプット互換フォーマット |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-new-epic` | CLI 仕様・ストーリードラフト完成後、成果物を引き渡して Epic 定義に進む |

## Step 0: 入力受付

1. 機能意図が未指定の場合はユーザーに確認する:

   ```
   /aidd-inception-cli を開始しますか？
   - 対象機能意図または機能名を入力してください
   ```

2. CLI フレームワーク・言語を確認する。詳細な推定手順は `references/tech-stack-detection.md` を参照。推定できない場合はユーザーに直接確認する。セッション開始を宣言する:

   ```
   🖥️ inception-cli セッション開始
   対象機能意図: [機能意図]
   CLI フレームワーク: [推定または指定されたフレームワーク]

   まず類似 CLI ツールをリサーチします...
   ```

## Step 1: 入力収集・AI リサーチ

**AC-E067-01: リサーチと見落とし補完:**

1. `/aidd-research` を呼び出して以下を調査する:
   - 類似 CLI ツール（`gh`・`docker`・`kubectl`・`npm` 等）の UX パターン
   - Unix 哲学・POSIX 準拠の設計原則
   - CLI フレームワーク（cobra/click/yargs 等）の慣習

2. リサーチ結果を以下の形式でまとめて提示し、**Step 2 でのコマンド体系設計に使用する**（AC-E070-03）:

   ```
   📚 リサーチ結果サマリー

   類似 CLI ツールの傾向: [主要な設計パターン・コマンド体系]
   Unix 哲学のポイント: [参考にすべき設計原則]
   フレームワーク慣習: [使用フレームワークの標準的な設計]
   補完候補（Step 2 で提案するコマンド）: [help・version・completion・config 等]

   情報源: [URL または文書名]
   ```

   > このサマリーは Step 2 のコマンド体系設計に直接使用する。
   > 特に「補完候補」セクションを Step 2 の AI 補完提案の根拠とする。

**AC-E067-02: 見落としがちなコマンドの補完提案:**

リサーチ結果をもとに以下のカテゴリのコマンドを自動提案する:

- `help` / `--help`（ヘルプ表示）
- `version` / `--version`（バージョン表示）
- `completion`（シェル補完スクリプト生成）
- `config`（設定ファイル管理）
- `init`（初期化）
- グローバルフラグ（`--verbose`・`--quiet`・`--output` 等）

## Step 2: コマンド体系設計・対話確定

**AC-E067-03: コマンド候補の承認フロー:**

1. 機能意図 + リサーチ結果からコマンド候補を設計する:

   ```
   🖥️ コマンド体系案

   メインコマンド: [コマンド名]

   | # | コマンド | サブコマンド | 概要 | 対応機能意図 |
   |---|---------|------------|------|------------|
   | 1 | [cmd] | [subcmd] | [説明] | [機能意図] |
   ...

   AIが補完提案したコマンド:
   - help: [理由]
   - version: [理由]
   - completion: [理由]

   Unix 哲学チェック（AC-E070-11）:
   - 単一責任: 各コマンドが1つの目的に特化しているか
   - POSIX 準拠: フラグ形式が `--long-form` / `-s` 形式か
   - 問題があれば候補提示時に指摘し、修正してから承認を取る

   この構成でよいですか？追加・削除があればお知らせください。
   ```

2. ユーザーが承認するまで次ステップに進まない

## Step 3: フラグ定義・使用例生成

詳細テンプレートは `references/command-template.md` を参照して実行する。

**AC-E067-04: フラグ定義:**

各コマンドのフラグ・オプションを定義する。
フォーマットは `references/command-template.md` に従うこと。

**AC-E067-05: 使用例生成:**

主要ユースケースのコマンド例を生成する:

```
📋 使用例

# [ユースケース1の説明]
$ [コマンド] [サブコマンド] [引数] [フラグ]

# [ユースケース2の説明]
$ [コマンド] [サブコマンド] --[フラグ]=[値]
```

**AC-E067-09: Unix 哲学・設計品質フィードバック:**

生成後、以下の観点で設計品質フィードバックを提示する:

```
💡 設計品質フィードバック

- 命名規則: [コマンド名・フラグ名の一貫性確認]
- Unix 哲学: [単一責任・パイプライン・テキスト処理の確認]
- POSIX 準拠: [フラグ形式の確認]
- エラー処理: [終了コードの定義確認]
- 改善提案: [具体的な提案]
```

## Step 4: ストーリードラフト導出

詳細フォーマットは `references/story-draft-format.md` を参照して実行する。

**AC-E067-06: ストーリー逆算:**

確定した CLI 操作からストーリーを逆算する:

```
📝 ストーリードラフト

機能意図から導出したストーリー:
- S1: As a **[ペルソナ]**, I want to [行動], so that [理由].
  根拠: [コマンド名] [サブコマンド]

AIが補完提案したストーリー:
- S2: As a **[ペルソナ]**, I want to [行動], so that [理由].
  根拠: [補完理由: シェル補完 / 設定管理 / ヘルプ表示 等]
```

## Step 5: 引き渡し準備

詳細フォーマットは `references/handoff-format.md` を参照して実行する。

**AC-E067-07: /aidd-new-epic 互換フォーマットで出力:**

CLI 仕様・ストーリードラフト・未決定事項を `/aidd-new-epic` Step 0 インプット互換フォーマットで出力する。

出力後:

```
🎉 inception-cli セッション完了

成果物を /aidd-new-epic に引き渡す準備ができました。

次のステップ:
→ /aidd-new-epic [Epic名] を実行してください。
→ Step 0（ストーリー詳細化）で上記の成果物を参照します。
```

## 重要なルール

- **発散フェーズ**: このスキルは発散フェーズに属する。アイデア・ストーリー候補・制約・対立点を積極的に増やすことが目的。成果物は「叩き台」であり、`/aidd-new-epic` の収束フェーズで精緻化される
- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う**
- **ユーザー確認なしの自動実行禁止** — コマンド体系の承認は必ずユーザーに委ねる
- **Unix 哲学に従う** — 単一責任・パイプライン対応・テキスト出力を基本とする
- **情報源付記必須** — リサーチ結果には必ず参照元を付記する
- ブリーフィング省略禁止
