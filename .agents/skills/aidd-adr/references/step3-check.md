# Step 3: 既存 ADR との矛盾チェック — 詳細手順

新規 ADR と `docs/architecture/adr/` 配下の既存 ADR を比較し、矛盾を検出・分類してユーザーに提示する。

## 入力

- 新規 ADR（Step 2 で保存済み）
- `docs/architecture/adr/` 配下の全既存 ADR

## 手順

### 1. 既存 ADR の確認

```bash
ls docs/architecture/adr/
```

既存 ADR が 0 件の場合はこの Step をスキップし、Step 4 に進む。

### 2. 矛盾チェックの実行

以下の観点で既存 ADR と新規 ADR を比較し、矛盾を検出する:

| 観点 | チェック内容 |
|------|------------|
| 技術選定の矛盾 | 同一の技術要素（認証方式・DB・フレームワーク等）について、異なる決定がされていないか |
| 方針の矛盾 | 既存 ADR の「決定事項」と新規 ADR の「決定事項」が相互に矛盾していないか |
| 廃止されるべき ADR | 新規 ADR によって既存 ADR が無効化される場合、廃止処理が必要か |

### 3. 矛盾の分類と提示

検出した矛盾を重大度で分類し、ユーザーに提示する:

| 重大度 | 定義 | 対応 |
|--------|------|------|
| **MUST FIX** | 矛盾が明確で、そのまま実装すると不整合が生じる | ADR を修正してから確定する |
| **SHOULD FIX** | 将来的に矛盾が顕在化する可能性がある | 修正を推奨するが、ユーザーが判断する |

**矛盾なしの場合:**

```
✅ 矛盾チェック: 問題なし
既存 ADR N 件と比較しました。矛盾は検出されませんでした。
Step 4（ドキュメント連動）に進みます。
```

**矛盾ありの場合:**

```
⚠️ 矛盾チェック: N 件の問題を検出しました

| # | 重大度 | 矛盾する ADR | 矛盾の内容 | 推奨対応 |
|---|--------|------------|-----------|---------|
| 1 | MUST FIX | ADR-NNN | [矛盾の説明] | [修正案] |

MUST FIX がある場合、ADR を修正してから確定することを推奨します。
このまま確定しますか？（MUST FIX がある場合は修正を強く推奨）
```

### 4. ユーザーの承認

- MUST FIX がある場合: ADR を修正してから確定する（強く推奨）
- ユーザーが「このまま確定」を選択した場合は ADR に `<!-- AI-UNCERTAIN: 既存 ADR との矛盾の可能性 -->` を付記して続行する

## 成果物

矛盾チェック結果の提示（問題がある場合のみ）
