# /aidd-research — リサーチエンジン

外部情報を調査し、`docs/research/*.md` にリサーチドキュメントを生成して根拠付き推薦を返す。他スキルからの内部呼び出し、単独実行の両方に対応する。

## インターフェース

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| 調査対象 | 引数 | 必須 | 調査する対象（自然言語） |
| 調査の目的 | prompt 内 | 必須 | なぜこの調査が必要か |
| 出力先 | prompt 内 | 任意 | `docs/research/*.md`（省略時は自動命名） |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| STATUS | テキスト | PASS / ERROR |
| サマリー | テキスト | 調査結果の要約と推薦 |
| 成果物 | ファイル | `docs/research/*.md` |

## Step 1: 入力受付

引数から調査対象を取得する。引数なしの場合は自然言語で指定するよう案内する。

## Step 2: リサーチ種別の自動判定

調査対象の記述から種別を自動判定する:

| 種別 | 例 |
|------|-----|
| 業界・ドメイン調査 | 「広告配信の業界標準ワークフロー」 |
| 技術選定 | 「Helm chart vs 手書き manifest」 |
| 仕様調査 | 「Claude Code の agents/ 仕様」 |
| ベストプラクティス | 「monorepo のテスト戦略」 |
| 類似プロダクト調査 | 「プロジェクト管理ツールの UX 比較」 |

判定が曖昧な場合はユーザーに確認する。

## Step 3: 既存リサーチの鮮度チェック

`docs/research/` の既存リサーチを確認する:

- 同一テーマのリサーチが **7 日以内** → 既存結果を返却（再調査スキップ）
- 7 日超、または存在しない → 新規調査を実行
- ユーザーが「最新で再調査」と指示した場合は鮮度チェックをスキップ

## Step 4: 情報収集（並列実行）

以下の 2 つの Agent を **1 メッセージ内で同時起動** し、結果を集約する:

```
Agent(prompt="""
  以下の調査対象に関連するリポジトリ内の情報を収集し、要点をサマリーとして返してください。

  調査対象: [調査対象]
  調査目的: [調査目的]

  Grep/Glob を使ってリポジトリ全体を自由に検索し、関連するファイルを特定・読み込んでください。
  特定のディレクトリに限定せず、調査対象に応じて適切な範囲を判断してください。

  出力: 関連する情報の要点リスト（ソースファイルパス付き）
""")

Agent(prompt="""
  以下の調査対象について外部ソースを調査し、要点をサマリーとして返してください。

  調査対象: [調査対象]
  調査目的: [調査目的]
  調査種別: [種別]

  手順:
  1. WebSearch で複数クエリを実行（公式ドキュメント優先）
  2. 上位の公式ドキュメントを WebFetch で取得
  3. 複数ソースを突合して要点を整理

  出力: 事実・比較・推薦の要点リスト（ソース URL と参照日付き）
""")
```

両 Agent の結果を受け取り Step 5 で統合する。

WebFetch/WebSearch が使えない環境では外部 Agent を省略し、LLM 学習知識で代替して `<!-- AI-UNCERTAIN: LLM依存 -->` マークを付与する。

## Step 5: 調査結果の構造化

調査結果を以下の構造でドキュメント化する:

```markdown
# リサーチ: {タイトル}

| 項目 | 内容 |
|------|------|
| 調査日 | YYYY-MM-DD |
| 調査目的 | {目的} |
| リサーチ種別 | {種別} |
| 依頼元 | {スキル名 or ユーザー直接} |

## 調査結果

### 事実
{ソース付きの事実整理}

### 比較表（技術選定の場合）

### 推薦
{根拠付き推薦}

### リスク・注意点

## ソース
- [ソース名](URL) — 参照日: YYYY-MM-DD
```

## Step 6: ドキュメント保存と結果返却

`docs/research/{slug}.md` に保存し、サマリーと推薦を返却する。

## 重要なルール

- 日本語で出力する
- **FRAMEWORK.md の意思決定基準に従う** — 問題の後回しを推奨しない
- 公式ドキュメントを最優先ソースとする — 理由: LLM 学習知識は古い可能性がある
- 複数ソースを突合し、単一ソースに依存しない — 理由: 情報の信頼性を担保する
- 推薦には必ず根拠を付ける — 理由: 根拠なしの推薦は意思決定に使えない
- 外部アクセス不可時は正直に AI-UNCERTAIN マークを付与する — 理由: 情報の確度を人間が判断できるようにする
- 既存リサーチの鮮度チェック（7 日）を必ず実施する — 理由: 重複調査を防ぐ
- リサーチドキュメントにソース URL と参照日を必ず記載する — 理由: 後から検証可能にする
- 成果物生成・実装完了時にコミットを自動実行する。ユーザーの指摘反映後・ステップ承認後などの状態遷移ごとにもコミットする。push と PR 作成はユーザーの確認を待つ
