# /aidd-feedback-recorder — 訂正イベント記録

実装中に訂正・指摘が発生したとき、修正の実施に加えて feedback Issue の作成と AI エージェント指示ファイルへの追記を行う。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| GitHub リポジトリが存在 | `gh repo view` | GitHub リポジトリを作成してください |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| 訂正・指摘の内容 | 対話 | Yes | ユーザーからの指摘フレーズで発動。種別・内容・正しい対応を対話で特定する |

### capability 呼び出し

該当なし

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| feedback Issue | GitHub Issue | `feedback` ラベル付きの Issue（ユーザー承認後に作成） |
| AI エージェント指示ファイル追記 | ファイル更新 | CLAUDE.md / AGENTS.md への発見事項追記（種別に応じて） |

### 後続スキル

該当なし

## 発動条件

ユーザーから実装の誤りを指摘された、AC と実装の矛盾が発見された、規約に記載されていないパターンが発生した、AI が同じ間違いを繰り返した、等の場合に発動する。

## Step 1: 入力受付（訂正内容の特定）

まず指摘された内容を修正する（修正が最優先）。

修正と並行して、以下を特定する:

- **種別**: 仕様変更 / 規約追記 / AI ミスパターン / AI レビュー漏れ
- **内容**: 何が間違っていたか
- **正しい対応**: 本来どうすべきだったか
- **関連 Task**: 現在のブランチから推定

## Step 2: feedback Issue の作成

修正完了後、ユーザーに「この指摘を feedback Issue として記録しますか？」と確認する。

承認された場合:

```bash
gh issue create \
  --title "FB: [種別の短縮] [概要]" \
  --label feedback \
  --body "[種別・関連Task・内容・対応をテンプレートに沿って記載]"
```

## Step 3: 完了処理（AI エージェント指示ファイルへの追記）

種別に応じて追記を提案する:

- **AI ミスパターン**: 「プロジェクト固有の発見事項」セクションに追記
- **AI レビュー漏れ**: 検出すべきだったパターンを「プロジェクト固有の発見事項」セクションに追記
- **規約追記**: 該当する規約ドキュメントに追記
- **仕様変更**: Epic 仕様書の該当箇所を更新

ユーザーに追記内容を確認してから実行する。

## 重要なルール

- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う** — 問題の後回しを推奨しない
- 修正が最優先。Issue 作成は修正完了後に行う — 理由: ユーザーの作業を止めない
- ユーザーへの確認なしに Issue を作成しない — 理由: 不要な Issue が増えると管理コストが上がる
- 軽微な指摘（typo、フォーマット等）は Issue 化しない。「同じミスが再発しうるパターン」のみ対象 — 理由: ノイズを減らす
- AI ミスパターンの追記は具体的に記述する。「注意する」ではなく「X の場合は Y を確認する」の形式 — 理由: 曖昧な記述は再発防止にならない
- 規約追記は既存の規約ドキュメントの記述スタイルに合わせる — 理由: 一貫性の維持
- 成果物生成・実装完了時にコミットを自動実行する。ユーザーの指摘反映後・ステップ承認後などの状態遷移ごとにもコミットする。push と PR 作成はユーザーの確認を待つ
