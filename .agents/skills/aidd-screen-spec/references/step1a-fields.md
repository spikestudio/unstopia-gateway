# Step 1a: フィールド定義・バリデーション生成 — 詳細手順

各 SCR-ID に対してフィールド定義とバリデーションルールを対話的に生成し、画面仕様書の対応セクションに記録する詳細手順を定義する。

## 対応 AC

| AC-ID | 内容 |
|-------|------|
| AC-E053-01 | 各 SCR-ID に対応する画面仕様書にフィールド定義（フィールド名・型・必須/任意）が含まれる |
| AC-E053-02 | フィールドを持たない画面（ナビゲーションのみ等）では「フィールドなし」と明記される |
| AC-E053-03 | 各フィールドのバリデーションルール（形式・文字数・依存条件）が画面仕様書に定義される |
| AC-E053-04 | バリデーションルールが未定義のフィールドには「TBD」と明記され、後から更新できる（スキルはエラー停止しない） |

## 入力

- Step 0 で読み込んだ screen-list ドキュメント（SCR-ID 一覧・画面名・種別）
- Epic 仕様書（存在する場合: バリデーションルール・フィールド定義の詳細）
- 既存の参考実装・設計書（存在する場合）

## 処理フロー

### 1. SCR-ID ループの開始

Step 0 で取得した全 SCR-ID に対して、以下の手順を順番に実行する。

```
処理対象: N 件の SCR-ID
処理順: screen-list の記載順
```

### 2. 各画面のフィールド定義生成（AC-E053-01, AC-E053-02）

各 SCR-ID ごとに以下を実行する:

#### 2-1. 画面種別の判定

screen-list から画面種別（一覧 / 詳細 / フォーム / ダッシュボード / モーダル / ウィザード / エラー / 未分類）を取得する。

#### 2-2. フィールドの存在確認

画面種別に応じて、入力フィールドを持つ可能性があるか判定する。

| 画面種別 | フィールドを持つ可能性 | 備考 |
|---------|-------------------|------|
| フォーム | 高い | 入力フィールドが主体 |
| ウィザード | 高い | ステップごとに入力フィールドを持つ |
| 一覧 | 低い（検索フィルターは含む場合あり） | 検索・フィルター項目があれば記録 |
| 詳細 | 低い（表示フィールドは記録する） | 表示専用フィールドは「任意」として記録 |
| ダッシュボード | 低い | 集計表示が主体のためフィールドは少ない |
| モーダル | 画面内容による | 確認ダイアログはフィールドなし、入力モーダルはあり |
| エラー | ない | 「フィールドなし」と記録 |
| 未分類 | 不明 | ユーザーに確認して判断 |

#### 2-3. フィールド定義の生成

**フィールドがある場合:**

screen-list・Epic 仕様書・画面名から推定し、以下の情報を生成する:

| フィールド名 | 型 | 必須/任意 | 説明 |
|------------|-----|----------|------|
| [フィールド名] | [型] | [必須/任意] | [フィールドの説明] |

フィールドの型は以下のいずれかを使用する:

| 型 | 説明 | 例 |
|----|------|-----|
| 文字列 | テキスト入力 | 名前、メールアドレス、説明文 |
| 数値 | 数値入力 | 金額、数量、年齢 |
| 真偽値 | チェックボックス、トグル | 有効/無効、同意フラグ |
| 日付 | 日付選択 | 生年月日、期限日 |
| 選択肢 | セレクトボックス、ラジオボタン | 都道府県、カテゴリー |
| ファイル | ファイルアップロード | 画像、PDF、添付書類 |
| 参照 | 外部キー参照（他エンティティの ID） | ユーザー ID、カテゴリー ID |
| その他 | 上記に該当しない型 | カスタム型、複合型 |

**フィールドがない場合（AC-E053-02）:**

```
フィールドなし（[画面種別の説明。例: ナビゲーションのみの画面]）
```

#### 2-4. ユーザーへの確認

生成したフィールド定義をユーザーに提示し、確認を求める。

```
## [SCR-ID] [画面名] — フィールド定義

| フィールド名 | 型 | 必須/任意 | 説明 |
|------------|-----|----------|------|
| [フィールド名] | [型] | [必須/任意] | [説明] |
...

フィールド定義の内容を確認してください。
- 追加・変更・削除があれば指摘してください
- 問題なければ「OK」で次のステップに進みます
```

ユーザーの指摘があれば修正して再提示する。「OK」等の明確な承認を得てから Step 2-5 に進む。

### 3. 各画面のバリデーションルール生成（AC-E053-03, AC-E053-04）

フィールド定義が承認されたら、各フィールドのバリデーションルールを生成する。

#### 3-1. バリデーションルールの推定

フィールド名・型・Epic 仕様書の内容からバリデーションルールを推定する。

**ルール種別:**

| ルール種別 | 説明 | 例 |
|----------|------|-----|
| 形式 | 入力値の形式制約 | メールアドレス形式（RFC 5322 準拠）、電話番号形式 |
| 文字数 | 最小・最大文字数 | 1〜100文字、最大255文字 |
| 数値範囲 | 最小・最大値 | 1〜999、0以上 |
| 必須 | 空欄を許可しない | 空欄不可 |
| 依存条件 | 他フィールドの値に依存する制約 | フィールドAが「はい」の場合のみ必須 |
| 選択肢制約 | 有効な選択肢の範囲 | 管理者 / 一般ユーザー のいずれか |

**未定義の場合（AC-E053-04）:**

バリデーションルールが Epic 仕様書や画面名から推定できない場合は「TBD」と記録して続行する。スキルはエラーで停止しない。

#### 3-2. バリデーションルールの生成フォーマット

| フィールド名 | ルール種別 | ルール内容 |
|------------|----------|----------|
| [フィールド名] | [ルール種別] | [ルール内容] |
| [フィールド名] | （全般） | TBD |

#### 3-3. ユーザーへの確認

生成したバリデーションルールをユーザーに提示し、確認を求める。

```
## [SCR-ID] [画面名] — バリデーションルール

| フィールド名 | ルール種別 | ルール内容 |
|------------|----------|----------|
| [フィールド名] | [ルール種別] | [ルール内容] |
...

TBD としたフィールド: [フィールド名一覧]（後から更新可能）

バリデーションルールを確認してください。
- 追加・変更・削除があれば指摘してください
- TBD のルールが判明した場合は今すぐ追記できます
- 問題なければ「OK」で次の画面に進みます
```

ユーザーの指摘があれば修正して再提示する。「OK」等の明確な承認を得てから次の SCR-ID の処理に進む。

### 4. 画面仕様書への記録

ユーザーの承認後、`references/screen-spec-template.md` のフォーマットに従い、以下のセクションを画面仕様書ファイルに記録する。

**出力先:** `docs/design/screen-spec-[slug]/[SCR-ID]-[screen-slug].md`

**記録するセクション（Step 1a のスコープ）:**

```markdown
## フィールド定義（AC-E053-01, AC-E053-02）

<!-- AC-E053-01: フィールド名・型・必須/任意を含める -->
<!-- AC-E053-02: フィールドがない場合は「フィールドなし」と明記する -->

[フィールド定義テーブル、または「フィールドなし（[理由]）」]

---

## バリデーションルール（AC-E053-03, AC-E053-04）

<!-- AC-E053-03: 各フィールドのバリデーションルール（形式・文字数・依存条件）を定義する -->
<!-- AC-E053-04: ルールが未定義のフィールドには「TBD」と明記する -->

[バリデーションルールテーブル]
```

### 5. 全 SCR-ID の処理完了後

全 SCR-ID のフィールド定義とバリデーションルールの生成・承認が完了したら、以下のサマリーを表示する。

```
## Step 1a 完了サマリー

- 処理した SCR-ID: N 件
- フィールドあり: N 件
- フィールドなし: N 件
- TBD バリデーション: N 件（後から更新可能）

### 次のステップ
Step 1b（状態設計・ロール・API参照生成）に進みます。
```

Step 1b の詳細手順は `references/step1b-state-role-api.md` を参照して実行する。

## エラー処理

| ケース | 条件 | 振る舞い |
|--------|------|---------|
| フィールド型が不明 | 型が7種類に該当しない | 「その他」として記録し続行（AC-E053-01 の「不明な型は『その他』として記録」に準拠） |
| バリデーション未定義 | Epic 仕様書・画面名から推定不可 | 「TBD」として記録し続行（AC-E053-04 に準拠） |
| フィールド定義が不明 | 画面種別・画面名からフィールドを推定できない | ユーザーに確認を求め、指示に従って記録 |

## TBD 管理

- Step 1a で「TBD」と記録したバリデーションルールは、仕様が決まり次第、画面仕様書を直接更新して TBD を解消できる
- TBD の一覧は Step 3（ブリーフィング）でユーザーに提示する
- TBD が残っていても画面仕様書の生成・後続スキルの実行は可能

## 注意事項

- **フィールドなし画面でも画面仕様書ファイルは作成する**（`references/screen-spec-template.md` を使用し、フィールド定義セクションに「フィールドなし」と記録する）
- **1画面ずつユーザーの承認を得てから次へ進む**（自動的に全 SCR-ID を処理しない）
- **AC トレーサビリティを維持する**（各画面仕様書の「AC トレーサビリティ」セクションに AC-E053-01〜04 を記録する）
- **フィールド定義の承認とバリデーションルールの承認は別のステップで行う**（一度に提示して混乱を避けるため）
