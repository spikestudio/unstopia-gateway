# Step 1b: 状態設計・ロール・API参照生成 — 詳細手順

各 SCR-ID に対して状態設計・ロール定義・API エンドポイント参照を対話的に生成し、画面仕様書の対応セクションに記録する詳細手順を定義する。

## 対応 AC

| AC-ID | 内容 |
|-------|------|
| AC-E053-05 | 各画面仕様書に空状態・ローディング状態・エラー状態のデザイン記述が含まれる |
| AC-E053-06 | 利用者が対話的に各状態の設計内容を確認・追記・変更できる |
| AC-E053-07 | 各画面仕様書に閲覧可能なロールと操作可能なロールが明記される |
| AC-E053-08 | ロールが未定義の場合は「全ロール」または「要確認」と記録され、スキルはエラーで停止しない |
| AC-E053-09 | 画面が参照する API エンドポイント（パス・HTTP メソッド）が仕様書に記録される（任意項目） |
| AC-E053-10 | API エンドポイントが未確定の場合は「TBD」と記録し、スキルはエラーで停止しない |

## 入力

- Step 1a で生成済みの画面仕様書（フィールド定義・バリデーションルールが記録済み）
- Step 0 で読み込んだ screen-list ドキュメント（SCR-ID 一覧・画面名・種別）
- Epic 仕様書（存在する場合: ロール定義・API 仕様の詳細）
- API 仕様書（存在する場合: `docs/design/api-spec.md` 等）

## 処理フロー

### 1. SCR-ID ループの開始

Step 1a で処理した全 SCR-ID に対して、以下の手順を順番に実行する。

```
処理対象: Step 1a で処理した N 件の SCR-ID
処理順: screen-list の記載順（Step 1a と同じ順）
```

### 2. 各画面の状態設計生成（AC-E053-05, AC-E053-06）

各 SCR-ID ごとに以下を実行する:

#### 2-1. 画面種別に応じた状態の有無判定

画面種別に応じて、各状態が発生し得るか判定する。

| 画面種別 | 空状態 | ローディング状態 | エラー状態 |
|---------|--------|----------------|----------|
| 一覧 | あり（データ 0 件） | あり（一覧取得中） | あり（取得失敗） |
| 詳細 | なし（存在しないリソースは 404） | あり（詳細取得中） | あり（取得失敗） |
| フォーム | なし（入力待ち状態が初期状態） | あり（送信中） | あり（バリデーションエラー / 送信失敗） |
| ダッシュボード | あり（データが揃っていない場合） | あり（集計中） | あり（取得失敗） |
| モーダル | 画面内容による | あり（処理中） | あり（処理失敗） |
| ウィザード | なし（ステップ初期状態が初期） | あり（各ステップ送信中） | あり（ステップ失敗） |
| エラー | なし（エラー画面自体がエラー状態） | なし | なし |
| 未分類 | 不明（ユーザーに確認） | 不明（ユーザーに確認） | あり（一般的に発生し得る） |

#### 2-2. 各状態の設計内容を推定

画面名・画面種別・Epic 仕様書・フィールド定義から各状態の設計内容を推定する。

**空状態（Empty State）:**

| 項目 | 推定内容 |
|------|---------|
| 発生条件 | データが 0 件 / 検索結果が 0 件 / 初回利用時など |
| 表示内容 | 状況を説明するメッセージ + 必要に応じてアクションボタン |
| アクション | 新規作成 / フィルタクリア / その他の誘導 |

**ローディング状態（Loading State）:**

| 項目 | 推定内容 |
|------|---------|
| 発生条件 | API リクエスト中 / データ処理中 |
| 表示内容 | スピナー / スケルトンスクリーン / プログレスバー のいずれか |
| タイムアウト | 30秒でエラー状態に遷移（不明な場合は「TBD」） |

**エラー状態（Error State）:**

| 項目 | 推定内容 |
|------|---------|
| 発生条件 | API エラー（4xx / 5xx）/ ネットワーク切断 / バリデーションエラー |
| 表示内容 | エラーメッセージ + 再試行 / 前画面に戻る などのアクション |
| アクション | 再試行 / 前画面に戻る / サポートへの問い合わせ など |

#### 2-3. ユーザーへの確認（AC-E053-06）

推定した状態設計をユーザーに提示し、確認を求める。

```
## [SCR-ID] [画面名] — 状態設計

### 空状態
| 項目 | 内容 |
|------|------|
| 発生条件 | [推定内容] |
| 表示内容 | [推定内容] |
| アクション | [推定内容] |

### ローディング状態
| 項目 | 内容 |
|------|------|
| 発生条件 | [推定内容] |
| 表示内容 | [推定内容] |
| タイムアウト | [推定内容 / TBD] |

### エラー状態
| 項目 | 内容 |
|------|------|
| 発生条件 | [推定内容] |
| 表示内容 | [推定内容] |
| アクション | [推定内容] |

状態設計の内容を確認してください。
- 追加・変更・削除があれば指摘してください
- 「この画面に空状態はない」等、発生しない状態がある場合はお知らせください
- 問題なければ「OK」でロール定義のステップに進みます
```

ユーザーの指摘があれば修正して再提示する（AC-E053-06）。「OK」等の明確な承認を得てから手順 3 に進む。

### 3. 各画面のロール定義生成（AC-E053-07, AC-E053-08）

フィールド定義・画面種別・Epic 仕様書からロール情報を推定する。

#### 3-1. ロール情報の取得

以下の優先順位でロール情報を取得する:

1. Epic 仕様書のロール定義（明示的なロール要件があればそちらを優先）
2. 画面種別・画面名からの推定（管理画面であれば「管理者のみ」等）
3. ロール情報が取得できない場合は「全ロール」または「要確認」とする（AC-E053-08）

#### 3-2. 操作の列挙

画面に存在するフィールド定義とアクション（Step 1a で確認済み）から、ロール制御が必要な操作を列挙する。

**操作の例:**

| 操作 | 発生場面 |
|------|---------|
| 画面の閲覧 | 全画面共通 |
| 作成 | フォーム・モーダルで新規作成時 |
| 編集 | フォーム・詳細画面で既存データを変更時 |
| 削除 | 一覧・詳細画面でデータを削除時 |
| エクスポート | 一覧画面でデータを出力時 |
| 承認 | ワークフロー画面で承認操作を行う時 |
| [その他] | 画面固有のアクション |

#### 3-3. ユーザーへの確認

生成したロール定義をユーザーに提示し、確認を求める。

```
## [SCR-ID] [画面名] — ロール定義

| 操作 | 許可ロール |
|------|----------|
| 画面の閲覧 | [推定ロール / 全ロール / 要確認] |
| [操作名] | [推定ロール / 全ロール / 要確認] |
...

注: ロールが未定義の操作は「全ロール」または「要確認」と記録しています（スキルはエラー停止しません）。

ロール定義を確認してください。
- 追加・変更・削除があれば指摘してください
- ロールが決まっていない場合は「要確認」のままで問題ありません
- 問題なければ「OK」で API 参照のステップに進みます
```

ユーザーの指摘があれば修正して再提示する。「OK」等の明確な承認を得てから手順 4 に進む。

### 4. 各画面の API エンドポイント参照生成（AC-E053-09, AC-E053-10）

> **API 参照は任意項目（推奨）。** API 仕様書が存在しない場合や API が未確定の場合は「TBD」と記録して続行する。スキルはエラーで停止しない（AC-E053-10）。

#### 4-1. API 仕様書の確認

以下の優先順位で API 情報を取得する:

1. `docs/design/api-spec.md` が存在する場合: 画面名・フィールド定義に対応するエンドポイントを参照
2. Epic 仕様書に API 定義がある場合: 対応するエンドポイントを参照
3. どちらも存在しない / 対応するエンドポイントが不明な場合: 「TBD」として記録

#### 4-2. エンドポイントの推定

API 情報が取得できた場合、画面種別・フィールド定義から使用する可能性が高いエンドポイントを推定する。

| 画面種別 | 典型的なエンドポイント |
|---------|-------------------|
| 一覧 | GET /api/v1/[resource]（一覧取得） |
| 詳細 | GET /api/v1/[resource]/:id（詳細取得） |
| フォーム（作成） | POST /api/v1/[resource]（作成） |
| フォーム（編集） | PUT または PATCH /api/v1/[resource]/:id（更新） |
| ダッシュボード | 複数の GET エンドポイント（集計・統計等） |
| モーダル（確認） | DELETE /api/v1/[resource]/:id 等（確認後の操作） |
| ウィザード | 複数ステップに対応する POST/PUT |

#### 4-3. ユーザーへの確認

推定した API エンドポイント参照をユーザーに提示し、確認を求める。

```
## [SCR-ID] [画面名] — API エンドポイント参照

| 目的 | HTTP メソッド | パス | 備考 |
|------|-------------|------|------|
| [目的] | [メソッド] | [パス] | [備考 / TBD] |
...

注: API が未確定の項目は「TBD」と記録しています（後から更新できます）。

API エンドポイント参照を確認してください。
- 追加・変更・削除があれば指摘してください
- API がまだ決まっていない場合は「TBD」のままで問題ありません
- この画面で API を使用しない場合は「（API なし）」とお知らせください
- 問題なければ「OK」で仕様書への記録に進みます
```

ユーザーの指摘があれば修正して再提示する。「OK」等の明確な承認を得てから手順 5 に進む。

### 5. 画面仕様書への記録

ユーザーの承認後、Step 1a で作成した画面仕様書ファイルに Step 1b の内容を追記する。

**出力先:** `docs/design/screen-spec-[slug]/[SCR-ID]-[screen-slug].md`（Step 1a で作成済みのファイルを更新）

**記録するセクション（Step 1b のスコープ）:**

```markdown
## 状態設計（AC-E053-05, AC-E053-06）

<!-- AC-E053-05: 空状態・ローディング状態・エラー状態の記述を含める -->
<!-- AC-E053-06: ユーザーが対話的に各状態を確認・追記・変更できる -->

### 空状態

| 項目 | 内容 |
|------|------|
| 発生条件 | [確認済みの内容] |
| 表示内容 | [確認済みの内容] |
| アクション | [確認済みの内容] |

### ローディング状態

| 項目 | 内容 |
|------|------|
| 発生条件 | [確認済みの内容] |
| 表示内容 | [確認済みの内容] |
| タイムアウト | [確認済みの内容 / TBD] |

### エラー状態

| 項目 | 内容 |
|------|------|
| 発生条件 | [確認済みの内容] |
| 表示内容 | [確認済みの内容] |
| アクション | [確認済みの内容] |

---

## ロール定義（AC-E053-07, AC-E053-08）

<!-- AC-E053-07: 閲覧可能なロールと操作可能なロールを明記する -->
<!-- AC-E053-08: ロール未定義の場合は「全ロール」または「要確認」と記録する -->

| 操作 | 許可ロール |
|------|----------|
| 画面の閲覧 | [確認済みのロール / 全ロール / 要確認] |
| [操作名] | [確認済みのロール / 全ロール / 要確認] |

<!-- AC-E053-09: 参照する API エンドポイント（パス・HTTP メソッド）を記録する（任意項目） -->
<!-- AC-E053-10: 未確定の場合は「TBD」と記録する（エラー停止しない） -->

| 目的 | HTTP メソッド | パス | 備考 |
|------|-------------|------|------|
| [目的] | [メソッド] | [パス] | [備考 / TBD] |
```

### 6. 全 SCR-ID の処理完了後

全 SCR-ID の状態設計・ロール定義・API 参照の生成・承認が完了したら、以下のサマリーを表示する。

```
## Step 1b 完了サマリー

- 処理した SCR-ID: N 件
- TBD ローディングタイムアウト: N 件（後から更新可能）
- 「要確認」ロール: N 件（後から更新可能）
- TBD API エンドポイント: N 件（後から更新可能）

### 次のステップ
Step 2（画面遷移図生成）に進みます。
```

Step 2 の詳細手順は `references/step2-flow.md` を参照して実行する。

## エラー処理

| ケース | 条件 | 振る舞い |
|--------|------|---------|
| 状態が発生しない画面種別 | エラー画面等、特定の状態が発生しない | 発生しない状態を「該当なし」として記録し続行 |
| ロール情報が不明 | Epic 仕様書・画面名からロールを特定できない | 「全ロール」または「要確認」として記録し続行（AC-E053-08） |
| API 仕様書が存在しない | `docs/design/api-spec.md` が未存在 | 「TBD」として記録し続行（AC-E053-10） |
| API エンドポイントが不明 | 画面種別から対応エンドポイントを推定できない | 「TBD」として記録し続行（AC-E053-10） |
| 画面に API が不要 | 静的コンテンツのみの画面等 | 「（API なし）」と記録し続行 |

## TBD 管理

- Step 1b で「TBD」と記録した項目は、仕様が決まり次第、画面仕様書を直接更新して TBD を解消できる
- TBD の一覧は Step 3（ブリーフィング）でユーザーに提示する
- TBD が残っていても画面仕様書の生成・後続スキルの実行は可能

## 注意事項

- **状態設計はユーザー確認後に記録する**（推定内容が画面の意図と異なる場合があるため、AC-E053-06 に従い対話的に確認する）
- **1画面ずつユーザーの承認を得てから次へ進む**（自動的に全 SCR-ID を処理しない）
- **ロール未定義でエラー停止しない**（「全ロール」または「要確認」として記録し続行する。AC-E053-08 に準拠）
- **API 参照は任意項目**（API 仕様が未確定でも「TBD」として記録し続行する。AC-E053-10 に準拠）
- **Step 1a で作成した画面仕様書を更新する**（新規ファイルを作成するのではなく、既存の仕様書に追記する）
- **AC トレーサビリティを維持する**（各画面仕様書の「AC トレーサビリティ」セクションに AC-E053-05〜10 を追記する）
