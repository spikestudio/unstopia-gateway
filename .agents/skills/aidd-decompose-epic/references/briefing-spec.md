# ブリーフィング仕様 --- /aidd-decompose-epic

各ステップ完了時にこの仕様に従ってブリーフィングを実行する。5 要素ブリーフィングを前提とし、`/task` スキル固有の品質基準を追加定義する。

## 5 要素チェックリスト

ブリーフィングは以下の 5 要素を**全て**含むこと。1 つでも欠けている場合、ブリーフィングは不完全とみなす。

| # | 要素 | 内容 | 欠落時のリスク |
|---|------|------|-------------|
| 1 | 成果物の全内容の説明 | 要約ではなく網羅的に説明する。Task 件数・各 Task の概要・AC マッピング・依存関係を省略しない | 人間が成果物を読まないと承認できない |
| 2 | AI が行った判断の根拠と代替案 | 「なぜこうしたか」「他にどんな選択肢があったか」を明示する | 判断の妥当性を評価できない |
| 3 | 暗黙の前提・行間の説明 | 成果物に直接書かれていないが、前提としている知識・制約を説明する | 前提の食い違いに気づけない |
| 4 | 未決定事項と論点の洗い出し | 決めきれなかったこと、後続で決定が必要なことを列挙する | 未決定事項が放置される |
| 5 | 人間に求める意思決定事項の明示 | 「何を判断してほしいか」を具体的に問いかける | 承認が形骸化する |

## ステップ別ブリーフィング重点

`/task` スキルは Step 0（入力受付）→ Step 1（Task 分解）→ Step 2（ファイル生成・Issue 作成）の 3 ステップで動作する。ブリーフィングは主に Step 1 完了時に実行される。以下に重点項目を定義する。

### Step 1: Task 分解完了時

Epic 全体を Task に分解するため、分割の妥当性と網羅性が最重要となる。

| 重点項目 | 説明内容 |
|---------|---------|
| **全 Task 一覧（全件・全文）** | Task 番号 / Task 名 / 対応 AC / 複雑度 / 依存 Task を全件・全文で提示する。件数のみ・タイトルのみ・「概要」への省略は禁止（aidd-framework/references/briefing-format.md 参照） |
| 分割判断 | なぜこの粒度・この分割方法にしたか。分割の優先順（ドメイン概念 > 機能 > レイヤー）のどれを適用したか。代替の分割案があればその比較を説明する |
| 依存グラフ | Task 間の依存関係を明示する。並行実行可能な Task と直列実行が必要な Task を区別し、クリティカルパスを示す |
| AC カバレッジ | Epic の全 AC が Task にマッピングされているかの一覧。カバレッジ率（目標: 100%）を明示する。未カバーの AC がある場合はその理由を説明する |
| E2E 検証 Task | E2E 検証 Task の対象範囲（どの AC を検証するか）と検証方法（どのようなシナリオで確認するか）を説明する |
| 複雑度の見積根拠 | 各 Task の複雑度レベル（S/M/L）の判断根拠。特に L と判定した Task について、分割を検討した結果と分割しなかった理由を説明する |

### 設計判断確定内容（Phase 3）

| 重点項目 | 説明内容 |
|---------|---------|
| **設計判断一覧（全件）** | 設計判断クラスで検討した全判断の一覧（Task 番号・category・選択内容・根拠・対案）を全件提示 |
| **要検討マークが残った場合** | 残存マークがある場合は FAIL として Phase 2 再実行を依頼する旨を説明 |
| **ユーザーへの確認** | 各設計判断に対してユーザーが変更・上書きできることを明示する |

## 品質基準

**「この説明だけ読めば成果物の承認可否を判断できる」品質を担保すること。**

具体的には:

- 成果物ファイルを別途開いて読まなくても、ブリーフィングだけで全体像と判断ポイントが把握できる
- AI 補完した箇所が明示され、人間の入力との区別がつく
- 自信度が低い箇所（`<!-- AI-UNCERTAIN -->` マーク付き）が重点的に説明されている
- 「はい/いいえ」で承認判断できる程度に論点が整理されている
- AC カバレッジの網羅性（全 AC が Task にマッピングされているか）が検証可能である
- 依存グラフに循環がないことが確認できる
- 各 Task の粒度が目安（S: ~200 行, M: ~400 行, L: ~800 行）に収まっていることが確認できる

## 出力フォーマット

```markdown
## ブリーフィング: Task 分解 --- ES-NNN [Epic名]

### 成果物の全体像
- Task 数: N 件（実装 Task: X 件、E2E 検証 Task: 1 件）
- 複雑度分布: S: X 件, M: Y 件, L: Z 件

| # | Task 名 | 対応 AC | 複雑度 | 依存 |
|---|---------|---------|--------|------|
| TASK-NNN | [Task 名（省略禁止・全件表示）] | AC-XXX-NN | S/M/L | -- or TASK-MMM |
| ... | ... | ... | ... | ... |

### 分割判断
- 分割方法: [ドメイン概念/機能/レイヤー のどれを主軸にしたか]
- 分割理由: [なぜこの方法を選んだか]
- 代替案: [検討した別の分割方法とその比較]

### 依存グラフ
- 並行実行可能: [Task 番号の組み合わせ]
- 直列実行が必要: [依存チェーンの説明]
- クリティカルパス: [最長の依存チェーン]

### AC カバレッジ
| AC | Task | カバー状況 |
|----|------|----------|
| AC-NNN-01 | TASK-XXX | covered |
| ... | ... | ... |

カバレッジ率: XX/YY (100%)

### 暗黙の前提
[成果物に直接書かれていないが前提としている事項]

### 未決定事項・論点
[決めきれなかった事項、後続で決定が必要な事項]

### AI 自信度
[自信度が低い箇所の説明。全体の自信度表明]

### 確認事項
[人間に判断を求める具体的な問いかけ]

この内容で承認しますか？
```
