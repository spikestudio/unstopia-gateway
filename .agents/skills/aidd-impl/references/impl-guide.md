# Step 1: 実装 -- 詳細手順

Task 定義の AC に従って対話的に実装を進める。各ステップで AC との対応を明示し、規約準拠・テスト方針の適用を確認しながら進行する。

## 入力

- Task 定義（`docs/tasks/TASK-NNN-*.md`）— AC・完了条件・テスト方針
- Epic 仕様書（Task が参照する Epic）— AC の詳細・ビジネスコンテキスト
- 設計成果物（Task が参照する設計書）— ドメインモデル・DB スキーマ・API spec
- ADR（Task が参照する ADR）— 技術的決定
- 規約ドキュメント（`docs/conventions/`）— 命名・構造・テスト・禁止事項
- リサーチドキュメント（`docs/research/`、設計書から参照がある場合）— 外部依存の動作仕様・URL・バージョン
- 既存コード（リポジトリ内の関連コード）— 実装パターンの参考

## 対話的実装の進め方

### 実装方針の提示

実装に着手する前に、Task 定義の内容をもとに実装方針を構造化して提示する:

```
## 実装方針: TASK-NNN [Task名]

### AC（完了条件）
1. [AC-1]: [内容]
2. [AC-2]: [内容]
...

### 設計との対応
- ドメインモデル: [どのエンティティ・バリューオブジェクトを実装するか]
- DB スキーマ: [どのテーブル・カラムを追加/変更するか]
- API: [どのエンドポイントを実装するか]

### テスト方針
- ユニットテスト: [対象]
- 結合テスト: [対象]
- E2E テスト: [該当する場合]

### 実装ステップ（提案）
1. [ステップ1]: [実装内容] + テスト → AC-1 に対応
2. [ステップ2]: [実装内容] + テスト → AC-2 に対応
3. ローカルでテストを実行（`task test`）— 全テスト PASS・Lint エラー 0（CI は push 後に自動実行される）

> **各ステップはコードとテストをセットで実装する。** テストを「後でまとめて書く」形式は取らない。ステップ完了 = コード + テストが揃った状態。

### 注意点
- [Task スコープの境界: これ以上は実装しない]
- [依存する他の Task: 前提条件の確認]
- [規約で特に注意すべき点]

この方針で進めてよいですか？
```

**ユーザーの承認を得てからステップ形式の実装に進む。**

> **Step 0 ブリーフィングとの役割の違い:** Step 0 の未決定事項ブリーフィングは「設計判断の迷い・未決定事項の解消」が目的。ここでの実装方針提示は「AI がこれから何をどう実装するか」の確認が目的。方針に誤りがあればこの時点で修正を指示すること。

### ステップ形式

実装ステップに沿って、各ステップを対話的に進める。ステップごとに以下の形式で実装内容を明示する:

```
## ステップ N/M: [ステップ名]

**対応 AC:** AC-X
**実装内容:** [具体的に何を実装するか]
**対象ファイル:** [新規作成/変更するファイル]

[実装を進める]
```

- 各ステップは 1 つ以上の AC に対応させる
- ステップの粒度は「レビュー可能な単位」を目安とする
- ステップ間の依存関係がある場合は順序を守る

### 各ステップ完了時の確認項目

各ステップの実装が完了したら、以下を確認する:

| # | 確認項目 | 説明 |
|---|---------|------|
| 1 | **テスト実装** | このステップの AC に対応するテストが書かれており、通過しているか（テストなし完了は禁止） |
| 2 | AC 対応 | AC の対応部分が実装されているか |
| 3 | 規約準拠 | 命名・構造・実装パターンが規約に従っているか |
| 4 | suppress 禁止 | `eslint-disable`, `@ts-ignore`, `@ts-expect-error`, `#[allow(...)]`, `// nolint`, `# noqa` 等の lint suppress が新たに追加されていないか |

**suppress が追加されている場合は根本原因を修正すること。** 修正不可能な場合はユーザーに相談する。

### AC との対応確認

全ステップの実装が完了したら、Task 定義との最終突合を行う:

```
## 実装完了確認: TASK-NNN

### AC 充足チェック
- [x] AC-1: [実装内容の概要]
- [x] AC-2: [実装内容の概要]
...

### テスト
- [x] ユニットテスト: N 件
- [x] 結合テスト: N 件

### スコープ確認
- [x] Task スコープ外の実装が含まれていない
- [x] 規約に準拠している

### CI
- [x] テスト通過
- [x] Lint 通過
```

## 外部依存値の転記ルール（MUST）

外部 URL、バージョン番号、設定値（環境変数名、ポート番号等）をコードに記述する際は、**必ず** 以下のいずれかからコピーする:

| 転記元 | 例 |
|--------|-----|
| 設計書 | `docs/design/` または `docs/requirements/` 内のドキュメント |
| リサーチドキュメント | `docs/research/` 内のドキュメント |
| 公式ドキュメント | `curl` / `npm info` / `helm show` 等で取得した最新情報 |

**記憶やトレーニングデータに基づいて外部依存値を書いてはならない。**

具体的な適用例:

- **ライブラリバージョン**: `npm info <package> version` や公式リリースページで最新安定版を確認してから記述する
- **外部 API の URL**: 設計書またはリサーチドキュメントに記載された URL をそのままコピーする
- **設定値・ポート番号**: 設計書や ADR で決定された値を転記する。推測で記述しない
- **環境変数名**: 規約ドキュメントまたは設計書の定義に従う

## テスト方針の適用方法

Task 定義のテスト方針セクションに基づいてテストを実装する。

### 手順

1. **テスト方針の確認**: Task 定義の「テスト方針」セクションを読み、どのテストレイヤー（UT / 統合 / E2E）が求められているかを確認する
2. **テスト対象の特定**: 各 AC に対応するテストケースを洗い出す
3. **テスト名に AC-ID を含める**: テストの describe/it ブロックに AC-ID（`AC-ENNN-NN`）を含め、CI での AC-ID カバレッジ自動検証を可能にする
4. **テスト実装**: 実装コードに対応するテストを作成する
5. **テスト実行**: 全テストがパスすることを確認する

### テストレイヤーごとの指針

| レイヤー | 対象 | 方針 |
|---------|------|------|
| ユニットテスト | ドメインロジック、ユーティリティ | 入力・出力の境界値、正常系・異常系を網羅 |
| 結合テスト | リポジトリ、API ハンドラ | 実際の DB / 外部サービスとの接続を含むテスト |
| E2E テスト | ユーザーシナリオ | Epic の AC に対応するシナリオテスト |

### テスト名の形式

```typescript
// 例: TypeScript / Jest
describe("AC-E015-01: ユーザー登録", () => {
  it("AC-E015-01: 有効な入力で登録が成功する", () => { ... });
  it("AC-E015-01: メールアドレスが重複する場合エラーを返す", () => { ... });
});
```

## 完了確認

全ステップの実装完了後、`references/selfcheck.md` を読み込んでセルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）。最大 3 ラウンドで MUST FIX を解消してからコミットする。

## 完了時

`references/briefing-spec.md` に従ってブリーフィングを実行する。

**ブリーフィングで必ず提示する情報:**

- 実装方針と AC の対応関係
- 設計判断のポイント（トレードオフがあった箇所）
- テストカバレッジの範囲

承認を得てから完了処理に進む。
