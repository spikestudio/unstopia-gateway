# Step 2 (code): 並列コードレビュー — /aidd-review

種別 `code` 専用。4 サブエージェントを並列で起動し、独立したコードレビューを実施する。

## 入力（Step 1 取得済み）

- PR diff（`/tmp/pr-diff.txt` として保存済み）
- CLAUDE.md 内容
- `docs/conventions/` ファイル群
- PR タイトル・本文（変更の意図の把握）

サブエージェントはこれらのデータを引数として受け取る。**サブエージェントが独自に `gh` コマンドやファイル読み込みを実行することを明示的に禁止する。**

## 並列サブエージェントの起動

以下の 4 サブエージェントを **1 メッセージで同時起動** する（順番ではなく並列）:

### Sub-A: Sonnet — 規約/命名チェック

**モデル:** Sonnet

**観点:** CLAUDE.md 準拠・`docs/conventions/` 準拠・命名規則

**指示:**

- Step 1 で取得した CLAUDE.md と `docs/conventions/` 配下の規約ドキュメントを確認する
- CLAUDE.md のルールが明確に違反されている箇所をフラグする（ルールを直接引用できない場合はフラグしない）
- `docs/conventions/` の aidd フレームワーク固有の規約（命名・構造・禁止事項等）への違反をチェックする
- 命名規則の逸脱（関数名・変数名・ファイル名）をチェックする
- pre-existing issues（PR の変更外の問題）はフラグしない

### Sub-B: Sonnet — バグ/ロジックチェック

**モデル:** Sonnet

**観点:** 明らかなバグ・ロジックエラー（diff のみ）

**指示:**

- PR diff のみを読む（diff 外のコンテキストは読まない）
- 明らかなバグのみをフラグする:
  - コンパイルエラー・構文エラー
  - 特定の入力で確実に失敗するロジックエラー
  - null/undefined の未処理
- スタイル・ニュアンス・潜在的な問題はフラグしない
- linter・型チェッカーが検出する問題はフラグしない

### Sub-C: Opus — セキュリティ/lint suppress チェック

**モデル:** Opus

**観点:** セキュリティ問題・lint suppress 検出

**指示:**

- 変更されたコードの範囲のみを対象にする
- セキュリティ問題（SQL インジェクション・XSS・認証バイパス・機密情報のハードコード等）を検出する
- lint/型チェッカーの suppress 指示を検出する:
  - `eslint-disable`, `@ts-ignore`, `@ts-expect-error`, `#[allow(...)]`
  - `// nolint`, `#pragma warning disable`, `@SuppressWarnings`, `# noqa`, `# type: ignore`
  - `suppress-approved:` マーカーのない suppress は必ずフラグする
- 推測に基づく問題はフラグしない（確実に問題と判断できる場合のみ）

### Sub-D: Opus — 技術的負債/設計チェック

**モデル:** Opus

**観点:** 技術的負債・設計の改善余地・将来拡張時の問題

**指示:**

- 変更されたコードの設計品質を評価する
- 技術的負債の兆候をフラグする:
  - 重複コード・コピーペースト
  - 単一責任原則の明確な違反
  - 将来の拡張を困難にする設計（モジュール結合度が高い等）
- 「改善余地がある」程度の問題もフラグする（non-critical として分類）
- 推測・主観に基づく問題はフラグしない

## HIGH SIGNAL 原則

**フラグしてよい問題:**

- コードがコンパイル/パースに失敗する（構文エラー・型エラー）
- 特定の入力で確実に誤った結果を返す（明確なロジックエラー）
- CLAUDE.md/conventions のルールを直接引用して証明できる違反
- `suppress-approved:` マーカーのない lint suppress
- セキュリティ上の脅威が確実に存在する

**フラグしてはならないもの（フォールスポジティブ）:**

- pre-existing issues（PR の変更外の問題）
- スタイル・品質の懸念（linter が検出するものを含む）
- 潜在的・仮定的な問題（「〜かもしれない」）
- テストカバレッジ不足・セキュリティの一般的懸念（CLAUDE.md に明記なし）
- 既に `suppress-approved:` マーカーで承認済みの問題

## critical / non-critical 分類基準（code 固有）

各サブエージェントは検出した問題を以下の基準で分類する:

### critical

以下のいずれかに該当する問題:

- バグ（特定の入力で確実に失敗するロジックエラー・コンパイルエラー）
- セキュリティ脆弱性（SQL インジェクション・XSS・認証バイパス・機密情報のハードコード等）
- `suppress-approved:` マーカーのない lint suppress（CLAUDE.md 禁止事項）
- CLAUDE.md / conventions の明示的な禁止事項への違反
- 技術的負債のうち将来の拡張を確実にブロックするもの

### non-critical

以下の問題:

- 命名改善（規約に準拠しているが改善余地がある）
- 可読性・コメントの改善
- 設計改善余地（将来的に問題になる可能性があるが現時点では動作する）
- 技術的負債の軽微なもの（重複コード等）

## 成果物

各サブエージェントが返す問題リスト（以下の形式）:

```
[問題の説明] / 観点: [規約・命名/バグ・ロジック/セキュリティ・lint suppress/技術的負債・設計] / ファイル: [パス:行番号] / 分類: [critical/non-critical]
```

集約後、重複を除去した全問題一覧（critical/non-critical 分類済み）を Step 3 (step3-agree-fix.md) に渡す。
