# Step 2: Epic 仕様書レビュー — 3 並列サブエージェント — /aidd-review

Epic 仕様書の品質・トレーサビリティ・設計整合を 3 つのサブエージェントで並列検査する。各サブエージェントは Step 1 で取得済みのデータのみを使用し、独立して外部データを取得しない。

## 入力（Step 1 から受け取るデータ）

| データ | 説明 |
|--------|------|
| Epic 仕様書（全文） | `docs/requirements/ES-NNN-*.md` の内容 |
| Phase 定義書（全文） | `docs/requirements/phase-*.md` の内容（機能意図照合用） |

## 並列サブエージェントの起動

以下の 3 サブエージェントを **1 メッセージで同時起動** する（順番ではなく並列）。

---

### Sub-A: Sonnet — AC 品質チェック

**目的:** Epic 仕様書の受け入れ基準（AC）が INVEST 原則を満たしているか、曖昧表現が含まれていないかを検査する。

**指示:**

- Epic 仕様書の全 AC を対象に、以下を検査する

**INVEST 原則チェック（AC 単位）:**

| 原則 | 意味 | 違反例 |
|------|------|--------|
| Independent | 他 AC との依存が最小化されている | 「AC-3 が完了した後に実施する」など |
| Negotiable | 実装の詳細が AC に含まれていない | 「React の useEffect で〜する」など |
| Valuable | ユーザーまたはシステムに価値がある | 技術タスクのみで価値が不明瞭 |
| Estimable | 実装規模をおよそ見積もれる | 範囲が不明確で見積もり不能 |
| Small | 1 Task 相当の範囲に収まっている | 複数機能を 1 AC に詰め込んでいる |
| Testable | 合否判定できる具体的な条件が記述されている | 合否基準が存在しない |

**曖昧表現の検出:**

以下の表現を含む AC は曖昧として検出する（例示であり、類似表現も対象）:

- 「適切に」「正しく」「必要に応じて」「十分に」「できる限り」「スムーズに」「問題なく」「うまく」「わかりやすく」「きちんと」「なるべく」

**成果物:**

```
[問題の説明] / 観点: AC品質 / AC-ID: [AC-ID または「AC-ID未付与」] / 詳細: [問題箇所の引用]
```

---

### Sub-B: Sonnet — トレーサビリティチェック

**目的:** ストーリーと AC の対応関係・AC-ID の付与状態・AI 補完 AC の根拠明記を検査する。

**指示:**

- Epic 仕様書の全ストーリーと全 AC を対象に、以下を検査する

**ストーリー↔AC 対応チェック:**

- 各ストーリーに対応する AC が存在するか確認する
- AC がどのストーリーに対応するか追跡できるか確認する（ストーリー番号との紐づけが明示されているか）
- ストーリーに書かれた価値・条件が AC に反映されているか確認する（ストーリーの内容が AC に落とし込まれていない場合は検出する）

**AC-ID 付与チェック:**

- 全 AC に一意の AC-ID（例: `AC-001`、`AC-1` 等の連番形式）が付与されているか確認する
- AC-ID が欠落している AC、または AC-ID の体系が不統一な場合は検出する

**AI 補完 AC の根拠チェック:**

- AI が補完・追記した AC に根拠（ユーザーヒアリング結果・仕様書参照・設計成果物参照等）が明記されているか確認する
- 根拠なしに「想定される AC」として追記された AC は検出する

**成果物:**

```
[問題の説明] / 観点: トレーサビリティ / ストーリー: [ストーリー番号または「不明」] / AC-ID: [AC-ID または「AC-ID未付与」] / 詳細: [問題箇所の引用]
```

---

### Sub-C: Opus — 設計整合・仕様の抜け・拡張制約チェック

**目的:** Epic 仕様書と設計成果物との整合性・仕様の抜け・将来拡張時の制約リスクを検査する。

**指示:**

- Epic 仕様書全体・Phase 定義書（機能意図）を対象に、以下を検査する

**設計成果物との整合チェック:**

- Epic 仕様書が参照している設計成果物（ADR・アーキテクチャ図・ドメインモデル・API 仕様等）の内容と AC が矛盾していないか確認する
- 設計成果物で定義されたインターフェース・制約・前提が AC に正しく反映されているか確認する
- 設計成果物を参照すべき箇所で参照が欠落している場合は検出する

**仕様の抜けチェック:**

- Phase 定義書の機能意図（やりたいこと）を Epic 仕様書の AC が網羅しているか確認する
- 機能意図のうち AC に落とし込まれていない項目を検出する
- 非機能要件（パフォーマンス・セキュリティ・エラーハンドリング・ログ等）が仕様から漏れていないか確認する

**将来拡張時の制約リスクチェック:**

- 現在の AC が将来の拡張（スケールアップ・新機能追加・マルチテナント化等）を著しく困難にする設計判断を含んでいないか確認する
- 制約リスクが存在する場合、問題として検出する（ただし、過度な先読み設計を要求する意図ではない。明らかなリスクのみを対象とする）

**成果物:**

```
[問題の説明] / 観点: 設計整合・仕様の抜け・拡張制約 / 該当箇所: [セクション名またはAC-ID] / 詳細: [問題箇所の引用または説明]
```

## epic-spec 固有の critical/non-critical 分類基準

Step 3 の集約前に各サブエージェントが以下の基準で分類する。

### critical（必ず修正する）

以下のいずれかに該当する問題は **critical**:

| 分類基準 | 例 |
|---------|---|
| 曖昧 AC（「適切に」「正しく」等の不検証表現を含む） | 「ユーザーに適切なエラーを表示する」 |
| AC-ID が未付与（全 AC または一部 AC に ID がない） | AC-ID なしの受け入れ基準が 1 件でも存在する |
| ストーリーと AC の不整合（ストーリーの価値・条件が AC に反映されていない） | ストーリーに書かれた「〇〇のとき〜できる」が AC に存在しない |
| 設計成果物との重大な不整合（AC が設計で定義された制約・インターフェースに矛盾する） | ADR で決定した API レスポンス形式と AC の記述が異なる |
| Phase 定義書の機能意図が AC に反映されていない（仕様の抜け） | Phase の機能意図「〇〇機能」に対応する AC が Epic 仕様書に存在しない |

### non-critical（改善推奨）

以下の問題は **non-critical**:

| 分類基準 | 例 |
|---------|---|
| 表現の改善余地（曖昧ではないが、より明確にできる） | テスト条件の粒度が粗い |
| 追加推奨 AC（必須ではないが品質向上に貢献する） | エラーログの詳細出力・境界値テストの AC |
| INVEST 原則の軽微な逸脱（Small・Independent の改善余地） | 1 AC が 2 Task 相当の範囲になっている |
| AI 補完 AC の根拠が薄い（完全に欠落はしていないが不十分） | 「設計検討時に追記」という記述のみ |

## 成果物の集約

3 サブエージェントの検出結果を `step3-agree-fix.md` に渡す。重複（同一 AC-ID の同一問題）は除去する。

```
## Sub-A 検出結果（AC 品質）
[Sub-A の成果物リスト]

## Sub-B 検出結果（トレーサビリティ）
[Sub-B の成果物リスト]

## Sub-C 検出結果（設計整合・仕様の抜け・拡張制約）
[Sub-C の成果物リスト]
```

サブエージェントが独自に `cat docs/requirements/...` を実行することを **明示的に禁止する**。Step 1 の取得結果を使い回すこと。
