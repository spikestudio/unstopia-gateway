# Step 2: Epic 総合レビュー（3 並列サブエージェント）— /aidd-review

Epic の全成果物を 3 観点で並列チェックし、検出問題を集約する。

## 入力（Step 1 取得済みデータ）

Step 1（`step1-data-fetch.md`）が取得済みの以下を受け取る。サブエージェントは独自に外部 I/O を実行しない。

| データ | 変数/ファイル | 説明 |
|--------|-------------|------|
| PR diff | `/tmp/pr-diff.txt` | 変更ファイル一覧・変更内容 |
| Epic 仕様書 | 文字列として引き渡し | AC・ストーリー・非機能要件 |
| Task 定義群 | 文字列として引き渡し | Epic に紐づく全 Task の完了条件・スコープ |
| 変更ファイル一覧 | `/tmp/pr-meta.json` | ファイル分類・変更量 |
| 設計成果物（ADR 等） | 文字列として引き渡し | Epic 仕様書が参照するもの |
| 規約ドキュメント | パス一覧として引き渡し | `docs/conventions/` |

## 並列サブエージェント設計

以下の 3 サブエージェントを **同時に** 起動する。各サブエージェントは独立して動作し、結果を返却する。

| Sub | モデル | 観点 | 担当チェックドキュメント |
|-----|--------|------|------------------------|
| **Sub-A** | Sonnet | ビジネス要件 | 下記「Sub-A チェック手順」 |
| **Sub-B** | Sonnet | ドキュメント整合 | 下記「Sub-B チェック手順」 |
| **Sub-C** | Opus | コード品質 | 下記「Sub-C チェック手順」 |

---

## Sub-A: ビジネス要件チェック

Epic 仕様書の受入基準（AC）が実装で充足されているかを検証する。

### Sub-A チェック手順

#### 1. AC 充足判定

Epic 仕様書から全 AC-ID（`AC-ENNN-NN` 形式）を抽出し、PR diff と突合して実装エビデンスを確認する。

| AC-ID | 内容 | Task | 充足ファイル | 判定 |
|-------|------|------|------------|------|
| AC-ENNN-NN | [AC の内容] | TASK-NNN | [実装ファイル] | OK / NG |

**NG の判定種別:**

| 条件 | 判定 |
|------|------|
| AC に対応する実装が PR diff に存在しない | NG — 未実装 |
| AC に対応する Task 定義が存在しない | NG — Task 未定義 |
| AC の条件が部分的にしか満たされていない | NG — 部分充足 |
| 実装は存在するが AC の意図と異なる | NG — 仕様不一致 |

#### 2. ストーリー整合確認

Epic 仕様書の各ストーリーの意図（"so that" 句）が、紐づく AC 群の実装によって実現されているかを検証する。

| Story | 意図（so that） | 紐づく AC | 実現状況 | 判定 |
|-------|----------------|-----------|---------|------|
| Sn | [意図] | AC-ENNN-NN, ... | [実現状況] | OK / NG |

#### 3. デモシナリオ検証

Epic 仕様書に定義されたデモシナリオ（またはユーザーフロー）が、実装によって実現できるかを検証する。シナリオが未定義の場合はスキップ。

### Sub-A 出力フォーマット

```markdown
## Sub-A: ビジネス要件チェック結果

### AC 充足判定
| AC-ID | 内容 | Task | 充足ファイル | 判定 |
|-------|------|------|------------|------|
| ... | ... | ... | ... | ... |

**未充足 AC:** [なし / AC-ID 一覧と理由]

### ストーリー整合確認
| Story | 意図（so that） | 紐づく AC | 実現状況 | 判定 |
|-------|----------------|-----------|---------|------|
| ... | ... | ... | ... | ... |

**整合性の問題:** [なし / 問題一覧と理由]

### デモシナリオ検証
[実施結果または「定義なし — スキップ」]

### 検出問題一覧
| # | 問題 | critical / non-critical | 修正提案 |
|---|------|------------------------|---------|
| 1 | ... | critical | ... |
```

## Sub-B: ドキュメント整合チェック

Epic 完了時に関連ドキュメント群の整合性・正確性・完全性を検証する。

### Sub-B チェック手順

#### 1. ドキュメント間の整合性

| チェック対象 | 検証内容 |
|------------|---------|
| Epic 仕様書 ↔ Task 定義 | AC の転記が正確か。AC-ID・内容が一致しているか |
| Epic 仕様書 ↔ 設計成果物 | 設計成果物テーブルのステータスが正しいか。成果物が実在するか |
| Task 定義 ↔ 実装 | スコープ（対象ファイル）と実際の変更が一致するか |
| Epic 仕様書 ↔ Phase 定義書 | Epic の目的・スコープが Phase の目標と整合しているか |
| Task 定義間 | Task 間で重複・矛盾するスコープがないか |

#### 2. 論理的矛盾の検出

| チェック対象 | 検証内容 |
|------------|---------|
| AC 間の矛盾 | AC 同士が矛盾する要件を定義していないか |
| 設計成果物間の矛盾 | DB スキーマ ↔ API spec ↔ ドメインモデル間で型・フィールド名・制約が一致しているか |
| 規約と設計の矛盾 | 設計成果物が規約（命名規則・レイヤールール等）に違反していないか |

#### 3. 記載漏れの検出

| チェック対象 | 検証内容 |
|------------|---------|
| Epic 仕様書の完全性 | テンプレートの必須セクションが全て記載されているか |
| Task 定義の完全性 | テンプレートの必須セクションが全て記載されているか |
| AC の検証可能性 | 曖昧語（「適切に」「必要に応じて」等）が検証基準に含まれていないか |
| エラーケース・異常系 | 正常系だけでなくエラーケース・異常系の AC が定義されているか |

#### 4. 参照整合

| チェック対象 | 検証内容 |
|------------|---------|
| ファイルパス参照 | 参照先のファイルが実在するか |
| AC-ID 参照 | Task 定義内の AC-ID が正しい AC を指しているか |
| ADR 参照 | ADR 番号が `docs/architecture/adr/` 配下の実在する ADR を指しているか |
| Issue 番号参照 | ドキュメント内の `#NNN` 参照が正しい Issue を指しているか |

#### 5. ドキュメント更新漏れ確認

PR diff（`git diff main...HEAD --name-only`）と照合し、未更新ドキュメントを特定する。

| 照合観点 | チェック内容 |
|---------|------------|
| 新規 API / インターフェース | API spec・ドメインモデル・README の対応箇所が更新されているか |
| スキーマ変更 | DB スキーマドキュメントが変更に追従しているか |
| 規約・命名変更 | `docs/conventions/` の関連規約が更新されているか |
| アーキテクチャ変更 | ADR が作成・更新されているか |
| セットアップ手順変更 | README / セットアップガイドが更新されているか |

### Sub-B 出力フォーマット

```markdown
## Sub-B: ドキュメント整合チェック結果

### 1. 整合性問題一覧
| # | 対象ドキュメント | 問題 | 重大度 |
|---|----------------|------|--------|
| 1 | [ファイルパス] | [問題内容] | MUST FIX / SHOULD FIX |

### 2. 矛盾箇所一覧
| # | ドキュメント A | ドキュメント B | 矛盾内容 | 重大度 |
|---|--------------|--------------|---------|--------|
| 1 | [パス:箇所] | [パス:箇所] | [矛盾の説明] | MUST FIX / SHOULD FIX |

### 3. 記載漏れ一覧
| # | 対象ドキュメント | 欠落項目 | 重大度 |
|---|----------------|---------|--------|
| 1 | [ファイルパス] | [欠落セクション/項目] | MUST FIX / SHOULD FIX |

### 4. 参照不整合一覧
| # | 参照元 | 参照先 | 問題 | 重大度 |
|---|--------|-------|------|--------|
| 1 | [パス:行] | [参照先パスまたは ID] | [不整合内容] | MUST FIX / SHOULD FIX |
```

## Sub-C: コード品質チェック

PR diff に含まれるコード変更を 6 つの観点でチェックし、問題を検出する。

### Sub-C チェック手順

#### 1. 規約違反の検出

`docs/conventions/` の各規約ドキュメントと PR diff を突合する。

| 規約 | チェック内容 |
|------|------------|
| 命名規則 | 変数・関数・ファイル・ディレクトリの命名が規約に従っているか |
| ディレクトリ構造 | 新規ファイルが規約のディレクトリ構造に配置されているか |
| レイヤー間ルール | 依存方向・呼び出しパターンが規約に従っているか |
| エラーハンドリング | エラー処理が規約の方針に従っているか |
| テスト規約 | テストの命名・構造・配置が規約に従っているか |
| UI 規約（該当時） | UI コンポーネントの設計・実装パターンが規約に従っているか |

#### 2. 技術的負債の検出

| チェック項目 | 検出方法 |
|------------|---------|
| TODO/FIXME コメント | 変更差分に新規追加された `TODO`、`FIXME`、`HACK`、`XXX` コメントを検出する |
| マジックナンバー | ハードコードされた数値・文字列リテラルが定数化されずに使用されていないか |
| エラーハンドリング不備 | try-catch で握りつぶし（空の catch ブロック）、エラーの無視がないか |
| 型安全性の欠如 | `any` 型の使用、型アサーションの乱用がないか |
| 非同期処理の不備 | await 忘れ、Promise の未処理、競合状態の可能性がないか |

検出コマンド（PR diff から）:

```bash
# TODO/FIXME コメントの検出
grep -E '^\+.*\b(TODO|FIXME|HACK|XXX)\b' /tmp/pr-diff.txt

# any 型の使用検出（TypeScript の場合）
grep -E '^\+.*:\s*any\b' /tmp/pr-diff.txt
```

#### 3. lint suppress 構文の検出

CLAUDE.md の禁止事項に基づき、リンター・型チェッカー・静的解析の suppress 指示が新規追加されていないかを検出する。**新規追加された suppress は全て critical として報告する。**

```bash
# PR diff から suppress 構文を検出する（追加行のみ）
grep -E '^\+' /tmp/pr-diff.txt | grep -E '(eslint-disable|@ts-ignore|@ts-expect-error|@ts-nocheck|#\[allow\(|// nolint|#pragma warning disable|@SuppressWarnings|# noqa|# type: ignore)'
```

| 言語/ツール | 禁止構文 |
|------------|---------|
| ESLint | `eslint-disable`, `eslint-disable-next-line`, `eslint-disable-line` |
| TypeScript | `@ts-ignore`, `@ts-expect-error`, `@ts-nocheck` |
| Rust | `#[allow(...)]` |
| Go | `// nolint` |
| Java | `@SuppressWarnings` |
| Python | `# noqa`, `# type: ignore` |

例外: `suppress-approved: [理由]` マーカー付きかつ人間の承認済みの場合のみ許容。

#### 4. 仕様書とコードの不整合検出

AC 定義と実装の微妙なズレを検出する。

| リスク分類 | 検出方法 |
|-----------|---------|
| もっともらしい誤り | AC との突合。境界値・エラーケースの実装確認 |
| 過剰実装 | Task スコープとの突合。YAGNI チェック |
| ハルシネーション | 存在しない API・ライブラリ・設定の参照確認 |
| テストの品質不足 | 仕様ではなく実装をテストしていないか（AC に基づくテストかの確認） |

#### 5. 冗長コードの検出

| チェック項目 | 検出方法 |
|------------|---------|
| 重複コード | 同一ロジックが複数箇所に存在しないか |
| 未使用インポート | 追加されたインポート文のうち未使用のものがないか |
| 未使用変数・関数 | 追加された変数・関数のうちどこからも参照されていないものがないか |
| デッドコード | 到達不能なコードブロックがないか |

#### 6. セキュリティ観点（認可・外部入力・データ永続化を含む変更がある場合）

| チェック項目 | 検出方法 |
|------------|---------|
| 認可 | 権限が Epic 仕様書の定義通りに実装されているか |
| 入力検証 | 外部入力に対するサニタイズ・バリデーションが実装されているか |
| データ整合性 | トランザクション境界、同時実行制御が適切か |
| 機密情報 | ログ・レスポンスに機密情報が含まれていないか |

### Sub-C 出力フォーマット

## 問題の critical / non-critical 分類基準（epic 固有）

3 つのサブエージェントが検出した問題は以下の基準で分類する。Step 3 の修正フローに引き渡す。

### critical（修正必須）

| 分類 | 具体例 |
|------|--------|
| AC 未充足 | AC に対応する実装が存在しない / 部分充足 / 仕様不一致 |
| バグ | 到達不能コード、await 忘れ、空の catch ブロック等 |
| lint suppress | `eslint-disable`、`@ts-ignore` 等の新規追加（`suppress-approved` なし） |
| 技術的負債 | TODO/FIXME コメントの新規追加、マジックナンバー、型安全性の欠如 |
| 仕様書とコードの不整合 | ドキュメント定義とコード実装の乖離、参照リンク切れ、AC-ID 誤参照 |
| 規約違反 | `docs/conventions/` に明示された規則への違反 |
| セキュリティ問題 | 認可漏れ、入力検証不備、機密情報の露出 |

### non-critical（推奨対応）

| 分類 | 具体例 |
|------|--------|
| 改善余地 | リファクタリング候補、可読性向上案 |
| 冗長コード | 未使用インポート、デッドコード、過剰抽象化 |
| ドキュメント改善 | 説明文の明瞭化、コメント追加 |
| テスト改善 | カバレッジ向上提案（AC は充足している場合） |

## 集約と出力

3 サブエージェントの結果を集約し、Step 3（`step3-agree-fix.md`）に引き渡す。

### 集約フォーマット

```markdown
## Step 2: Epic レビュー結果サマリー

| 観点 | critical | non-critical | 判定 |
|------|---------|-------------|------|
| Sub-A: ビジネス要件 | N 件 | N 件 | PASS / FAIL |
| Sub-B: ドキュメント整合 | N 件 | N 件 | PASS / FAIL |
| Sub-C: コード品質 | N 件 | N 件 | PASS / FAIL |
| **合計** | **N 件** | **N 件** | **PASS / FAIL** |

### 全検出問題一覧（critical 優先）

| # | 観点 | 問題 | critical / non-critical | 修正提案 |
|---|------|------|------------------------|---------|
| 1 | Sub-A | ... | critical | ... |
| 2 | Sub-B | ... | critical | ... |
| 3 | Sub-C | ... | non-critical | ... |
```

> critical が 1 件以上存在する場合は **FAIL**。critical が 0 件であれば **PASS**。
> non-critical の件数は PASS/FAIL 判定に影響しない。

### Gate 処理（PASS 時）

Step 2 の判定が PASS の場合、Step 3 での修正フローは不要（問題なし）となり、Step 4（`step4-completion.md`）が以下を実行する:

- `gate:reviewed` ラベル付与
- Epic PR の draft 解除
- PR タイトルから `[WIP]` 除去

FAIL の場合は Step 3 で全 critical 問題を修正した後に Step 4 で再判定する。
