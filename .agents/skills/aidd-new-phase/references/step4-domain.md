# Step 4: 詳細分析 — 詳細手順

コンテキストマップ・ドメインイベント・用語集を生成する。

## 入力

- Step 3 で承認された BC キャンバス群
- Phase 定義書（BC 一覧・Epic 一覧を参照）
- `references/domain-templates/context-map.md`（コンテキストマップテンプレート — TASK-061 で作成）
- `references/domain-templates/domain-events.md`（ドメインイベントテンプレート — TASK-061 で作成）
- `references/domain-templates/glossary-section.md`（用語集テンプレート — TASK-061 で作成）

## 手順

### 1. コンテキストマップ生成

`references/domain-templates/context-map.md` テンプレートを読み込み、以下を記述する:

| セクション | 内容 |
|-----------|------|
| BC 一覧 | 全 BC の責務サマリー（BC キャンバスから転記） |
| 統合パターン | BC 間の統合パターンを定義する |
| Mermaid 図 | BC 間の関係を図示する |

**統合パターンの選択基準:**

| パターン | 使い分け |
|---------|---------|
| ACL（腐敗防止層） | 下流が上流のモデルに汚染されたくない場合 |
| OHS（公開ホストサービス） | 上流が公開 API を提供する場合 |
| CF（適合者） | 下流が上流のモデルをそのまま受け入れる場合 |
| SK（共有カーネル） | 2 つの BC が同じモデルを共有する場合（慎重に使用） |
| PL（パブリッシュ言語） | イベント駆動で疎結合に連携する場合 |

### 2. ドメインイベント一覧生成

`references/domain-templates/domain-events.md` テンプレートを読み込み、BC 間をまたぐ主要なドメインイベントを列挙する:

- イベント名は過去形（「〜された」）で記述する
- ペイロードは Epic に委譲する。ここではイベント名・発行元・購読先・トリガーのみ定義する
- ワークフローのステップ間の状態遷移をイベントとして捕捉する

### 3. 用語集生成

`references/domain-templates/glossary-section.md` テンプレートを読み込み、BC ごとに用語集を生成する:

- BC 内で使用する全てのドメイン用語を網羅する（最低 5 語以上）
- 同じ用語が BC ごとに異なる意味を持つ場合、その違いを明記する
- `docs/glossary.md` に BC ごとのセクションとして追記する

### 4. ファイル書き出し

- コンテキストマップを `docs/domain/context-map.md` に書き出す
- ドメインイベント一覧を `docs/domain/domain-events.md` に書き出す
- 用語集を `docs/glossary.md` に追記する
- Phase 定義書のドメイン分析成果物テーブルを最終更新する

## 成果物

- コンテキストマップ（`docs/domain/context-map.md`）
- ドメインイベント一覧（`docs/domain/domain-events.md`）
- 用語集（`docs/glossary.md` に追記）
- Phase 定義書のドメイン分析成果物テーブル最終更新

## 完了時

`briefing-spec.md` に従ってブリーフィングを実行する。

**ブリーフィングで必ず提示する情報（成果物を見なくても判断できるレベル）:**

<!-- 括弧内は briefing-spec.md 5 要素との対応 -->

- コンテキストマップの全体像（全 BC と統合パターンの Mermaid 図またはテキスト表現）（要素1: 成果物全内容）
- 統合パターンの一覧表（上流 BC・下流 BC・パターン名・選択理由）（要素1, 2: 成果物全内容 + 判断根拠）
- ドメインイベントの全件一覧（イベント名・発行元 BC・購読先 BC・トリガー）（要素1: 成果物全内容）
- 主要なイベントフロー（ワークフローに沿った時系列でのイベント連鎖）（要素3: 暗黙の前提）
- 用語集の全件一覧（BC ごとに用語・定義。特に BC 間で意味が異なる用語を強調）（要素1: 成果物全内容）
- Phase 定義書のドメイン分析成果物テーブルの最終状態（要素1: 成果物全内容）
- 統合パターン選択の判断根拠（なぜ ACL で OHS ではないか等）（要素2: 判断根拠）
- AI 自信度が低い箇所（ドメイン知識不足で統合パターンやイベント設計の妥当性を保証できない部分）（要素4: 未決定事項）
- 人間に判断を求める事項（統合パターンの選択、イベント粒度、用語の定義）（要素5: 意思決定事項）

承認を得て Phase 定義完了とする。
