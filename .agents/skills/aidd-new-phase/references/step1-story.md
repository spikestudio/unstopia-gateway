# Step 1: ストーリー補完 — 詳細手順

ユーザー入力のストーリーを AI が補完・具体化する。

## 入力

- ユーザーが入力したストーリー（不完全でよい）
- Step 0 で作成した Phase 定義書ファイル（空の状態）

## 手順

### 1. ストーリーの受付と整理

ユーザー入力のストーリーを As a / I want to / so that 形式に整理する。入力が口語・箇条書きの場合は AI が形式を補完する。原文の意図を変えない。

### 2. 情報収集（/aidd-research 呼び出し — 必要時のみ）

以下の観点でストーリーの抜け漏れを調査する:

| 観点 | 調査内容 |
|------|---------|
| 類似プロダクト | 競合・類似サービスの機能からユーザーが言い忘れているストーリーを発見する |
| 非機能要件 | パフォーマンス・セキュリティ・可用性など、暗黙に前提とされている要件を検出する |
| エラーケース | 正常系だけでなく、失敗時・例外時のユーザー体験を検出する |
| 管理系機能 | 管理者向けの設定・監視・運用系機能が漏れていないか確認する |
| 認証認可 | ペルソナごとのアクセス権限に関するストーリーが定義されているか確認する |

調査が不要な場合（ストーリーが十分に具体的、ドメインが明確）はスキップしてよい。

### 3. CHARTER・ビジョンとの突合

- `docs/PROJECT-CHARTER.md` が存在する場合、ストーリーがビジネスゴール・ビジョンと整合しているか確認する
- 既存の Phase 定義書（`docs/requirements/PD-*.md`）との重複・依存を確認する

### 4. AI 補完ストーリーの生成

調査結果に基づき、AI がストーリーを補完する。補完したストーリーには以下を付記する:

- `（AI 補完: [補完理由]）` — なぜこのストーリーを追加したか
- 情報源（類似プロダクト名、CHARTER のどの項目に対応するか等）

### 5. 補完結果の提示

人間入力と AI 補完を区別して提示する:

```markdown
### 人間の入力
- S1: As a **[ペルソナ]**, I want to [行動], so that [理由].
- S2: ...

### AI 補完
- S3: As a **[ペルソナ]**, I want to [行動], so that [理由].（AI 補完: [理由]）
- S4: ...
```

## 成果物

補完済みストーリー一覧。Phase 定義書ファイルにはまだ書き出さない（Step 2 で書き出す）。

## 完了時

`briefing-spec.md` に従ってブリーフィングを実行する。

**ブリーフィングで必ず提示する情報（成果物を見なくても判断できるレベル）:**

<!-- 括弧内は briefing-spec.md 5 要素との対応 -->

- 人間入力ストーリーの全文一覧（S1, S2, ...）（要素1: 成果物全内容）
- AI 補完ストーリーの全文一覧（各ストーリーに補完理由・情報源を付記）（要素1, 2: 成果物全内容 + 判断根拠）
- 人間入力と AI 補完の件数内訳（例: 人間 3 件 + AI 補完 5 件 = 計 8 件）（要素1: 成果物全内容）
- ペルソナの一覧（ストーリーに登場する全ペルソナ）（要素1: 成果物全内容）
- 調査で発見した抜け漏れの観点（非機能要件・エラーケース・管理系等、どの観点から何件追加したか）（要素3: 暗黙の前提）
- AI 自信度が低いストーリー（ドメイン知識不足で妥当性を保証できないもの）（要素4: 未決定事項）
- 人間に判断を求める事項（追加したストーリーの要否、優先度の感覚）（要素5: 意思決定事項）

承認を得てから Step 2 に進む。
