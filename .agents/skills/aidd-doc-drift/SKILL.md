# /aidd-doc-drift — git 履歴起点のドキュメント乖離検出・修正

PR 履歴または最近のコミットを走査し、実装・プロセス変更に追いついていないドキュメントを検出・修正する。ES-028（作業中の予防的更新）の補完として、後追いで乖離を一括回収する。

> 共有スクリプト: `scripts/scan-docs.sh`（`${CLAUDE_PLUGIN_ROOT}/scripts/scan-docs.sh` 経由で呼び出し）

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| git リポジトリ | `.git/` の存在 | git リポジトリ内で実行してください |
| `.md` ファイルが存在する | プロジェクト内を確認 | ドキュメントが存在しないため実行できません |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| 走査範囲 | 引数 | 任意 | 省略時は PR 直近10件（または未マージコミット） |

**引数パターン:**

| パターン | 例 | 動作 |
|---------|-----|------|
| 省略 | `/aidd-doc-drift` | PR 履歴 直近10件（未マージ作業があればコミット履歴） |
| 件数指定 | `/aidd-doc-drift 5` | PR 履歴 直近5件 |
| 日付指定 | `/aidd-doc-drift --since 2026-04-01` | 指定日付以降 |
| コミット指定 | `/aidd-doc-drift --commits HEAD~10` | コミット履歴を明示指定 |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| 乖離レポート | 対話 | 更新・新規作成が必要なドキュメントの一覧（根拠付き） |
| 修正済みファイル | ファイル | ユーザー承認後に更新・作成されたドキュメント |
| コミット | git commit | 修正内容をまとめたコミット |

### capability 呼び出し

該当なし

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-next` | 修正完了後の次アクション確認 |

## Step 1: スコープ決定

`$ARGUMENTS` を解析し、走査対象を決定する。

詳細手順: `references/step1-scope.md`

---

## Step 2: コンテキスト収集・意味理解（並列）

以下の 2 つの Agent を **1 メッセージ内で同時起動** する:

```
Agent(prompt="""
  プロジェクトのドキュメント構造を収集し、鳥瞰図をサマリーとして返してください。

  手順:
  1. `bash "${CLAUDE_PLUGIN_ROOT}/scripts/scan-docs.sh"` を実行して全 .md ファイル一覧を取得
  2. CLAUDE.md / README 等のエントリポイントドキュメントを読み込む
  3. ドキュメントの構造・役割マップをサマリーとして返す
""")

Agent(prompt="""
  以下の走査対象の変更内容を読み込み、「何がどう変わったか」をサマリーとして返してください。

  走査対象: [Step 1 で決定したスコープ（PR一覧 or コミット範囲）]

  手順:
  1. 対象 PR の diff または git log の変更ファイルを読み込む
  2. 変更の意味（新機能追加・廃止・リネーム・プロセス変更等）を把握する
  3. 変更の概要リストを返す（ファイルパスと変更の意味を対にする）
""")
```

両 Agent の結果を受け取り、主スキルが影響を受けるドキュメントを根拠付きで推論する（インライン実行）。

詳細手順: `references/step2-context.md`

---

## Step 3: 乖離確認（並列）

Step 2 の影響推論で特定した候補ドキュメントを、**候補ごとに Agent を並列起動** して検証する:

```
# 候補が N 件ある場合、1 メッセージ内で N 件を同時起動
Agent(prompt="[候補ドキュメント1のパス] を読み、[変更内容サマリー] との乖離を特定して返してください")
Agent(prompt="[候補ドキュメント2のパス] を読み、[変更内容サマリー] との乖離を特定して返してください")
...
```

各 Agent は「何が古いか / 何が抜けているか」を返す。主スキルが結果を集約してブリーフィング用の乖離レポートを生成する。

詳細手順: `references/step3-verify.md`

---

## Step 4: ブリーフィング・修正実行

検出した問題をユーザーにブリーフィングし、承認を得てから修正を実行する。

詳細手順: `references/step4-fix.md`

**問題の2分類:**

- **更新**: 既存ドキュメントの古い参照・追記漏れ・死リンクを修正
- **新規作成**: 変更に対応するドキュメントが存在しない場合に作成

---

## Step 5: セルフレビュー・コミット

`aidd-framework/references/self-review-loop.md` に従いセルフレビューを実施し、問題がなければコミットする。

---

## 重要なルール

- **日本語で対話する**
- **FRAMEWORK.md の意思決定基準に従う**
- **自動修正禁止** — 修正前に必ずユーザーの承認を得る。自動で変更を加えてはならない
- **根拠を必ず提示する** — 「なぜそのドキュメントが影響を受けるか」を推論根拠とともに提示する
- **精度の透明性** — コンテキストが不完全な場合（docs/ なし・エントリポイント doc なし）は、その旨と精度低下の可能性を明示する
- **各 Step でユーザーの確認を得てから次に進む**
