# /aidd-design — インタラクティブ設計探索

未決定の設計課題を「既存資料収集 → 動的質問 → リサーチ → ドラフト生成 → フィードバック反映 → 保存」の 6 ステップで解決する。特定のテンプレートに縛られず、AI が設計対象に合わせて動的に質問を生成する。

> **このスキルの目的は「設計の不確実性を解消し、後続の `/aidd-new-phase`・`/aidd-new-epic` に渡せる品質のドキュメントを作成する」こと。**

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| git リポジトリが初期化済み | `git status` | プロジェクトの初期化後に実行してください |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| 設計対象 | 引数 | 任意 | 「プロジェクトチャーターを作りたい」等の自然言語。省略時はユーザーに聞く |

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| /aidd-research | Step 2 | 業界事例・ベストプラクティス調査（必要時のみ） |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| 設計ドキュメント | ファイル | `docs/` 配下（設計対象に応じたパス） |
| コミット | git commit | `docs: /aidd-design [ドキュメント名]` 形式 |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-new-phase` | Phase 定義が設計対象の場合 |
| `/aidd-new-epic` | Epic 設計が設計対象の場合 |

## Step 0: 入力受付・テンプレート選択

`references/template-catalog.md` を読み込んで実行する。

1. 引数から設計対象を取得する。引数なしの場合はユーザーに設計したいドキュメントを質問する
2. `references/template-catalog.md` を参照し、最も適切な書式テンプレートを特定してユーザーに提示する
3. ユーザーが書式を確認・選択したら Step 1 に進む

**バリデーション:**

- 設計対象が空の場合 → 「設計対象を指定してください（例: プロジェクトチャーター、機能一覧）」と案内
- 既存ファイルに同名が存在する場合 → 上書き確認・バックアップ提案

## Step 1: 既存資料収集と不足情報の特定

設計対象に関連する既存資料を自動収集し、何が不足しているかを特定する。

**収集対象:**

| 種別 | 収集方法 |
|------|---------|
| README・プロジェクト概要 | `ls README*`, `cat README.md` |
| 既存ドキュメント群 | `ls docs/` の一覧、関連ファイルの読み込み |
| Epic/Phase 定義書 | `docs/requirements/` のスキャン |
| 規約・ADR | `docs/conventions/`, `docs/architecture/` |
| コード概要 | プロジェクト構造の把握（`ls src/`, `ls app/` 等） |

**収集後の提示:**

```
## 収集した既存資料

| 資料 | 内容の概要 |
|------|----------|
| README.md | ... |
| docs/requirements/... | ... |

## 設計に必要だが不足している情報

1. [不足情報1]
2. [不足情報2]
...
```

収集に失敗したファイルはスキップし、その旨をユーザーに伝えて続行する。

## Step 2: 動的質問生成とリサーチ

選択した書式テンプレートの内容と収集済み情報を照合し、不足している情報を埋めるための質問を動的に生成する。テンプレートの機械的な穴埋めではなく、**AI が設計対象の特性に合わせて問いかける内容を判断する**。

**質問生成の原則:**

- 1 回の質問は 3〜5 件まで（認知負荷を下げる）
- 既存資料から判明した情報は質問しない（重複排除）
- 「はい/いいえ」で答えられる質問は避け、具体的な情報を引き出す質問にする
- 業界事例やベストプラクティスの調査が有効と判断した場合は `/aidd-research` を呼び出す

**リサーチが必要なケース:**

- 設計対象に類似した先行事例が有効な場合
- 技術選定の判断材料が必要な場合
- 業界標準・ベストプラクティスの確認が必要な場合

```
## 設計のための質問

（既存資料で判明した情報のサマリー）

以下の情報を教えてください:

1. [質問1]
2. [質問2]
3. [質問3]

不明な項目はスキップしてください。後から追記できます。
```

ユーザーの回答を受け取ったら Step 3 に進む。複数ラウンドの質問が必要な場合は繰り返す。

## Step 3: ドラフト生成とブリーフィング

ユーザーの回答・収集資料・リサーチ結果を統合し、選択した書式テンプレートに従ってドキュメントのドラフトを生成する。

**ブリーフィング形式:**

```
## ドラフト: [ドキュメント名]

[生成したドキュメントの全文]

---

## ブリーフィング

### 判断根拠
- [主要な設計判断とその根拠]

### 不確実箇所（<!-- AI-UNCERTAIN -->）
- [ドメイン知識不足・情報不足で判断した箇所]

### 確認事項
1. [ユーザーに確認が必要な事項]
2. [...]

フィードバックをください（修正・追加・削除の指示、または「承認」でファイル保存に進みます）。
```

`<!-- AI-UNCERTAIN -->` マーカーを不確実箇所に付与する（CLAUDE-base.md の自信度表明基準に従う）。

## Step 4: フィードバック反映（反復）

ユーザーのフィードバックを受け取り、指摘を反映した改訂版を生成する。

- **修正・追加・削除の指示** → 反映して改訂版を提示。Step 3 のブリーフィング形式で再提示
- **「承認」「OK」「進めて」** → Step 5 に進む
- 反復回数に制限はない

## Step 5: ファイル保存・コミット

ユーザーが承認したら、ドキュメントを `docs/` 配下の適切なパスに保存してコミットする。

**保存パスの推定:**

| 設計対象 | 推奨パス |
|---------|---------|
| プロジェクトチャーター | `docs/PROJECT-CHARTER.md` |
| 機能仕様書 | `docs/requirements/[slug].md` |
| 技術スタック | `docs/architecture/tech-stack.md` |
| アーキテクチャ概要 | `docs/architecture/architecture-overview.md` |
| NFR 詳細 | `docs/architecture/nfr.md` |
| 画面フロー | `docs/design/screen-flow.md` |
| 外部連携仕様 | `docs/design/external-integrations.md` |
| その他 | `docs/[設計対象を表すスラグ].md` |

保存パスをユーザーに確認してから保存する。既存ファイルが存在する場合は上書き確認。

```bash
git add [保存ファイルパス]
git commit -m "docs: /aidd-design [ドキュメント名]"
```

**完了通知:**

```
設計ドキュメントが保存されました:
- ファイル: [パス]
- コミット: [ハッシュ]

次のステップ:
→ Phase 定義に進む: `/aidd-new-phase`
→ Epic 定義に進む: `/aidd-new-epic`
→ 別の設計ドキュメントを作成: `/aidd-design [対象]`
```

## 重要なルール

- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う** — 優先順序 1〜5 に該当する問題の見送り・後回しを推奨しない
- **テンプレートの機械的な穴埋めをしない** — AI が設計対象の特性を判断して問いかける内容を動的に決定する
- **不確実箇所には必ず `<!-- AI-UNCERTAIN -->` マーカーを付与する** — CLAUDE-base.md の自信度表明基準に従う
- **既存資料から判明した情報は質問しない** — 重複を避け、ユーザーの負担を下げる
- **1 質問回で 3〜5 件に絞る** — 一度に大量の質問をしない
- **承認なしにファイル保存しない** — Step 4 で明示的な承認を得てから Step 5 を実行する
- 各ステップの参照ファイルは `references/` を読んで実行する
