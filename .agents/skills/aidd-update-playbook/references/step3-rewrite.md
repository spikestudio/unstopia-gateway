# Step 3: セクション再執筆 — 詳細手順

影響セクションごとに、品質基準に従って更新案を生成し、ユーザー承認を得て反映する。

## 入力

- Step 2 で承認された影響セクション一覧
- Step 1 で読み込み済みの FW 全体像・プレイブック全体像
- FW 変更サマリー

## 手順

### 1. 再執筆品質基準の読み込み

`references/rewrite-quality.md` を読み込み、再執筆時に遵守する品質基準を把握する。

> 品質基準が未作成の場合（TASK-318 未完了）、以下の最低限の基準を適用する:
>
> - FRAMEWORK.md の記述を複製しない（参照リンクで代替）
> - プレイブック固有価値（対話例・つまずきポイント・Before/After）を維持する
> - 既存の文体・トーンを維持する

### 2. セクション単位で更新案を生成

影響セクションごとに、以下の形式で更新案を生成する:

```markdown
## [章名]（[ファイル名]）更新案

### 変更箇所 N: [変更箇所の名前]

**現在の記述:**

> [現在のプレイブックの該当箇所を引用]

**更新案:**

> [更新後の記述]

**変更理由:** [FW のどの変更に対応するか]

**品質チェック:**

- [ ] FRAMEWORK.md の記述を複製していない
- [ ] 固有価値（対話例・つまずきポイント等）を維持している
- [ ] 既存の文体・トーンと一貫している
- [ ] 前後の文脈と整合している
```

### 3. 更新案の品質セルフレビュー

全セクションの更新案を生成した後、以下の観点でセルフレビューする:

| 観点 | チェック内容 |
|------|------------|
| FRAMEWORK.md との重複 | 更新案が FRAMEWORK.md の記述をそのまま転記していないか。転記している場合は参照リンク（`詳細は [FRAMEWORK.md](../../aidd-framework/FRAMEWORK.md#セクション名) を参照`）に置き換える |
| 固有価値の維持 | 対話例やつまずきポイントが削除・希釈されていないか。FW 変更に伴い対話例を新たに追加すべき箇所はないか |
| 章間整合性 | ある章の更新が他の章の記述と矛盾を生まないか |
| リンクの有効性 | 更新案内で新たに追加したリンクが有効か（Step 4 で検証されるが、明らかに無効なリンクはここで修正） |

### 4. 更新案の提示とユーザー承認

全セクションの更新案をまとめてユーザーに提示する。

**セクションが複数ある場合:**

- 全セクションを一括で提示し、セクション単位で承認・修正指示を受け付ける
- 一括承認も可能

**ユーザーの応答パターン:**

| 応答 | 処理 |
|------|------|
| 「承認」「OK」「進めて」 | 全更新案を反映して Step 4 へ |
| セクション単位の修正指示 | 指示に従い更新案を修正 → 再提示 |
| 「この変更は不要」 | 該当箇所の更新をスキップ |
| 質問 | 回答した上で再度承認を求める |

### 5. ファイル更新

承認された更新案を `docs/playbook/` の該当ファイルに反映する。

```bash
# 変更を加えた後
git add docs/playbook/
```

## 成果物

更新済みのプレイブックファイル（ステージング済み）。Step 4（リンク検証・コミット）の入力となる。

## 完了時

`references/briefing-spec.md` の Step 3 完了時の要素に従ってブリーフィングを実行する。ユーザーの承認を得てから Step 4 に進む。
