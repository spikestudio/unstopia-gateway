# 再執筆品質基準

プレイブックのセクション再執筆時に遵守する品質基準。Step 3（再執筆）から参照される。

## 基本原則

**プレイブックは FRAMEWORK.md の「人間向け翻訳」ではない。** プレイブックは FRAMEWORK.md とは独立した固有の価値を持つ。再執筆時にこの固有価値を削除・希釈してはならない。

## プレイブック固有価値の定義

以下はプレイブックにのみ存在し、FRAMEWORK.md には含まれない内容である。再執筆時に必ず維持する:

| 固有価値 | 説明 | 具体例 |
|---------|------|--------|
| **対話例** | スキルとの対話のやり取り（プロンプト→AI応答→ユーザー承認） | 「`/aidd-new-phase` を実行すると、AI がストーリーの入力を求めてきます...」 |
| **つまずきポイント** | 導入者が実際に経験した困りごと（ブロック引用形式 `>` で記述） | 「> ⚠️ つまずきポイント: スラッシュコマンドが動かない場合は...」 |
| **Before/After** | AIDD 導入前後の比較 | 「Before: 手動で要件を整理 → After: `/aidd-new-phase` で対話的に構造化」 |
| **リカバリー手順** | 問題発生時の具体的な復旧手順 | 「症状: CI が赤い → 原因: gate ラベル未付与 → 対処: `gh pr edit --add-label`」 |
| **チートシート** | 緊急時に参照する簡潔な手順リスト | 緊急リカバリーチートシート（第4章末尾） |

## FRAMEWORK.md との重複回避ルール

| やってよいこと | やってはならないこと |
|--------------|-------------------|
| FRAMEWORK.md の定義を **要約テーブル** + **参照リンク** で言及する | FRAMEWORK.md の記述を **そのまま転記** する |
| FRAMEWORK.md の概念を **対話例や具体的シナリオ** で説明する | FRAMEWORK.md の原則を **再説明** する |
| 「詳細は [FRAMEWORK.md](../../aidd-framework/FRAMEWORK.md#セクション名) を参照」でリンクする | FRAMEWORK.md の表やリストを **複製** する |

**判断基準:** 「この記述を削除して FRAMEWORK.md へのリンクに置き換えたとき、読者がプレイブックの文脈を見失うか？」

- 見失う → プレイブック固有の文脈が必要。記述を維持する
- 見失わない → FRAMEWORK.md への参照リンクに置き換えてよい

## 文体・トーン基準

| 基準 | 説明 |
|------|------|
| **主語** | 「あなた」（導入者）を想定。「ユーザー」ではなく「あなた」 |
| **語調** | 丁寧語（です・ます）。堅すぎない、フレンドリーなトーン |
| **技術用語** | 初出時に簡潔な説明を付ける。FRAMEWORK.md 準拠の用語を使用 |
| **コマンド例** | 実行可能な形で記述。省略記号（`...`）は避け、具体的な値を使う |
| **見出しレベル** | 各章内の既存の見出し構造に合わせる。不用意に見出しレベルを変えない |

## セクション追加時の基準

既存セクションの更新ではなく、新しいセクション（サブセクション）を追加する場合:

1. **追加する章の既存構造を確認する** — 見出しレベル、記述パターン、情報の粒度を把握
2. **既存パターンに合わせる** — 同じ章内の類似セクションと同じ形式・粒度で記述
3. **前後の文脈を確認する** — 追加位置の前後のセクションと論理的に繋がるか確認

## チェックリスト（再執筆完了時）

再執筆した各セクションに対して、以下を確認する:

- [ ] FRAMEWORK.md の記述をそのまま転記していない
- [ ] プレイブック固有価値（対話例・つまずきポイント等）が維持されている
- [ ] 新たに追加した固有価値コンテンツ（対話例等）がある場合、具体的で実用的な内容になっている
- [ ] 既存の文体・トーンと一貫している
- [ ] 前後のセクションとの論理的な繋がりが保たれている
- [ ] 参照リンクのパスが正しい（Step 4 で検証されるが、明らかに無効なものはここで修正）
