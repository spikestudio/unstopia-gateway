# /aidd-update-playbook — プレイブック差分更新

FW 変更（スキル追加・プロセス変更）または Issue/フィードバックを起点に、プレイブック（`docs/playbook/`）の影響セクションを自動検出し、差分更新を行う。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| プレイブック初版が存在 | `docs/playbook/index.md` の存在確認 | プレイブック初版を作成してください |
| git リポジトリ内 | `.git/` の存在確認 | git リポジトリ内で実行してください |
| gh CLI 認証済み | `gh auth status` | `gh auth login` を実行してください |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| （引数なし） | — | — | FW 差分起点: 直近の main マージ以降の diff を解析 |
| `--issue <番号>` | 引数 | 任意 | Issue 起点: 指定 Issue の内容を起点に更新 |

### capability 呼び出し

該当なし

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| 影響セクション一覧 | 対話出力 | 影響を受けるプレイブックの章とその根拠 |
| 更新案 | 対話出力 | 各セクションの「現在の記述」と「更新案」の対比 |
| 修正済みファイル | ファイル | ユーザー承認後に更新されたプレイブック |
| リンク検証結果 | 対話出力 | 全リンク有効 or リンク切れ一覧 |
| コミット | git commit | 更新内容をまとめたコミット |

| `/aidd-next` | 更新完了後の次アクション確認 |

## Step 0: 入力受付

`$ARGUMENTS` を解析し、実行モードを決定する。

**引数なし:**

- FW 差分起点モード → Step 1 へ

**`--issue <番号>`:**

- Issue 起点モード → Step 1 へ（Issue 番号を記録）

**前提条件チェック:**

- `docs/playbook/index.md` が存在しない → エラー終了（「プレイブック初版を作成してください」）
- `.git/` が存在しない → エラー終了（「git リポジトリ内で実行してください」）
- `--issue` の番号が正の整数でない → エラー終了（「有効な Issue 番号を指定してください」）

## Step 1: 差分解析・入力収集

詳細手順: `references/step1-diff-analysis.md`（FW 差分起点）/ `references/issue-input.md`（Issue 起点）

### FW 差分起点モード（引数なし）

1. **FW 全体像を読み込む**:
   - `aidd-framework/FRAMEWORK.md` を読み込む
   - `skills/` 配下の全スキルの frontmatter（name, description）を収集する
   - 主要ガイド（`aidd-framework/guides/` 配下）の一覧を把握する

2. **プレイブック全体像を読み込む**:
   - `docs/playbook/` 配下の全 5 章を読み込む（index.md, 01〜05）

3. **FW 変更差分を取得する**:

   ```bash
   # プレイブック最終更新コミットを検出
   git log --oneline -1 -- docs/playbook/

   # そのコミット以降の FW 関連ファイルの変更を取得
   git diff <playbook-last-commit>..HEAD -- aidd-framework/ skills/
   ```

4. diff が空の場合 → 「FW に変更がありません」と表示して終了

### Issue 起点モード（`--issue <番号>`）

1. **FW 全体像を読み込む**（同上）
2. **プレイブック全体像を読み込む**（同上）
3. **Issue 内容を取得する**:

   ```bash
   gh issue view <番号> --json title,body,labels
   ```

4. Issue が存在しない → エラー終了（「Issue #xxx が見つかりません」）

ユーザーの確認を得てから Step 2 に進む。

---

## Step 2: 影響セクション判定・ブリーフィング

**FW 全体像 + プレイブック全体像を把握した上で**、差分（FW diff または Issue 内容）がプレイブックのどの章に影響するかを俯瞰的に判定する。

詳細手順: `references/step2-impact-assessment.md`（TASK-316/317 で作成）

### 判定の観点

- **直接的影響**: diff のファイルパスと章の対応（例: `skills/` の変更 → 第3章・第5章）
- **間接的影響**: FW の変更がプレイブックの暗黙の前提に影響しないか（例: 新スキル追加 → 第1章の「AI の役割」の範囲が変わる）
- **影響なし判定**: テストコードのみの変更、CI 設定変更等、プレイブックに影響しない場合は「影響なし」と判定理由を提示

### プレイブック章と FW の対応マップ

| 章 | 主な対応 FW ファイル | 暗黙の前提 |
|----|-------------------|-----------|
| 01-introduction | FRAMEWORK.md（原則・ビジョン） | AI の役割範囲、トレードオフ |
| 02-setup | guides/, aidd-setup スキル | 最小構成に必要なもの |
| 03-development | スキルパイプライン定義、各スキル SKILL.md | 標準フローの流れ |
| 04-recovery-faq | 全スキル（エラーケース・FAQ 観点） | よくある困りごとのカバー範囲 |
| 05-reference | 全スキル一覧、ゲート定義、用語集 | スキル網羅性、ゲート完全性 |

### ブリーフィング

影響セクションを一覧で提示する:

```
## 影響セクション判定

### 影響あり
- 第3章（03-development.md）: [根拠]
- 第5章（05-reference.md）: [根拠]

### 影響なし
- 第1章: [影響しない理由]
- 第2章: [影響しない理由]
- 第4章: [影響しない理由]

この判定でよいですか？追加・除外したいセクションがあればお知らせください。
```

**Issue 起点で該当セクションなしの場合:**

- 「この Issue はプレイブックのどのセクションにも該当しません。理由: [理由]」と提示して終了

ユーザーの確認を得てから Step 3 に進む。

## Step 3: セクション再執筆・承認

影響セクションごとに、再執筆品質基準（`references/rewrite-quality.md`、TASK-318 で作成）に従って更新案を生成する。

詳細手順: `references/step3-rewrite.md`（TASK-316 で作成）

### 提示形式

各セクションに対して「現在の記述」と「更新案」を並べて提示する:

```
## 第5章（05-reference.md）更新案

### 変更箇所 1: スキル早見表

**現在の記述:**
> | `/aidd-adr` | ADR の対話的作成... |
> （なし）

**更新案:**
> | `/aidd-adr` | ADR の対話的作成・採番・保存・既存 ADR との矛盾チェック |

**変更理由:** 新スキル `/aidd-adr` が追加されたため

---

この更新案で進めてよいですか？
```

ユーザーの承認を得てから Step 4 に進む。

---

## Step 4: リンク検証・コミット

### リンク整合性検証

詳細手順: `references/step4-link-verification.md`（TASK-319 で作成）

1. プレイブック全 5 章内の Markdown リンク（`[text](path)` 形式）を抽出する
2. 各リンクの相対パスを解決し、リンク先ファイルが実際に存在するかを検証する
3. 結果を報告する:
   - リンク切れあり → リンク切れ一覧 + 修正案を提示 → ユーザー承認後に修正
   - リンク切れなし → 「リンク整合性検証: 全リンク有効」と報告

### コミット

```bash
git add docs/playbook/
git commit -m "docs: プレイブック更新（[更新起点の概要]）"
```

```
プレイブックの更新が完了しました:
- 更新セクション: [一覧]
- リンク検証: 全リンク有効 / リンク切れ N 件修正
- コミット: [ハッシュ]

次のステップ:
→ `/aidd-next` で次のアクションを確認
```

## ゲート制御ルール

**各ステップ間の承認ゲートは構造的に必須である。以下のルールに例外はない。**

1. **Step N の承認なしに Step N+1 を開始してはならない。** 承認なしに次ステップを進めようとした場合、「Step N の承認が必要です」と表示しスキルを停止する。
2. **ブリーフィングなしに承認を求めてはならない。** 各ステップ完了時に影響判定結果・更新案を必ず提示する。
3. **回答 ≠ 承認。** ユーザーの問いかけへの回答は回答内容の反映のみに留める。次ステップへの遷移には「OK」「進めて」「承認」等の明確な承認表現が必要。

## 重要なルール

- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う** — 問題の後回しを推奨しない
- **FW 全体像 + プレイブック全体像を必ず読み込む** — 差分だけを見て影響判定しない。全体像を俯瞰した上で、直接的影響と間接的影響の両方を判定する
- **プレイブック固有価値を維持する** — 再執筆時に FRAMEWORK.md の複製を生まない。対話例・つまずきポイント・Before/After・リカバリー手順はプレイブックの固有価値であり、これらを削除・希釈しない（`references/rewrite-quality.md` 参照）
- **自動修正禁止** — 修正前に必ずユーザーの承認を得る
- **根拠を必ず提示する** — 影響判定の根拠、更新案の変更理由を明示する
- 成果物生成・実装完了時にコミットを自動実行する。ユーザーの指摘反映後・ステップ承認後などの状態遷移ごとにもコミットする。push と PR 作成はユーザーの確認を待つ
