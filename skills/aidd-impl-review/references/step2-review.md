# Step 2: 並列コードレビュー — 詳細手順

4 エージェントを並列で起動し、独立したコードレビューを実施する。

## 入力

- PR diff（`gh pr diff <PR番号>`）
- CLAUDE.md ファイルパス一覧（Step 1 で取得）
- `docs/conventions/` 配下の規約ドキュメント
- PR タイトル・本文（変更の意図の把握）

## 並列エージェントの起動

以下の 4 エージェントを **1 メッセージで同時起動** する（順番ではなく並列）:

### Agent #1: Sonnet — CLAUDE.md 準拠チェック

**指示:**

- PR diff と対象ファイルの CLAUDE.md を確認する
- CLAUDE.md のルールが明確に違反されている箇所のみをフラグする
- ルールを直接引用できない場合はフラグしない（フォールスポジティブ禁止）
- pre-existing issues（PR の変更外の問題）はフラグしない

### Agent #2: Sonnet — docs/conventions/ 準拠チェック

**指示:**

- PR diff と `docs/conventions/` 配下の規約ドキュメントを確認する
- aidd フレームワーク固有の規約（命名・構造・禁止事項等）への違反をチェックする
- ルールを直接引用できる場合のみフラグする

### Agent #3: Opus — 明らかなバグ検出

**指示:**

- PR diff のみを読む（diff 外の コンテキストは読まない）
- 明らかなバグのみをフラグする:
  - コンパイルエラー・構文エラー
  - 特定の入力で確実に失敗するロジックエラー
  - null/undefined の未処理
- スタイル・ニュアンス・潜在的な問題はフラグしない
- linter・型チェッカーが検出する問題はフラグしない

### Agent #4: Opus — セキュリティ・ロジック問題

**指示:**

- 変更されたコードの範囲のみを対象にする
- セキュリティ問題（SQL インジェクション・XSS・認証バイパス等）と重大なロジック問題をチェックする
- 推測に基づく問題はフラグしない（確実に問題と判断できる場合のみ）

## 追加観点（任意）

並列エージェントのいずれかが以下も確認することが望ましい（リソースが許す場合）:

| 観点 | 方法 | 注意 |
|------|------|------|
| git blame/履歴 | `gh api repos/.../git/blames` または `git blame` | 変更箇所の歴史的コンテキスト |
| 過去 PR コメント | `gh search prs` で同ファイルの過去 PR を取得 | 同じ問題の繰り返しを防ぐ |
| コードコメント整合 | diff 内のコメントと変更の整合性確認 | 乖離しているコメントの検出 |

## HIGH SIGNAL 原則

**フラグしてよい問題:**

- コードがコンパイル/パースに失敗する（構文エラー・型エラー）
- 特定の入力で確実に誤った結果を返す（明確なロジックエラー）
- CLAUDE.md/conventions のルールを直接引用して証明できる違反

**フラグしてはならないもの（フォールスポジティブ）:**

- pre-existing issues（PR の変更外の問題）
- スタイル・品質の懸念（linter が検出するものを含む）
- 潜在的・仮定的な問題（「〜かもしれない」）
- テストカバレッジ不足・セキュリティの一般的懸念（CLAUDE.md に明記なし）
- 既にコード内でシレンスされている問題（lint ignore コメント等）

## 成果物

各エージェントが返す問題リスト:

```
[問題の説明] / 観点: [CLAUDE.md準拠/conventions/バグ/セキュリティ] / ファイル: [パス:行番号]
```
