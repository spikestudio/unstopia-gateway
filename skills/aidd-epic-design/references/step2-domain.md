# Step 2: ドメインモデル詳細 — 詳細手順

Step 1 で承認された Epic 仕様書を起点に、Phase レベルのドメイン分析成果物をドリルダウンして集約の詳細設計を行う。

## 入力

- Step 1 承認済み Epic 仕様書（`docs/requirements/ES-NNN-slug.md`）
- Phase ドメイン分析成果物:
  - BC キャンバス（`docs/domain/bc-{name}.md`）
  - ドメインイベント一覧（`docs/domain/domain-events.md`）
  - 用語集（`docs/glossary.md`）
- `references/design-templates/domain-model.md`（出力テンプレート）

## Phase ドメイン分析が存在しない場合

`/aidd-new-phase` が Step 3-4 をスキップしたケースでは、上記の Phase 成果物が存在しない。

- Epic レベルからドメインモデルを新規定義する（品質は維持するが入力の粒度が粗い）
- ユーザーにこの状況を説明し、確認を得てから進める

## 手順

### 1. BC キャンバス読み込みと集約候補の確認

- `docs/domain/bc-{name}.md` から Epic の所属 BC を特定する
- BC キャンバスの妥当性を検証する（Phase 定義後に BC 構造が変更されている可能性がある）。不整合があれば人間に報告する
- 集約候補名一覧を取得し、Epic AC との対応を確認する
- AC がカバーする集約候補を洗い出し、スコープ外の候補を除外する

### 2. 集約詳細化（エンティティ単位で構造＋ロジックを一括定義）

集約候補をエンティティ・値オブジェクト・ドメインサービスに展開する。各エンティティは以下の形式で定義する。

---

```markdown
## Entity: <名前>

### Fields（属性）

| フィールド | 型 | 制約 | 必須 | 対応 AC |
|-----------|---|------|------|--------|
| | | | | |

### Invariants（不変条件）[必須: 1件以上、根拠 AC-ID を明記]

- [不変条件の内容] — 根拠: [AC-ID]

### State Machine（状態遷移）[状態数 3 以上の場合必須]

| 遷移元 | イベント | ガード条件 | 遷移先 | 根拠 AC-ID |
|--------|---------|----------|--------|-----------|
| | | | | |

※「常に可」の場合もガード条件列に「常に可」と明示する

### Domain Logic（計算式・決定表）[「〜によって決まる」パターンの AC がある場合必須]

| ロジック名 | 入力 | 出力 | 計算式 / 決定表 | 根拠 AC-ID |
|-----------|-----|------|---------------|-----------|
| | | | | |

### Operations（操作の事前/事後条件）

| 操作名 | 事前条件 | 事後条件 | 失敗時の挙動 | 根拠 AC-ID |
|-------|---------|---------|------------|-----------|
| | | | | |
```

---

**各セクションの適用条件:**

| セクション | 適用条件 |
|-----------|---------|
| Fields | 全エンティティ必須 |
| Invariants | 全エンティティ必須（1件以上） |
| State Machine | 状態数 3 以上のエンティティに必須 |
| Domain Logic | 「〜によって決まる」パターンの AC がある場合に必須 |
| Operations | 全エンティティ（操作が存在する場合） |

### 3. ドメインイベントへのペイロード追加

- `docs/domain/domain-events.md` のイベント名にペイロード（フィールド名・型）を追加する
- Phase では「イベント名まで」、Epic で「ペイロードまで」がドリルダウン境界

### 4. 用語集更新

- `docs/glossary.md` に Epic 固有の用語を追記する
- 既存用語との重複チェックを行い、BC 間で意味が異なる用語は差分を明記する

### 4-1. 論点別議論（設計の透明化）

ドメインモデルの設計判断を透明化するため、以下の 3 カテゴリについて複数案とトレードオフを提示してユーザーと議論する。全カテゴリについて議論が完了するか、ユーザーが「このカテゴリは議論不要」と判断した後、次の手順に進む。

#### カテゴリ 1: 集約境界

どこで集約を分けるか（例: 注文と注文明細を 1 集約にするか別にするか）について、複数案のトレードオフを提示して合意する。

提示フォーマット:

```
**集約境界の判断: [集約名]**

| 案 | 構造 | メリット | デメリット |
|----|------|---------|----------|
| A（推奨）| [構造] | [メリット] | [デメリット] |
| B | [構造] | [メリット] | [デメリット] |

推奨は案 A です。理由: [1〜2 文]
どの案で進めますか？
```

#### カテゴリ 2: 主要な不変条件

ビジネスルールの制約（例: 「注文金額は 0 より大きい」「在庫数はマイナスにならない」）が正しいかユーザーと確認する。

提示フォーマット:

```
**不変条件の確認: [集約名]**

以下の不変条件を定義しました:
1. [不変条件の内容] — 根拠: [ビジネスルール / AC-ID]
2. ...

追加すべき不変条件や修正が必要な条件はありますか？
```

#### カテゴリ 3: 状態遷移の妥当性

ステートマシンの遷移が業務要件を満たしているかユーザーと確認する。

提示フォーマット:

```
**状態遷移の確認: [エンティティ名]**

| 遷移元 | イベント | ガード条件 | 遷移先 |
|--------|---------|----------|--------|
| [状態] | [イベント] | [条件 or 常に可] | [状態] |

この遷移で業務フローが表現できていますか？不足している遷移や不要な遷移があれば指摘してください。
```

**議論をスキップできる条件:** ユーザーが「このカテゴリは議論不要」と明示した場合のみスキップ可能。スキップした場合はブリーフィングでその旨を明示する。

### 5. セルフレビュー

セルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）。以下の 2 観点で検証する。

#### consistency 観点

- ドメインモデルと Epic 仕様書の AC の整合性
- Phase ドメイン分析成果物との整合性（集約候補の変更がある場合は理由を明記）
- 不変条件がビジネスルールとして妥当か
- 状態遷移が不変条件と矛盾しないか
- Fields の制約列がドメインモデルの属性制約と一致しているか

#### completeness 観点

- **Step 1 の確認（G3 ゲート前の最終確認）**: 全 AC が `docs/domain/[epic-slug]-ac-mapping.md` にマップされているか（未マップ AC が 0 件）
- 全エンティティに Invariants が 1 件以上定義されているか
- ステートフルエンティティ（状態数 3 以上）の全遷移にガード条件が定義されているか（「常に可」含む）
- 「〜によって決まる」パターンの AC に Domain Logic（計算式または決定表）があるか
- 全操作に Operations（事後条件・失敗時の挙動）が定義されているか

#### FAIL 条件（G3 完了不可）

以下に該当する場合、セルフレビューは FAIL とし、修正後に再確認する:

- Step 1 のマッピングマトリクスに未マップ AC が存在する（Step 1 に戻る）
- Invariants が 0 件のエンティティが存在する
- ガード条件未定義の状態遷移が存在する

### 6. ファイル書き出し

- `references/design-templates/domain-model.md` テンプレートに従い `docs/domain/[epic-slug]-domain-model.md` に書き出す
- ドメインイベント一覧（`docs/domain/domain-events.md`）を更新する
- 用語集（`docs/glossary.md`）を更新する
- Epic 仕様書の設計成果物テーブルを更新する

## 成果物

- 集約モデル詳細（`docs/domain/[epic-slug]-domain-model.md`）
- ドメインイベント更新（`docs/domain/domain-events.md`）
- 用語集更新（`docs/glossary.md`）
- Epic 仕様書の設計成果物テーブル更新

## 完了時

`references/briefing-spec.md` に従ってブリーフィングを実行する。

**ブリーフィングで必ず提示する情報（成果物を見なくても判断できるレベル）:**

<!-- 括弧内は briefing-spec.md 5 要素との対応 -->

- Phase 分析からの変更点（集約候補をどう展開したか）（要素1: 成果物全内容）
- 全集約の一覧表（集約名・ルートエンティティ・値オブジェクト・不変条件数）（要素1: 成果物全内容）
- 不変条件の全件リスト（ビジネスルール・根拠 AC-ID）（要素1: 成果物全内容）
- 状態遷移の全件表（遷移元・イベント・ガード条件・遷移先）（要素1: 成果物全内容）
- Domain Logic の定義一覧（ロジック名・計算式/決定表）（要素1: 成果物全内容）
- Operations の定義一覧（操作名・事前/事後条件・失敗時の挙動）（要素1: 成果物全内容）
- ドメインイベントペイロードの定義（イベント名・ペイロードフィールド・発行条件）（要素1: 成果物全内容）
- 追加した用語一覧（用語・定義・所属 BC）（要素1: 成果物全内容）
- 設計判断の根拠と代替案（集約境界・不変条件の選定理由）（要素2: 判断根拠）
- 状態遷移の設計判断（なぜこの状態を選んだか、代替案）（要素2: 判断根拠）
- AI 自信度が低い箇所（ドメイン知識不足で集約設計の妥当性を保証できない部分）（要素4: 未決定事項）
- 人間に判断を求める事項（集約境界、不変条件の妥当性、ドメインサービスの切り出し）（要素5: 意思決定事項）

承認を得てから Step 3 に進む。
