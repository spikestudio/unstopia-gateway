# /aidd-epic-design — ドメイン設計 + インターフェース設計（G3 専用）

G2 承認済みの Epic 仕様書を入力として、AC パターン分類・AC-ドメイン要素マッピングマトリクス生成から始まり、ドメインモデル詳細（集約構造・不変条件・ロジックセット）とインターフェース設計骨格（DB スキーマ・API spec）まで一貫して実行する。各ステップ完了時にブリーフィング→承認ゲートを設け、スキップを構造的に防止する。`/aidd-new-epic` の G2 通過後に常にこのスキルを呼び出す。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| Epic 仕様書（`docs/requirements/ES-NNN-slug.md`）が存在する | `ls docs/requirements/ES-*.md` | `/aidd-new-epic` を先に実行 |
| G2（AC 承認）ゲートが通過済み | Epic 仕様書の「ゲート状態」フィールドが `G2: 通過` | `/aidd-new-epic` Step 1 を先に完了 |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| Epic 名または Epic 番号 | 引数 | 任意 | 指定なしなら進行中 Epic を自動推定してユーザーに確認 |

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| /aidd-research | Step 2, 3 | 技術仕様・外部 API 調査（必要時のみ） |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| AC マッピングマトリクス | ファイル | `docs/domain/[epic-slug]-ac-mapping.md` |
| ドメインモデル詳細 | ファイル | `docs/domain/[epic-slug]-domain-model.md` |
| DB スキーマ骨格 | ファイル | `docs/design/[epic-slug]-db-schema.md`（データ永続化を伴う場合） |
| API spec 骨格 | ファイル | `docs/design/[epic-slug]-api-spec.md`（API を含む場合） |
| インフラ設計骨格 | ファイル | `docs/design/[epic-slug]-infra.md`（インフラ変更を伴う場合） |
| バッチ/ジョブ設計骨格 | ファイル | `docs/design/[epic-slug]-batch-job.md`（バッチ/非同期ジョブを含む場合） |
| Epic 仕様書（更新） | ファイル | `docs/requirements/ES-NNN-slug.md`（設計成果物テーブル・ステータス更新） |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-decompose-epic` | G3 通過後、Task 分解に進む |
| `/aidd-screen-plan` | 画面系プロジェクトの場合、G3 通過後に G3 Section 2 へ進む |

## 不変条件

- **G2 未通過の Epic には実行できない。** Step 0 の前提チェックで停止し、`/aidd-new-epic` の実行を案内する。
- **セルフレビューで未マップ AC が残る場合、G3 を完了できない。** 全 AC が少なくとも 1 つのドメイン要素にマッピングされるまで Step 1 を再実行する。

## Step 0: 入力受付・前提確認

`$ARGUMENTS` から Epic 名または番号（`ES-NNN`）を取得する。

### 0-1. 引数解決

- 引数あり → `docs/requirements/ES-*.md` から一致する Epic を特定する
- 引数なし → `status:in-progress` ラベルの epic Issue を検索し、対象 Epic を自動推定してユーザーに確認する
- 特定失敗 → エラーメッセージを表示して停止する:

  ```
  対象 Epic を特定できませんでした。
  引数に Epic 名または番号（例: ES-042）を指定して再実行してください。
  ```

### 0-2. G2 通過確認

Epic 仕様書の「ゲート状態」フィールドを確認する。`G2: 通過` でない場合は以下を表示して停止する:

```
この Epic は G2（AC 承認）を通過していません。
/aidd-new-epic を実行して Step 1（AC 導出）を完了・承認してから、このスキルを実行してください。
```

### 0-3. スキル実行開始の確認

対象 Epic、入力ファイルパス、実行するステップ（Step 1〜3）をユーザーに提示し、実行開始の確認を得る:

```
以下の内容で /aidd-epic-design を開始します:

  Epic: ES-NNN [Epic名]
  仕様書: docs/requirements/ES-NNN-slug.md
  ゲート状態: G2 通過済み

  実行ステップ:
    Step 1: AC パターン分類 + AC-ドメイン要素マッピングマトリクス生成
    Step 2: ドメインモデル詳細（構造 + ロジックセット）
    Step 3: インターフェース設計骨格（DB スキーマ・API spec 等）

開始してよいですか？
```

## Step 1: AC パターン分類 + AC-ドメイン要素マッピングマトリクス生成

> **前提: Step 0 の確認が完了していること。**

`references/step1-ac-pattern.md` を読み込んで実行する。

**概要:** Epic 仕様書の全 AC をパターン分類（CRUD・状態遷移・バリデーション・通知・集計 等）し、AC-ドメイン要素（集約・エンティティ・値オブジェクト・ドメインサービス）のマッピングマトリクスを生成する。全 AC に少なくとも 1 つのドメイン要素が紐付くことを保証する。

**成果物:** `docs/domain/[epic-slug]-ac-mapping.md`

**capability 呼び出し:**

- セルフレビュー反復ループ — `coverage` 観点（未マップ AC がゼロになるまで反復）

**追加レビュー観点:**

| 観点 | チェック内容 |
|------|------------|
| AC 全件マッピング | 全 AC に少なくとも 1 つのドメイン要素が紐付いているか |
| パターン分類の妥当性 | 分類が AC の本質を正しく表しているか |
| ドメイン要素の粒度 | 集約・エンティティ・値オブジェクト・ドメインサービスの区別が適切か |
| 新規ドメイン要素の識別 | 既存ドメインモデルにない要素が登場した場合に明示されているか |

**完了時:** `references/briefing-spec.md` に従ってブリーフィングを実行し、承認を得る。

**ブリーフィング重点:**

| 重点項目 | 説明内容 |
|---------|---------|
| AC パターン分布 | パターンごとの AC 件数と設計上の含意 |
| 新規ドメイン要素 | 既存ドメインモデルから追加・変更が必要な要素 |
| 未解決の曖昧性 | マッピングが複数候補に分かれた AC とその判断理由 |
| AI 自信度 | パターン分類・マッピングの判断根拠と不確かな箇所 |

ユーザーの明示的な承認を得てから Step 2 に進む。

## Step 2: ドメインモデル詳細

> **前提: Step 1 の承認が完了していること。**

`references/step2-domain.md` を読み込んで実行する。

**概要:** Step 1 で生成した AC-ドメイン要素マッピングを基に、Epic レベルで集約の不変条件・エンティティ属性・値オブジェクト制約・ドメインサービスのロジックセットを定義する。Phase レベルのドメイン分析成果物（BC キャンバス・ドメインイベント・用語集）が存在する場合は入力として参照する。成果物は `docs/domain/` に配置する。

**成果物:**

- 集約モデル詳細（`docs/domain/[epic-slug]-domain-model.md`）
- ドメインイベント詳細（`docs/domain/domain-events.md` に追記、既存ファイルがある場合）
- 用語集更新（`docs/glossary.md` に追記、新規ドメイン用語がある場合）
- Epic 仕様書の設計成果物テーブル更新

**capability 呼び出し:**

- `/aidd-research` — 技術仕様・外部 API 調査（必要時のみ）
- セルフレビュー反復ループ — `consistency` 観点（aidd-framework/references/self-review-loop.md 参照）

**完了時:** `references/briefing-spec.md` に従ってブリーフィングを実行し、承認を得る。

**ブリーフィング重点:**

| 重点項目 | 説明内容 |
|---------|---------|
| 集約境界の判断根拠 | なぜその境界で集約を切ったか、トランザクション整合性との関係 |
| 不変条件の導出根拠 | どのビジネスルールを不変条件として定義したか |
| Step 1 マッピングからの変更点 | マッピングマトリクスをどう展開・精緻化したか |
| 技術選択への影響 | ドメインモデルが技術スタックに与える制約 |
| AI 自信度 | 集約境界・ドメインサービス設計の不確かな箇所 |

ユーザーの明示的な承認を得てから Step 3 に進む。

## Step 3: インターフェース設計骨格

> **前提: Step 2 の承認が完了していること。**

`references/step3-interface.md` を読み込んで実行する。

**概要:** DB スキーマ骨格と API spec 骨格を**骨格レベル**（テーブル一覧・カラム名・エンドポイント一覧・リクエスト/レスポンス型定義）で生成する。インデックス設計・詳細バリデーション実装は Task に委ねる。成果物は `docs/design/` に配置する。

**成果物:**

- DB スキーマ骨格（`docs/design/[epic-slug]-db-schema.md`）— データ永続化を伴う場合
- API spec 骨格（`docs/design/[epic-slug]-api-spec.md`）— API を含む場合
- インフラ設計骨格（`docs/design/[epic-slug]-infra.md`）— インフラ変更を伴う場合
- バッチ/ジョブ設計骨格（`docs/design/[epic-slug]-batch-job.md`）— バッチ/非同期ジョブを含む場合
- Epic 仕様書の設計成果物テーブル更新

**完了ゲート:** 「AI が実装できるレベル」の 5 問に全て回答できること:

1. エンティティの属性は何か
2. 適用されるビジネスルールは何か
3. データの保存方法は何か
4. インターフェースの入出力は何か
5. 失敗時の挙動は何か

**完了時:** `references/briefing-spec.md` に従ってブリーフィングを実行し、承認を得る。承認 = G3 通過。

> Step 3 の承認 = G3（設計承認）相当。

**ブリーフィング重点:**

| 重点項目 | 説明内容 |
|---------|---------|
| 骨格の範囲判断 | 何を骨格に含め、何を Task に委ねたか |
| スキップ判断 | DB/API をスキップした場合、その理由 |
| 設計トレードオフ | 複数の選択肢があった設計判断 |
| 依存関係 | 他 Epic・外部システムとの依存 |

## 設計不要時のスキップ

以下の条件に該当する場合、Step 2-3 をスキップできる:

- ドキュメント群のプロジェクト（ドメインモデルが存在しない）

**手順:**

1. Step 1 完了後に「設計ステップは該当なし」と人間に説明
2. 人間の承認を得てスキップ（承認なしのスキップは禁止）
3. Epic 仕様書の設計成果物テーブルに「該当なし」と記載

**部分スキップ:**

- データ永続化を伴わない Epic → DB スキーマ生成をスキップ
- API を含まない Epic → API spec 生成をスキップ
- 各スキップの判断理由はブリーフィングで説明する

## 完了処理

> **タイミング**: Step 3 の最終ブリーフィングをユーザーが承認した直後に実行する。Step 2-3 をスキップした場合は Step 1 承認後に実行する。

**1. Epic 仕様書のステータスを「設計済み」に更新する:**

`docs/requirements/ES-NNN-slug.md` の「ゲート状態」フィールドを `G3: 通過` に更新し、設計成果物テーブルに生成したファイルパスを記載する。

**2. セルフレビューと確定コミット:**

セルフレビュー観点（design-consistency）:

- 全 AC が Step 1 マッピングマトリクスに存在するか
- ドメインモデルの全集約が DB スキーマ（またはスキップ理由）に対応しているか
- API spec の全エンドポイントがドメインサービスまたは集約操作に対応しているか
- Epic 仕様書の設計成果物テーブルに全成果物ファイルパスが記載されているか

指摘があれば修正してから確定コミットを行う:

```bash
git add docs/domain/ docs/design/ docs/requirements/ES-NNN-slug.md
git commit -m "feat: ES-NNN [Epic名] — G3 設計成果物確定"
```

**3. G3 通過コメントを GitHub Issue に投稿する:**

```bash
gh issue comment <Epic Issue番号> --body "G3 通過: /aidd-epic-design 完了

設計成果物:
- AC マッピング: docs/domain/[epic-slug]-ac-mapping.md
- ドメインモデル: docs/domain/[epic-slug]-domain-model.md
- DB スキーマ: docs/design/[epic-slug]-db-schema.md（生成した場合）
- API spec: docs/design/[epic-slug]-api-spec.md（生成した場合）
"
```

**4. 次のステップ案内:**

```
設計が完成しました（G3 通過）:
- AC マッピング: docs/domain/[epic-slug]-ac-mapping.md
- ドメインモデル: docs/domain/[epic-slug]-domain-model.md
- 設計成果物: docs/design/ 配下（生成したもの）
- Epic 仕様書: docs/requirements/ES-NNN-slug.md（G3 通過済み）

次のステップ（条件に応じて選択）:
→ Task 分解に進む: `/aidd-decompose-epic ES-NNN`（G4 へ）
→ 画面系プロジェクトの場合: `/aidd-screen-plan ES-NNN`（G3 Section 2 へ）
→ 次の Epic の設計に進む: `/aidd-epic-design [次の Epic名]`
```

## ゲート制御ルール

**各ステップ間の承認ゲートは構造的に必須である。以下のルールに例外はない。**

1. **Step N の承認なしに Step N+1 を開始してはならない。** 承認なしに次ステップを進めようとした場合、「Step N の承認が必要です」と表示しスキルを停止する。
2. **ブリーフィングなしに承認を求めてはならない。** 各ステップ完了時に `references/briefing-spec.md` に従ったブリーフィングを必ず実行する。1-2 行のサマリーのみでの承認要求は禁止。
3. **回答 ≠ 承認。** ユーザーの問いかけへの回答は回答内容の反映のみに留める。次ステップへの遷移には「OK」「進めて」「承認」等の明確な承認表現が必要。
4. **設計スキップも承認が必要。** Step 2-3 のスキップは人間の承認なしに行えない。
5. **未マップ AC が残る場合、Step 1 を完了できない。** セルフレビューで未マップ AC が検出された場合は再度マッピングを行い、ゼロになるまで反復する。

## 重要なルール

- **収束フェーズ**: このスキルは収束フェーズに属する。G2 承認済みの Epic 仕様書を入力として設計を確定させる
- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う** — 優先順序 1〜5 に該当する問題の見送り・後回しを推奨しない
- **ブリーフィング省略禁止** — 各ステップ完了時に `references/briefing-spec.md` に従ったブリーフィングを必ず実行する
- 業務ルール・ドメイン知識は推測で埋めない。不明な点は人間に質問する
- AI が新たに識別したドメイン要素（集約・値オブジェクト等）は必ず人間に提示し、承認を得る
- 各ステップの詳細手順は `references/` を読んで実行する。SKILL.md 内に手順を複製しない
- 成果物生成・ステップ承認後などの状態遷移ごとにコミットする。push はユーザーの確認を待つ
