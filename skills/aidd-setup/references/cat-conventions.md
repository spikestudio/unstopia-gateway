# カテゴリ: conventions（コーディング規約）

技術スタックを検出し、言語・フレームワーク固有のコーディング規約をリサーチして `docs/conventions/` に反映する。

> **このカテゴリの目的は「TODO だらけの規約を実用的なルールで埋める」こと。** 一般論ではなく、プロジェクトの技術スタックに具体化した規約を生成する。

## 前提条件

- `project` カテゴリが実行済み（`docs/conventions/` の 7 ファイルが存在すること）
- 技術スタックが選定済み（未選定の場合は引数で指定）

## 手順

### 1. 技術スタック検出

引数に言語/FW 名が指定されている場合はそれを使用する。指定がない場合は以下から自動検出:

| 検出元 | 検出内容 |
|--------|---------|
| `docs/architecture/tech-stack.md` | 技術スタックの公式定義（最優先） |
| `package.json` | Node.js / TypeScript / フレームワーク |
| `go.mod` | Go / フレームワーク |
| `pyproject.toml` / `requirements.txt` | Python / フレームワーク |
| `Cargo.toml` | Rust / フレームワーク |
| `tsconfig.json` | TypeScript 設定 |
| `pnpm-workspace.yaml` / `turbo.json` | monorepo 構成 |

検出結果をユーザーに提示し、確認を得る。推定失敗時は引数での指定を案内する。

### 2. リサーチ

検出した技術スタックに基づき、以下の情報を広範囲にリサーチする:

| リサーチ対象 | 情報源 |
|------------|--------|
| 公式スタイルガイド | 言語・FW の公式ドキュメント |
| コミュニティのベストプラクティス | 主要なスタイルガイド（Google, Airbnb, Effective Go 等） |
| 言語固有のアンチパターン | 公式ドキュメント、コミュニティの集 |
| デザインパターン | 言語固有の推奨パターン |
| DI パターン | 言語固有の推奨 DI 手法 |
| エラーハンドリングパターン | 言語固有の推奨エラー処理手法 |
| テスト戦略 | FW 推奨のテスト手法・ライブラリ |
| ディレクトリ構成 | FW 推奨のプロジェクト構成 |
| セキュリティプラクティス | OWASP、言語固有のガイドライン |
| パフォーマンスプラクティス | 言語・FW 固有の最適化パターン |

**リサーチの原則:**

- 公式ドキュメントを最優先するが、それに限定しない
- 広く採用されているパターンを優先し、ニッチな手法は避ける
- リサーチ結果の出典を記録する

**WebSearch 不可時のフォールバック:** AI の学習済み知識ベースで規約を生成し、冒頭に注意コメントを付記する。

### 3. 具体化テンプレート突合チェック

`aidd-framework/guides/convention-bootstrap.md`「汎用原則の具体化ガイド」セクションを読み込み、各原則が対象ファイルに反映されているかを確認する。

1. 「汎用原則の具体化ガイド」から各原則の `→ 対象ファイル:` を抽出
2. 各対象ファイルの内容を読み、原則に対応するルール・正例・誤例が含まれているか確認
3. 不足があれば補完対象として記録

### 4. 規約生成

`aidd-framework/guides/convention-bootstrap.md` のガイドに準拠し、各規約ファイルを生成する。

**生成ルール:**

- 既存のプロジェクト固有記述は上書きしない
- `<!-- TODO -->` コメント箇所を補完する
- 突合チェックで不足と判定された原則を対象ファイルに追加する
- 各ルールに正例・誤例のペアを必ず付ける
- 強度（MUST / SHOULD / MAY）を明記する

**各ファイルの生成内容:**

| ファイル | 主な内容 |
|---------|---------|
| `naming.md` | 言語固有の命名慣習、FW 固有の命名パターン、Boolean プレフィックス、定数命名、ファイル名規則 |
| `directory-structure.md` | FW 推奨のディレクトリ構成、新規ファイルの配置判断基準、禁止される配置パターン |
| `layer-rules.md` | 依存方向、import 可能範囲、テスタビリティ（DI）、データ受け渡し |
| `error-handling.md` | 言語固有のエラー処理、Fail Fast、エラーレスポンスフォーマット、ログ出力ルール |
| `testing.md` | テストフレームワーク、テストファイル配置、テストピラミッド、モック方針、カバレッジ基準 |
| `prohibitions.md` | アンチパターン、外部依存の直接インスタンス化禁止、YAGNI |
| `git-workflow.md` | 変更なし（言語非依存）。プレースホルダーのままの場合のみデフォルトで補完 |

### 5. ユーザー確認

生成した規約をファイルごとにユーザーに提示し、承認・修正を受け付ける。全ファイル一括ではなく、1ファイルずつ確認する。

### 6. 規約間整合性チェック

全ファイルの生成・更新が完了した後、以下の整合性を検証する:

| チェック | 対象 |
|---------|------|
| パス参照が directory-structure.md の定義と一致しているか | 全ファイルのコード例内パス |
| コード例内の命名が naming.md の規約に従っているか | 全ファイルの正例・誤例コード |
| 禁止事項が error-handling.md / layer-rules.md と矛盾していないか | prohibitions.md の項目 |
| コード例が実際のプロジェクト構成と整合しているか | 全ファイルのコード例コメント内のパス |

不整合が見つかった場合、ユーザーに提示し修正を行う。

### 7. コミット

- ブランチ名: `chore/setup-conventions`
- コミットメッセージ: `chore: 技術スタックに基づくコーディング規約を生成`
- PR 作成するか確認する

## 完了条件

- `docs/conventions/` 配下の 7 ファイルが技術スタックに具体化されている（TODO コメントが解消済み）
- 全ルールに正例・誤例のペアが付いている
- 強度（MUST / SHOULD / MAY）が全ルールに明記されている
- convention-bootstrap.md の「汎用原則の具体化ガイド」の全原則がカバーされている
- 規約間の整合性チェックが完了し、矛盾がない
- ユーザーがファイルごとに承認済み
