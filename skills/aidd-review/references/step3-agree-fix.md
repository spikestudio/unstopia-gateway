# Step 3: critical/non-critical 分類 → 1 件ずつ合意・修正 — /aidd-review

Step 2 の全サブエージェント結果を集約し、問題を critical / non-critical に分類する。critical から処理を開始し、全件完了後に non-critical へ進む。各問題をユーザーに 1 件ずつ提示し、合意を得た上で修正を実行する。

## 3-1. 問題の集約と分類

全サブエージェントの検出結果をマージし、重複を除去する。各問題を以下の基準で分類する。

### critical の判断基準

以下のいずれかに該当する問題は **critical**:

- AC が未充足（ビジネス要件の欠落）
- バグ・セキュリティ脆弱性（特定の入力で確実に失敗する）
- CLAUDE.md / FRAMEWORK.md の明示的な禁止事項への違反
- データ不整合・データ損失リスク
- 仕様書と実装の直接矛盾

### non-critical の判断基準

以下の問題は **non-critical**:

- コードスタイル・命名規約の軽微な逸脱
- ドキュメントの補足・改善（内容に誤りはないが不完全）
- テスト品質の向上（テストは存在するが網羅性に改善余地）
- 設計判断の記録が不足している（コメントまたは ADR 追記推奨）

## 3-2. 問題一覧の提示（処理開始前）

処理を開始する前に、分類済み全問題一覧をユーザーに提示する。

```
## レビュー結果サマリー: [種別] [対象ID]

### critical（優先処理）
| # | 問題タイトル | 影響 |
|---|------------|------|
| C1 | [問題概要] | [影響範囲] |
| C2 | [問題概要] | [影響範囲] |

### non-critical
| # | 問題タイトル | 影響 |
|---|------------|------|
| N1 | [問題概要] | [影響範囲] |
| N2 | [問題概要] | [影響範囲] |

合計: critical [M] 件 / non-critical [N] 件

critical から 1 件ずつ確認します。進んでよいですか？
```

問題が 0 件の場合は「検出問題なし」を明示し、Step 4 への進行承認を求める。

## 3-3. 1 件ずつ合意・修正フロー

### 処理順序

1. critical を番号順に全件処理する
2. critical が全件完了したら non-critical を番号順に処理する
3. 各問題について「修正する / 見送る（理由必須）」をユーザーが選択する

### 各問題の提示形式

```
## 問題 [C/N][番号]/[合計]: [問題タイトル]

**種別**: critical / non-critical
**影響**: [影響範囲・深刻度]
**問題詳細**: [具体的な問題の説明]
**修正提案**: [具体的な修正内容・変更ファイル・変更箇所]

修正しますか？（修正する / 見送る ※見送りは理由が必要）
```

### ユーザーの選択に応じた処理

**「修正する」を選択した場合:**

1. 提案した修正を実行する
2. 修正完了後に「修正完了: [問題タイトル]」を確認表示する
3. 次の問題へ進む

**「見送る」を選択した場合:**

1. 見送り理由をユーザーに入力してもらう（理由なし見送りは不可）

   ```
   見送り理由を入力してください（例: 今 Phase では対象外・将来の拡張で対応予定・意図的な設計選択）:
   ```

1. 理由を受け取ったら、以下のいずれかの方法で記録する:

   **コードコメントとして記録する場合:**

   ```
   # suppress-approved: [見送り理由]（/aidd-review 見送り - [日付]）
   ```

   または関連コードの近くに:

   ```typescript
   // NOTE: [見送り理由]（/aidd-review 見送り決定 [日付]）
   ```

   **ADR として記録する場合（設計判断の場合）:**

   `/aidd-adr` を呼び出して見送り理由を ADR に記録する。

1. 記録完了後に「見送り記録完了: [問題タイトル]（理由: [理由の概要]）」を確認表示する
1. 次の問題へ進む

## 3-4. 全件完了後のサマリー提示

全問題の処理が完了したら以下のサマリーを提示する。

```
## 処理完了サマリー

| 区分 | 修正済み | 見送り（理由記録済み） |
|------|---------|---------------------|
| critical | [M] 件 | [N] 件 |
| non-critical | [P] 件 | [Q] 件 |
| **合計** | **[M+P] 件** | **[N+Q] 件** |

### 見送り一覧（理由付き）
| # | 問題タイトル | 見送り理由 | 記録場所 |
|---|------------|---------|---------|
| 1 | [タイトル] | [理由] | [ファイルパス / ADR番号] |

Step 4（完了処理）に進みますか？
```

## 3-5. ルール

- **理由なし見送りは不可**: ユーザーが見送りを選択した場合、必ず見送り理由を入力してもらう。理由が入力されるまで次の問題に進まない
- **critical の先行処理必須**: critical が残っている状態で non-critical を先に処理しない
- **修正実行後の確認**: 各修正は実行後にユーザーに確認表示を行う
- **問題の追加**: 修正作業中に新たな問題が発見された場合は一覧に追加し、同じフローで処理する
