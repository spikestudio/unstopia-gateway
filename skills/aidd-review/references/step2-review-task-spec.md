# Step 2: Task 定義レビュー（3 並列サブエージェント） — /aidd-review

3 つのサブエージェントを並列起動し、独立した観点から Task 定義群を検査する。

## 入力

Step 1 で取得済みの以下のデータを受け取る（サブエージェントが独自に取得することを禁止する）:

- Task 定義群（`docs/tasks/TASK-*.md` の内容）
- Epic 仕様書（`docs/requirements/ES-NNN-*.md` の内容）— AC 参照用

## 並列サブエージェントの起動

以下の 3 サブエージェントを **1 メッセージで同時起動** する（順番ではなく並列）:

### Sub-A: Sonnet — AC→Task カバレッジ

**モデル:** Sonnet

**観点:** AC→Task カバレッジ

**指示:**

- Epic 仕様書の全 AC（Acceptance Criteria）を列挙する
- 各 AC が少なくとも 1 つの Task にマップされているかを確認する
- **100% カバレッジが必須**: どの AC も 1 つ以上の Task に対応していなければならない
- カバレッジ未達の AC を具体的に列挙し、対応 Task が存在しない理由を分析する
- 逆マッピング（Task に紐づく AC が存在しない孤立 Task）も検出する

**フラグしてよい問題:**

- 対応する Task が存在しない AC（カバレッジ未達）
- Epic 仕様書の AC に紐づかない孤立 Task

**フラグしてはならないもの:**

- Task の実装詳細（実装方針・コード設計等）
- AC の品質・記述スタイル

### Sub-B: Sonnet — スコープ・完了条件

**モデル:** Sonnet

**観点:** スコープ・完了条件

**指示:**

- 各 Task のスコープ境界が明確かを確認する（何を含み何を含まないかが定義されているか）
- 完了条件が検証可能かを確認する（「動作すること」ではなく「〇〇テストが PASS すること」のように具体的か）
- Task 間の依存関係を分析し、循環依存（A→B→A のような依存ループ）が存在しないかを確認する

**フラグしてよい問題:**

- スコープ境界が曖昧で実装範囲が判断できない Task
- 完了条件が主観的・検証不可能な Task（例: 「きれいに実装すること」）
- 循環依存（A が B に依存し、B が A に依存する）

**フラグしてはならないもの:**

- スコープ境界の記述スタイル（記述方法の好みの差異）
- 完了条件の文言の細かい改善余地（意味は明確だが表現が冗長、など）

### Sub-C: Opus — 実装可能性・設計判断

**モデル:** Opus

**観点:** 実装可能性・設計判断

**指示:**

- 各 Task の実装可能性を評価する（技術的制約・依存ライブラリ・環境の前提が明示されているか）
- 設計判断漏れを検出する: Task 内の「要検討」「TBD」「TODO」等の未解決マークを全て列挙する
- Task サイズの適切性を評価する（大きすぎる Task は分割推奨、細かすぎる Task は統合推奨）
  - 目安: 1 Task = 1 コミット相当（数時間〜半日程度）
- 未解決の要検討マークがある Task は、設計判断が確定しないと実装を開始できないリスクを評価する

**フラグしてよい問題:**

- 実装に必要な技術的前提・依存が不明で着手できない Task
- 「要検討」「TBD」「TODO」等の未解決マークが残っている Task
- 1 Task のスコープが大きすぎて複数のコミットが必要になることが明確な Task

**フラグしてはならないもの:**

- 実装方針の好み（複数の有効なアプローチが存在するケース）
- コード設計の詳細（Task 定義のスコープ外）

## 問題の critical/non-critical 分類基準（task-spec 固有）

各サブエージェントは検出した問題を以下の基準で分類して報告する。

### critical

以下のいずれかに該当する問題は **critical**:

- AC→Task カバレッジ未達（Epic の AC に対応する Task が存在しない）
- 完了条件が検証不可能（主観的・客観的な検証手段がない）
- 循環依存（Task 間の依存関係にループが存在する）
- 未解決の要検討マーク（「要検討」「TBD」「TODO」等が Task に残っている）

### non-critical

以下の問題は **non-critical**:

- スコープ境界の記述に改善余地がある（意味は通じるが明確化できる）
- Task サイズ調整が推奨される（大きすぎる / 細かすぎる Task の分割・統合推奨）

## 成果物

各サブエージェントが返す問題リスト:

```
[問題の説明] / 観点: [ACカバレッジ/スコープ完了条件/実装可能性設計判断] / 対象Task: [TASK-NNN] / critical/non-critical
```

## ルール

- **サブエージェントは Step 1 の取得データのみを使用する** — 独自に `cat docs/tasks/...` や `cat docs/requirements/...` を実行することを禁止する
- **HIGH SIGNAL 原則** — 確実に問題と判断できる場合のみフラグする。推測・仮定に基づく問題はフラグしない
- **重複検出の排除** — 複数のサブエージェントが同一問題を検出した場合、`step3-agree-fix.md` への引き渡し前に重複を除去する
