# Step 2: サブエージェント並列起動（phase）— /aidd-review

`phase` 種別のレビューを実行する。Step 1 で取得済みの以下のデータを入力として受け取り、3 つのサブエージェントを並列起動する。

**入力（Step 1 取得済み）:**

| データ | 説明 |
|--------|------|
| Phase 定義書 | 機能意図・成功基準・非機能要件（`docs/requirements/phase-*.md`） |
| 全 Epic 仕様書 | Phase に紐づく全 AC・ストーリー（`docs/requirements/ES-*.md`） |
| マスタドキュメント群 | `docs/architecture/tech-stack.md`・`docs/architecture/architecture-overview.md`・DB 設計等 |
| Phase Issue 一覧 | `gh issue list --milestone "Phase N"` の結果（JSON） |
| feedback Issue 一覧 | `gh issue list --label feedback --state open` の結果（JSON） |

> **サブエージェントは Step 1 で取得済みのデータのみを使用する。** 独自に `gh` コマンドやファイル読み込みを実行することを禁止する。

## サブエージェント並列起動

以下の 3 つのサブエージェントを**同時に**起動する。

---

### Sub-A: 成功基準・完了確認（Sonnet）

**観点:** Phase の成果が定義された成功基準を満たしているか、全 Epic が完了しているかを確認する。

**チェック内容:**

#### A-1. 全 Epic 完了確認

Phase Milestone に紐づく全 Epic Issue が closed であることを確認する。

```
| Epic ID | タイトル | ステータス |
|---------|---------|----------|
| ES-NNN | [タイトル] | ✅ closed / ❌ open |
```

#### A-2. 成功基準充足確認

Phase 定義書の成功基準を 1 件ずつ確認し、充足状況を判定する。**全件を全文で表示する（省略・要約禁止）。**

```
| 基準 | 内容（全文・省略禁止） | 充足状況 |
|------|---------------------|---------|
| 基準-1 | [Phase 定義書の成功基準の全文をそのまま転記] | ✅ / ❌ |
```

各基準の充足根拠を示すこと（例: 対応する Epic・実装・テスト結果）。

#### A-3. E2E 検証状況確認

Phase の主要ユースケースが E2E で動作確認されているかを確認する。Phase 定義書に主要ユースケースが定義されている場合、各ユースケースの確認状況を提示する。

**出力フォーマット:**

```
### Sub-A 検証結果: 成功基準・完了確認

#### 全 Epic 完了
[Epic 完了確認テーブル]

#### 成功基準
[成功基準充足テーブル（全文・省略禁止）]

#### E2E 検証
[主要ユースケース確認状況]

**問題一覧:**
| # | 種別 | 問題 | critical/non-critical |
|---|------|------|----------------------|
| 1 | [観点] | [問題概要] | critical / non-critical |

問題なし の場合: 「検出問題なし」と明示する。
```

---

### Sub-B: マスタドキュメント最新化（Sonnet）

**観点:** Phase の実装成果がマスタドキュメントに正確に反映されているかを確認する。

**チェック内容:**

#### B-1. 技術スタック（tech-stack.md）

Phase で追加・変更したライブラリ・ツール・フレームワークが `docs/architecture/tech-stack.md` に反映されているか確認する。

#### B-2. アーキテクチャ概要（architecture-overview.md）

Phase で追加・変更したコンポーネント・サービス・インフラ構成が `docs/architecture/architecture-overview.md` に反映されているか確認する。

#### B-3. DB 設計

Phase でテーブル・カラム・インデックスの追加・変更があった場合、DB 設計ドキュメント（`docs/architecture/db-design.md` 等）が最新化されているか確認する。DB 変更がない Phase では「DB 変更なし」と明示する。

#### B-4. その他ドメイン成果物

`docs/domain/` 配下のドメインモデル・ユビキタス言語等が Phase の成果を反映しているか確認する。

**出力フォーマット:**

```
### Sub-B 検証結果: マスタドキュメント最新化

| ドキュメント | 最新化状態 | 備考 |
|------------|----------|------|
| tech-stack.md | ✅ 最新 / ❌ 未更新 | [未更新内容の概要] |
| architecture-overview.md | ✅ 最新 / ❌ 未更新 | [未更新内容の概要] |
| DB 設計 | ✅ 最新 / ❌ 未更新 / — 変更なし | [未更新内容の概要] |
| ドメイン成果物 | ✅ 最新 / ❌ 未更新 / — 対象外 | [未更新内容の概要] |

**問題一覧:**
| # | 種別 | 問題 | critical/non-critical |
|---|------|------|----------------------|
| 1 | [観点] | [問題概要] | critical / non-critical |

問題なし の場合: 「検出問題なし」と明示する。
```

---

### Sub-C: 非機能要件・feedback 確認（Opus）

**観点:** Phase の非機能要件が達成されているか、全 feedback Issue が解決されているかを確認する。

**チェック内容:**

#### C-1. 非機能要件の目標値達成確認

Phase 定義書に非機能要件（性能・可用性・セキュリティ・保守性等）が定義されている場合、目標値の達成状況を確認する。

```
| 非機能要件 | 目標値 | 達成状況 | 根拠 |
|----------|--------|---------|------|
| [要件名] | [目標値] | ✅ / ❌ / 未測定 | [根拠・測定結果] |
```

非機能要件が未定義の Phase では「非機能要件未定義」と明示する。

#### C-2. feedback Issue 解決確認

全 feedback Issue がクローズまたは判断済み（対応済み / 次 Phase 送り（理由付き） / 見送り（理由付き））であることを確認する。

```
| Issue # | タイトル | 状態 | 判断内容 |
|---------|---------|------|---------|
| #NNN | [タイトル] | ✅ closed / ❌ open | [判断内容] |
```

open の feedback Issue が残存する場合は critical 問題として報告する。

**出力フォーマット:**

```
### Sub-C 検証結果: 非機能要件・feedback 確認

#### 非機能要件
[非機能要件達成確認テーブル]

#### feedback Issue
[feedback Issue 解決確認テーブル]

**問題一覧:**
| # | 種別 | 問題 | critical/non-critical |
|---|------|------|----------------------|
| 1 | [観点] | [問題概要] | critical / non-critical |

問題なし の場合: 「検出問題なし」と明示する。
```

---

## 並列実行後の集約

Sub-A・Sub-B・Sub-C の結果が全件揃ったら、以下のフォーマットで集約結果を提示する。

```
## Step 2 集約結果: Phase Review

### Sub-A: 成功基準・完了確認
[Sub-A の検証結果全文]

### Sub-B: マスタドキュメント最新化
[Sub-B の検証結果全文]

### Sub-C: 非機能要件・feedback 確認
[Sub-C の検証結果全文]

---
全問題一覧（Step 3 へ引き渡し）:
| # | Sub | 観点 | 問題 | critical/non-critical |
|---|-----|------|------|----------------------|
| 1 | A | 成功基準 | [問題概要] | critical |
| 2 | B | マスタドキュメント | [問題概要] | critical |
...

合計: critical [M] 件 / non-critical [N] 件
```

## critical/non-critical 分類基準（phase 固有）

| 分類 | 該当する問題 |
|------|------------|
| **critical** | 成功基準の未達・全 Epic が完了していない・未解決の feedback Issue（open のまま）・マスタドキュメントの未更新（tech-stack・architecture-overview・DB 設計）|
| **non-critical** | 軽微なドキュメント改善（内容は正しいが補足が不足）・ドメイン成果物の細部更新 |

> **IMPORTANT:** このスキルは PASS/FAIL 判定のみを担う。G6 通過記録・Milestone クローズ・Phase Issue クローズは `/aidd-phase-closing` が担う。Step 4 完了後に `/aidd-phase-closing` の実行を案内すること。
