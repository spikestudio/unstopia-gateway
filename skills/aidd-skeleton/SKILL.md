# /aidd-skeleton — Skeleton 構築ガイド（G0 マイルストーン）

技術基盤定義で定義した技術スタック・アーキテクチャを、最小限の実装で E2E 動作検証する。規約ドキュメントの実地検証も兼ねる。G0 マイルストーンの Skeleton 検証カテゴリに対応する。

```
ゴール確認 → E2E パス選定 → 実装ガイド → 規約検証 → ローカルテスト確認
```

> **このスキルの目的は「技術基盤が実際に動作することを最小コストで証明する」こと。** 機能の完成度ではなく、レイヤー間の結合と規約の妥当性を検証する。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| 技術スタックが選定済み | `docs/architecture/tech-stack.md` の存在確認 | 技術スタック ADR を先に作成 |
| 規約ドキュメントが具体化済み | `docs/conventions/` の TODO が解消済み | `/aidd-setup conventions` を先に実行 |
| 開発環境が構築済み | mise, Docker 等が動作する | `/aidd-setup env` を先に実行 |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| （なし） | — | — | 対話的に E2E パスを選定する |

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| /aidd-research | Step 2 | 技術選定（必要時のみ） |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| Skeleton 実装 | コード | DB → ドメインロジック → 外部インターフェース の最小 E2E パス |
| テスト | コード | ユニットテスト + 結合テスト |
| 規約検証結果 | レポート | 規約と実装の整合性検証結果 |

| スキル | 条件 |

## Step 1: 入力受付

### ゴール確認

G0 マイルストーンの Skeleton 検証チェックリスト（`aidd-framework/checklists/g0-initialization.md` Section 3）を読み込み、ゴールを提示する:

```
## Skeleton 構築のゴール

G0 マイルストーン（Skeleton 検証）の通過には以下が必要です:

1. 最低 1 つの機能が DB → ドメインロジック → 外部インターフェース → テストで E2E 動作
2. サンプル実装が規約ドキュメント群に準拠
3. テストが存在し CI を通過
4. 規約ドキュメントが実装と整合していることを確認（乖離があれば規約修正）

これから E2E パスの選定に入ります。
```

### E2E パス選定

#### コンテキスト読み込み

- `docs/PROJECT-CHARTER.md` — ビジネスゴール・主要機能
- `docs/architecture/tech-stack.md` — 技術スタック
- `docs/architecture/architecture-overview.md` — アーキテクチャ方針
- `docs/conventions/` — 規約ドキュメント群
- `docs/architecture/adr/` — 技術的決定

#### E2E パスの提案

技術スタックの全レイヤーを通過する最小機能を提案する:

```
## E2E パス提案

**機能:** [最小機能の説明]
**選定理由:** [なぜこの機能が最小の E2E パスとして適切か]

通過するレイヤー:
- [ ] DB（テーブル作成 + CRUD）
- [ ] ドメインロジック（ビジネスルール 1 つ）
- [ ] 外部インターフェース（API エンドポイント 1 つ）
- [ ] テスト（ユニット + 結合テスト）
- [ ] CI パイプライン

この E2E パスで進めてよいですか？
```

ユーザーの承認を得てから Step 2 に進む。

## Step 2: 実装

### 成果物（生成）

選定した E2E パスを、レイヤーごとにステップバイステップで実装する。各ステップでユーザーに確認しながら進める:

#### DB レイヤー

- スキーマ定義（マイグレーション）
- 規約に従ったテーブル命名・カラム命名

#### ドメインロジック

- エンティティ / バリューオブジェクトの定義
- 最小のビジネスルール実装
- 規約に従った命名・ディレクトリ配置

#### 外部インターフェース

- API エンドポイント（またはその他のインターフェース）
- リクエスト / レスポンスの型定義
- エラーハンドリング（規約準拠）

#### テスト

- ユニットテスト（ドメインロジック）
- 結合テスト（DB → API）
- テスト規約に従った命名・構造

#### 規約検証

Skeleton 実装を通じて、規約ドキュメントの妥当性を検証する:

```
## 規約検証結果

### 規約通りに実装できた項目
- [項目1]: OK
- [項目2]: OK

### 規約と実装の乖離（要修正）
- [項目]: 規約では X だが実装では Y が自然。規約を修正しますか？

### 規約に不足していた項目
- [項目]: 規約に記載がなかったが、実装時に判断が必要だった。規約に追記しますか？
```

乖離や不足があった場合、規約ドキュメントの修正をユーザーと相談して実施する。

### 補完（情報収集）

技術選定に関する詳細な調査が必要な場合、`/aidd-research` に `技術選定` を依頼する。例:

- 特定のライブラリやフレームワークの設定方法
- マイグレーションツールの選定
- テスト戦略の詳細

### レビュー観点（追加）

セルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）。スキル固有の追加観点:

- Skeleton 実装が規約ドキュメント群に準拠しているか
- 全レイヤー（DB → ドメイン → インターフェース → テスト）を通過しているか
- テストが存在し CI を通過するか
- 規約と実装の乖離が解消されているか

### ブリーフィング重点（Step 5）

- E2E パスの全体像（どの機能でどのレイヤーを通過したか）
- 規約検証結果（乖離・不足の有無と対応内容）
- ローカルテスト通過状況

## Step 3: 完了処理

### ローカルテスト確認

ローカルでテストを実行して動作を確認する:

```bash
# ローカルテスト実行
task test
```

> CI は push 後に自動実行される。ローカルテスト通過後に push すること。

### 完了案内

ローカルテストが通過したら、次のステップを案内:

```
✅ Skeleton 構築完了

- E2E パス: [機能名]
- テスト: 通過
- ローカルテスト: 通過
- 規約修正: N 件

次のステップ:
→ ブリーフィングを実行（aidd-framework/references/briefing-format.md 参照）
→ `/aidd-next` でマイルストーン判定に進む（プロジェクト初期化の他の項目も含む）
```

## 重要なルール

- **日本語で対話する** — フレームワーク全体の対話言語が日本語であるため
- **FRAMEWORK.md の意思決定基準に従う** — 優先順序 1〜5 に該当する問題の見送り・後回しを推奨しない
- **PR 作成後は必ずレビューを提案する** — 「PR #XX を作成しました。`/aidd-epic-review` を実行しますか？」と問いかける。品質担保のプロセスを確実に実行するため
- 成果物生成・実装完了時にコミットを自動実行する。ユーザーの指摘反映後・ステップ承認後などの状態遷移ごとにもコミットする。push と PR 作成はユーザーの確認を待つ
- **最小限の実装に留める** — Skeleton の目的はレイヤー間結合と規約検証。機能の完成度は求めない。過剰な実装は後続タスクの設計判断を先取りするため
- **各ステップでユーザーに確認を求める** — 勝手に実装を進めない。技術選定の判断はユーザーにあるため
- **規約との乖離を必ず報告する** — 規約を修正するか実装を修正するかはユーザーが判断する。AI が独断で解決すると規約の意図を損なうため
- **技術基盤の問題を発見したら記録する** — ADR の追加・修正が必要な場合は明示する。決定事項のトレーサビリティを維持するため
