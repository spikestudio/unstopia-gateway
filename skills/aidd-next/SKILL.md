# /aidd-next — プロセスオーケストレーター

プロジェクト状態とスキルパイプラインを照合し、次に実行すべきスキルを提案する。マイルストーン前の前提条件チェックも担う。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| aidd-fw がインストール済み | `aidd-framework/` の存在確認 | `/aidd-setup fw` を先に実行 |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| `--phase N` | 引数 | 任意 | 指定 Phase に絞り込んで分析 |
| `--epic ES-NNN` | 引数 | 任意 | 指定 Epic に絞り込んで分析 |
| （引数なし） | — | — | プロジェクト全体を分析 |

### capability 呼び出し

該当なし

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| 現在の状態 | 対話出力 | パイプライン上の現在位置 |
| 次のアクション | 対話出力 | 優先度付きの具体的なアクション一覧 |
| 前提条件 | 対話出力 | 次に進む前に解消すべき項目（該当時のみ） |

### 後続スキル

| スキル | 条件 |
|--------|------|
| 提案されたスキル | 分析結果に基づき動的に決定 |

## Step 1: 情報収集

以下を並列で実行する:

```bash
gh issue list --state open --label epic --json number,title,state,labels
gh issue list --state open --label task --json number,title,state,labels
gh issue list --state open --label feedback --json number,title
gh pr list --state open --json number,title,headBranch
```

併せて以下のファイルを確認する:

- `docs/requirements/PD-*.md` — Phase 定義書の存在とステータス
- `docs/requirements/ES-*.md` — Epic 仕様書の存在とステータス
- `docs/tasks/TASK-*.md` — Task 定義の存在とステータス
- git 状態（現在のブランチ、未コミット変更）

引数 `--phase N` または `--epic ES-NNN` が指定されている場合は、該当するドキュメントと Issue に絞り込む。

## Step 2: 次のアクション提案

CLAUDE.md の「スキルパイプライン」セクションとプロジェクト状態を照合し、次に実行すべきアクションを優先度付きで提案する。

**優先度の基準:**

- **高**: オープン PR のレビュー、未対応 feedback、マイルストーン確認
- **中**: 次に着手すべき Task の実装、ドキュメント作成
- **低**: 将来の Phase/Epic の準備

**Epic 最終 Task の検出:**

残り 1 Task で Epic 完了と判定された場合、実装完了後に `/aidd-review epic` を実行する旨を付記する。

**全作業完了時:**

「全作業完了。次の Phase の検討を開始しますか？」と提案する。

## Step 3: 前提条件チェック

次のアクションに進む前にやるべきことがあれば提示する:

- **マイルストーン前提**: [milestones.md](../../aidd-framework/process/milestones.md) のチェックリストを参照し、前提スキルの完了状況を確認
- **ブロッカー**: 未対応 feedback Issue、放置中の PR、依存 Task 未完了
- **Issue 状態の異常**: open であるべき Issue が closed、またはその逆

## Step 4: アクション実行（承認後のみ）

以下のアクションはユーザーに提案し、承認を得てから実行する:

- Issue ラベル更新（`status:in-progress` 付与・除去等）
- ラベル付与（`status:in-progress` 等）

> **注意:** マイルストーン通過記録は各スキル（`/aidd-new-phase` が G1、`/aidd-review epic` が G5、`/aidd-review phase` が G6 等）が自身の完了処理で行う。`/aidd-next` はゲート判定・通過記録を行わない。

### 出力フォーマット

```
## 現在の状態
[Phase名]: Epic N/M 完了
現在: [パイプライン上の位置]

## 次のアクション
1. [具体的なアクション]（`/aidd-xxx` で実行）
2. ...

## 前提条件（該当する場合のみ）
- ⚠️ [未達項目] → [解決方法]

## マイルストーン（該当する場合のみ）
[G{N}] の前提が揃っています。チェックリストを確認して通過判定を実行しますか？
```

## 重要なルール

- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う** — 問題の後回しを推奨しない
- アクションは具体的に記述する — Issue 番号・スキル名・対象ファイルを明記する。「Issue を確認」ではなく「feedback #38 に対応し ES-002 の AC-3 を更新する」のように書く。理由: 曖昧な指示は次の行動に移れない
- マイルストーン通過記録・ラベル付与は提案のみ行い、ユーザーの承認後に実行する — 理由: AI が勝手にプロジェクト状態を変更すると人間が追跡できなくなる
- ブロッカーがある場合は必ず明示する — 理由: 見落としたブロッカーは手戻りの原因になる
- マイルストーン判定は [milestones.md](../../aidd-framework/process/milestones.md) のチェックリストに従う — 理由: /aidd-next 独自の判定ロジックを持つと二重管理になる
- 成果物生成・実装完了時にコミットを自動実行する。ユーザーの指摘反映後・ステップ承認後などの状態遷移ごとにもコミットする。push と PR 作成はユーザーの確認を待つ
