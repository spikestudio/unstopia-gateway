# /aidd-discuss — 議論・意思決定ファシリテーター

成果物の論点抽出・議論促進、レビュー指摘の対応方針、課題リストの整理など、議論と意思決定が必要な場面で専門家ペルソナを並行起動して多角的に分析し、1件ずつユーザーと合意形成しながらネクストアクションを確定するオーケストレーター。内部で Agent(subagent_type="aidd-analyst/architect/builder") を並行起動し、/aidd-research（技術調査）に委譲する。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| 意思決定が必要な項目群 | ユーザー入力 / GitHub Issue / PR | 項目を提示してください |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| 入力ソース | 引数 | 任意 | GitHub Issue URL、PR URL、テキスト。なければ会話履歴から抽出 |

### 委譲先

| コンポーネント | タイプ | 起動条件 |
|-------------|--------|---------|
| Agent(subagent_type="aidd-analyst") | ペルソナ | Step 2（業務視点分析、並行起動） |
| Agent(subagent_type="aidd-architect") | ペルソナ | Step 2（設計視点分析、並行起動） |
| Agent(subagent_type="aidd-builder") | ペルソナ | Step 2（実装視点分析、並行起動） |
| /aidd-research | capability | Step 3（技術調査、必要時） |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| 決定事項サマリー | テーブル | 全項目の決定内容・対策概要 |
| ネクストアクション | チェックリスト | 即時対応・次回対応・対応不要に分類 |

### 後続スキル

| スキル | 条件 |
|--------|------|
| /aidd-impl | 即時対応の実装が必要な場合 |
| /aidd-decompose-epic | Task 化が必要な場合 |
| /aidd-feedback-recorder | 規約追記・AI ミスパターンが発見された場合 |

## フロー概要

```
入力収集（成果物・課題・指摘）→ 論点抽出・分類 → 専門家ペルソナ並行分析 → 一覧提示
  → [1件ずつ] 分析・推奨案提示 → ユーザー判断 → 記録
  → 全件完了 → ネクストアクション一覧提示 → 合意形成 → 完了
```

---

## Step 1: 入力収集と項目抽出

### 入力ソースの特定

ユーザーの指示に応じて、以下のいずれか（または複合）から項目を収集する:

1. **会話履歴** — 現在の会話で既に言及されている課題・指摘を抽出
2. **成果物** — 指定された成果物（仕様書・設計書・SKILL.md 等）を読み込み、論点・トレードオフ・未決定事項を抽出
3. **GitHub PR** — `gh pr view <URL> --json reviews,comments` と `gh api repos/{owner}/{repo}/pulls/{number}/comments` でレビューコメントを取得
4. **GitHub Issues** — `gh issue view <URL> --json body,comments` で課題を取得
5. **テキスト入力** — ユーザーが直接リストを提示
6. **ファイル** — 指定されたファイルから課題リストを読み取り

### 項目の構造化

各項目を以下の形式で整理する:

```
ID: #1, #2, ...（通し番号）
タイトル: 簡潔な1行要約
ソース: どこから来た項目か（PR #123 のレビューコメント、Issue #456、会話内の指摘 等）
カテゴリ: 後述の分類
原文: 元の指摘・課題テキスト（省略せず引用）
```

### カテゴリ分類

各項目を以下のカテゴリに分類する（複数該当可）:

| カテゴリ | 説明 |
|---------|------|
| security | セキュリティに関する指摘 |
| performance | パフォーマンスに関する指摘 |
| architecture | 設計・アーキテクチャに関する指摘 |
| correctness | バグ・ロジック誤りに関する指摘 |
| maintainability | 保守性・可読性に関する指摘 |
| testing | テストに関する指摘 |
| convention | 規約・スタイルに関する指摘 |
| documentation | ドキュメントに関する指摘 |
| ux | ユーザー体験に関する指摘 |
| operations | 運用・監視・デプロイに関する指摘 |
| other | 上記に該当しないもの |

ユーザーの確認を得てから Step 2 に進む。

---

## Step 2: 専門家ペルソナ並行分析 + 合意形成

### 2-1. 一覧サマリー提示

全項目を一覧表で提示し、全体像をユーザーに把握してもらう:

```
## 項目一覧（全 N 件）

| # | タイトル | カテゴリ | 重要度 | ソース |
|---|---------|---------|--------|--------|
| 1 | ... | security | 対応必須 | PR #123 |
| 2 | ... | architecture | 対応推奨 | 会話内 |
| ...

**重要度（FRAMEWORK.md「全ての問題は必ず解決する」原則に基づく）:**
- 対応必須: 安全性・正確性・規約・整合性・技術的負債に該当。AI は見送りを推奨しない
- 対応推奨: 品質向上に該当。対応が望ましいが見送り可能
- 確認済み: 検討の結果、現状が正しいと判断。対応不要

検討順序は重要度の高い順を推奨しますが、変更できます。
この順序で進めてよいですか？
```

### 2-2. 各項目の専門家分析（1件ずつ、Agent を並行起動）

各項目について、3 つの専門家ペルソナを **1 メッセージ内で並行起動** して分析する:

```
Agent(subagent_type="aidd-analyst", prompt="""
  以下の問題を業務視点で分析してください。

  ## 問題
  [項目の原文]

  ## 分析観点
  - ユーザー影響: この問題がユーザーの業務フローにどう影響するか
  - 業務リスク: 放置した場合の業務上のリスク
  - 業務要件との整合性: 既存の要件・AC と矛盾しないか

  ## 参照
  - Epic 仕様書: docs/requirements/ES-*.md
  - プロジェクト憲章: docs/PROJECT-CHARTER.md
""")

Agent(subagent_type="aidd-architect", prompt="""
  以下の問題を設計視点で分析してください。

  ## 問題
  [項目の原文]

  ## 分析観点
  - 構造的影響: この問題がアーキテクチャにどう影響するか
  - 設計整合性: 既存の ADR・設計方針と矛盾しないか
  - 技術的負債: 放置した場合の技術的リスク

  ## 参照
  - ADR: docs/architecture/adr/
  - 技術スタック: docs/architecture/tech-stack.md
  - アーキテクチャ概要: docs/architecture/architecture-overview.md
""")

Agent(subagent_type="aidd-builder", prompt="""
  以下の問題を実装視点で分析してください。

  ## 問題
  [項目の原文]

  ## 分析観点
  - 実装コスト: 対応に必要な作業量（低/中/高）
  - 影響範囲: 変更が波及するファイル・コンポーネント
  - 分割可能性: 段階的に対応できるか、一括対応が必要か

  ## 参照
  - 規約: docs/conventions/
  - 既存コード: [関連ファイルパス]
""")
```

### 2-3. 結果集約・推奨案生成（自身が実行）

3 つのペルソナの分析結果を集約し、以下のフォーマットで提示する:

```
---
## 項目 #X / N: [タイトル]

**カテゴリ:** [カテゴリ] | **重要度:** [対応必須/対応推奨/確認済み] | **ソース:** [ソース]

### 原文
> [元の指摘・課題テキストをそのまま引用]

### 分析

**業務視点（aidd-analyst）:**
[業務影響・リスク分析]

**設計視点（aidd-architect）:**
[構造的影響・設計整合性分析]

**実装視点（aidd-builder）:**
[実装コスト・影響範囲・分割可能性]

### 推奨対策案

| 案 | 内容 | コスト | 効果 | リスク |
|----|------|--------|------|--------|
| A（推奨） | [具体的な対策] | [低/中/高] | [低/中/高] | [リスク説明] |
| B | [代替案] | ... | ... | ... |
| C（対応しない） | 理由を記録して見送り | なし | なし | [見送りリスク] |

推奨は **案 A** です。理由: [1-2文で根拠]

---
どの案で進めますか？（A / B / C / 他の案を提案 / 質問）
```

### 2-4. 各項目での対話ルール

- ユーザーが案を選択 → 決定として記録し、次の項目へ
- ユーザーが質問 → 該当ペルソナが回答。納得を得てから再度判断を求める
- ユーザーが別案を提案 → ペルソナで評価し、コスト・効果・リスクを提示
- ユーザーが「まとめて対応」→ 類似項目をグルーピングして一括対策を提案

決定のステータス:

<!-- レビュー指摘: 見送り・後回しのハードルが低く、技術的負債を許容する判断が容易にできてしまっていた -->

| ステータス | ユーザーの意思表示 | AI が推奨可能か | 処理 |
|-----------|-----------------|---------------|------|
| **対応** | 案 A/B を選択、または別案を提示 | 常に可能 | 決定事項として記録。ネクストアクションに含める |
| **見送り** | 「対応しない」「不要」 | 対応推奨/確認済みのみ。対応必須には推奨不可 | 見送りコメントを記録: `<!-- 見送り: [指摘内容] — 理由: [判断理由] -->` |
| **後回し** | 「後で」「今じゃない」「スキップ」 | AI は後回しを推奨しない | 人間が判断した場合のみ。ADR に記録 + 解消 Issue を必ず作成。`<!-- TODO: [指摘内容] -->` |

**対応必須の項目に対する見送り・後回し:**

AI は対応必須の項目に「見送り」「後回し」を推奨してはならない。人間がそれでも後回しを選択した場合は、以下を必ず実施する:

1. リスクを明示的に警告する
2. ADR に判断理由を記録する
3. 解消 Issue を作成し、対応期限を設定する

## Step 3: 情報収集（/aidd-research に委譲、必要時）

技術的な調査が必要な場合（ライブラリの仕様、パフォーマンス特性、セキュリティベストプラクティス等）:

```
Agent(prompt="""
  /aidd-research を実行してください。

  ## 調査対象
  [具体的な調査内容]

  ## 調査種別
  技術選定 / 仕様調査 / ベストプラクティス

  ## 期待する出力
  [判断に必要な情報]
""")
```

---

## Step 4: 整合性チェック・結果提示

全項目の検討完了後、決定事項の整合性をセルフチェックする:

- 決定事項間で矛盾がないか
- 既存の ADR・規約と矛盾しないか
- 見送り・後回しの判断が FRAMEWORK.md の意思決定基準に合致しているか
- 後回しの項目に解消 Issue が作成されているか

矛盾が見つかった場合はユーザーに報告して解消する。

**提示内容:**

### 決定事項サマリー

```
## 決定事項サマリー

| # | タイトル | 決定 | 対策概要 |
|---|---------|------|---------|
| 1 | ... | 案 A | [概要] |
| 2 | ... | 案 B | [概要] |
| 3 | ... | 見送り | [理由] |
```

### ネクストアクション提案

```
## ネクストアクション（提案）

以下の順序で対応することを提案します:

### 即時対応（このセッション内）
1. [ ] [具体的なアクション] — 対象: 項目 #1, #3
2. [ ] [具体的なアクション] — 対象: 項目 #2

### 次回対応
3. [ ] [具体的なアクション] — 対象: 項目 #5（要再検討）

### 対応不要（記録のみ）
- 項目 #4: [見送り理由]

この計画で進めてよいですか？調整したい点があればお知らせください。
```

---

## Step 5: 承認・合意形成

- ユーザーが承認 → 「即時対応」の項目から作業を開始するか確認
- ユーザーが調整を求める → 順序・内容を修正して再提示
- ネクストアクションの各項目についても、必要に応じて詳細を議論する

---

## 重要なルール

- **日本語で対話する**
- **FRAMEWORK.md の意思決定基準に従う** — 優先順序 1〜5 に該当する問題の見送り・後回しを推奨しない。コストが高い場合は分割・段階的対応を提案する
- **1件ずつ確実に合意を得る** — ユーザーの明示的な判断なしに次の項目に進まない
- **原文を省略しない** — 指摘・課題の原文は必ずそのまま引用する
- **推奨案には必ず根拠を付ける** — 「なんとなく」の推奨はしない
- **対応必須の項目に「見送り」を推奨しない** — FRAMEWORK.md の「全ての問題は必ず解決する」原則に従う。指摘が誤りの場合（確認済み）を除き、安全性・正確性・規約・整合性・技術的負債に該当する項目は対応を推奨する
- **スキップされた項目は最後に再確認する** — 忘れない
- **ペルソナの発言は専門性に基づく** — 各ペルソナは自分の専門領域の視点で発言する。汎用的なコメントは避ける
- **ペルソナ未配置時のフォールバック** — `.claude/agents/` にペルソナファイルが存在しない、または `subagent_type` が利用できない場合は、`Agent(prompt="あなたは [analyst/architect/builder] です。...")` のプレーン呼び出しで代替する
- **過剰な儀式化を避ける** — 項目が少ない場合（3件以下）はフォーマットを簡略化してよい
- **コンテキストを活用する** — プロジェクトの AI エージェント指示ファイル（`CLAUDE.md` / `AGENTS.md`）、規約、設計書がある場合は参照し、それに沿った推奨を行う

<!-- レビュー指摘: プラグマティストが「コストが高い」を理由に技術的負債の見送りを推奨していた -->
> **コスト制約:** コストを理由に品質を犠牲にする提案はしない。コストが高い場合は「どう分割すればコストを下げられるか」「段階的にどう対応するか」を提案する。

- **Agent ツールが利用できない環境では、3 ペルソナの分析をインラインで実行する**（フォールバック）。その場合も業務・設計・実装の 3 視点を必ずカバーする
