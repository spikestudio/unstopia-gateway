# /aidd-screen-design — 画面仕様書の対話的生成（廃止済み）

モックコードを読み取り、画面設計テンプレートに基づいて画面仕様書を自動生成するオーケストレーター。内部で専門家ペルソナ（解析・ドラフト生成）に委譲する。

> 画面設計テンプレート: `aidd-framework/templates/design/screen-design.md`
> 画面設計チェックリスト: `aidd-framework/checklists/g3-design-approval.md` Section 2（画面設計承認）
> 記入例: `aidd-framework/examples/design/blog-screen-design.md`

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| モック構築完了 | `mocks/app/[epic]/` ディレクトリの存在 | `/aidd-mock` を実行してください |
| Epic 仕様書 | `docs/requirements/ES-*.md` または `docs/epics/` | `/aidd-new-epic` で作成してください |
| デザインシステム定義 | `docs/design/design-system.md` の存在 | `/aidd-setup mocks` を実行してください |
| API 仕様書（推奨） | `docs/design/api-spec*.md` の存在 | 基盤設計で作成してください（なくても進行可能） |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| Epic ID | 引数 | 任意 | 指定なしならモック構築済み・画面仕様書未作成の Epic を自動検出 |

### 委譲先

| コンポーネント | タイプ | 起動条件 |
|-------------|--------|---------|
| Agent(subagent_type="aidd-analyst") | ペルソナ | Step 2（画面解析） |
| Agent(subagent_type="aidd-builder") | ペルソナ | Step 2（仕様書ドラフト生成） |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| 画面仕様書 | Markdown | `docs/design/screen-design-[epic-slug].md` |
| AC カバレッジ | テーブル | 全 AC が画面・インタラクションにマッピング済み |

### 後続スキル

| スキル | 条件 |
|--------|------|
| /aidd-next | G3 マイルストーン判定 |
| /aidd-decompose-epic | G3 通過後 |

## Step 1: 対象 Epic の特定

`$ARGUMENTS` に応じて対象 Epic を決定する:

- **Epic ID 指定あり**（例: `/aidd-screen-design ES-001`）: 指定された Epic 仕様書を読み込む
- **引数なし**: `docs/requirements/ES-*.md` からモック構築済み（`mocks/app/` にディレクトリが存在）で画面仕様書が未作成の Epic を推定し、推定結果をユーザーに確認してから進める
- **推定失敗時**:

```
対象の Epic が特定できませんでした。以下を確認してください:
- `/aidd-screen-design ES-NNN` — Epic ID を指定して実行
- モックが未構築の場合は `/aidd-mock` を先に実行してください
```

対象 Epic が特定できたら以下を読み込む:

1. **Epic 仕様書** — `docs/requirements/ES-*.md` から対象を読み込む
2. **モックディレクトリ** — `mocks/app/[epic-slug]/` のディレクトリ構造を走査
3. **AC の抽出** — Epic 仕様書の AC セクションを読み取る

ユーザーの確認を得てから Step 2 に進む。

---

## Step 2: モック解析・画面仕様書ドラフト生成

### 2-1. モック解析（インライン実行）

以下の手順でモック解析をインラインで実行する:

1. `mocks/app/[epic-slug]/` 配下のページファイル（`page.tsx`, `layout.tsx`）を走査する
2. 各ページの import 文からコンポーネント構成を抽出する
3. イベントハンドラ（onClick, onSubmit 等）からインタラクション定義を抽出する
4. 条件分岐から状態遷移を推定する
5. 画面一覧テーブル・画面遷移図・インタラクション定義・AC カバレッジを生成する

### 2-2. 画面解析（Agent に委譲）

```
Agent(subagent_type="aidd-analyst", prompt="""
  モック解析結果を業務視点で補完してください。

  ## 入力
  - モック解析結果: [Step 2-1 の出力]
  - Epic 仕様書: [Epic 仕様書パス]

  ## 検証・補完項目
  1. 画面遷移がユーザーの業務フローと整合しているか
  2. AC でカバーされていない暗黙のユースケースがないか
  3. エラー系の画面遷移が網羅されているか
  4. 各画面のペルソナ（利用者）が正しく特定されているか
""")
```

### 2-3. 仕様書ドラフト生成（Agent に委譲）

```
Agent(subagent_type="aidd-builder", prompt="""
  画面仕様書のドラフトを生成してください。

  ## 入力
  - モック解析結果: [Step 2-1 の出力]
  - 業務視点補完: [aidd-analyst の出力]
  - 画面設計テンプレート: aidd-framework/templates/design/screen-design.md
  - 記入例: aidd-framework/examples/design/blog-screen-design.md
  - デザインシステム: docs/design/design-system.md
  - API 仕様書: [あれば参照]

  ## 出力要件
  テンプレートの全セクションを充填する:
  1. 画面一覧テーブル
  2. 画面遷移図（Mermaid）
  3. 各画面の詳細（モック参照、コンポーネント構成、インタラクション定義、状態管理、条件分岐）
  4. フォームバリデーション（該当画面のみ）
  5. 共通コンポーネント
  6. 画面間のデータフロー
  7. AI が迷うポイント（全 8 項目に方針記入）
  8. AC カバレッジ

  ## 注意
  - 推定で記入した箇所は [推定] マーカーを付ける
  - API 呼び出し先が不明な場合は空欄にし、フラグを付ける
""")
```

ドラフトを画面ごとにユーザーに提示し、確認・修正を得る。**全画面の確認が完了してから** Step 3 に進む。

---

## Step 3: セルフチェック（自身が実行）

テンプレート末尾のセルフチェックリストを全項目確認する:

```
## セルフチェック（G3 マイルストーン — 画面設計）

- [✅/❌] 全画面にモック参照（実働モックまたはスクリーンショット）が含まれている
- [✅/❌] 画面遷移図が全画面・全遷移パスを網羅している
- [✅/❌] 全画面のコンポーネント構成が定義されている
- [✅/❌] 全コンポーネントの API 呼び出し先が明記されている
- [✅/❌] 全インタラクションに正常系・エラー系の結果が定義されている
- [✅/❌] フォーム入力がある画面でバリデーション定義が記載されている
- [✅/❌] 画面固有の条件分岐が全画面で定義されている
- [✅/❌] レスポンシブ対応のモバイルスクリーンショットが含まれている
- [✅/❌] 画面間のデータフローが定義されている
- [✅/❌] コンポーネントがデザインシステムのコンポーネントカタログと整合している
- [✅/❌] Epic 仕様書の全 AC が AC カバレッジでカバーされている
- [✅/❌] 「AI が迷うポイント」の全項目にプロジェクトの方針が記入されている
- [✅/❌] ADR の決定事項と矛盾していない
```

未完了項目がある場合、対応方法を提案する。

## Step 4: セルフレビュー

セルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）。

## レビュー対象

- 画面仕様書: docs/design/screen-design-[epic-slug].md
- Epic 仕様書: [Epic 仕様書パス]

## 追加検証項目

- 全 AC が画面・インタラクションにマッピングされているか
- テンプレートの全セクションが充填されているか
- 「AI が迷うポイント」全 8 項目に方針が記入されているか
- デザインシステムとの整合性
- モックコードとの整合性（画面名・コンポーネント構成・状態）

問題が見つかった場合は修正する。

---

## Step 5: ブリーフィング

`aidd-framework/references/briefing-format.md` に定義されたブリーフィングを実行する。

**重点項目:**

- 画面数と AC カバレッジ結果
- 「AI が実装できるレベル」5 問の確認
- 推定で記入した箇所のリスト
- セルフチェック結果

### 「AI が実装できるレベル」確認

画面仕様書を読んだ AI が以下に答えられるか確認する:

1. この画面にどのコンポーネントがあるか?
2. 各コンポーネントはどの API を呼ぶか?
3. ユーザーがこの操作をしたとき、何が起きるか?
4. データが 0 件 / ローディング中 / エラー時に何が表示されるか?
5. この画面から他のどの画面に遷移できるか?

```
**出力ファイル:** docs/design/screen-design-[epic-slug].md
**画面数:** N 画面
**セルフチェック:** N/13 項目通過

### 次のステップ
- [ ] ブリーフィングを実行（aidd-framework/references/briefing-format.md 参照）

```

## 重要なルール

- **日本語で対話する** — フレームワーク全体の対話言語が日本語であるため
- **FRAMEWORK.md の意思決定基準に従う**
- **各ステップでユーザーの確認を得てから次に進む** — 自動的に全ステップを実行しない。ユーザーの意図と成果物のずれを早期に検出するため
- **モックコードが主要なデータソース** — モックから読み取れる情報は自動充填し、推定部分は明示する。仕様書の信頼性を担保するため
- **ユーザーの判断が優先** — AI の推定よりユーザーの修正指示を優先する。業務ドメイン知識はユーザー側にあるため
- **セクションをスキップしない** — 情報が不足していても空欄のままフラグを付けて進む。欠落を可視化して人間が補完できるようにするため
- **G3 マイルストーンの画面設計品質基準を常に意識する** — 「AI が実装できるレベル」の 5 問に答えられる仕様書を目指す
- **出力ファイルはテンプレートベース** — `aidd-framework/templates/design/screen-design.md` の構造を維持する。テンプレートからの逸脱はレビューコストを増加させるため
- **モック解析はインラインで実行する** — ページファイルの静的解析はスキル内で完結する
