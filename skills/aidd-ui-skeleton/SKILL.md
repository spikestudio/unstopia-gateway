# /aidd-ui-skeleton — UI Skeleton 構築ガイド（G0 マイルストーン）

UI 基盤定義で定義したデザインシステムを、最小限の画面実装で動作検証する。モック環境（`/aidd-setup mocks` で構築済み）上で実施する。G0 マイルストーンの UI Skeleton カテゴリ（画面系プロジェクトのみ）に対応する。

```
ゴール確認 → 画面選定 → モック構築 → デザインシステム検証 → 共有確認
```

> **このスキルの目的は「UI 基盤（テーマ・コンポーネント・レスポンシブ）が実際に動作することを最小コストで証明する」こと。**

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| `/aidd-setup mocks` が実行済み | モック環境の存在確認 | `/aidd-setup mocks` を先に実行 |
| UI 基盤定義が完了 | テーマ・デザインシステム設定の存在確認 | UI 基盤のセットアップを先に完了 |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| （なし） | — | — | 対話的に画面を選定する |

### capability 呼び出し

該当なし

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| UI Skeleton 画面 | コード | テーマプリセット適用済みの最小画面モック |
| デザインシステム検証結果 | レポート | コンポーネント・レスポンシブ・ダークモードの動作検証結果 |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-next` | G0 マイルストーン判定 |

## Step 1: 入力受付

### 前提条件確認

前提が満たされていない場合:

```
⚠️ 前提条件が未達です:
- モック環境が未構築 → `/aidd-setup mocks` を先に実行してください
- UI 基盤定義が未完了 → UI 基盤のセットアップを先に完了してください
```

### ゴール確認

G0 マイルストーンの UI Skeleton チェックリスト（`aidd-framework/checklists/g0-initialization.md` Section 4）を読み込み、ゴールを提示する:

```
## UI Skeleton 構築のゴール

G0 マイルストーン（UI Skeleton 検証）の通過には以下が必要です:

1. テーマプリセットを使用した 1 画面モックが構築されている
2. shadcn/ui コンポーネント（Button, Card, Table 等）が正常に表示される
3. レスポンシブ表示（モバイル / デスクトップ）が正常に動作する
4. ダークモード切り替えが正常に動作する
5. モック共有方法で顧客にモックを提示できることが確認されている

これから画面の選定に入ります。
```

### 画面選定

#### コンテキスト読み込み

- `docs/PROJECT-CHARTER.md` — 主要機能・画面
- デザインシステム定義（G0 マイルストーン — UI 基盤の成果物）
- モック環境の設定（テーマプリセット）

#### 画面の提案

デザインシステムの検証に最適な画面を提案する:

```
## 画面提案

**画面:** [画面名]
**選定理由:** [多くの UI コンポーネントを含み、検証効果が高い理由]

検証対象コンポーネント:
- [ ] Button（プライマリ / セカンダリ / デストラクティブ）
- [ ] Card / コンテナ
- [ ] Table / リスト表示
- [ ] Form / 入力フィールド
- [ ] ナビゲーション
- [ ] レスポンシブレイアウト
- [ ] ダークモード

この画面で進めてよいですか？
```

## Step 2: 実装

### 成果物（生成）

#### モック構築

選定した画面を、テーマプリセットを適用してモック環境上に構築する。

##### ページコンポーネント作成

- テーマプリセットの CSS 変数を使用（色のハードコード禁止）
- shadcn/ui コンポーネントを活用
- レスポンシブデザインを適用

##### 動作確認

```bash
# dev server 起動
task mocks:dev
```

ユーザーに画面を確認してもらい、フィードバックを受ける。修正が必要な場合は対話的に修正する。

#### デザインシステム検証

モック画面を通じて、デザインシステム定義の妥当性を検証する:

```
## デザインシステム検証結果

### 正常に動作した項目
- テーマプリセット: OK
- コンポーネント: OK
- レスポンシブ: OK
- ダークモード: OK

### 問題が見つかった項目
- [項目]: [問題の説明]。修正しますか？

### デザインシステムに不足していた定義
- [項目]: モック構築時に判断が必要だった。定義を追加しますか？
```

### レビュー観点（追加）

セルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）。スキル固有の追加観点:

- テーマプリセットの CSS 変数が正しく使用されているか（色のハードコードがないか）
- shadcn/ui コンポーネントが正しく活用されているか
- レスポンシブデザインが適切に実装されているか
- ダークモード切り替えが正常に動作するか
- 検証対象の全コンポーネントがカバーされているか

### ブリーフィング重点（Step 5）

- 構築した画面の概要と使用コンポーネント
- デザインシステム検証結果（問題・不足の有無）
- 共有方法の確認結果

## Step 3: 完了処理

### 共有確認

モックを顧客（PO）に提示する準備ができているかを確認する:

```bash
# Storybook でも確認
task ui:storybook
```

共有方法（URL・スクリーンショット等）が機能することを確認する。

### 完了案内

```
✅ UI Skeleton 構築完了

- 画面: [画面名]
- テーマプリセット: 動作確認済み
- コンポーネント: N 種類動作確認済み
- レスポンシブ: OK
- ダークモード: OK
- 共有方法: [方法]

次のステップ:
→ ブリーフィングを実行（aidd-framework/references/briefing-format.md 参照）
→ `/aidd-next` でマイルストーン判定に進む（プロジェクト初期化の他の項目も含む）
```

## 重要なルール

- **日本語で対話する** — フレームワーク全体の対話言語が日本語であるため
- **FRAMEWORK.md の意思決定基準に従う** — 優先順序 1〜5 に該当する問題の見送り・後回しを推奨しない
- **PR 作成後は必ずレビューを提案する** — 「PR #XX を作成しました。`/aidd-epic-review` を実行しますか？」と問いかける。品質担保のプロセスを確実に実行するため
- 成果物生成・実装完了時にコミットを自動実行する。ユーザーの指摘反映後・ステップ承認後などの状態遷移ごとにもコミットする。push と PR 作成はユーザーの確認を待つ
- **CSS 変数を使用し、色のハードコードを禁止する** — デザインシステムのテーマプリセットに従う。ハードコードはテーマ変更時に破綻するため
- **最小限の画面に留める** — 1 画面でデザインシステムの動作検証を完了させる。Skeleton の目的は機能完成ではなく基盤検証であるため
- **各ステップでユーザーに確認を求める** — モック画面の見た目はユーザーの承認が必要。UI の妥当性はユーザーが判断するため
- **デザインシステムとの乖離を必ず報告する** — 定義を修正するか実装を修正するかはユーザーが判断する。AI が独断で判断すると設計意図を損なうため
