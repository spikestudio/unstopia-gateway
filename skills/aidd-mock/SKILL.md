# /aidd-mock — 対話的モック構築（廃止済み）

Epic の AC を読み取り、テンプレートカタログから最適なテンプレートを推薦し、対話的にモック画面を構築するオーケストレーター。内部で /aidd-research（仕様調査）に委譲する。

> 前提ガイド: `aidd-framework/guides/mock-development.md`
> テンプレートカタログ: `mocks/templates/catalog.json`
> テーマ: `ui/globals.css`（Catalyst-like 単一テーマ、ADR-006）

## Step 1: Epic AC の読み込み

`$ARGUMENTS` に応じて対象 Epic を決定する:

- **Epic ID 指定あり**（例: `/aidd-mock ES-001`）: 指定された Epic 仕様書を読み込む
- **引数なし**: `docs/requirements/ES-*.md` から G2・G3 設計完了で未モック構築の Epic を推定し、推定結果をユーザーに確認してから進める。該当が複数あれば一覧を提示して選択してもらう
- **推定失敗時**:

```
対象の Epic が特定できませんでした。以下を確認してください:
- `/aidd-mock ES-NNN` — Epic ID を指定して実行
- Epic が未作成の場合は `/aidd-new-epic` を先に実行してください
```

対象 Epic が特定できたら AC を読み込む:

1. **Epic 仕様書の特定** — `docs/requirements/ES-*.md` から対象 Epic を読み込む
2. **AC の抽出** — Epic 仕様書の AC セクションを読み取る
3. **画面の洗い出し** — 各 AC がどの画面で実現されるかをマッピングし、画面一覧を提示する

提示フォーマット:

```
## 画面一覧（Epic: [Epic名]）

| # | 画面名 | URL パス | 対応 AC | 画面の性質 |
|---|--------|---------|--------|-----------|
| 1 | ダッシュボード | /dashboard | AC-1, AC-2 | データ概要表示 |
| 2 | プロジェクト一覧 | /projects | AC-3 | データ一覧（カード） |
| ...

この画面一覧で進めますか？追加・修正があればお知らせください。
```

ユーザーの確認を得てから Step 2 に進む。

## Step 2: テンプレート推薦・カスタマイズ・カバレッジ検証

### 2-1. テンプレート推薦

以下の手順でテンプレート推薦をインラインで実行する:

1. `mocks/templates/catalog.json` を読み込む（存在しない場合はテンプレートディレクトリを直接スキャン）
2. AC から画面要件を抽出する（画面タイプ・必要コンポーネント・インタラクション要件）
3. catalog.json のテンプレートと画面要件をマッチングする
4. 各画面に対して推薦テンプレートを選定する（第 1 候補 + 代替案）
5. 推薦理由を AC と紐付けて説明し、similarTo がある場合は diffSummary を使って代替案を提示する

推薦結果をユーザーに提示し、確認・修正を得る。

### 2-2. 対話的カスタマイズ（自身が実行）

ユーザーが承認したテンプレート構成に基づいて、モック環境にファイルを構築する:

1. **テンプレートのコピー** — `{mock_dir}/templates/` からモック画面にコピー
2. **モックデータの差し替え** — ドメインモデルに合わせたサンプルデータに置換
3. **ナビゲーション・カラム・フィルタ・テキスト** — プロジェクトに合わせて変更
4. **デザインシステムの適用** — `docs/design/design-system.md` の方針を遵守

コピー先のパス規則:

- レイアウト → `{mock_dir}/app/` 直下（共通レイアウトとして使用）
- ページ → `{mock_dir}/app/[epic-slug]/[screen-name]/page.tsx`
- コンポーネント → `{mock_dir}/components/` 配下（複数画面で共用する場合）

### 2-3. Dev Server 起動と対話的修正ループ（自身が実行）

```bash
cd {mock_dir} && pnpm dev
```

ユーザーのフィードバックに基づいて修正を繰り返す:

1. ユーザーが修正箇所を指示（テキスト or スクリーンショット）
2. AI がコードを修正
3. ブラウザで確認
4. 満足するまで繰り返す

### 2-4. カバレッジ検証（Agent に委譲）

```
Agent(subagent_type="aidd-builder", prompt="""
  モックのカバレッジを検証してください。

  ## 検証対象
  - モックディレクトリ: {mock_dir}/app/[epic-slug]/
  - 画面一覧: [Step 1 で確定した画面一覧]

  ## 検証項目
  1. 全画面の 4 状態（data/empty/loading/error）が実装されているか
  2. 状態切替 UI（モック専用）が配置されているか
  3. レスポンシブ対応（375px / 768px / 1280px）
  4. ダークモード対応（CSS 変数ベースの色切り替え）

  ## 出力
  - 状態カバレッジチェックリスト（画面ごと）
  - 不足項目のリスト
""")
```

不足がある場合は自身が修正を実行し、ユーザーに確認を得る。

## 重要なルール

- **日本語で対話する** — フレームワーク全体の対話言語が日本語であるため
- **FRAMEWORK.md の意思決定基準に従う**
- **各 Step でユーザーの確認を得てから次に進む** — 自動的に全 Step を実行しない。ユーザーの意図と成果物のずれを早期に検出するため
- **catalog.json を必ず参照する** — テンプレート推薦時は catalog.json のメタデータ（useWhen, avoidWhen, similarTo, combinesWith）を根拠として提示する。根拠のない推薦はレビュー不能になるため
- **推薦理由を明示する** — なぜそのテンプレートを選んだかを AC と紐付けて説明する。人間が判断を検証できるようにするため
- **類似テンプレートの比較を提示する** — similarTo がある場合は diffSummary を使って判断材料を提供する。選択肢が複数ある場合に人間が最終判断できるようにするため
- **デザインシステムに準拠する** — CSS 変数を使用し、色のハードコードをしない。ステータスバッジは `bg-{color}-500/10 text-{color}-700 dark:text-{color}-400` パターン
- **モック品質基準を守る** — `aidd-framework/guides/mock-development.md` の品質基準チェックリストに準拠
- **テンプレートの冒頭コメントを活用する** — 各テンプレートの「カスタマイズポイント」「必要な shadcn/ui コンポーネント」を参照してカスタマイズする
- **テンプレート推薦はインラインで実行する** — catalog.json のマッチング処理はスキル内で完結する
