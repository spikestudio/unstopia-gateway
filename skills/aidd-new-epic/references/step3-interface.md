# Step 3: インターフェース設計 -- 詳細手順

ドメインモデルから DB スキーマ骨格と API spec 骨格を生成する。

## 入力

- Step 2 承認済みドメインモデル（`docs/domain/[epic-slug]-domain-model.md`）
- Epic 仕様書（AC 一覧）
- `references/design-templates/db-schema.md`（DB テンプレート）
- `references/design-templates/api-spec.md`（API テンプレート）

## 前提条件

| 前提 | 確認方法 | 未確定時の案内 |
|------|---------|--------------|
| 横断設計事項が Phase/ADR で確定済み | Phase 定義書・ADR を確認 | 「ADR を先に作成してください」と案内。未確定項目を具体的に列挙 |

横断設計事項:

- 認証・認可方針（認証方式、ロール定義）
- エラーコード体系（命名規則、HTTP ステータス対応表）
- セキュリティ方針（CORS・CSRF・レート制限の要否）
- DB マイグレーション戦略（ツール選定）

## 骨格レベルの範囲定義

**DB スキーマ骨格に含めるもの:**

- テーブル一覧とカラム名・型・主要制約
- テーブル間リレーション（FK、Mermaid ER 図）

**実装詳細（How）— Task に委ねる:**

- インデックス設計、詳細バリデーション制約、パーティション設計、マイグレーションファイル

**API spec 骨格に含めるもの:**

- エンドポイント一覧（メソッド・パス・対応 AC）
- リクエスト/レスポンス型定義
- 認証要否・認可ロール（エンドポイント別）— Phase/ADR の認証方針を各エンドポイントに適用
- エラーコード体系（命名規則・HTTP ステータス対応表）— Phase/ADR のエラーコード体系に基づいて Epic 固有コードを定義
- セキュリティ要件概要（CORS・CSRF・レート制限の要否）— Phase/ADR のセキュリティ方針を Epic に適用

**実装詳細（How）— Task に委ねる:**

- 認証ミドルウェアの実装詳細
- レート制限の具体的な閾値設定
- 詳細なバリデーションルール（正規表現等）
- エラーレスポンスのボディ構造

> 再分類の基準: 「複数 Task に影響する決定」「後から変更すると手戻りが大きい決定」= What（Epic で確定）、「1 Task 内で完結する決定」= How（Task に委ねる）

## スキップ条件

- データ永続化を伴わない Epic → DB スキーマ生成をスキップ
- API を含まない Epic → API spec 生成をスキップ
- インフラ変更を伴わない Epic → インフラ設計生成をスキップ
- バッチ/非同期ジョブを含まない Epic → バッチ/ジョブ設計生成をスキップ
- 各スキップの判断理由はブリーフィングで説明する

## 手順

### 1. DB スキーマ骨格生成（該当する場合）

ドメインモデルの集約・エンティティからテーブルを導出する:

- `references/design-templates/db-schema.md` テンプレートに従い骨格を定義
- 集約ルートをテーブル一覧の起点とし、値オブジェクトはカラムとして展開する
- テーブル間リレーション（FK）を定義し、Mermaid ER 図を生成する

### 2. API spec 骨格生成（該当する場合）

Epic AC から CRUD 操作を抽出しエンドポイントを定義する:

- `references/design-templates/api-spec.md` テンプレートに従い骨格を定義
- 各エンドポイントと AC の対応を明記する（対応 AC 列）
- リクエスト/レスポンス型をドメインモデルのエンティティ属性から導出する

### 3. インフラ設計骨格生成（該当する場合）

Epic の AC から以下のいずれかを含む場合に生成する:
新規サービス/コンテナの追加、クラウドリソースの新設・変更、外部サービス連携の追加、デプロイ構成の変更。

- `references/design-templates/infra.md` テンプレートに従い骨格を定義する
- クラウドリソース一覧・デプロイアーキテクチャ・環境差異を骨格レベルで定義する
- 詳細なパラメータ（インスタンスサイズ・タイムアウト値等）は Task に委ねる

**骨格に含めるもの:**

- 新設・変更するリソース種別と用途
- Mermaid コンポーネント図（主要フローが読み取れるレベル）
- 環境ごとの構成差異（dev/stg/prod）
- 外部依存・連携サービス一覧

**Task に委ねるもの:**

- 具体的なインスタンスサイズ・スペック選定
- 詳細なネットワーク設定（サブネット・セキュリティグループルール）
- コスト見積もり・最適化

### 4. バッチ/ジョブ設計骨格生成（該当する場合）

Epic の AC にバッチ処理・非同期ジョブ・スケジュール実行が含まれる場合に生成する。

- `references/design-templates/batch-job.md` テンプレートに従い骨格を定義する
- ジョブ名・トリガー種別・入出力型・べき等性方針・失敗時挙動を骨格レベルで定義する
- 具体的なcron式・リトライ上限値・タイムアウト値は Task に委ねる

### 5. 「AI が実装できるレベル」5 問検証 + 論点別議論

#### 5-1. 5 問チェック

全てに回答できるまで完了としない:

1. エンティティの属性は何か → ドメインモデル参照
2. 適用されるビジネスルールは何か → 不変条件参照
3. データの保存方法は何か → DB スキーマ参照
4. インターフェースの入出力は何か → API spec 参照
5. 失敗時の挙動は何か → エラーケース参照

**5 問全てに回答できた場合:** 論点別議論をスキップして手順 6 に進む。

#### 5-2. 論点別議論（5 問未回答論点のみ）

5 問の回答が曖昧・未確定な論点について、複数案とトレードオフを提示してユーザーと議論する。回答が確定している論点は議論しない。

提示フォーマット:

```
**未回答論点: [問 N] [内容]**

現時点では以下の選択肢があります:

| 案 | 内容 | メリット | デメリット |
|----|------|---------|----------|
| A（推奨）| [内容] | [メリット] | [デメリット] |
| B | [内容] | [メリット] | [デメリット] |

推奨は案 A です。理由: [1〜2 文]
どの案で進めますか？
```

議論が完了して 5 問全てに回答できたら手順 6 に進む。

### 6. ファイル書き出し

- DB スキーマ骨格を `docs/design/[epic-slug]-db-schema.md` に書き出す
- API spec 骨格を `docs/design/[epic-slug]-api-spec.md` に書き出す
- インフラ設計骨格を `docs/design/[epic-slug]-infra.md` に書き出す（該当する場合）
- バッチ/ジョブ設計骨格を `docs/design/[epic-slug]-batch-job.md` に書き出す（該当する場合）
- Epic 仕様書の設計成果物テーブルを更新する

## 成果物

- DB スキーマ骨格（`docs/design/[epic-slug]-db-schema.md`）— データ永続化を伴う場合
- API spec 骨格（`docs/design/[epic-slug]-api-spec.md`）— API を含む場合
- インフラ設計骨格（`docs/design/[epic-slug]-infra.md`）— インフラ変更を伴う場合
- バッチ/ジョブ設計骨格（`docs/design/[epic-slug]-batch-job.md`）— バッチ/非同期ジョブを含む場合
- Epic 仕様書の設計成果物テーブル更新

## 完了時

`references/briefing-spec.md` に従ってブリーフィングを実行する。

**ブリーフィングで必ず提示する情報（成果物を見なくても判断できるレベル）:**

<!-- 括弧内は briefing-spec.md 5 要素との対応 -->

- 骨格の範囲判断（何を含め、何を Task に委ねたか）（要素2: 判断根拠）
- スキップした場合、その判断理由（要素2: 判断根拠）
- テーブル一覧とリレーション図（要素1: 成果物全内容）
- エンドポイント一覧（AC 対応付き）（要素1: 成果物全内容）
- インフラ変更概要（新設・変更リソース一覧）（要素1: 成果物全内容）
- 「AI が実装できるレベル」5 問の回答（要素1: 成果物全内容）
- 設計トレードオフ（複数の選択肢があった場合の選定理由）（要素3: 暗黙の前提）
- AI 自信度が低い箇所（ドメイン知識不足で設計の妥当性を保証できない部分）（要素4: 未決定事項）
- 人間に判断を求める事項（テーブル構造・エンドポイント粒度・型定義・インフラ構成）（要素5: 意思決定事項）

**ADR 作成要否の確認（承認前に必ず実施）:**

以下のいずれかに該当する設計判断があった場合、承認前に ADR を作成すること:

- アーキテクチャ選択（レイヤー構造・通信パターン・データフローの決定）
- ライブラリ・ツール採用（複数の選択肢を検討してトレードオフで選んだ場合）
- 既存設計原則からの逸脱
- 他 Epic・将来 Phase に影響する重要なトレードオフ

ADR が必要な場合は `aidd-framework/templates/adr.md` を参照して作成し、Epic 仕様書ヘッダーの ADR 参照フィールドを更新してから承認を求める。

承認を得て Epic 定義完了（G3 相当）とする。
