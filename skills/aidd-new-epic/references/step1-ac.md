# Step 1: AC 導出 -- 詳細手順

Phase 定義書のストーリーから AC を導出し、Epic 仕様書ドラフトを生成する。

## 入力

- Phase 定義書（対象 Epic のストーリーと BC 情報）
- `references/epic-spec-template.md`

## 手順

### 1. ストーリーの取得と確認

Phase 定義書から対象 Epic に割り当てられたストーリーを取得する。以下を確認する:

- 各ストーリーが As a / I want to / so that 形式で記述されているか
- BC キャンバス・ドメイン分析成果物の存在（Step 2 の入力になる）
- 既存 Epic 仕様書（`docs/requirements/ES-*.md`）との重複がないか

### 2. AC 導出

`aidd-framework/process/story-to-epic.md` に従い、各ストーリーから AC を導出する:

- AC はユーザー視点の E2E シナリオとして記述する（「ユーザー操作 --> システム応答」の対）
- API レスポンスや DB 状態の検証は AC に含めない（Task レベルのテスト観点）
- AC-ID 形式: `AC-ENNN-NN`（E = Epic 番号, NN = 連番）
- ストーリートレース: 各 AC の末尾に `<- Sn` で導出元ストーリーを明示
- AI 補完 AC には `（AI 補完: [理由]）` を付記し、情報源を明示する
- 正常系だけでなく、異常系・境界値・エッジケースを網羅する

#### 2-1. AC 根拠・代替案の生成

各 AC を導出したら、以下を付記して透明性を確保する:

| 付記項目 | 内容 |
|---------|------|
| **根拠** | なぜこの AC を導出したか（ストーリーの "so that" 句との対応・ビジネスルールの根拠） |
| **代替案** | この AC の代わりに取りうる別の定義（例: 粒度を変える・条件を追加する等） |
| **省略/変更した場合の影響** | この AC を省略またはより弱い条件にした場合に発生するリスク |

ブリーフィング時に、全 AC の根拠と主要な代替案を一覧として提示する（ユーザーが AC の妥当性を評価できる情報を提供する）。

#### 2-2. リサーチ委譲の判断

AC 導出中に以下のいずれかに該当する論点が見つかった場合、`/aidd-research` を呼び出して調査を実施し、結果を AC に反映する:

| 判断基準 | 例 |
|---------|-----|
| 技術選定・外部仕様の確認が必要 | 「この外部 API の仕様は？」「このライブラリの制約は？」 |
| ベストプラクティスの確認が必要 | 「このドメインの業界標準の AC パターンは？」 |
| 類似実装の調査が必要 | 「同種の機能で一般的に採用されている AC 設計は？」 |

調査が不要な場合（ドメインが明確・既存コードで確認済み）はスキップしてよい。スキップした場合はブリーフィングでその理由を明示する。

### 3. AC 品質基準（6 基準）

全 AC が以下の 6 基準を満たすことを検証する:

| 基準 | 説明 |
|------|------|
| テスト可能 | 合否を機械的に判定できる具体的な条件が記述されている |
| 境界明確 | 入力値・条件の境界が曖昧でなく、数値・範囲で定義されている |
| 独立 | 他の AC に依存せず、単独で検証できる |
| 完全 | ストーリーの意図を漏れなくカバーしている |
| 観測可能 | 外部から観測できる振る舞い（UI 表示・レスポンス等）で記述されている |
| E2E 検証可能 | 自動 E2E テストまたは手動デモで検証できるシナリオになっている |

### 4. Epic 仕様書ドラフト生成

`references/epic-spec-template.md` に従い Epic 仕様書を生成する。全セクションを埋める:

- 対応ストーリー（Phase 定義書から転記）
- ストーリーと受入基準（導出した AC を配置）
- バリデーションルール（入力フィールドごとのルールとエラー時の振る舞い）
- ステータス遷移（該当する場合は mermaid 図を作成）
- エラーケース（条件・期待する振る舞いを網羅）
- 非機能要件（パフォーマンス・セキュリティ等）
- デリバリーする価値（対象ユーザー・価値・デモシナリオ）
- E2E 検証計画（検証シナリオ・検証環境・前提条件）
- 他 Epic への依存・影響
- 未決定事項
- 完全性チェック（全項目を検証）

### 4-1. ステップ間整合性チェック

Epic 仕様書ドラフト生成後、Step 2-3 で作成される設計成果物との整合性を事前に確認する。

以下のチェックリストを実施し、矛盾・漏れを検出したらユーザーに提示して合意してから次のステップに進む:

| チェック対象 | 確認内容 |
|------------|---------|
| AC ↔ ドメインモデル（Step 2 入力） | 各 AC に対応するエンティティ・集約・ドメインイベントが導出可能か。AC で言及している業務概念がドメイン設計に反映できるか |
| AC ↔ DB スキーマ（Step 3 入力） | 各 AC が必要とするデータが DB に永続化できるか。データ取得・更新の操作が AC から導出可能か |
| AC ↔ API spec（Step 3 入力） | 各 AC に対応するエンドポイントが定義できるか。入出力の型が AC から導出可能か |
| AC 間の依存・矛盾 | 複数の AC が互いに矛盾する要件を定義していないか |

**矛盾・漏れを検出した場合の対応:**

1. 具体的な矛盾箇所（AC-ID とその内容）をユーザーに提示する
2. 推奨修正案を提示する（AC を修正するか、設計制約として記録するか）
3. ユーザーと合意してから当該 AC を修正する

**問題なしの場合:** 「整合性チェック: 問題なし」とブリーフィングに明示する。

### 5. セルフレビュー

セルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）。`ac, coverage` 観点で以下の追加観点を適用する:

| 観点 | チェック内容 |
|------|------------|
| AC<-ストーリーのトレーサビリティ | 全ストーリーに対して AC が導出されているか |
| AC の品質 | 6 基準を満たしているか |
| AC に複合条件がないか | 「かつ/または」を含む AC は分割する |
| 1 ストーリーの AC 数 | 8 を超えていないか |
| Phase 定義書との乖離 | ストーリー概要から逸脱していないか |

### 6. ファイル書き出し

`docs/requirements/ES-NNN-slug.md` に書き出す。

## 成果物

Epic 仕様書ドラフト（`docs/requirements/ES-NNN-slug.md`）

## 完了時

`references/briefing-spec.md` に従ってブリーフィングを実行する。

**ブリーフィングで必ず提示する情報（成果物を見なくても判断できるレベル）:**

<!-- 括弧内は briefing-spec.md 5 要素との対応 -->

- 全ストーリーの背景と意義（なぜこのストーリーが必要か）（要素1: 成果物全内容）
- **全 AC の一覧（AC-ID + 全文 + ← Sn トレース。人間入力 / AI 補完を区別して全件提示。省略禁止）**（要素1: 成果物全内容）
- **各 AC の根拠・代替案一覧**（「なぜこの AC か」「別の定義にした場合の影響」）（要素2: 判断根拠）
- **リサーチ委譲の結果**（実施した場合は調査内容と反映箇所、スキップした場合はその理由）（要素2: 判断根拠）
- AI 補完 AC 一覧（補完理由・情報源付記）（要素1, 2: 成果物全内容 + 判断根拠）
- エッジケースの判断（含めた/含めなかった理由）（要素3: 暗黙の前提）
- AC 品質基準の自己評価（6 基準ごとの達成状況）（要素2: 判断根拠）
- AI 自信度が低い AC（ドメイン知識不足で妥当性を保証できないもの）（要素4: 未決定事項）
- 人間に判断を求める事項（AC の要否・粒度・優先度）（要素5: 意思決定事項）

承認を得てから Step 2 に進む。
