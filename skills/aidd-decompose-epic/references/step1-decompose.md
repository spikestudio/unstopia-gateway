# Step 1: Task 分解 -- 詳細手順

Epic 仕様書の AC と設計成果物を入力として、AC を実装単位（コミット単位）の Task に分解する。E2E 検証 Task を末尾に必ず配置する。

## 入力

- Epic 仕様書（全ストーリー、全 AC）
- 設計成果物（ドメインモデル、スキーマ、API spec 等）
- 関連 ADR
- 規約ドキュメント群（`docs/conventions/`）
- 既存の Task 定義（あれば）

## 前提条件チェック

Step 0 で読み込んだ入力に対して、以下の前提条件を検証する。未充足の場合はスキルを停止し、案内メッセージを表示する。

### 1. Epic 仕様書の承認確認

- Epic Issue が `status:in-progress` ラベル付きでオープンであること（`gh issue list --label "status:in-progress" --label epic`）
- Epic Issue のコメントに G2 通過記録が存在すること（`gh issue view <Epic番号> --comments` で `gate:g2` または G2 通過コメントを確認）
- 設計成果物（ドメインモデル、スキーマ、API spec）が存在していること（該当なしの場合は Epic 仕様書に明記されていること）

**未充足時:**

```
Epic Issue が status:in-progress でない、または G2 通過コメントが存在しません。
→ `/aidd-new-epic` を先に実行し、Epic 定義を完了してください。
```

### 2. 横断設計事項の確定確認

以下の横断設計事項が Phase 定義書または ADR で確定されていることを確認する:

| 横断設計事項 | 確認先 | 説明 |
|------------|--------|------|
| 認証・認可方針 | Phase 定義書 / ADR | 認証方式、ロール定義 |
| エラーコード体系 | Phase 定義書 / ADR | 命名規則、HTTP ステータス対応表 |
| セキュリティ方針 | Phase 定義書 / ADR | CORS・CSRF・レート制限の要否 |
| DB マイグレーション戦略 | ADR | ツール選定 |

**未確定の項目がある場合:**

```
以下の横断設計事項が Phase/ADR で未確定です:
- [未確定の項目を列挙]

ADR を先に作成してください。横断設計事項は Epic 横断で影響するため、
Task 分解の前に確定が必要です。
```

**ドキュメント群プロジェクト（ドメインモデル不要）の場合:**

横断設計事項の該当がない場合はこのチェックをスキップする。Epic 仕様書の所属 BC が「該当なし」の場合にスキップ条件と判断する。

## 引数なし時の自動推定

引数が指定されなかった場合、以下のロジックで分解対象の Epic を自動推定する:

1. `gh issue list --label "status:in-progress" --label epic` でオープンな Epic Issue を確認する
2. **G2 通過コメント済みかつ Task 未定義の Epic がある場合** → 「ES-NNN を Task に分解しますか？」と提案し、ユーザーの確認を得る
3. **該当する Epic がない場合** → 以下のエラーメッセージを表示する:

```
Task 分解の対象となる Epic が見つかりませんでした。以下を確認してください:
- `/aidd-decompose-epic ES-NNN` — 分解対象の Epic を直接指定
- Epic が未作成の場合は `/aidd-new-epic` を先に実行してください
- 設計が未完了の場合は `/aidd-new-epic` を先に実行してください
```

**推定の判定基準:**

| 条件 | 判定方法 |
|------|---------|
| Epic Issue が承認済み | Epic Issue が `status:in-progress` ラベル付きでオープンかつ G2 通過コメントが存在する |
| 設計成果物が完了 | 設計成果物ファイル（ドメインモデル、スキーマ等）の存在確認 |
| Task 未定義 | `docs/tasks/TASK-*.md` に該当 Epic の Task が存在しない |

## 分割ルール

### 分割の優先順

以下の優先順で分割方法を選択する。上位の基準で十分に分割できる場合、下位の基準は適用しない:

1. **ドメイン概念で分割**（最優先）— エンティティ A の CRUD → エンティティ B の CRUD → 統合
2. **機能で分割** — 正常系 → 異常系/エッジケース
3. **レイヤーで分割** — ドメインモデル → リポジトリ → ハンドラ — **最終手段**

### 1 Task = 1 責務の原則

全ての Task は **「1 Task = 1 責務」** の原則に従う。

- **責務の定義**: 変更理由が 1 つであること。1 つの Task が複数の理由で変更される場合、分割が必要
- **判断基準**: 「この Task を修正する理由は何か？」と問い、答えが複数ある場合は分割する
- **単一責務チェック**: 1 Task 内に独立した判断・承認が 2 つ以上含まれる場合は分割する
- **例**:
  - OK: 「ユーザーエンティティの CRUD 実装」（変更理由: ユーザーモデルの仕様変更）
  - NG: 「ユーザーとロールの CRUD 実装」（変更理由: ユーザー仕様変更 or ロール仕様変更）
  - NG: 「スキル A〜D の内容精査」（各スキルの判断が独立 → 1 スキル = 1 Task に分割）

#### アンチパターン: 効率のためにまとめる

「類似した作業だからまとめた方が効率的」という判断は禁止する。追跡可能性・個別承認・差し戻し容易性を優先する。AI が効率を理由にグループ化した場合、ユーザーから分割指示が来る前にセルフチェックで検出すること。

### 分割判断基準

以下の閾値のいずれかに該当する場合、Task の分割を検討する:

| 指標 | 閾値 | アクション |
|------|------|-----------|
| ドメイン概念数 | 4 以上 | ドメイン概念ごとに Task を分割 |
| 対象ファイル数 | 20 以上 | 関連ファイル群ごとに Task を分割 |
| 条件分岐数 | 15 以上 | 正常系/異常系で Task を分割 |
| 参照コード行数 | 300 行以上 | 機能単位で Task を分割 |

### 複雑度レベル

各 Task に複雑度レベルを付与する。目安は以下の通り:

| レベル | 変更行数目安 | 説明 |
|--------|------------|------|
| S | ~200 行 | 単純な CRUD、設定変更、テスト追加 |
| M | ~400 行 | 複数ファイルにまたがるロジック実装 |
| L | ~800 行 | 複雑なドメインロジック、統合処理 |

**800 行超は分割必須。** L を超える見積もりになった場合、1 Task = 1 責務の原則に立ち返り、さらに分割する。

## Task の構成要素

各 Task には以下の情報を含める:

| 要素 | 説明 | 必須 |
|------|------|------|
| Task 名と概要 | 何を実装するかの簡潔な説明 | Yes |
| 対応する Epic AC | AC 番号で明記（例: AC-E015-01, AC-E015-02） | Yes |
| テスト方針 | どのテストレイヤーを含むか（UT / 統合 / E2E） | Yes |
| 複雑度レベル | S / M / L | Yes |
| 依存関係 | 先行 Task（なければ「--」） | Yes |
| スコープ | 主要な対象ファイル・ディレクトリ | Yes |
| 完了条件（機能面） | 対応する Epic AC を満たすこと | Yes |
| 完了条件（品質面） | テスト・レビュー・CI パスの基準 | Yes |
| AI への指示コンテキスト | 参照すべきドキュメント・既存コード | Yes |

## E2E 検証 Task の自動配置

Task 一覧の末尾に必ず E2E 検証 Task を配置する。E2E 検証 Task の仕様:

```
TASK-NNN-e2e-verification -- [Epic名] E2E 検証
  役割: 全 E2E テストの一括パス確認 + AC カバレッジ検証
  対応 AC: Epic の全 AC
  完了条件:
    - 全 E2E テストがパスしている
    - Epic 仕様書の全 AC が少なくとも 1 つの E2E テストでカバーされている
    - 未カバーの AC がある場合、E2E テストを追加して全件パスを確認
    - デモシナリオが実行可能な状態である
  依存: 全実装 Task
```

**E2E 検証 Task の省略は、ユーザーが明示的に不要と判断した場合のみ許容する。**

## Task テーブル形式

分解結果は以下のテーブル形式で提示する:

```
## Task 分解案（全 N 件）

| # | Task 名 | 対応 AC | テスト方針 | 複雑度 | 依存 |
|---|---------|--------|----------|--------|------|
| 1 | ... | AC-1, AC-2 | UT + E2E | M | -- |
| 2 | ... | AC-3 | UT + 統合 | S | #1 |
| ... | | | | | |
| N | E2E 検証 | 全 AC | E2E 全件パス確認 | S | 全 Task |

AC カバレッジ: [カバー済み AC / 全 AC]（100% であること）
```

**AC カバレッジは 100% を必須とする。** 全 AC が少なくとも 1 つの Task にマッピングされていない場合、Task を追加する。

## セルフレビュー

セルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）。`ac, consistency` 観点で以下の追加観点を適用する:

| 観点 | チェック内容 |
|------|------------|
| AC→Task の全カバレッジ | Epic の全 AC が少なくとも 1 つの Task にマッピングされているか |
| 依存関係 | Task 間の依存関係に循環がないか |
| 1 Task = 1 責務 | 各 Task の変更理由が 1 つに限定されているか。独立した判断・承認が 2 つ以上含まれていないか |
| グループ化禁止 | 「類似作業だからまとめた」Task がないか。対象が N 個あれば N Task に分割されているか |
| 実装粒度 | 分割判断基準（ドメイン概念 4+、ファイル 20+、分岐 15+、参照 300 行+）に該当しないか |
| 複雑度レベルの妥当性 | 見積もりサイズが目安（S: ~200 行, M: ~400 行, L: ~800 行）に収まるか。800 行超は分割必須 |
| E2E 検証 Task の存在 | E2E 検証 Task が末尾に配置されているか |

## 成果物

Task 分解ドラフト（Task テーブル + 各 Task の詳細定義）

## 完了時

`references/briefing-spec.md` に従ってブリーフィングを実行する。

**ブリーフィングで必ず提示する情報（成果物を見なくても判断できるレベル）:**

<!-- 括弧内は briefing-spec.md 5 要素との対応 -->

- Epic の全体像と分解の方針（なぜこの分割方法を選んだか）（要素1: 成果物全内容）
- Task 一覧テーブル（AC カバレッジ付き）（要素1: 成果物全内容）
- 分割判断の根拠（代替の分割案との比較）（要素2: 判断根拠）
- 依存関係グラフ（並行実行可能な Task と直列実行が必要な Task）（要素2: 判断根拠）
- 暗黙の前提（技術的制約、環境依存等）（要素3: 暗黙の前提）
- AI 自信度が低い Task（粒度や複雑度の見積もりに不安があるもの）（要素4: 未決定事項）
- 人間に判断を求める事項（分割粒度・優先順序・並行可否）（要素5: 意思決定事項）

承認を得てから Step 2 に進む。
