# ドキュメントチェック -- 詳細手順

Epic 完了時に、関連ドキュメント群の整合性・正確性・完全性を検証する。問題が見つかった場合は MUST FIX として報告し、修正完了まで次のステップに進まない。

> **位置づけ:** セルフレビューの `consistency` 観点をドキュメント横断に特化・拡張したものである。コードレビューでは検出しにくいドキュメント間の矛盾・記載漏れを早期に発見する。

## 入力

| 入力 | 取得元 | 用途 |
|------|--------|------|
| Epic 仕様書 | `docs/requirements/ES-NNN-*.md` | 整合性の基点（AC・設計成果物テーブル・ストーリー） |
| Task 定義群 | `docs/tasks/TASK-NNN-*.md` | AC 転記・スコープの検証 |
| 設計成果物 | ドメインモデル・DB スキーマ・API spec 等 | 設計間の矛盾検出 |
| 規約ドキュメント | `docs/conventions/` | 規約との突合 |
| Phase 定義書 | `docs/requirements/PD-NNN-*.md` | 上位目的・スコープとの整合 |

## チェック手順

### 1. ドキュメント間の整合性

Epic 仕様書を基点として、関連ドキュメント間の整合性を検証する。

| チェック対象 | 検証内容 | 検証方法 |
|------------|---------|---------|
| Epic 仕様書 ↔ Task 定義 | AC の転記が正確か。AC-ID・内容が一致しているか | Epic の AC 一覧と各 Task の AC マッピングを突合し、漏れ・不一致を検出する |
| Epic 仕様書 ↔ 設計成果物 | 設計成果物テーブルのステータスが正しいか。成果物が実在するか | テーブルのパスに対して `ls` でファイル存在を確認し、ステータスと実態を照合する |
| Task 定義 ↔ 実装 | スコープ（対象ファイル）と実際の変更が一致するか | Task の対象範囲と `git diff` の変更ファイルを照合し、過不足を検出する |
| Epic 仕様書 ↔ Phase 定義書 | Epic の目的・スコープが Phase の目標と整合しているか | Phase 定義書のスコープ・成功基準と Epic の範囲を突合する |
| Task 定義間 | Task 間で重複・矛盾するスコープがないか | 全 Task の対象ファイルと AC を横断比較する |

### 2. 論理的矛盾の検出

ドキュメント内およびドキュメント間で、論理的に矛盾する記述を検出する。

| チェック対象 | 検証内容 | 検証方法 |
|------------|---------|---------|
| AC 間の矛盾 | AC 同士が矛盾する要件を定義していないか | 全 AC を列挙し、ペアワイズで矛盾がないことを確認する（例: AC-A が「必須」としたフィールドを AC-B が「任意」としていないか） |
| 設計成果物間の矛盾 | DB スキーマ ↔ API spec ↔ ドメインモデル間で型・フィールド名・制約が一致しているか | エンティティ・フィールドの名前・型・制約を設計成果物間で突合する |
| 規約と設計の矛盾 | 設計成果物が規約（命名規則・レイヤールール等）に違反していないか | `docs/conventions/` の各規約と設計成果物を照合する |
| 前提条件の矛盾 | Task の前提条件が、依存先 Task の出力と整合しているか | Task の依存関係グラフに沿って、前提条件と成果物を突合する |

### 3. 記載漏れの検出

必須項目が欠落していないかを検証する。

| チェック対象 | 検証内容 | 検証方法 |
|------------|---------|---------|
| Epic 仕様書の完全性 | テンプレートの必須セクションが全て記載されているか | `aidd-framework/templates/` の Epic テンプレートと突合し、欠落セクションを検出する |
| Task 定義の完全性 | テンプレートの必須セクションが全て記載されているか | Task テンプレートと突合し、欠落セクションを検出する |
| AC の検証可能性 | 全 AC に具体的な検証方法が記載されているか | 「適切に」「必要に応じて」等の曖昧語が検証基準に含まれていないことを確認する |
| 設計成果物テーブル | Epic で参照される設計成果物が全て記載されているか | Epic 本文で言及される設計成果物とテーブルのエントリを突合する |
| エラーケース・異常系 | 正常系だけでなくエラーケース・異常系の AC が定義されているか | AC 一覧にエラー・バリデーション・境界条件に関する AC が存在することを確認する |

### 4. ドキュメント間の参照整合

ドキュメント内のリンク・参照が正しい対象を指しているかを検証する。

| チェック対象 | 検証内容 | 検証方法 |
|------------|---------|---------|
| ファイルパス参照 | 参照先のファイルが実在するか | ドキュメント内のパス参照（`docs/...`, `src/...` 等）を抽出し、`ls` でファイル存在を確認する |
| AC-ID 参照 | AC-ID が正しい AC を指しているか | Task 定義内の AC-ID 参照を抽出し、Epic 仕様書の AC 定義と突合する |
| Issue 番号参照 | Issue 番号が正しい Issue を指しているか | ドキュメント内の `#NNN` 参照を抽出し、`gh issue view` で存在と内容を確認する |
| ドキュメント間リンク | 相互参照リンクが双方向で整合しているか | Epic → Task、Task → Epic の参照が双方向で存在することを確認する |
| ADR 参照 | ADR 番号が正しい ADR を指しているか | ADR-NNN 参照を抽出し、`docs/architecture/adr/` 配下のファイルと突合する |

### 5. ドキュメント更新漏れ確認

Epic 実装（`/aidd-impl` または adhoc コミット）で変更されたファイルと、プロジェクト内のドキュメント一覧を照合し、更新が必要なドキュメントが漏れていないかを最終確認する。

#### 手順

1. スキルに同梱の `references/scan-docs.sh` を実行してドキュメント一覧と冒頭を取得する。

   ```bash
   bash "${SKILL_DIR}/references/scan-docs.sh"
   ```

   > `SKILL_DIR` = スキル起動時に "Base directory for this skill:" として提供されるパス。

2. Epic の変更内容（`git diff main...HEAD --name-only`）と照合し、以下の観点で未更新ドキュメントを特定する。

   | 照合観点 | チェック内容 |
   |---------|------------|
   | 新規 API / インターフェース | API spec・ドメインモデル・README の対応箇所が更新されているか |
   | スキーマ変更 | DB スキーマドキュメントが変更に追従しているか |
   | 規約・命名変更 | `docs/conventions/` の関連規約が更新されているか |
   | アーキテクチャ変更 | ADR が作成・更新されているか |
   | セットアップ手順変更 | README / セットアップガイドが更新されているか |

3. 未更新ドキュメントが見つかった場合は、レビュー指摘として Step 2（修正ステップ）で対応する。

   ```text
   [ドキュメント更新漏れ] <ファイルパス>: <更新が必要な理由>  重大度: MUST FIX / SHOULD FIX
   ```

## 修正種別の判定ルール

検出した問題に対して修正種別を判定する。全問題に修正提案を付ける（「修正しない」は選択肢にない）。

| 条件 | 修正種別 |
|------|---------|
| AC の転記ミス・欠落 | ドキュメント修正 |
| 設計成果物間の型・フィールド不一致 | ドキュメント修正 |
| 必須セクションの欠落 | ドキュメント修正 |
| 参照先ファイルが存在しない | ドキュメント修正 |
| AC-ID・Issue 番号の参照ミス | ドキュメント修正 |
| 曖昧語が検証基準に含まれる | ドキュメント修正 |
| 設計成果物テーブルのステータス不一致 | ドキュメント修正 |
| 双方向リンクの片方が欠落 | ドキュメント修正 |
| 意図的な不整合（トレードオフ選択等） | 設計判断の記録 |
