# ビジネス要件チェック -- 詳細手順

Epic 仕様書の受入基準（AC）が実装で充足されているかを検証する。3 つのチェック（AC 充足判定・ストーリー整合確認・要件カバレッジ検証）を順に実施し、未充足の AC やストーリーとの乖離を漏れなく検出する。

> **位置づけ:** `/aidd-epic-review` Step 1 の 3 観点チェックのうち「ビジネス要件」観点を担当する。Epic 仕様書を正とし、実装（コード変更 + ドキュメント変更）との突合を行う。

## 入力

| 項目 | 取得元 | 説明 |
|------|--------|------|
| Epic 仕様書 | `docs/requirements/ES-NNN-*.md` | 全 AC、対応ストーリー、バリデーションルール、エラーケース、非機能要件 |
| Task 定義群 | `docs/tasks/TASK-NNN-*.md` | Epic に紐づく全 Task の完了条件・スコープ |
| PR 差分 | `gh pr diff` / `git diff main...HEAD` | 変更ファイル一覧・変更内容 |

## チェック手順

### 1. AC 充足判定

Epic 仕様書の全 AC を 1 件ずつ確認し、実装エビデンスと突合する。

#### AC 充足判定の手順

1. Epic 仕様書から全 AC-ID（`AC-ENNN-NN` 形式）を抽出する
2. 各 AC に対応する Task 定義を特定する（Task 定義の「完了条件」セクションに AC-ID が記載されている）
3. Task 定義の対象ファイルと PR 差分を照合し、AC を充足する実装（コード変更またはドキュメント変更）が存在するか判定する
4. AC エビデンス表を作成する

#### AC エビデンス表

```markdown
| AC-ID | 内容 | Task | 充足ファイル | 判定 |
|-------|------|------|------------|------|
| AC-E017-01 | context: fork が設定され... | TASK-086 | skills/aidd-epic-review/SKILL.md | OK |
| AC-E017-02 | 独自 4 ステップ構成で... | TASK-087 | skills/aidd-epic-review/SKILL.md | OK |
| AC-E017-06 | ビジネス要件チェック... | TASK-089 | skills/aidd-epic-review/references/check-business.md | OK |
```

#### AC 充足の判定基準

| 条件 | 判定 |
|------|------|
| AC に対応する実装（コード変更またはドキュメント変更）が PR 差分に存在する | OK |
| AC に対応する Task が存在するが、PR 差分に実装が見つからない | NG -- 未実装 |
| AC に対応する Task 自体が存在しない | NG -- Task 未定義 |
| AC の内容と実装が部分的にしか一致しない（一部の条件が未充足） | NG -- 部分充足 |
| AC に対応する実装は存在するが、AC の意図と異なる実装になっている | NG -- 仕様不一致 |

### 2. ストーリー整合確認

各ストーリーの意図が AC を通じて実現されているかを検証する。

#### ストーリー整合確認の手順

1. Epic 仕様書の「対応ストーリー」セクションから全ストーリーを抽出する
2. 各ストーリーに紐づく AC を特定する（AC の末尾に `← Sn` のストーリートレースが記載されている）
3. ストーリーの意図（"so that" 句）が、紐づく AC 群の実装によって実現されているか判定する
4. ストーリー整合表を作成する

#### ストーリー整合表

```markdown
| Story | 意図（so that） | 紐づく AC | 実現状況 | 判定 |
|-------|----------------|-----------|---------|------|
| S1 | コンテキスト問題によるルール無視が解消される | AC-E017-01, 02, 03, ... | fork 内チェック + 500 行制限で実現 | OK |
| S3 | コンテキスト消費が減り遵守率が上がる | AC-E017-03, 05, 15 | 行数制限 + references/ 分離で実現 | OK |
```

#### ストーリー整合の判定基準

| 条件 | 判定 |
|------|------|
| ストーリーの意図が紐づく AC 群の実装で実現されている | OK |
| ストーリーに紐づく AC が 1 件も存在しない | NG -- AC 未定義 |
| ストーリーの意図が AC の実装では実現されていない（AC はあるが意図と乖離） | NG -- 意図未実現 |
| ストーリーの一部の意図のみが実現されている | NG -- 部分実現 |

### 3. 要件カバレッジ検証

Epic 仕様書に定義された要件（バリデーションルール・エラーケース・非機能要件）が実装でカバーされているかを検証する。

#### 要件カバレッジ検証の手順

1. Epic 仕様書の以下のセクションを抽出する:
   - 「バリデーションルール」セクション
   - 「エラーケース」セクション
   - 「非機能要件」セクション
2. 各要件に対応する実装が PR 差分に存在するか確認する
3. 要件カバレッジ表を作成する

#### バリデーションルール・カバレッジ表

```markdown
| フィールド | ルール | エラー時の振る舞い | 実装ファイル | 判定 |
|-----------|--------|------------------|------------|------|
| SKILL.md 行数 | 500 行以下 | レビューで MUST FIX | SKILL.md（実測 N 行） | OK |
```

#### エラーケース・カバレッジ表

```markdown
| ケース | 条件 | 期待する振る舞い | 実装ファイル | 判定 |
|--------|------|----------------|------------|------|
| Epic の PR が存在しない | PR がオープンでない | エラーメッセージ表示 | SKILL.md:NN | OK |
```

#### 非機能要件・カバレッジ表

```markdown
| 項目 | 基準 | 実装エビデンス | 判定 |
|------|------|--------------|------|
| コンテキスト効率 | context: fork でメインコンテキストへの影響ゼロ | SKILL.md frontmatter | OK |
```

#### 要件カバレッジの判定基準

| 条件 | 判定 |
|------|------|
| 要件に対応する実装が存在し、定義通りに動作する | OK |
| 要件に対応する実装が存在しない | NG -- 未実装 |
| 要件の定義と実装が異なる（境界値の不一致、エラーメッセージの相違等） | NG -- 定義不一致 |
| 要件が Epic 仕様書に定義されているが曖昧で検証不能 | NG -- 要件曖昧（Epic 仕様書の修正が必要） |

### 問題検出時の出力

検出した問題には全て修正提案を付ける（AC-E017-09 準拠）。修正の種類は以下のいずれか:

| 修正種類 | 説明 | 例 |
|---------|------|-----|
| コード修正 | 実装の追加・変更が必要 | AC 未充足のため機能を追加実装する |
| ドキュメント修正 | ドキュメントの追加・変更が必要 | Task 定義に AC-ID の記載漏れを追加する |
| 設計判断の記録 | 意図的な選択であることを明記 | 非機能要件を現時点では対象外とした理由をコメントで記録する |
