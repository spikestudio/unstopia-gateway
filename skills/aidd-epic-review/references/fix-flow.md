# 合意 + 修正フロー — 詳細手順

Step 1 で検出された全問題に対して、1 件ずつ合意 → 修正を実行する。

AI は問題の分析・影響評価・具体的な修正提案まで責任を持つ。人間に「どうしますか？」と丸投げしない。

## フロー概要

```
1 件ずつ繰り返し:
  問題提示（内容・影響・修正提案）→ 人間の判断 → 修正実行 → 次の問題へ
全問題完了 → Step 3 に進む
```

## 問題提示の原則

各問題に対して、AI は以下の 3 要素を**必ず**含めて提示する:

| 要素 | 説明 | AI の責任 |
|------|------|----------|
| **(a) 問題の内容** | 何が問題か、具体的な箇所を示す | 根拠となるコード/ドキュメントを引用する |
| **(b) 影響範囲と深刻度** | 誰に・どこに・どの程度影響するか | 影響を受けるファイル・機能・ユーザーを特定する |
| **(c) 具体的な修正提案** | どう修正すべきか、修正種別を明示する | ファイルパス・変更内容を具体的に示す |

**禁止パターン:**

- 「この問題をどう修正しますか？」（修正提案なしの丸投げ）
- 「問題があります」のみで影響評価なし
- 「適切に修正してください」のような曖昧な提案

## 問題提示の形式

### 修正種別: コード修正

```
## 問題 #X / N: [タイトル]

**問題:** ○○という問題があります。
  該当箇所: `path/to/file.ts:42`
  問題のあるコード: `// 問題のコード（引用）`

**影響:** △△に影響し、□□のリスクがあります。
  影響範囲: [影響を受けるファイル・機能の一覧]
  深刻度: 高 / 中 / 低

**修正提案（コード修正）:**
  ファイル: `path/to/file.ts:42`
  変更内容: [具体的な修正内容]
  修正後: `// 修正後のコード`

この修正を実行しますか？
```

### 修正種別: ドキュメント修正

```
## 問題 #X / N: [タイトル]

**問題:** ○○という整合性の問題があります。
  該当箇所: `docs/xxx.md` の「セクション名」

**影響:** ドキュメントの信頼性が低下します。
  影響範囲: [このドキュメントを参照する他のドキュメント・スキル]
  深刻度: 高 / 中 / 低

**修正提案（ドキュメント修正）:**
  ファイル: `docs/xxx.md`
  変更内容: [具体的な修正内容]

この修正を実行しますか？
```

### 修正種別: 設計判断の記録

```
## 問題 #X / N: [タイトル]

**問題:** ○○の実装が一見△△に見えます。
  該当箇所: `path/to/file.ts:42`

**影響:** 機能上の問題はありません。ただし将来の開発者が意図を誤解するリスクがあります。
  深刻度: 低

**修正提案（設計判断の記録）:**
  ファイル: `path/to/file.ts`
  変更内容: コメントを追加 — 「この実装は○○の理由で選択した」
  追加コメント: `// この実装は○○の理由で△△パターンを採用（参照: ADR-NNN / Issue #NNN）`

この修正を実行しますか？
```

## 人間の判断パターンと AI の対応

### パターン 1: 同意 — 「はい」「修正して」

1. 提案した修正を即座に実行する
2. 修正結果を簡潔に報告する
3. 次の問題に進む

### パターン 2: 問題ではない — 「問題ではない」「意図的」

AI は無条件に受け入れず、以下の対話で認識を合わせる:

1. **理由を質問する** — 「なぜ問題ではないと判断されましたか？」
2. **理由を評価する** — 人間の説明に基づき、以下のいずれかに分類:
   - **正当な設計判断**: AI が見落としていた文脈がある → 設計判断の記録として残すことを提案
   - **リスクの許容**: 問題は存在するが許容範囲 → リスクを明示した上で記録を提案
   - **認識の相違**: AI と人間で前提が異なる → 前提を確認し、修正提案を調整
3. **最終的な修正方針を確定する** — 必ず何らかの形で解消する（修正内容の変更 / 設計判断の記録 / 影響の明示的な受容記録）

**重要:** 「問題ではない」で終了にしない。最低でも設計判断の記録として、なぜ問題ではないかの根拠をコードまたはドキュメントに残す。

### パターン 3: 別の修正案 — 「こうした方がいい」「別のアプローチで」

1. 人間の提案を受け取る
2. AI が提案を評価する（実現可能性・副作用・他の問題との整合性）
3. 評価結果を提示する:
   - 問題なし → 人間の提案で修正を実行
   - 懸念あり → 懸念点を説明し、調整案を提示
4. 合意に達したら修正を実行する

### パターン 4: 保留 — 「後で」「今は対応しない」

FRAMEWORK.md の意思決定基準に従い、AI は後回しを選択肢に含めない:

1. **保留の理由を確認する**
2. **リスクを明示する** — 「この問題を保留にした場合、○○のリスクがあります」
3. **最低限の対応を提案する** — 全く対応しないのではなく、最低限の対応（TODO コメント + Issue 作成）を提案
4. 人間が最終判断した場合のみ、理由・リスクを明示した Issue を即時作成する

## 修正実行の手順

修正が合意されたら、以下の手順で実行する:

1. 対象ファイルを Read で読み込む
2. Edit で修正を実行する
3. 修正が他のファイルに波及する場合は、関連ファイルも修正する
4. 修正結果を簡潔に報告する: 「修正完了: `path/to/file.ts:42` — [1 行サマリー]」
5. 次の問題に進む

## 全問題完了時

全問題に対して修正が実行されたことを確認し、以下を提示する:

```
## 修正サマリー

| # | 問題 | 修正種別 | 対応 |
|---|------|---------|------|
| 1 | [タイトル] | コード修正 | 修正完了 |
| 2 | [タイトル] | ドキュメント修正 | 修正完了 |
| 3 | [タイトル] | 設計判断の記録 | 記録追加 |

全 N 件の問題に対して修正が完了しました。Step 3 に進みます。
```

未解消の問題が残っている場合は Step 3 に進まず、残問題の対応を継続する。
