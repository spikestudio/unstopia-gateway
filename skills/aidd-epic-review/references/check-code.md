# コードチェック — 詳細手順

PR 差分に含まれるコード変更を 6 つの観点でチェックし、問題を検出する。

## 入力

| 項目 | 説明 |
|------|------|
| PR 差分 | 変更ファイル一覧（`gh pr diff` または `git diff` で取得） |
| 規約ドキュメント | `docs/conventions/` 配下の全ファイル |
| CLAUDE.md | 禁止事項セクション |
| 変更されたソースコード | PR に含まれる全変更ファイルの内容 |

## チェック手順

### 1. 規約違反の検出

`docs/conventions/` の各規約ドキュメントと PR 差分を突合する。

| 規約 | 参照ドキュメント | チェック内容 |
|------|----------------|------------|
| 命名規則 | `docs/conventions/naming.md` | 変数・関数・ファイル・ディレクトリの命名が規約に従っているか |
| ディレクトリ構造 | `docs/conventions/directory-structure.md` | 新規ファイルが規約のディレクトリ構造に配置されているか |
| レイヤー間ルール | `docs/conventions/layer-rules.md` | 依存方向・呼び出しパターンが規約に従っているか |
| エラーハンドリング | `docs/conventions/error-handling.md` | エラー処理が規約の方針に従っているか |
| テスト規約 | `docs/conventions/testing.md` | テストの命名・構造・配置が規約に従っているか |
| 禁止事項 | `docs/conventions/prohibitions.md` | 禁止されたパターンが使用されていないか |
| UI 規約（該当時） | `docs/conventions/ui-*.md` | UI コンポーネントの設計・実装パターンが規約に従っているか |

**検出手順:**

1. PR の変更ファイル一覧を取得する
2. 各ファイルの変更内容を読み込む
3. ファイル種別に応じた規約ドキュメントを参照する
4. 規約の各ルールと変更内容を突合し、違反箇所を記録する

### 2. 冗長コードの検出

変更されたコードに不要な重複・未使用要素・過剰な抽象化がないかを検出する。

| チェック項目 | 検出方法 |
|------------|---------|
| 重複コード | 同一ロジックが複数箇所に存在しないか。PR 内の変更ファイル間、および既存コードとの重複を確認する |
| 未使用インポート | 追加されたインポート文のうち、実際に使用されていないものがないか |
| 未使用変数・関数 | 追加された変数・関数のうち、どこからも参照されていないものがないか |
| デッドコード | 到達不能なコードブロック（常に false の条件分岐等）がないか |
| 過剰な抽象化 | YAGNI 原則に反する抽象化がないか。現時点で 1 箇所からしか呼ばれない汎用インターフェース等 |

**検出手順:**

1. 追加・変更された関数・クラスの一覧を抽出する
2. 各要素の参照箇所を検索する（`Grep` ツールで確認）
3. 未参照の要素を冗長コードとして記録する
4. 類似ロジックの重複パターンを検出する

### 3. 技術的負債の検出

将来のバグ・パフォーマンス問題の種となるコードを検出する。

| チェック項目 | 検出方法 |
|------------|---------|
| TODO/FIXME コメント | 変更差分に新規追加された `TODO`、`FIXME`、`HACK`、`XXX` コメントを検出する |
| マジックナンバー | ハードコードされた数値・文字列リテラルが定数化されずに使用されていないか |
| エラーハンドリング不備 | try-catch で握りつぶし（空の catch ブロック）、エラーの無視がないか |
| 型安全性の欠如 | `any` 型の使用、型アサーションの乱用がないか |
| 非同期処理の不備 | await 忘れ、Promise の未処理、競合状態の可能性がないか |

**検出コマンド:**

```bash
# TODO/FIXME コメントの検出（PR 差分内）
gh pr diff <番号> | grep -E '^\+.*\b(TODO|FIXME|HACK|XXX)\b'

# any 型の使用検出（TypeScript の場合）
gh pr diff <番号> | grep -E '^\+.*:\s*any\b'
```

### 4. lint suppress 構文の検出

CLAUDE.md の禁止事項に基づき、リンター・型チェッカー・静的解析の suppress 指示が新規追加されていないかを検出する。**新規追加された suppress は全て問題として報告する。**

| 言語/ツール | 禁止構文 |
|------------|---------|
| ESLint | `eslint-disable`, `eslint-disable-next-line`, `eslint-disable-line` |
| TypeScript | `@ts-ignore`, `@ts-expect-error`, `@ts-nocheck` |
| Rust | `#[allow(...)]` |
| Go | `// nolint` |
| C/C++ | `#pragma warning disable` |
| Java | `@SuppressWarnings` |
| Python | `# noqa`, `# type: ignore` |

**検出コマンド:**

```bash
# PR 差分から suppress 構文を検出する（追加行のみ）
gh pr diff <番号> | grep -E '^\+' | grep -E '(eslint-disable|@ts-ignore|@ts-expect-error|@ts-nocheck|#\[allow\(|// nolint|#pragma warning disable|@SuppressWarnings|# noqa|# type: ignore)'
```

**判定ルール:**

- 出力が空: suppress の新規追加なし（PASS）
- 出力あり: 該当箇所を全て 🔴 MUST FIX として報告する
- `suppress-approved: [理由]` マーカー付きかつ人間の承認済みの場合のみ例外とする

### 5. AI 生成コード特有のリスク

AI が生成したコードに特有のリスクパターンを検出する。

| リスク分類 | 具体例 | 検出方法 |
|-----------|--------|---------|
| **もっともらしい誤り** | 構文的に正しく動作もするが、仕様と微妙に異なるロジック | AC との突合。境界値テストの確認 |
| **過剰実装** | 指示されていない機能の追加、不要な抽象化 | Task スコープとの突合。YAGNI チェック |
| **規約からの逸脱** | 学習データに引きずられた一般的なパターンの適用 | 規約ドキュメントとの突合 |
| **ハルシネーション** | 存在しない API・ライブラリ・設定の参照 | ビルド・型チェックで検出 |
| **テストの品質不足** | 仕様ではなく実装をテストしている | AC に基づいてテストが書かれているかの確認 |

### 7. 実行時検証

静的解析のパスは必要条件であって十分条件ではない。変更ファイルのパターンから作業 Tier を自動判定し、対応する実行時検証を実施する。

> ローカルファースト原則に従い、全 Tier の確認はローカル環境で実施する。詳細は `aidd-framework/guides/local-verification.md` を参照。

#### Tier 自動判定

変更ファイル一覧（`gh pr diff --name-only` または `gh pr view --json files`）を分析し、最高 Tier を適用する:

| Tier | 判定条件（変更ファイルのパターン） | 検証レベル |
|------|--------------------------------|----------|
| 0 | `*.md`, `skills/**`, `docs/**` のみ | 実行時検証不要 |
| 1 | `app/**/route.*`, `api/**`, `src/**`（非 UI ファイル）を含む | API エンドポイント呼び出し確認 |
| 2 | `app/**/page.*`, `components/**`, `mocks/**` 等の UI ファイルを含む | dev server 起動・画面確認 |
| 3 | `*.tf`, `migrations/**`, `*.sql` 等の IaC/DB ファイルを含む | plan/diff 分析 + 人間による apply 承認 |

複数 Tier に該当する場合は最高 Tier を適用する。

#### Tier 0: 実行時検証不要

```
実行時検証: Tier 0（ドキュメント・スキル変更のみ）
→ 実行時検証をスキップします。理由: 実行環境に影響する変更がありません。
```

#### Tier 1: API エンドポイント確認

1. ローカル dev server を起動する（`task dev`）。staging への先行デプロイは禁止
2. PR 差分から変更された API エンドポイントを特定する
3. 各エンドポイントに実際のリクエストを送信し、期待するレスポンス（ステータスコード・内容）を確認する
4. 結果を記録する

#### Tier 2: フロントエンド画面確認

**ローカル dev server** で確認する。dev server 確認方法をユーザーに選択させる:

```
Tier 2（UI変更）を検出しました。ローカル dev server 確認を以下のどちらで行いますか？
→ 自動: agent-browser で dev server 起動・画面確認を実行します
→ 手動: ローカル dev server を起動して確認後、結果を報告してください
```

**自動の場合（agent-browser）:**

`agent-browser` を使用して dev server を起動し、変更された画面の動作・表示を確認する:

- dev server を起動（`npm run dev` / `next dev` 等）
- 変更されたページ・コンポーネントを対象に画面を確認する
- コンソールエラー・表示崩れがないことを確認する

**手動の場合:**

- ユーザーが dev server を起動して確認し、結果を報告する

#### Tier 3: IaC/DB 変更確認

1. AI が plan/migration diff を分析し、変更内容・影響範囲を報告する
2. apply / migrate の実行は AI が自動実行しない。必ず人間の承認を求める:

```
Tier 3（IaC/DB変更）を検出しました。
AI による plan/diff 分析結果:
[分析結果]

apply / migrate の実行は人間が行う必要があります。
実行・確認が完了したら結果を報告してください。
```

#### 検証失敗時の処理

実行時検証が失敗または完了不可能な場合:

- `gate:reviewed` を付与しない
- 失敗理由と解消方法を報告する
- 解消後に再検証を実施する

### 6. セキュリティ観点（該当する変更がある場合）

認可・外部入力処理・データ永続化を含む変更がある場合に実施する。

| チェック項目 | 検出方法 |
|------------|---------|
| 認可 | 権限が Epic 仕様書の定義通りに実装されているか |
| 入力検証 | 外部入力に対するサニタイズ・バリデーションが実装されているか |
| データ整合性 | トランザクション境界、同時実行制御が適切か |
| 機密情報 | ログ・レスポンスに機密情報が含まれていないか |

## 出力

チェック結果を以下の形式で構造化する。

```
## コードチェック結果

### 規約違反
| # | ファイル:行番号 | 違反規約 | 問題内容 | 修正指示 |
|---|---------------|---------|---------|---------|
| 1 | src/xxx.ts:42 | naming.md — 関数命名規則 | camelCase でない | [具体的な修正内容] |

### 冗長コード
| # | ファイル:行番号 | 種別 | 問題内容 | 修正指示 |
|---|---------------|------|---------|---------|
| 1 | src/xxx.ts:15 | 未使用インポート | `import { foo }` が未使用 | インポートを削除する |

### 技術的負債
| # | ファイル:行番号 | 種別 | 問題内容 | 修正指示 |
|---|---------------|------|---------|---------|
| 1 | src/xxx.ts:88 | TODO コメント | `// TODO: 後で修正` | 本 PR 内で解決するか、Issue を作成する |

### suppress 検出結果
| # | ファイル:行番号 | 検出構文 | 修正指示 |
|---|---------------|---------|---------|
| （なし: PASS） |
```

全カテゴリで検出件数 0 の場合も、明示的に「検出なし」と記載する。

実行時検証結果は `gate:reviewed` 付与時の PR コメントに以下のセクションを含める:

```
### 実行時検証
| 項目 | 内容 |
|------|------|
| Tier 判定 | Tier N（判定根拠: [該当ファイルパターン]） |
| 実施内容 | [実施したコマンド・確認内容] |
| 結果 | ✅ PASS / ❌ FAIL |
| 出力サマリー | [コマンド出力の要約または「手動確認済み」] |
```
