# /aidd-impl — Task 実装（対話的実装 → セルフレビュー → コミット）

Task 定義を読み込み、AC・設計成果物・テスト方針に基づいて対話的に実装する。セルフレビューを経てコミットし、次のステップを案内する。

> **このスキルの目的は「Task 定義の AC を確実に満たす実装を、規約に従って進める」こと。** 実装のガイドに徹し、Task スコープ外の実装を防ぐ。

## インターフェース

### 前提条件

| 前提 | 確認方法 | 未完了時の案内 |
|------|---------|-------------|
| Task 定義が作成済み | `docs/tasks/TASK-NNN-*.md` の存在確認 | `/aidd-decompose-epic` を先に実行 |
| Epic 仕様書が承認済み | Epic Issue が `status:in-progress` ラベル付きでオープンかつ G2 通過コメントが存在する（`gh issue view <Epic番号> --comments`） | `/aidd-new-epic` を先に実行 |

### 入力

| 項目 | 形式 | 必須 | 説明 |
|------|------|------|------|
| Task 番号 | 引数 | 任意 | 指定なしなら GitHub Issues から自動推定 |

### capability 呼び出し

| capability | Step | 引数 |
|-----------|------|------|
| /aidd-research | Step 1 | 技術選定, 仕様調査（必要時のみ） |

### 出力

| 項目 | 形式 | 説明 |
|------|------|------|
| 実装コード | ファイル | AC に対応するコード変更 |
| テストコード | ファイル | Task 定義のテスト方針に基づくテスト |
| コミット | git commit | `feat: TASK-NNN [Task名]` 形式 |

### 後続スキル

| スキル | 条件 |
|--------|------|
| `/aidd-impl` | Epic に残 Task がある場合、次の Task を実装 |
| `/aidd-epic-review` | Epic 全 Task 完了時 |

## Step 0: 入力受付

`$ARGUMENTS` から Task を特定する。指定がなければ自動推定し、推定結果をユーザーに確認してから進行する。

### 0-1. Task の特定

- **引数あり**（例: `/aidd-impl TASK-005`）: `docs/tasks/TASK-005-*.md` を読み込み
- **引数なし**: オープンな Task Issue（`gh issue list --label task --state open`）から自動推定。並行可能 Task が複数ある場合は「自動並行実装」セクションに分岐する。単一 Task の場合は推定結果をユーザーに確認
- **推定失敗時**:

```
実装対象の Task を自動推定できませんでした。以下のいずれかで指定してください:
- `/aidd-impl TASK-005` — Task 番号を指定して実行
```

### 0-2. コンテキスト読み込み

| 入力 | 取得元 | 用途 |
|------|--------|------|
| Task 定義 | `docs/tasks/TASK-NNN-*.md` | AC・完了条件・テスト方針 |
| Epic 仕様書 | Task が参照する Epic | AC の詳細・ビジネスコンテキスト |
| 設計成果物 | Task が参照する設計書 | ドメインモデル・DB スキーマ・API spec |
| ADR | Task が参照する ADR | 技術的決定 |
| 規約ドキュメント | `docs/conventions/` | 命名・構造・テスト・禁止事項 |
| リサーチドキュメント | `docs/research/`（設計書から参照がある場合） | 外部依存の動作仕様・URL・バージョン |
| 既存コード | リポジトリ内の関連コード | 実装パターンの参考 |

### 0-3. Epic ブランチの確認

[gitflow ガイド](../../aidd-framework/guides/gitflow.md)に従い、現在のブランチが Epic ブランチ（`feature/ES-NNN-slug`）であることを確認する。

```bash
git branch --show-current
```

- **ブランチが `feature/ES-NNN-slug` の場合**: そのまま作業を継続する
- **ブランチが異なる場合**: 「Epic ブランチ（feature/ES-NNN-slug）で作業してください。他ブランチへの切り替えは禁止されています」と警告してスキルを停止する
- **Epic ブランチが存在しない場合**: 先に `/aidd-new-epic` を実行してワークツリーを作成する
- Task Issue に `status:in-progress` ラベルを付与する（`gh issue edit <番号> --add-label "status:in-progress"`）

### 0-4. 未決定事項ブリーフィング

Task 定義と設計成果物を突合し、実装を進めるうえで未決定・曖昧な事項がないか確認する。

- **未決定事項がない場合** → その旨を伝え、Step 1 に進む
- **未決定事項がある場合** → 未決定事項の一覧と AI の推奨案を提示し、ユーザーと合意してから Step 1 に進む

```
## 実装前ブリーフィング: TASK-NNN [Task名]

### Task 定義の概要
- AC: [AC-ENNN-01: ..., AC-ENNN-02: ...]
- スコープ: [対象ファイル一覧]
- テスト方針: [UT / 統合 / E2E の対象]

### 未決定事項
1. [事項]: [内容] → 推奨: [AI の推奨案]
2. [事項]: [内容] → 推奨: [AI の推奨案]

### 確認済み事項
- [Task 定義の AC は明確]
- [設計成果物との整合性は問題なし]

上記の内容で実装に進んでよいですか？
```

## Step 1: 実装

> **前提: Step 0 が完了していること。**

`references/impl-guide.md` を読み込んで実行する。

**概要:** Task 定義の AC に従い、対話的に実装を進める。各ステップで AC との対応を明示し、規約に準拠したコードを生成する。実装完了後、セルフレビューを実施する。

**対話的実装の流れ:**

1. Task 定義の AC・テスト方針に基づき、実装ステップを構築して提示する
2. 各ステップを順に実装し、AC との対応を都度確認する
3. 全ステップ完了後、`references/selfcheck.md` に従ってセルフレビュー反復ループを実施する（aidd-framework/references/self-review-loop.md 参照）
4. 最大 3 ラウンドで MUST FIX を解消してから完了とする

**capability 呼び出し:**

- `/aidd-research` — 技術選定や仕様の不明点がある場合（必要時のみ）

**完了時:** `references/briefing-spec.md` に従ってブリーフィングを実行し、承認を得る。

## Step 2: 完了処理

> **前提: Step 1 の承認が完了していること。**

### 2-0. ドキュメント更新確認

実装内容に応じて関連ドキュメントを更新する。

#### ドキュメントスキャン

共有スクリプト `scripts/scan-docs.sh` を実行し、変更内容と照合して更新が必要なドキュメントを特定・更新・コミットする。

```bash
bash "${CLAUDE_PLUGIN_ROOT}/scripts/scan-docs.sh"
```

スキャン結果と今回の実装変更を照合し、更新が必要なドキュメントがあれば更新してコミットに含める。

#### ADR 作成ガイド

以下のいずれかに該当する場合は、`aidd-framework/templates/adr.md` を参照して ADR を作成する:

- フレームワーク設計の変更（スキル構造・プロセス定義・ゲート制御の変更など）
- プロセスルールの変更（規約・ガイドライン・禁止事項の追加・変更など）
- 重要な技術的意思決定（アーキテクチャ選択・ライブラリ採用・破壊的変更など）

```bash
# ADR テンプレートを確認
cat aidd-framework/templates/adr.md
```

ADR を作成した場合は、対応するマスタドキュメントを同一コミット内で更新すること（CLAUDE.md の「ドキュメント連動ルール」参照）。

### 2-1. コミット・Task 完了処理

Step 1 のセルフレビューがクリアになり承認を得たら、コミットして Task Issue を完了する。**PR はここでは作成しない。PR は `/aidd-epic-review` PASS 後に作成する。**

```bash
git add [変更ファイル]
git commit -m "feat: TASK-NNN [Task名]"

# Task Issue を完了処理する（#699 #732）
gh issue edit <Task Issue番号> --remove-label "status:in-progress"
gh issue close <Task Issue番号>
```

### 2-2. 次のステップ案内

```
Task 実装が完了しました:
- Task: TASK-NNN [Task名]（Epic ES-NNN の N/M Task 目）
- コミット: [コミットハッシュ]
- 残 Task: N 件

次のステップ:
→ 残 Task あり: `/aidd-impl TASK-NNN` で [次の Task名] を実装
→ 全 Task 完了: `/aidd-epic-review ES-NNN` でレビュー → PASS 後 `/aidd-next` で G5 判定
```

## 自動並行実装

引数なしで起動した場合に、Epic のオープンな Task の依存グラフを自動解析し、並行可能 Task を検出・提案・Agent 並行起動する。

> **ES-015 実績パターン:** TASK-071〜075 の 5 Task が全て独立ファイルで依存なし。Agent ツールで 5 サブエージェントを 1 メッセージ内で並行起動し、各エージェントに完全なコンテキスト（Task 定義の内容、参照パターン、実行指示）を渡して実装。全エージェント完了後にまとめてレビュー・修正・コミット。

### P-1. 依存グラフ解析

1. 対象 Epic の全 Task 定義（`docs/tasks/TASK-*.md`）を読み込む
2. 各 Task の `## 依存 > 先行 Task` セクションから依存関係を抽出する
3. 以下の条件を**すべて**満たす Task を「並行可能」と判定する:
   - 先行 Task が「なし」「—」、または先行 Task が全て完了済み（Issue が closed）
   - Task Issue がオープン（未着手 or 進行中）
4. 判定結果に応じて分岐する:
   - **並行可能 Task が 2 件以上** → P-2（競合検出）に進む
   - **並行可能 Task が 1 件** → 逐次実装（Step 0-2 以降）に進む
   - **並行可能 Task が 0 件** → 「依存関係のある先行 Task を先に完了してください」と案内

### P-2. 競合検出

並行可能グループ内の Task が同一ファイルを変更する場合、並行実行すると競合する。事前に検出して回避する。

1. 並行可能グループ内の各 Task の `## スコープ > 対象ファイル` を比較する
2. **同一ファイルが複数 Task に含まれる場合:**
   - 競合として警告を表示する
   - 競合する Task のうち、依存が浅い（先行 Task が少ない）方を並行グループに残し、他方を逐次実行に回す
   - 判断をユーザーに提示し、確認を得る

```
## 競合検出: 並行実装の調整

以下の Task 間でファイル競合を検出しました:
- TASK-NNN と TASK-MMM: `path/to/file.ts` が重複

調整案:
- 並行実装: TASK-NNN, TASK-PPP, TASK-QQQ（競合なし）
- 逐次実行: TASK-MMM（TASK-NNN 完了後に実装）

この調整で進めてよいですか？
```

1. **競合がない場合** → P-3 に進む

### P-3. 並行実装の提案

並行可能 Task グループをユーザーに提案し、承認を得る。

```
## 自動並行実装の提案

Epic ES-NNN の依存グラフを解析しました。以下の N 件を並行実装できます。進めますか？

| # | Task | 名前 | 先行 Task | 対象ファイル |
|---|------|------|----------|------------|
| 1 | TASK-NNN | [名前] | なし | path/to/file1.ts |
| 2 | TASK-MMM | [名前] | なし | path/to/file2.ts |
| ... | ... | ... | ... | ... |

並行実装しますか？（y: 並行実装 / n: 1 件ずつ逐次実装）
```

### P-4. Agent 並行起動

ユーザーの承認後、Agent ツールで並行可能 Task を同時起動する。

1. 各 Task について、以下のコンテキストを収集する:
   - Task 定義（`docs/tasks/TASK-NNN-*.md`）の全文
   - Epic 仕様書の該当 AC
   - 参照する設計成果物・ADR
   - 規約ドキュメント（`docs/conventions/`）
   - 既存コードのパターン（参考実装）
2. **1 メッセージ内で全 Agent を同時起動する**（`run_in_background: true` は使用しない。Agent ツールを複数回呼び出す）
3. 各 Agent への指示には以下を含める:

```
以下の Task を実装してください。

## Task 定義
[Task 定義の全文]

## Epic AC
[対応する AC の全文]

## 規約
[関連する規約の要約]

## 参照コード
[既存の実装パターン]

## 実行指示
1. Task 定義の AC に従って実装する
2. テスト方針に従ってテストを実装する
3. セルフレビュー反復ループ（aidd-framework/references/self-review-loop.md + references/selfcheck.md）を実施する
4. 実装結果のサマリーを返す（AC 充足状況、変更ファイル一覧、問題点）
```

1. 全 Agent の完了を待ち合わせる

### P-5. 一括ブリーフィング

全 Task の実装結果をまとめてブリーフィングする。個別ブリーフィングでは並行実装の効率メリットが失われるため、一括で提示する。

```
## 並行実装 一括ブリーフィング

### 1. 全体概要
- 並行実装した Task 数: N 件
- 全体の AC 充足状況: [全て充足 / 一部未充足]

### 2. 各 Task の結果

| Task | AC 充足 | テスト | 変更ファイル | 問題点 |
|------|---------|--------|------------|--------|
| TASK-NNN | ✅ 全 AC 充足 | ✅ PASS | file1.ts, file1.test.ts | なし |
| TASK-MMM | ✅ 全 AC 充足 | ✅ PASS | file2.ts, file2.test.ts | なし |
| ... | ... | ... | ... | ... |

### 3. 重要な判断
- [実装中に行った設計判断があれば記載]

### 4. AI 補完箇所
- [Task 定義にない判断を AI が補完した箇所があれば記載]

### 5. 自信度
- [各 Task の実装に対する自信度]

### 6. 承認
上記の内容で問題なければ、コミットに進みます。承認をお願いします。
```

承認後、Step 2（完了処理）で各 Task をまとめてコミットする。Task ごとに個別コミットする場合は `git add [Task-N の変更ファイル] && git commit -m "feat: TASK-NNN [名前]"` を Task 数分繰り返す。

## ゲート制御ルール

**ゲートは構造的に必須である。以下のルールに例外はない。**

1. **実装開始前に対応 Issue の存在を確認すること。** Task Issue が存在しない場合は先に Issue を作成し、PR description に必ず `Closes #N` を含めること。Issue なし・参照なし PR は作成してはならない。
2. **Step 0 の未決定事項が合意されるまで Step 1 を開始してはならない。** 未決定事項がある状態で実装を開始しようとした場合、「未決定事項の合意が必要です」と表示しスキルを停止する。
3. **Step 1 の承認なしに Step 2 を開始してはならない。** ブリーフィング→承認を経ずにコミットを開始してはならない。
4. **ブリーフィングなしに承認を求めてはならない。** Step 1 完了時に `references/briefing-spec.md` に従ったブリーフィングを必ず実行する。1-2 行のサマリーのみでの承認要求は禁止。
5. **回答 ≠ 承認。** ユーザーの問いかけへの回答は回答内容の反映のみに留める。次ステップへの遷移には「OK」「進めて」「承認」等の明確な承認表現が必要。

## 重要なルール

- 日本語で対話する
- **FRAMEWORK.md の意思決定基準に従う** — 優先順序 1〜5 に該当する問題の見送り・後回しを推奨しない
- **ブリーフィング省略禁止** — Step 1 完了時に `references/briefing-spec.md` に従ったブリーフィングを必ず実行する
- 成果物生成・実装完了時にコミットを自動実行する。ユーザーの指摘反映後・ステップ承認後などの状態遷移ごとにもコミットする。push と PR 作成はユーザーの確認を待つ
- **Task スコープを厳守する** — AC に含まれない機能を実装しない。改善アイデアがあっても別 Issue として提案する
- **規約に従って実装する** — 規約ドキュメントに従ったコードを生成する
- **各ステップでユーザーに確認を求める** — 特に設計判断が必要な箇所では必ず確認
- **テスト方針に従う** — Task 定義のテスト方針セクションに記載されたレイヤーのテストを実装する
- **実装の参考コードを活用する** — リポジトリ内の既存コードパターンに合わせる
- **過剰な抽象化を避ける** — YAGNI 原則。今必要なものだけを実装する
- **lint suppress を使用してはならない** — `eslint-disable`, `@ts-ignore`, `@ts-expect-error`, `#[allow(...)]`, `// nolint`, `# noqa` 等の suppress 指示で警告を回避してはならない。根本原因を修正すること。修正不可能な場合はユーザーに相談する
- **テスト名に AC-ID を含める** — テストの describe/it ブロックに AC-ID（`AC-ENNN-NN`）を含め、CI での AC-ID カバレッジ自動検証を可能にする
- **外部依存値の転記ルール** — 外部 URL、バージョン番号、設定値をコードに記述する際は、設計書・リサーチドキュメント・公式ドキュメントからコピーする。記憶やトレーニングデータに基づいて外部依存値を書いてはならない
- 各ステップの詳細手順は `references/` を読んで実行する。SKILL.md 内に手順を複製しない
