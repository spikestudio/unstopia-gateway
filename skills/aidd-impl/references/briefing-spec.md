# ブリーフィング仕様 --- /aidd-impl

`/aidd-impl` は実装前（Step 0 完了時）と実装完了（Step 1 完了時）の 2 回ブリーフィングを行う。5 要素ブリーフィングを前提とし、各パターンで固有の品質基準を追加定義する。

## 5 要素チェックリスト

ブリーフィングは以下の 5 要素を**全て**含むこと。1 つでも欠けている場合、ブリーフィングは不完全とみなす。

| # | 要素 | 内容 | 欠落時のリスク |
|---|------|------|-------------|
| 1 | 成果物の全内容の説明 | 要約ではなく網羅的に説明する。実装方針・変更ファイル・AC との対応を省略しない | 人間が成果物を読まないと承認できない |
| 2 | AI が行った判断の根拠と代替案 | 「なぜこうしたか」「他にどんな選択肢があったか」を明示する | 判断の妥当性を評価できない |
| 3 | 暗黙の前提・行間の説明 | 成果物に直接書かれていないが、前提としている知識・制約を説明する | 前提の食い違いに気づけない |
| 4 | 未決定事項と論点の洗い出し | 決めきれなかったこと、後続で決定が必要なことを列挙する | 未決定事項が放置される |
| 5 | 人間に求める意思決定事項の明示 | 「何を判断してほしいか」を具体的に問いかける | 承認が形骸化する |

## パターン 1: 実装前ブリーフィング（Step 0 完了時）

Task 定義・設計成果物・規約を読み込んだ後、実装に入る前に行うブリーフィング。目的は **Task 定義時点で解決しきれなかった未決定事項・設計判断の迷いを人間と解消する** こと。

### 重点項目

| 重点項目 | 説明内容 |
|---------|---------|
| 未決定事項の解消 | Task 定義の「未決定事項」セクション、または Task 分解時に明示的に記載されなかった暗黙の判断ポイントを列挙し、AI の推奨案を提示する |
| 設計判断の迷い | 複数のアプローチがあり得る箇所を洗い出し、各選択肢のトレードオフ（実装コスト・保守性・パフォーマンス・規約との整合）を比較する |
| 技術的リスク | 外部依存・パフォーマンス・互換性など、実装中に問題化しうるリスクを特定し、対策案を提示する |
| スコープの確認 | 何を実装し、何を実装しないかの境界を明示する。Task スコープ外だが関連する改善点があれば別 Issue として報告する旨を伝える |
| 実装ステップの妥当性 | 提案する実装ステップの順序・粒度が適切かどうか。各ステップと AC の対応関係を明示する |

### 品質基準

**「この説明だけ読めば実装方針の承認可否を判断できる」品質を担保すること。**

具体的には:

- Task 定義の未決定事項が全て列挙され、AI の推奨案が提示されている
- 設計判断の迷いに対して、各選択肢の比較が「判断可能な粒度」で説明されている
- 技術的リスクの対策案が具体的である（「注意が必要」のような曖昧な表現ではなく、具体的なアクション）
- スコープの境界が明確で、過剰実装の防止に役立つ
- 自信度が低い箇所（`<!-- AI-UNCERTAIN -->` マーク付き）が重点的に説明されている
- 「はい/いいえ」で承認判断できる程度に論点が整理されている

### 出力フォーマット

```markdown
## ブリーフィング（実装前）: TASK-NNN [Task名]

### 実装方針の全体像
- 対応 AC（全件・全文。省略禁止）: AC-ENNN-01 [全文] / AC-ENNN-02 [全文] / ...（件数のみ・タイトルのみへの省略禁止）
- 変更対象: [新規作成/変更するファイルの一覧]
- 実装ステップ数: N ステップ
- [各ステップの概要と AC との対応]

### 未決定事項と推奨案
| # | 未決定事項 | AI の推奨案 | 理由 |
|---|----------|-----------|------|
| 1 | [Task 定義の未決定事項] | [推奨する選択肢] | [判断根拠] |
| ... | ... | ... | ... |

### 設計判断の迷い
| # | 判断ポイント | 選択肢 A | 選択肢 B | 推奨 |
|---|------------|---------|---------|------|
| 1 | [判断が必要な箇所] | [選択肢とトレードオフ] | [選択肢とトレードオフ] | [推奨と理由] |
| ... | ... | ... | ... | ... |

### 技術的リスクと対策
- [リスク 1]: [対策案]
- [リスク 2]: [対策案]

### スコープ確認
- **実装する:** [スコープ内の項目]
- **実装しない:** [スコープ外の項目と理由]

### 暗黙の前提
[成果物に直接書かれていないが前提としている事項]

### AI 自信度
[自信度が低い箇所の説明。全体の自信度表明]

### 確認事項
[人間に判断を求める具体的な問いかけ]

この方針で実装を進めてよいですか？
```

## パターン 2: 実装完了ブリーフィング（Step 1 完了時）

実装が完了した後、コミット前に行うブリーフィング。目的は **AC 充足の確認と実装結果の報告**。

### 実装完了ブリーフィングの重点項目

| 重点項目 | 説明内容 |
|---------|---------|
| AC 充足状況 | 各 AC の充足チェック表を**全件・全文**で提示する（AC-ID + AC 全文 + 充足状況 + 実装概要）。省略禁止。全 AC がカバーされていることを明示する |
| テスト結果 | テスト件数・種別（ユニット/結合/E2E）・カバレッジを報告する。テスト名に AC-ID が含まれているかを確認する |
| 実装中の設計判断 | 実装前ブリーフィングで承認された方針からの逸脱があれば報告する。実装中に新たに行った設計判断とその根拠を説明する |
| 規約準拠の確認 | lint/型チェック/静的解析の通過状況を報告する。lint suppress が新たに追加されていないことを確認する |
| 発見した問題・改善点 | 実装中に発見した問題・改善点を報告する。スコープ外のものは別 Issue として報告する旨を伝える |

### 実装完了ブリーフィングの品質基準

**「この説明だけ読めば実装結果の承認可否を判断できる」品質を担保すること。**

具体的には:

- 全 AC に対する充足チェックが一覧化され、各 AC の実装内容が具体的に説明されている
- テスト結果が数値で報告されている（件数・通過状況）
- 実装前ブリーフィングで承認された方針との差分（あれば）が明示されている
- lint/型チェックの通過状況が報告されている
- スコープ外の実装が含まれていないことが確認できる
- 自信度が低い箇所（`<!-- AI-UNCERTAIN -->` マーク付き）が重点的に説明されている
- 「はい/いいえ」で承認判断できる程度に論点が整理されている

### 実装完了ブリーフィングの出力フォーマット

```markdown
## ブリーフィング（実装完了）: TASK-NNN [Task名]

### 実装結果の全体像
- 変更ファイル数: N 件（新規: X 件、変更: Y 件）
- 変更行数: 追加 +NNN 行 / 削除 -NNN 行
- テスト: N 件（全件通過）

### AC 充足チェック
| AC | 内容 | 充足 | 実装概要 |
|----|------|------|---------|
| AC-ENNN-01 | [AC の内容] | :white_check_mark: | [どのように実装したか] |
| AC-ENNN-02 | [AC の内容] | :white_check_mark: | [どのように実装したか] |
| ... | ... | ... | ... |

充足率: XX/YY (100%)

### テスト結果
| 種別 | 件数 | 結果 |
|------|------|------|
| ユニットテスト | N 件 | 全件通過 |
| 結合テスト | N 件 | 全件通過 |

- AC-ID カバレッジ: [テスト名に AC-ID が含まれているかの確認結果]

### 実装中の設計判断
[実装中に新たに行った判断。実装前ブリーフィングの方針からの逸脱があればその理由]

### 規約準拠
- [x] lint 通過
- [x] 型チェック通過
- [x] lint suppress 未追加
- [x] 命名規則・ディレクトリ構造が規約準拠

### 発見した問題・改善点
- [スコープ内で対応済みの問題]
- [スコープ外の改善点 → 別 Issue で報告予定]

### 暗黙の前提
[成果物に直接書かれていないが前提としている事項]

### AI 自信度
[自信度が低い箇所の説明。全体の自信度表明]

### 確認事項
[人間に判断を求める具体的な問いかけ]

この内容でコミットしてよいですか？
```
