# Step 1: AC 解析・画面候補提示 — 詳細手順

AC-E052-01（AC 解析・画面候補の自動抽出・提示）および AC-E052-02（追加/除外/統合の対話操作）を満たす詳細手順を定義する。

## 入力

- Epic 仕様書の AC 一覧（Step 0 で読み込み済み）
- ドメインモデル情報（Step 0 で読み込み済み、存在する場合）

## 1-1. AC の解析・画面候補の抽出（AC-E052-01）

### UI 操作キーワードの検出

以下のキーワードを含む AC を「画面候補の導出元」として特定する。

| カテゴリ | キーワード |
|---------|----------|
| 閲覧系 | 表示、一覧、確認、参照、表示する、一覧する、確認できる |
| 入力系 | 入力、登録、作成、追加、入力する、登録する、作成する |
| 更新系 | 更新、編集、変更、修正、更新する、編集する |
| 削除系 | 削除、除外、削除する、削除確認 |
| 選択系 | 選択、選ぶ、選択する、選択できる |
| 検索系 | 検索、絞り込み、フィルタ、検索する |
| 操作系 | 実行、承認、却下、完了、キャンセル、送信 |

### 抽出ルール

1. 各 AC を上記キーワードで検索する
2. キーワードを含む AC を「画面候補の導出元 AC」として記録する
3. 同一画面から複数の AC が導出される場合は統合する（例: 「ユーザー一覧の表示 AC」と「ユーザー検索 AC」は「ユーザー一覧画面」として統合）
4. 1 つの AC から複数の画面候補が導出される場合は全て記録する

### 画面名の命名規則

| パターン | 命名例 |
|---------|--------|
| 一覧表示 | 〇〇一覧画面 |
| 詳細表示 | 〇〇詳細画面 |
| 作成フォーム | 〇〇作成画面 |
| 編集フォーム | 〇〇編集画面 |
| 削除確認 | 〇〇削除確認画面 |
| ダッシュボード | 〇〇ダッシュボード |
| 検索・絞り込み | 〇〇検索画面 |

## 1-2. ドメインモデル起点の画面候補（AC-E052-03, 04）

ドメインモデルが存在する場合:

1. `docs/domain/` 配下のファイルから集約・エンティティを列挙する
2. 各集約に対して CRUD 画面候補を生成する

| 集約の操作 | 生成する画面候補 |
|-----------|---------------|
| 一覧取得 | 〇〇一覧画面 |
| 詳細取得 | 〇〇詳細画面 |
| 作成 | 〇〇作成画面 |
| 更新 | 〇〇編集画面 |
| 削除 | 〇〇削除確認画面 |

1. Step 1-1 の AC 起点の画面候補と重複するものは除外する
2. 追加提案として別セクションに掲載する

ドメインモデルが存在しない場合:

- 「`docs/domain/` が見つかりませんでした。AC 起点の画面のみで続行します。」と通知する
- エラーで停止せず、AC 起点の画面のみを提示する（AC-E052-04）

## 1-3. 画面候補の提示

以下の形式で提示する:

```markdown
## Step 1: 画面候補の提示

### AC 起点の画面候補（N 件）

| # | 画面名（案） | 導出元 AC | 備考 |
|---|------------|----------|------|
| 1 | 〇〇一覧画面 | AC-ENNN-01 | — |
| 2 | 〇〇作成画面 | AC-ENNN-02, AC-ENNN-03 | 複数 AC から導出 |

### ドメインモデル起点の追加候補（N 件）（※ドメインモデルが存在する場合）

| # | 画面名（案） | 対応集約 | 種別 |
|---|------------|---------|------|
| 1 | 〇〇詳細画面 | 〇〇集約 | 詳細 |

上記の内容を確認してください。追加・除外・統合の操作があれば指示してください。
変更がなければ「OK」または「進めて」で確定します。
```

## 1-4. 対話操作の受け付け（AC-E052-02）

ユーザーからの操作指示を受け付け、画面候補リストを更新する。

### 追加

```
例: 「ログイン画面を追加してください」
→ リストに新しい行を追加する
```

処理:

1. ユーザーが指定した画面名で新しい候補を追加する
2. 導出元 AC は「手動追加」として記録する
3. 更新後のリストを再提示する

### 除外

```
例: 「3 番を除外してください」
例: 「〇〇削除確認画面は不要です」
```

処理:

1. 指定された画面候補をリストから除外する
2. 除外理由をメモとして記録する（後のカバレッジ確認に使用）
3. 更新後のリストを再提示する

### 統合

```
例: 「2 番と 4 番を統合して「〇〇管理画面」にしてください」
```

処理:

1. 指定された画面候補を 1 つに統合する
2. 導出元 AC は統合前の全 AC を引き継ぐ
3. 統合後の画面名をユーザーが指定した名前にする
4. 更新後のリストを再提示する

### 名前変更

```
例: 「1 番の画面名を「ユーザー管理一覧」に変更してください」
```

処理:

1. 指定された画面候補の名前を更新する
2. 更新後のリストを再提示する

### 操作後の再提示

操作ごとに更新後のリストを再提示し、さらに変更があるか確認する。変更がなくなったら承認を求める。

```markdown
## 更新後の画面候補リスト（N 件）

| # | 画面名 | 導出元 AC | 変更 |
|---|--------|----------|------|
| 1 | ... | ... | — |
| 2 | ... | ... | [追加] / [名前変更] |

他に変更はありますか？なければ「OK」または「進めて」で確定します。
```

## 1-5. 確定・承認

ユーザーから「OK」「進めて」「承認」等の明示的な承認を受けた時点で画面候補リストを確定する。

確定後:

- 最終的な画面候補リスト（画面名・導出元 AC）を内部に記録する
- 除外した画面候補と除外理由も記録する（AC カバレッジ計算に使用）
- Step 2 に進む

## エラーケース

| ケース | 条件 | 対応 |
|--------|------|------|
| AC キーワードが見つからない | AC に UI 操作キーワードが全く含まれない | 「AC から画面候補を自動抽出できませんでした。手動で追加するか、Epic 仕様書の AC を確認してください。」と案内 |
| 画面候補がゼロ件 | AC 起点もドメインモデル起点も候補なし | エラーメッセージ + 入力 Epic の確認を促す（SKILL.md エラーケース参照） |

## 完了時

`references/briefing-spec.md` に従い、Step 1 のブリーフィングを実行する。

ブリーフィングで必ず提示する情報:

- 提示した画面候補の総数（AC 起点 / ドメインモデル起点の内訳）
- 除外した AC（UI 操作キーワードが見つからなかった AC 一覧）
- ユーザーが変更した内容（追加・除外・統合の操作履歴）
- 確定後の最終的な画面候補一覧（全件）
